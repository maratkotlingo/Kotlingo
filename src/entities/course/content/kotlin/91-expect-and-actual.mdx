## Зачем нужны `expect` и `actual`

`expect` и `actual` связывают common API с platform-specific implementation.

В common source set объявляется expectation:

```kotlin
// commonMain/kotlin/platform/PlatformName.kt
package platform

expect fun platformName(): String
```

В platform source set дается actual implementation:

```kotlin
// jvmMain/kotlin/platform/PlatformNameJvm.kt
package platform

actual fun platformName(): String {
    return "JVM ${System.getProperty("java.version")}"
}
```

```kotlin
// jsMain/kotlin/platform/PlatformNameJs.kt
package platform

actual fun platformName(): String {
    return "JavaScript"
}
```

Общий код вызывает `platformName()`, как если бы реализация уже существовала.

```kotlin
// commonMain/kotlin/lesson/Diagnostics.kt
package lesson

import platform.platformName

fun diagnosticsLine(): String {
    return "kotlingo работает на ${platformName()}"
}
```

<CatTip>
`expect` — это кошачий след в common code: «здесь будет platform paw». `actual` — сама лапа на JVM, JS, iOS или другой platform.
</CatTip>

## Matching rules

Объявление `actual` должно соответствовать объявлению `expect`.

Совпадать должны:

- package;
- name;
- вид декларации;
- type parameters;
- function parameters;
- return type;
- property type;
- модификаторы, влияющие на контракт;
- члены класса, ожидаемые общим кодом;
- Иерархия набора источников, где фактический набор источников зависит от ожидаемого набора источников.

```kotlin
// commonMain/kotlin/platform/Clock.kt
package platform

expect object PlatformClock {
    fun nowMillis(): Long
}
```

```kotlin
// jvmMain/kotlin/platform/ClockJvm.kt
package platform

actual object PlatformClock {
    actual fun nowMillis(): Long {
        return System.currentTimeMillis()
    }
}
```

Если `actual` находится в другом package, compiler не считает его implementation.

<MouseTrap>
`actual` — не просто похожая функция. Это точное выполнение common contract. Переименовал package, поменял nullability или забыл member — common module больше не имеет platform implementation.
</MouseTrap>

## Expected functions

Функции — это простейший случай ожидания/факта.

```kotlin
// commonMain/kotlin/platform/Uuid.kt
package platform

expect fun randomUuidString(): String
```

JVM:

```kotlin
// jvmMain/kotlin/platform/UuidJvm.kt
package platform

import java.util.UUID

actual fun randomUuidString(): String {
    return UUID.randomUUID().toString()
}
```

iOS:

```kotlin
// iosMain/kotlin/platform/UuidIos.kt
package platform

import platform.Foundation.NSUUID

actual fun randomUuidString(): String {
    return NSUUID().UUIDString()
}
```

Common usage:

```kotlin
// commonMain/kotlin/lesson/LessonDraft.kt
package lesson

import platform.randomUuidString

data class LessonDraft(
    val id: String = randomUuidString(),
    val title: String
)
```

Функции хороши для:

- platform constants;
- small platform operations;
- factories;
- wrappers around APIs like UUID, clock, environment;
- simple one-off platform bridges.

Для крупных сервисов интерфейс плюс внедрение зачастую проще протестировать и заменить.

## Expected properties

Свойства также можно ожидать.

```kotlin
// commonMain/kotlin/platform/RuntimeInfo.kt
package platform

expect val runtimeFamily: String
```

```kotlin
// jvmMain/kotlin/platform/RuntimeInfoJvm.kt
package platform

actual val runtimeFamily: String = "jvm"
```

```kotlin
// jsMain/kotlin/platform/RuntimeInfoJs.kt
package platform

actual val runtimeFamily: String = "js"
```

Common usage:

```kotlin
// commonMain/kotlin/analytics/AnalyticsContext.kt
package analytics

import platform.runtimeFamily

data class AnalyticsContext(
    val runtime: String = runtimeFamily,
    val appName: String = "kotlingo"
)
```

Если значение вычислено, используйте метод получения:

```kotlin
// jvmMain/kotlin/platform/RuntimeInfoJvm.kt
package platform

actual val runtimeFamily: String
    get() = "jvm:${System.getProperty("java.vm.name")}"
```

## Expected classes

`expect class` объявляет тип, который может использовать общий код, тогда как каждая платформа предоставляет фактический класс.

```kotlin
// commonMain/kotlin/platform/PlatformPath.kt
package platform

expect class PlatformPath(raw: String) {
    val raw: String

    fun fileName(): String
}
```

JVM:

```kotlin
// jvmMain/kotlin/platform/PlatformPathJvm.kt
package platform

import java.nio.file.Path

actual class PlatformPath actual constructor(
    actual val raw: String
) {
    private val path: Path = Path.of(raw)

    actual fun fileName(): String {
        return path.fileName.toString()
    }
}
```

JS:

```kotlin
// jsMain/kotlin/platform/PlatformPathJs.kt
package platform

actual class PlatformPath actual constructor(
    actual val raw: String
) {
    actual fun fileName(): String {
        return raw.substringAfterLast("/")
    }
}
```

Ожидаемые классы являются мощными, но объединяют общий код с типом в форме платформы. Предпочитайте простые ожидаемые функции или интерфейсы, если только сам тип действительно не зависит от платформы.

## Конструкторы и члены

Общий код может использовать только члены, объявленные в `expect`.

```kotlin
// commonMain/kotlin/platform/PlatformToken.kt
package platform

expect class PlatformToken(value: String) {
    val value: String

    fun normalized(): String
}
```

Фактический класс может иметь дополнительные члены, предназначенные только для платформы, но общий код их не увидит.

```kotlin
// jvmMain/kotlin/platform/PlatformTokenJvm.kt
package platform

actual class PlatformToken actual constructor(
    actual val value: String
) {
    actual fun normalized(): String {
        return value.trim().lowercase()
    }

    fun asSystemPropertyKey(): String {
        // Этот method доступен только JVM code, если type visible there.
        return "kotlingo.$value"
    }
}
```

Common code:

```kotlin
// commonMain/kotlin/platform/TokenUsage.kt
package platform

fun renderToken(token: PlatformToken): String {
    // Доступны только expected members.
    return token.normalized()
}
```

## Объекты и константы

`expect object` полезен для одноэлементных платформенных сервисов с небольшой поверхностью.

```kotlin
// commonMain/kotlin/platform/LineSeparator.kt
package platform

expect object PlatformText {
    val lineSeparator: String
}
```

```kotlin
// jvmMain/kotlin/platform/LineSeparatorJvm.kt
package platform

actual object PlatformText {
    actual val lineSeparator: String = System.lineSeparator()
}
```

Используйте объект осторожно: глобальный синглтон может скрыть зависимости. С точки зрения тестируемости часто выигрывает внедрение интерфейса.

## Введите псевдонимы как фактические значения

Фактическое объявление может быть псевдонимом существующего типа платформы, если оно соответствует ожидаемой форме API.

Common declaration:

```kotlin
// commonMain/kotlin/platform/DateFormatter.kt
package platform

expect class PlatformDateFormatter(pattern: String) {
    fun format(timestampMillis: Long): String
}
```

Вместо псевдонима типов JVM часто требуется оболочка, потому что `java.text.SimpleDateFormat` не соответствует этому конкретному контракту конструктора/члена безопасно для общих ожиданий.

Wrapper actual:

```kotlin
// jvmMain/kotlin/platform/DateFormatterJvm.kt
package platform

import java.text.SimpleDateFormat
import java.util.Date
import java.util.Locale

actual class PlatformDateFormatter actual constructor(
    pattern: String
) {
    private val delegate = SimpleDateFormat(pattern, Locale.US)

    actual fun format(timestampMillis: Long): String {
        return delegate.format(Date(timestampMillis))
    }
}
```

Фактический Typealias лучше всего подходит для простого повторного использования типов.

```kotlin
// commonMain/kotlin/platform/PlatformRegex.kt
package platform

expect class PlatformRegex(pattern: String) {
    fun matches(input: String): Boolean
}
```

```kotlin
// jvmMain/kotlin/platform/PlatformRegexJvm.kt
package platform

actual typealias PlatformRegex = kotlin.text.Regex
```

Это работает, поскольку `Regex` имеет совместимый конструктор и член `matches`.

<MouseTrap>
`actual typealias` — это не волшебный адаптер. Существующий тип платформы уже должен соответствовать ожидаемому контракту. Если имена, конструктор или члены различаются, напишите фактический класс оболочки.
</MouseTrap>

## Expected annotations

Также можно ожидать аннотаций.

```kotlin
// commonMain/kotlin/platform/SerializableName.kt
package platform

expect annotation class PlatformSerializableName(
    val name: String
)
```

Фактический JVM может сопоставляться с аннотацией платформы, если контракт соответствует, или определять реальную аннотацию:

```kotlin
// jvmMain/kotlin/platform/SerializableNameJvm.kt
package platform

actual annotation class PlatformSerializableName actual constructor(
    actual val name: String
)
```

Используйте ожидаемые аннотации экономно. Многие мультиплатформенные библиотеки уже предоставляют общие аннотации, например аннотации сериализации.

## Visibility

Видимость — это часть дизайна API.

```kotlin
// commonMain/kotlin/platform/InternalSecret.kt
package platform

internal expect fun machineFingerprint(): String
```

```kotlin
// jvmMain/kotlin/platform/InternalSecretJvm.kt
package platform

internal actual fun machineFingerprint(): String {
    return System.getProperty("os.name") + ":" + System.getProperty("os.arch")
}
```

Не публикуйте объявления ожидаемых значений, если только вы не хотите, чтобы они были включены в ваш общий API. Декларация публичных ожиданий становится публичным контрактом для каждой цели.

## Иерархия исходного набора и фактические данные

Фактическая реализация должна находиться в исходном наборе, который зависит от ожидаемого исходного набора.

Пример:

```text
commonMain
  nativeMain
    appleMain
      iosMain
        iosArm64Main
        iosSimulatorArm64Main
  jvmMain
```

Если ожидание находится в `commonMain`, то возможные фактические местоположения включают:

- `jvmMain` for JVM;
- `iosMain` для устройства iOS и симулятора;
- `appleMain` для всех целей Apple;
- `nativeMain` для всех Native целей;
- конкретный целевой исходный набор, если реализация действительно специфична для цели.

```kotlin
// commonMain/kotlin/platform/DeviceFamily.kt
package platform

expect fun deviceFamily(): String
```

```kotlin
// appleMain/kotlin/platform/DeviceFamilyApple.kt
package platform

actual fun deviceFamily(): String {
    return "apple"
}
```

Это фактическое значение может удовлетворить все цели в `appleMain`.

## Интерфейс или ожидаемое/фактическое

Отдавайте предпочтение интерфейсу плюс инъекция, когда:

- реализация является зависимостью;
- tests need фиктивный implementation;
- на одной платформе может существовать несколько реализаций;
- вызывающий код should choose implementation;
- lifecycle matters;
- задействована служба с отслеживанием состояния.

```kotlin
// commonMain/kotlin/platform/Logger.kt
package platform

interface Logger {
    fun info(message: String)

    fun error(message: String, throwable: Throwable?)
}
```

JVM implementation:

```kotlin
// jvmMain/kotlin/platform/JvmLogger.kt
package platform

class JvmLogger : Logger {
    override fun info(message: String) {
        println("INFO $message")
    }

    override fun error(message: String, throwable: Throwable?) {
        println("ERROR $message ${throwable?.message}")
    }
}
```

Предпочитайте ожидаемое/фактическое, когда:

- общему коду нужен примитив платформы;
- реализация естественно одна на платформу;
- внедрение зависимостей добавило бы шума;
- Поверхность API небольшая и стабильная;
- специфичная для платформы реализация неизбежна.

```kotlin
// commonMain/kotlin/platform/Platform.kt
package platform

expect fun defaultLocaleTag(): String
```

<CatTip>
Если common code спрашивает «какая сейчас platform-specific value», `expect/actual` удобен. Если common code просит «выполни сервисную работу», чаще лучше interface.
</CatTip>

## Боевой example: clock and UUID

Common:

```kotlin
// commonMain/kotlin/audit/AuditEventFactory.kt
package audit

import platform.PlatformClock
import platform.randomUuidString

class AuditEventFactory {
    fun lessonOpened(userId: String, slug: String): AuditEvent {
        return AuditEvent(
            id = randomUuidString(),
            userId = userId,
            slug = slug,
            createdAtMillis = PlatformClock.nowMillis()
        )
    }
}

data class AuditEvent(
    val id: String,
    val userId: String,
    val slug: String,
    val createdAtMillis: Long
)
```

Expected declarations:

```kotlin
// commonMain/kotlin/platform/PlatformServices.kt
package platform

expect object PlatformClock {
    fun nowMillis(): Long
}

expect fun randomUuidString(): String
```

JVM actuals:

```kotlin
// jvmMain/kotlin/platform/PlatformServicesJvm.kt
package platform

import java.util.UUID

actual object PlatformClock {
    actual fun nowMillis(): Long {
        return System.currentTimeMillis()
    }
}

actual fun randomUuidString(): String {
    return UUID.randomUUID().toString()
}
```

iOS actuals:

```kotlin
// iosMain/kotlin/platform/PlatformServicesIos.kt
package platform

import platform.Foundation.NSDate
import platform.Foundation.NSUUID
import platform.Foundation.timeIntervalSince1970

actual object PlatformClock {
    actual fun nowMillis(): Long {
        return (NSDate().timeIntervalSince1970 * 1_000).toLong()
    }
}

actual fun randomUuidString(): String {
    return NSUUID().UUIDString()
}
```

Эта конструкция сохраняет `AuditEventFactory` в общем коде, в то время как наборы исходных кодов платформы предоставляют информацию о времени и UUID.

## Common mistakes

Частые ошибки:

- объявление `expect`, когда интерфейс будет более тестируемым;
- предоставление имен в форме платформы в общем API;
- использование `actual typealias` там, где нужна обертка;
- размещение актуального в неправильном исходном наборе;
- забывание актуального для одной цели;
- изменение ожидаемого публичного заявления без обновления всех фактических данных;
- hiding complex platform lifecycle behind tiny global expected object;
- ожидание, что общий код увидит только фактические члены.

Хороший ожидаемый/фактический API небольшой, явный и стабильный.
