## Dispatcher решает, где выполняется coroutine

`CoroutineDispatcher` — элемент `CoroutineContext`, который определяет, на каком thread, thread pool или event loop будет выполняться coroutine.

```kotlin
import kotlinx.coroutines.Dispatchers
import kotlinx.coroutines.launch
import kotlinx.coroutines.runBlocking

fun main() = runBlocking {
    launch(Dispatchers.Default) {
        println("CPU work: ${Thread.currentThread().name}")
    }

    launch(Dispatchers.IO) {
        println("Blocking I/O: ${Thread.currentThread().name}")
    }
}
```

Dispatcher не меняет смысл structured concurrency. `launch(Dispatchers.IO)` все еще child текущего scope, если ты не подменил `Job`.

<CatTip>
Dispatcher — это маршрутная карта coroutine: работа та же, но лапы ставятся на подходящую дорожку.
</CatTip>

## Dispatcher inheritance

Если builder вызван без dispatcher, child coroutine inherits dispatcher from parent scope.

```kotlin
import kotlinx.coroutines.CoroutineName
import kotlinx.coroutines.Dispatchers
import kotlinx.coroutines.launch
import kotlinx.coroutines.runBlocking

fun inheritanceExample() = runBlocking(Dispatchers.Default + CoroutineName("parent")) {
    launch {
        // Child inherited Dispatchers.Default.
        println("Inherited dispatcher: ${Thread.currentThread().name}")
    }
}
```

Можно override dispatcher для отдельной child coroutine.

```kotlin
import kotlinx.coroutines.Dispatchers
import kotlinx.coroutines.launch
import kotlinx.coroutines.runBlocking

fun overrideExample() = runBlocking(Dispatchers.Default) {
    launch(Dispatchers.IO) {
        // Эта child coroutine выполняет blocking I/O на I/O dispatcher.
        println("I/O child: ${Thread.currentThread().name}")
    }
}
```

Не добавляй новый `Job()` ради смены dispatcher. Для dispatcher достаточно передать `Dispatchers.IO`, `Dispatchers.Default` или custom dispatcher.

## `Dispatchers.Default`

`Dispatchers.Default` предназначен для CPU-bound work.

Примеры:

- parsing JSON/Markdown без blocking I/O;
- сортировка больших коллекций;
- compression;
- cryptographic hashing;
- rendering templates;
- анализ текста;
- поиск по local in-memory index.

```kotlin
import kotlinx.coroutines.Dispatchers
import kotlinx.coroutines.withContext

suspend fun calculateSkillScore(answers: List<Answer>): Int =
    withContext(Dispatchers.Default) {
        answers.sumOf { answer ->
            if (answer.isCorrect) 10 else -2
        }.coerceAtLeast(0)
    }

data class Answer(
    val questionId: String,
    val isCorrect: Boolean
)
```

`Default` обычно использует shared pool, размер которого связан с количеством CPU cores. Поэтому не нужно отправлять туда blocking I/O: blocking call займет worker thread и ухудшит выполнение CPU tasks.

<MouseTrap>
`Dispatchers.Default` не означает «любой background». Для blocking database/JDBC/filesystem/synchronous HTTP используй `Dispatchers.IO` или driver с настоящей suspending поддержкой.
</MouseTrap>

## `Dispatchers.IO`

`Dispatchers.IO` предназначен для blocking I/O.

Примеры:

- `Files.readString`;
- old JDBC driver;
- synchronous HTTP client;
- blocking queue;
- legacy SDK, который blocks current thread;
- чтение больших файлов обычными Java APIs.

```kotlin
import kotlinx.coroutines.Dispatchers
import kotlinx.coroutines.withContext
import java.nio.file.Files
import java.nio.file.Path

suspend fun readLessonMarkdown(path: Path): String =
    withContext(Dispatchers.IO) {
        // Files.readString блокирует thread, поэтому уводим его на I/O dispatcher.
        Files.readString(path)
    }
```

`Dispatchers.IO` elastic и умеет расширяться под blocking I/O лучше, чем `Default`. Но это не разрешение запускать бесконечно много тяжелых operations. Для внешних сервисов часто нужен отдельный limit.

```kotlin
import kotlinx.coroutines.Dispatchers
import kotlinx.coroutines.async
import kotlinx.coroutines.awaitAll
import kotlinx.coroutines.coroutineScope
import kotlinx.coroutines.withContext

class BlockingAvatarClient {
    private val avatarDispatcher = Dispatchers.IO.limitedParallelism(8)

    suspend fun loadAll(userIds: List<String>): List<ByteArray> =
        withContext(avatarDispatcher) {
            coroutineScope {
                userIds.map { userId ->
                    async {
                        loadOneBlocking(userId)
                    }
                }.awaitAll()
            }
        }

    private fun loadOneBlocking(userId: String): ByteArray {
        // Представим synchronous SDK, который блокирует thread.
        return "avatar:$userId".encodeToByteArray()
    }
}
```

## `Dispatchers.Main`

`Dispatchers.Main` — dispatcher для main/UI thread.

Он доступен, когда подключен platform module:

- Android: `kotlinx-coroutines-android`;
- Swing: `kotlinx-coroutines-swing`;
- JavaFX: `kotlinx-coroutines-javafx`;
- platform-specific UI runtimes.

```kotlin
import kotlinx.coroutines.CoroutineScope
import kotlinx.coroutines.Dispatchers
import kotlinx.coroutines.launch
import kotlinx.coroutines.withContext

class LessonScreen(
    private val scope: CoroutineScope,
    private val repository: LessonRepository
) {
    fun show(slug: String) {
        scope.launch(Dispatchers.Main) {
            showLoading()

            val lesson = withContext(Dispatchers.IO) {
                repository.loadLessonBlocking(slug)
            }

            // После withContext coroutine возвращается на Main dispatcher.
            showLesson(lesson)
        }
    }

    private fun showLoading() {
        println("Показываем загрузку")
    }

    private fun showLesson(lesson: Lesson) {
        println("Показываем ${lesson.title}")
    }
}

class LessonRepository {
    fun loadLessonBlocking(slug: String): Lesson {
        return Lesson("Урок $slug", "Текст урока")
    }
}

data class Lesson(
    val title: String,
    val body: String
)
```

На backend JVM без UI module `Dispatchers.Main` обычно недоступен.

## `Dispatchers.Unconfined`

`Dispatchers.Unconfined` начинает выполнение в caller thread, но после suspension resumes там, где решит suspending function.

```kotlin
import kotlinx.coroutines.Dispatchers
import kotlinx.coroutines.delay
import kotlinx.coroutines.launch
import kotlinx.coroutines.runBlocking

fun unconfinedExample() = runBlocking {
    launch(Dispatchers.Unconfined) {
        println("До delay: ${Thread.currentThread().name}")
        delay(100)
        println("После delay: ${Thread.currentThread().name}")
    }
}
```

Это advanced mechanism для редких случаев:

- когда immediate execution важнее обычной dispatch scheduling;
- в отдельных library internals;
- в некоторых test scenarios.

Для application code `Unconfined` почти никогда не является хорошим default.

<MouseTrap>
`Dispatchers.Unconfined` не означает «быстрее». Он означает «без стабильной привязки к thread после suspension». Для UI, shared mutable state и predictable execution это опасный выбор.
</MouseTrap>

## `withContext`

`withContext` switches context for a block and returns result.

```kotlin
import kotlinx.coroutines.Dispatchers
import kotlinx.coroutines.withContext
import java.nio.file.Files
import java.nio.file.Path

suspend fun loadAndParseLesson(path: Path): ParsedLesson =
    withContext(Dispatchers.IO) {
        val text = Files.readString(path)

        ParsedLesson(
            slug = path.fileName.toString().substringBeforeLast("."),
            title = text.lineSequence().firstOrNull().orEmpty()
        )
    }

data class ParsedLesson(
    val slug: String,
    val title: String
)
```

Если block завершился успешно, coroutine resumes in previous context.

```kotlin
import kotlinx.coroutines.Dispatchers
import kotlinx.coroutines.withContext

suspend fun renderThenSave(slug: String): String {
    val html = withContext(Dispatchers.Default) {
        // CPU-bound rendering.
        "<h1>$slug</h1>"
    }

    withContext(Dispatchers.IO) {
        // Blocking save можно увести на I/O dispatcher.
        println("Сохраняем $html")
    }

    return html
}
```

`withContext` cooperates with cancellation. Если parent scope отменен до возврата value, caller не продолжит работать с уже неактуальным результатом.

## `limitedParallelism`

`limitedParallelism` создает view dispatcher-а с ограничением параллельных executions.

Это полезно, когда общий dispatcher подходит по типу работы, но конкретный external resource имеет лимит.

```kotlin
import kotlinx.coroutines.Dispatchers
import kotlinx.coroutines.async
import kotlinx.coroutines.awaitAll
import kotlinx.coroutines.coroutineScope
import kotlinx.coroutines.withContext

class CertificateRenderer {
    private val renderDispatcher = Dispatchers.Default.limitedParallelism(2)

    suspend fun renderAll(userIds: List<String>): List<String> =
        withContext(renderDispatcher) {
            coroutineScope {
                userIds.map { userId ->
                    async {
                        renderCertificate(userId)
                    }
                }.awaitAll()
            }
        }

    private fun renderCertificate(userId: String): String {
        // CPU-heavy rendering, ограничиваем parallelism до 2.
        return "certificate:$userId"
    }
}
```

Для blocking service:

```kotlin
import kotlinx.coroutines.Dispatchers
import kotlinx.coroutines.withContext

class LegacySearchClient {
    private val dispatcher = Dispatchers.IO.limitedParallelism(4)

    suspend fun search(query: String): List<String> =
        withContext(dispatcher) {
            searchBlocking(query)
        }

    private fun searchBlocking(query: String): List<String> {
        return listOf("Результат для $query")
    }
}
```

## Custom dispatchers

Custom dispatcher нужен, когда:

- есть dedicated executor;
- external API требует single thread;
- нужно изолировать legacy blocking SDK;
- нужно контролировать shutdown executor-а;
- framework предоставляет свой event loop.

```kotlin
import kotlinx.coroutines.asCoroutineDispatcher
import kotlinx.coroutines.runBlocking
import kotlinx.coroutines.withContext
import java.util.concurrent.Executors

fun customDispatcherExample() {
    val executor = Executors.newSingleThreadExecutor { runnable ->
        Thread(runnable, "kotlingo-legacy-sdk")
    }

    val dispatcher = executor.asCoroutineDispatcher()

    try {
        runBlocking {
            withContext(dispatcher) {
                println("Работаем на ${Thread.currentThread().name}")
            }
        }
    } finally {
        // Executor принадлежит нам, значит мы обязаны его закрыть.
        dispatcher.close()
        executor.shutdown()
    }
}
```

Dedicated thread is expensive. Не создавай new dispatcher per request.

## `newSingleThreadContext`

`newSingleThreadContext` создает dedicated thread. Это удобно для demos, но в production чаще лучше executor plus `asCoroutineDispatcher`, потому что lifecycle становится явным.

```kotlin
import kotlinx.coroutines.newSingleThreadContext
import kotlinx.coroutines.runBlocking
import kotlinx.coroutines.withContext

fun singleThreadContextExample() {
    newSingleThreadContext("lesson-writer").use { dispatcher ->
        runBlocking {
            withContext(dispatcher) {
                println("Пишем lesson file на dedicated thread")
            }
        }
    }
}
```

Если не закрыть dedicated dispatcher, thread останется жить.

## Production pattern: dispatcher ownership

```kotlin
import kotlinx.coroutines.CoroutineDispatcher
import kotlinx.coroutines.Dispatchers
import kotlinx.coroutines.withContext
import java.nio.file.Files
import java.nio.file.Path

class LessonFileRepository(
    private val ioDispatcher: CoroutineDispatcher = Dispatchers.IO
) {
    suspend fun load(path: Path): LessonFile =
        withContext(ioDispatcher) {
            val lines = Files.readAllLines(path)

            LessonFile(
                path = path.toString(),
                title = lines.firstOrNull().orEmpty(),
                body = lines.drop(1).joinToString(separator = "\n")
            )
        }
}

data class LessonFile(
    val path: String,
    val title: String,
    val body: String
)
```

Почему dispatcher передается через constructor:

- test can inject test dispatcher;
- class не hardcodes global dispatcher в каждом method;
- production code сохраняет sensible default;
- ownership и intent становятся видимыми.

<CatTip>
Хороший dispatcher design обычно читается по глаголам: считаем на `Default`, блокирующее читаем на `IO`, UI обновляем на `Main`, внешние лимиты выражаем через `limitedParallelism`.
</CatTip>
