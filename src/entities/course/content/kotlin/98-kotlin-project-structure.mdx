## Структура проекта — это архитектурный контракт

Project structure отвечает на вопросы:

- где живет domain logic;
- где platform adapters;
- где public API;
- где тесты;
- какие modules can depend on which modules;
- где границы features;
- как new developer найдет нужный code;
- как избежать свалки `utils`;
- как менять implementation without breaking callers.

Плохая структура обычно выглядит удобно первые две недели, а потом превращает каждое изменение в поиск по всему проекту.

<CatTip>
Хорошая структура проекта — это кошачья карта территории: понятно, где миска domain logic, где лежанка tests, где когтеточка platform adapters, и куда не надо тащить грязные зависимости.
</CatTip>

## Пакет по слоям и пакет по функциям

Layer-based layout:

```text
src/main/kotlin/dev/kotlingo/
  controllers/
  services/
  repositories/
  models/
  utils/
```

It looks simple, but feature code is spread across folders. To change lesson прогресс, you jump between `controllers`, `services`, `repositories`, `models`.

Feature-based layout:

```text
src/main/kotlin/dev/kotlingo/
  lesson/
    api/
    domain/
    persistence/
    web/
  progress/
    api/
    domain/
    persistence/
    web/
  shared/
    time/
    validation/
```

Feature-based layout keeps related code close.

```text
progress/
  domain/
    Progress.kt
    MarkLessonCompleted.kt
  persistence/
    ProgressRepository.kt
    JdbcProgressRepository.kt
  web/
    ProgressController.kt
    ProgressDto.kt
  api/
    ProgressFacade.kt
```

Используйте имена на основе слоев внутри объекта, а не во всем приложении.

## Package naming

Имена пакетов Kotlin должны быть строчными и стабильными.

Хорошо:

```text
dev.kotlingo.progress.domain
dev.kotlingo.progress.persistence
dev.kotlingo.lesson.api
dev.kotlingo.shared.time
```

Избегай:

```text
dev.kotlingo.ProgressDomain
dev.kotlingo.newFeature
dev.kotlingo.commonUtils
dev.kotlingo.misc
```

Имя пакета — это не просто метка папки. Это влияет на импорт, видимость, ограничения закрытой иерархии и читаемость API.

## API boundaries

Публичный API должен быть небольшим и целенаправленным.

```kotlin
package dev.kotlingo.progress.api

class ProgressFacade(
    private val markCompleted: MarkLessonCompleted,
    private val readProgress: ReadProgress
) {
    suspend fun complete(userId: String, slug: String): ProgressSnapshot {
        return markCompleted.execute(userId, slug)
    }

    suspend fun snapshot(userId: String): ProgressSnapshot {
        return readProgress.execute(userId)
    }
}
```

Internal domain сценарий использования:

```kotlin
package dev.kotlingo.progress.api

internal class MarkLessonCompleted(
    private val repository: ProgressRepository
) {
    suspend fun execute(userId: String, slug: String): ProgressSnapshot {
        require(userId.isNotBlank()) { "userId не должен быть пустым" }
        require(slug.isNotBlank()) { "slug не должен быть пустым" }

        repository.markCompleted(userId, slug)

        return repository.snapshot(userId)
    }
}
```

Репозиторий contract:

```kotlin
package dev.kotlingo.progress.api

internal interface ProgressRepository {
    suspend fun markCompleted(userId: String, slug: String)

    suspend fun snapshot(userId: String): ProgressSnapshot
}
```

Общедоступная поверхность — `ProgressFacade` и `ProgressSnapshot`; детали реализации остаются внутренними.

<MouseTrap>
Если every class is public, module has no architecture. Public по умолчанию is convenient locally and expensive after publication.
</MouseTrap>

## Domain models

Domain model should express business meaning, not база данных or JSON shape.

```kotlin
package dev.kotlingo.progress.domain

data class ProgressSnapshot(
    val userId: UserId,
    val completedLessons: Set<LessonSlug>,
    val totalLessons: Int
) {
    val percent: Int
        get() = if (totalLessons == 0) 0 else completedLessons.size * 100 / totalLessons
}

@JvmInline
value class UserId(
    val value: String
) {
    init {
        require(value.isNotBlank()) { "UserId не должен быть пустым" }
    }
}

@JvmInline
value class LessonSlug(
    val value: String
) {
    init {
        require(value.matches(Regex("[a-z0-9-]+"))) {
            "LessonSlug должен содержать только lower-case буквы, цифры и дефисы"
        }
    }
}
```

Persistence DTO:

```kotlin
package dev.kotlingo.progress.persistence

internal data class ProgressRow(
    val userId: String,
    val slug: String,
    val completedAtEpochMillis: Long
)
```

Маппер:

```kotlin
package dev.kotlingo.progress.persistence

import dev.kotlingo.progress.domain.LessonSlug
import dev.kotlingo.progress.domain.UserId

internal fun ProgressRow.toDomainUserId(): UserId {
    return UserId(userId)
}

internal fun ProgressRow.toDomainSlug(): LessonSlug {
    return LessonSlug(slug)
}
```

Не позволяй строке базы данных становиться доменной моделью только потому, что поля похожи.

## Avoiding anemic utility dumping

Пакет `utils` обычно означает, что разработка была остановлена ​​слишком рано.

Плохо:

```text
utils/
  DateUtils.kt
  StringUtils.kt
  ValidationUtils.kt
  ProgressUtils.kt
```

Better:

```text
shared/time/
  Clock.kt
  SystemClock.kt
lesson/slug/
  LessonSlug.kt
  LessonSlugNormalizer.kt
progress/domain/
  ProgressPolicy.kt
```

Bad utility:

```kotlin
package dev.kotlingo.utils

object StringUtils {
    fun normalizeSlug(value: String): String {
        return value.trim().lowercase().replace(" ", "-")
    }
}
```

Better domain type:

```kotlin
package dev.kotlingo.lesson.slug

@JvmInline
value class LessonSlug(
    val value: String
) {
    init {
        require(value.isNotBlank()) { "Slug не должен быть пустым" }
    }

    companion object {
        fun fromTitle(title: String): LessonSlug {
            val normalized = title.trim()
                .lowercase()
                .replace(Regex("\\s+"), "-")

            return LessonSlug(normalized)
        }
    }
}
```

Теперь поведение живет рядом с концепцией, которую оно защищает.

<CatTip>
Если хочется создать `CatUtils`, остановись и спроси: «какому понятию принадлежит эта функция?» Почти всегда у функции уже есть дом.
</CatTip>

## Module boundaries

Packages help organization. Gradle modules enforce dependencies.

Possible modules:

```text
kotlingo-domain
kotlingo-progress
kotlingo-lesson
kotlingo-api
kotlingo-jvm-server
kotlingo-android
kotlingo-shared-test
```

Dependency direction:

```text
jvm-server -> api -> progress -> domain
android    -> api -> progress -> domain
```

Домен не должен зависеть от сервера или Android.

```kotlin
// domain module
package dev.kotlingo.domain

interface Clock {
    fun nowMillis(): Long
}
```

```kotlin
// jvm-server module
package dev.kotlingo.server

import dev.kotlingo.domain.Clock

class SystemClock : Clock {
    override fun nowMillis(): Long {
        return System.currentTimeMillis()
    }
}
```

## Dependency rules

Healthy dependency direction:

- UI depends on сценарии использования;
- HTTP controllers depend on сценарии использования;
- сценарии использования depend on репозиторий interfaces;
- репозиторий implementations depend on база данных or network clients;
- домен не зависит от адаптеров;
- адаптеры преобразуют внешнюю форму в форму домена.

```kotlin
package dev.kotlingo.lesson.domain

class PublishLesson(
    private val repository: LessonRepository
) {
    suspend fun execute(draft: LessonDraft): PublishedLesson {
        draft.validate()

        val lesson = PublishedLesson(
            slug = draft.slug,
            title = draft.title,
            body = draft.body
        )

        repository.save(lesson)

        return lesson
    }
}

interface LessonRepository {
    suspend fun save(lesson: PublishedLesson)
}
```

Adapter:

```kotlin
package dev.kotlingo.lesson.persistence

import dev.kotlingo.lesson.domain.LessonRepository
import dev.kotlingo.lesson.domain.PublishedLesson

class InMemoryLessonRepository : LessonRepository {
    private val lessons = mutableMapOf<String, PublishedLesson>()

    override suspend fun save(lesson: PublishedLesson) {
        lessons[lesson.slug] = lesson
    }
}
```

## File organization

Kotlin допускает несколько объявлений для каждого файла, но используйте это сознательно.

Хороший файл с одной концепцией:

```kotlin
package dev.kotlingo.lesson.domain

data class LessonDraft(
    val slug: String,
    val title: String,
    val body: String
) {
    fun validate() {
        require(slug.isNotBlank()) { "Slug не должен быть пустым" }
        require(title.isNotBlank()) { "Title не должен быть пустым" }
        require(body.isNotBlank()) { "Body не должен быть пустым" }
    }
}
```

Хороший файл sealed-иерархии:

```kotlin
package dev.kotlingo.lesson.domain

sealed interface PublishResult {
    data class Published(
        val lesson: PublishedLesson
    ) : PublishResult

    data class Rejected(
        val reason: String
    ) : PublishResult
}
```

Избегайте огромных файлов с именами:

```text
Helpers.kt
Common.kt
Extensions.kt
AllModels.kt
Everything.kt
```

## Тестирует поведение зеркала, а не папок вслепую

Тесты должны соответствовать границам поведения.

```text
src/test/kotlin/dev/kotlingo/lesson/domain/
  PublishLessonTest.kt
src/test/kotlin/dev/kotlingo/lesson/persistence/
  InMemoryLessonRepositoryTest.kt
src/test/kotlin/dev/kotlingo/progress/domain/
  ProgressSnapshotTest.kt
```

Test example:

```kotlin
package dev.kotlingo.lesson.domain

import kotlin.test.Test
import kotlin.test.assertFailsWith

class LessonDraftTest {
    @Test
    fun rejectsBlankTitle() {
        val draft = LessonDraft(
            slug = "98-kotlin-project-structure",
            title = "",
            body = "Текст"
        )

        assertFailsWith<IllegalArgumentException> {
            draft.validate()
        }
    }
}
```

Name tests after behavior, not деталь реализации.

## Multiplatform structure

For KMP:

```text
shared/
  src/
    commonMain/kotlin/dev/kotlingo/
      lesson/domain/
      progress/domain/
      shared/time/
    androidMain/kotlin/dev/kotlingo/
      progress/android/
    jvmMain/kotlin/dev/kotlingo/
      progress/jdbc/
    iosMain/kotlin/dev/kotlingo/
      progress/ios/
```

Общий пакет не должен притворяться, что платформы не существует. Адаптеры должны четко соответствовать платформе.

Хорошо:

```text
progress/
  domain/
  android/
  jdbc/
  ios/
```

Плохо:

```text
progress/
  impl/
  impl2/
  platform/
  real/
```

## Проверочный список для боевого кода

Структура проекта Kotlin работоспособна, когда:

- функции легко найти;
- имена пакетов строчные и стабильные;
- доменные модели выражают бизнес-правила;
- DTO не просачиваются в домен;
- публичный API небольшой;
- `internal` используется намеренно;
- dependencies point inward toward domain;
- адаптеры платформы остаются на границах;
- тесты охватывают поведение на значимых границах;
- пакеты утилит редки и специфичны;
- границы модулей обеспечивают архитектуру, в которой пакетов недостаточно.

<MouseTrap>
Самый опасный запах структуры — когда разработчики говорят «я просто положу это рядом с похожим файлом». Похожесть по синтаксису не равна архитектурной принадлежности.
</MouseTrap>
