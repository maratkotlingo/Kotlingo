## Property — контракт доступа к значению

Property в Kotlin объединяет имя, тип и операции чтения/записи. Это не обязательно прямое поле памяти.

```kotlin
class Lesson(
    val title: String,
    var completed: Boolean
)
```

`title` можно читать, `completed` — читать и изменять:

```kotlin
val lesson = Lesson("Свойства", completed = false)
println(lesson.title)

lesson.completed = true
println(lesson.completed)
```

Синтаксис выглядит как доступ к полю, но логически вызывает getter, а для `var` — setter.

## `val` и `var`

`val` объявляет только для чтения property: после инициализации через этот контракт нельзя присвоить другое значение.

```kotlin
class Course(val lessons: MutableList<String>)

fun main() {
    val course = Course(mutableListOf("Классы"))
    course.lessons += "Свойства"
}
```

`val` не делает объект глубоко immutable. Ссылка на список неизменяема, но сам `MutableList` можно менять.

Для неизменяемого внешнего контракта возвращай только для чтения view/snapshot и не отдавай mutable storage:

```kotlin
class SafeCourse(initialLessons: List<String>) {
    private val mutableLessons = initialLessons.toMutableList()

    val lessons: List<String>
        get() = mutableLessons.toList()

    fun addLesson(title: String) {
        require(title.isNotBlank()) { "Название не должно быть пустым" }
        mutableLessons += title.trim()
    }
}
```

`var` разрешает присваивание через setter:

```kotlin
class Progress(var completed: Int)
```

Public `var` уместен для простых data holders, UI-состояние и явно mutable моделей. В domain objects setter часто должен защищать диапазон или быть private.

<CatTip title="Котосовет: начинай с val">
Выбирай `val`, пока изменение не является частью требований. Чем меньше mutable surface, тем проще reasoning, concurrency и сохранение инвариантов.
</CatTip>

## Тип property

Тип можно вывести из initializer:

```kotlin
val courseName = "kotlingo"
var activeLesson = 23
```

Без initializer тип обязателен:

```kotlin
class Lesson(rawTitle: String) {
    val title: String

    init {
        title = rawTitle.trim()
    }
}
```

Компилятор проверяет definite assignment: `val` должен получить значение ровно один раз на каждом успешном пути constructor.

Для public API явный тип защищает контракт от случайного изменения initializer:

```kotlin
val lessonCount: Long = 108
```

## Getter

У каждой property есть getter. Default getter возвращает сохранённое значение.

```kotlin
class Lesson(val title: String) {
    val displayTitle: String
        get() = title.trim().ifEmpty { "Без названия" }
}
```

`displayTitle` вычисляется при каждом чтении и не требует отдельного storage.

```kotlin
val lesson = Lesson("  Свойства  ")
println(lesson.displayTitle)
```

### Block getter

```kotlin
class CourseProgress(
    private val completed: Int,
    private val total: Int
) {
    val percent: Int
        get() {
            if (total == 0) return 0
            return completed * 100 / total
        }
}
```

Getter обязан вернуть значение declared type. Для нескольких шагов block body читается лучше expression body.

### Getter вызывается каждый раз

```kotlin
class Clock {
    val currentMillis: Long
        get() = System.currentTimeMillis()
}
```

Два чтения могут вернуть разные значения. Property-синтаксис обычно обещает дешёвое наблюдение. Для I/O, network или тяжёлого вычисления используй функцию `load...()`/`calculate...()`.

<MouseTrap title="Мышеловка: computed property не кешируется">
Getter выполняется при каждом обращении. Если результат дорогой и стабилен, используй lazy property или явный cache с правилами invalidation. Не прячь тяжёлую работу за невинным `object.value`.
</MouseTrap>

## Setter

У `var` есть setter с неявным параметром `value`.

```kotlin
class StudentLevel(initialLevel: Int) {
    var level: Int = initialLevel
        set(value) {
            require(value in 1..10) { "Уровень должен быть от 1 до 10" }
            field = value
        }

    init {
        require(initialLevel in 1..10) { "Начальный уровень должен быть от 1 до 10" }
    }
}
```

`field` — специальный backing field текущей property. Присваивание `field = value` сохраняет новое значение.

Если написать `level = value` внутри setter, setter вызовет сам себя бесконечно:

```kotlin
// set(value) {
//     level = value
// }
```

Такой код завершится переполнением стека.

### Имя параметра setter

Параметр можно переименовать:

```kotlin
var title: String = "Без названия"
    set(newTitle) {
        field = newTitle.trim()
    }
```

`value` — convention, а не обязательное имя.

### Visibility setter

Getter и property могут быть public, а setter — ограниченным.

```kotlin
class Progress {
    var completed: Int = 0
        private set

    fun completeNext() {
        completed += 1
    }
}
```

Снаружи `completed` только для чтения, внутри класса изменяется. Setter visibility нельзя сделать шире, чем property.

```kotlin
class PublishedLesson {
    var publishedAt: Long? = null
        internal set
}
```

## Проверка initial value

Пользовательский setter не вызывается автоматически для initializer property.

```kotlin
class Score(initial: Int) {
    var value: Int = initial
        set(newValue) {
            require(newValue in 0..100) { "Балл должен быть от 0 до 100" }
            field = newValue
        }
}
```

Если `initial = -1`, initializer напрямую установит backing field, обходя setter. Поэтому initial value нужно проверять отдельно:

```kotlin
class SafeScore(initial: Int) {
    var value: Int = initial
        set(newValue) {
            require(newValue in 0..100) { "Балл должен быть от 0 до 100" }
            field = newValue
        }

    init {
        require(initial in 0..100) { "Начальный балл должен быть от 0 до 100" }
    }
}
```

Либо нормализуй initializer через отдельную функцию, используемую и setter:

```kotlin
class NormalizedTitle(rawTitle: String) {
    var value: String = normalize(rawTitle)
        set(newValue) {
            field = normalize(newValue)
        }

    private fun normalize(raw: String): String {
        val result = raw.trim()
        require(result.isNotEmpty()) { "Название не должно быть пустым" }
        return result
    }
}
```

## Backing field `field`

Backing field создаётся, если property использует аксессор по умолчанию или пользовательский accessor обращается к `field`.

```kotlin
class Lesson {
    var title: String = "Без названия"
        get() = field
        set(value) {
            field = value.trim()
        }
}
```

`field` доступен только внутри accessor этой property. Нельзя использовать `field` в обычной member-функции или accessor другой property.

Computed property без `field` не хранит значение:

```kotlin
class Rectangle(val width: Int, val height: Int) {
    val area: Int
        get() = width * height
}
```

### Когда backing field отсутствует

```kotlin
val isComplete: Boolean
    get() = completed == total
```

Если оба accessors полностью вычисляют результат и не используют `field`, отдельное поле не нужно.

## Backing property

Backing property — обычная отдельная property, которая хранит mutable/internal representation. Это не специальное ключевое слово.

```kotlin
class LessonCatalog {
    private val _lessons = mutableListOf<String>()

    val lessons: List<String>
        get() = _lessons.toList()

    fun add(title: String) {
        require(title.isNotBlank()) { "Название обязательно" }
        _lessons += title.trim()
    }
}
```

Convention `_lessons` часто используется, когда public property и private storage логически связаны.

Варианты getter:

- `_lessons` как `List` view не позволяет обычную мутацию через public type, но может отражать последующие изменения;
- `_lessons.toList()` создаёт snapshot и дополнительную allocation;
- persistent immutable collections дают другие trade-offs.

Выбор зависит от требований к snapshot semantics и performance.

### Observable state backing property

```kotlin
class SessionState {
    private var _status: String = "idle"

    val status: String
        get() = _status

    fun start() {
        check(_status == "idle") { "Сессия уже запущена" }
        _status = "running"
    }
}
```

Здесь backing property позволяет читать status, но изменение выражено domain method.

## Top-level properties

Property можно объявить на уровне файла.

```kotlin
val supportedDifficulties: Set<String> = setOf(
    "Начальный",
    "Средний",
    "Продвинутый",
    "Эксперт"
)
```

Top-level `val` инициализируется в соответствии с бэкенд/runtime моделью файла. Не помещай туда mutable process-wide state без явной необходимости и concurrency policy.

```kotlin
private val slugSeparator = Regex("""[^a-zа-яё0-9]+""")

fun toSlug(value: String): String {
    return value.lowercase().replace(slugSeparator, "-").trim('-')
}
```

Private top-level property может хранить переиспользуемый immutable helper.

## Compile-time constants: `const val`

`const` сообщает, что значение известно во время компиляции и может быть встроено в call sites.

```kotlin
const val MAX_LESSON_TITLE_LENGTH: Int = 120
const val DEFAULT_CATEGORY: String = "Основы"
```

`const val` разрешён:

- на top level;
- внутри `object`;
- внутри `companion object`.

Он не может быть обычной instance property класса.

Тип должен быть primitive type или `String`, initializer — compile-time constant expression без пользовательский getter.

```kotlin
class LessonRules {
    companion object {
        const val MIN_ORDER: Int = 1
        const val DEFAULT_PUBLISHED: Boolean = false
    }
}
```

Compile-time constants можно использовать в annotations и `when` labels, где требуется константное выражение.

### ABI-риск const

Значение может быть встроено в bytecode клиента. Если библиотека изменит constant с `100` на `120`, уже скомпилированный клиент может продолжить использовать старое значение до recompilation.

<MouseTrap title="Мышеловка: public const встраивается в клиента">
Не публикуй как `const` значение, которое должно меняться независимо от версии клиента: timeout, feature flag, server limit. Для runtime-конфигурации нужен обычный getter или configuration source.
</MouseTrap>

## Обычный `val` против `const val`

```kotlin
val startedAt: Long = System.currentTimeMillis()
```

Это runtime value, поэтому `const` невозможен.

```kotlin
const val MILLIS_PER_SECOND: Long = 1_000L
```

Это compile-time constant.

Обычный `val` может иметь любой тип, вызывать функции и использовать пользовательский getter. `const val` имеет строгие ограничения ради compile-time использования.

## `lateinit`

`lateinit var` откладывает инициализацию non-null property. Компилятор разрешает не задавать значение в constructor, но ответственность переходит к программе.

```kotlin
class LessonController {
    lateinit var repository: LessonRepository

    fun load(id: Int): String {
        return repository.findTitle(id)
    }
}

interface LessonRepository {
    fun findTitle(id: Int): String
}
```

Если вызвать `load` до присваивания `repository`, будет `UninitializedPropertyAccessException`.

### Ограничения `lateinit`

Property должна:

- быть `var`;
- иметь non-null тип;
- не иметь primitive type вроде `Int`, `Boolean`, `Double`;
- не иметь пользовательские getter/setter, несовместимых с механизмом;
- быть instance или top-level property в допустимом контексте;
- не быть nullable, потому что `null` используется внутренне как маркер отсутствия инициализации на JVM-подобной модели.

Нельзя:

```kotlin
// lateinit val title: String
// lateinit var count: Int
// lateinit var title: String?
```

### Проверка `isInitialized`

```kotlin
class LessonController {
    lateinit var repository: LessonRepository

    fun isReady(): Boolean {
        return ::repository.isInitialized
    }
}
```

Проверка доступна через property reference в области, где backing field доступен. Внешний произвольный код не может всегда проверять private `lateinit` другого объекта.

```kotlin
fun loadIfReady(controller: LessonController, id: Int): String? {
    return if (controller.isReady()) controller.load(id) else null
}
```

### Когда `lateinit` уместен

- dependency injection framework устанавливает dependency после создания;
- test fixture инициализируется в настройка;
- Android/UI lifecycle гарантирует позднее появление view binding;
- двухфазный lifecycle задан внешним framework и не может быть выражен constructor.

### Когда лучше не использовать `lateinit`

Если dependency обязательна для любого корректного объекта, передай её constructor:

```kotlin
class SafeLessonController(
    private val repository: LessonRepository
) {
    fun load(id: Int): String {
        return repository.findTitle(id)
    }
}
```

Теперь невозможно создать неготовый controller.

Для optional value используй nullable type:

```kotlin
var selectedLesson: String? = null
```

Для дорогой initialization при первом доступе используй `lazy`, а не `lateinit`:

```kotlin
val lessonIndex: Map<Int, String> by lazy {
    buildMap {
        put(21, "Callable references")
        put(22, "Классы и экземпляры")
        put(23, "Свойства")
    }
}
```

`lazy` и delegated properties изучаются отдельно.

<CatTip title="Котосовет: lateinit — адаптер lifecycle, а не default">
Используй его, только если внешний lifecycle объективно не позволяет constructor injection. Добавь явную проверку готовности или ограничь доступ фазой, где инициализация гарантирована.
</CatTip>

## Properties в interfaces и abstract classes

Interface может объявить property без storage:

```kotlin
interface NamedLesson {
    val title: String
}
```

Реализация предоставляет значение:

```kotlin
class TextLesson(
    override val title: String
) : NamedLesson
```

Abstract class может объявить abstract property или concrete property с backing field. Эти вопросы подробно раскрываются в уроках о наследовании и interfaces.

## Accessor annotations и JVM names

Annotation можно направить на getter/setter use-site target:

```kotlin
class LessonMetadata {
    @get:Deprecated("Используй displayTitle")
    var title: String = "Свойства"

    val displayTitle: String
        get() = title
}
```

Для Java interop annotations вроде `@JvmField`, `@JvmStatic`, `@JvmName` меняют platform exposure, но их следует применять под конкретный interop-контракт. Обычный Kotlin-код должен работать через property accessors.

## Практический пример: безопасная модель прогресса

```kotlin
class CourseProgress(
    totalLessons: Int,
    completedLessons: Int = 0
) {
    companion object {
        const val MIN_TOTAL_LESSONS: Int = 1
    }

    val totalLessons: Int = totalLessons.also { total ->
        require(total >= MIN_TOTAL_LESSONS) {
            "Курс должен содержать хотя бы один урок"
        }
    }

    var completedLessons: Int = validateCompleted(
        completed = completedLessons,
        total = totalLessons
    )
        private set

    private val _events = mutableListOf<String>()

    val events: List<String>
        get() = _events.toList()

    val remainingLessons: Int
        get() = totalLessons - completedLessons

    val percent: Int
        get() = completedLessons * 100 / totalLessons

    val isComplete: Boolean
        get() = completedLessons == totalLessons

    fun completeNext(): Boolean {
        if (isComplete) return false

        completedLessons += 1
        _events += "Завершён урок $completedLessons"
        return true
    }

    fun reset() {
        completedLessons = 0
        _events += "Прогресс сброшен"
    }

    private fun validateCompleted(completed: Int, total: Int): Int {
        require(completed in 0..total) {
            "Завершённые уроки должны находиться в диапазоне от 0 до $total"
        }
        return completed
    }
}

fun main() {
    val progress = CourseProgress(totalLessons = 3)

    while (progress.completeNext()) {
        println("${progress.percent}% — осталось ${progress.remainingLessons}")
    }

    println("Курс завершён: ${progress.isComplete}")
    println(progress.events.joinToString(separator = "\n"))
}
```

Модель использует:

- `const val` для стабильного compile-time правила;
- public `val totalLessons`;
- public `var completedLessons` с private setter;
- backing property `_events` и snapshot getter;
- computed properties без лишнего storage;
- methods для допустимых переходов;
- проверки initial values до публикации объекта.

## Итоги

- property — accessor contract, а не обязательно поле;
- `val` запрещает переназначение через property, но не гарантирует deep immutability;
- `var` предоставляет getter и setter;
- пользовательский getter вычисляется при каждом чтении;
- пользовательский setter должен присваивать `field`, а не property самой себе;
- initializer не проходит через пользовательский setter;
- backing field `field` доступен только внутри accessors;
- backing property — отдельное private storage для controlled public view;
- setter можно сделать private/internal;
- `const val` ограничен compile-time значениями и может встраиваться в clients;
- `lateinit` откладывает обязательную initialization и может упасть при раннем доступе;
- constructor injection безопаснее `lateinit`, когда dependency обязательна;
- computed property подходит дешёвому наблюдению, а тяжёлую работу лучше выражать функцией.
