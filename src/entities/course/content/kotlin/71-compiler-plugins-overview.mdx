## Compiler plugin расширяет компилятор

Compiler plugin подключается к компиляции Kotlin и может менять или дополнять обработку кода: генерировать serializers, трансформировать calls, открывать classes для frameworks, добавлять synthetic constructors.

```kotlin
plugins {
    kotlin("jvm") version "2.4.0"
    kotlin("plugin.serialization") version "2.4.0"
}
```

Compiler plugin глубже KSP. KSP читает symbols и генерирует новые files, но не меняет existing source semantics. Compiler plugin может участвовать в анализе и трансформации компиляции.

<CatTip>
KSP — это кошачий писарь, который читает исходники и дописывает новые файлы. Compiler plugin — это помощник прямо у компилятора, который может менять правила обработки.
</CatTip>

## Когда нужен compiler plugin

Compiler plugin нужен, когда обычной библиотеки недостаточно.

Типичные причины:

- compiler должен сгенерировать code, недоступный через обычные APIs;
- нужны transformations на IR/backend level;
- framework ожидает JVM shape, не совпадающий с idiomatic Kotlin;
- нужно повлиять на bytecode или metadata;
- требуется compile-time integration с language semantics.

Если можно решить задачу обычной функцией, annotation processing или KSP, compiler plugin может быть избыточен.

## Serialization plugin

Kotlin serialization plugin генерирует serializers для classes, помеченных `@Serializable`.

```kotlin
plugins {
    kotlin("plugin.serialization") version "2.4.0"
}

dependencies {
    implementation("org.jetbrains.kotlinx:kotlinx-serialization-json:1.11.0")
}
```

Model:

```kotlin
import kotlinx.serialization.Serializable
import kotlinx.serialization.json.Json

@Serializable
data class LessonDto(
    val slug: String,
    val title: String,
    val minutes: Int
)

val lesson = LessonDto(
    slug = "01-introduction",
    title = "Введение",
    minutes = 14
)

val json = Json.encodeToString(lesson)
```

Plugin нужен, потому что serializer должен знать структуру class compile-time и корректно работать с Kotlin-specific features.

<MouseTrap>
Одной зависимости `kotlinx-serialization-json` недостаточно для `@Serializable` classes. Нужен и runtime library, и serialization compiler plugin.
</MouseTrap>

## Compose compiler

Compose compiler plugin обрабатывает `@Composable` functions и превращает declarative UI calls в код, который умеет recomposition, state tracking и efficient updates.

```kotlin
plugins {
    id("org.jetbrains.kotlin.plugin.compose") version "2.4.0"
}
```

С Kotlin 2.0 Compose compiler находится в Kotlin repository и выпускается совместимо с версией Kotlin.

```kotlin
@Composable
fun LessonTitle(title: String) {
    Text(text = title)
}
```

Этот код выглядит как обычная function, но compiler plugin меняет модель вызовов для Compose runtime.

## All-open plugin

Kotlin classes и members по умолчанию `final`. Некоторые JVM frameworks, например AOP/proxy frameworks, требуют `open` classes.

All-open plugin делает classes с заданными annotations open без явного `open`.

```kotlin
plugins {
    kotlin("plugin.allopen") version "2.4.0"
}

allOpen {
    annotation("com.example.OpenForProxy")
}
```

Annotation:

```kotlin
annotation class OpenForProxy

@OpenForProxy
class LessonService {
    fun publish(slug: String) {
        println("Публикуем $slug")
    }
}
```

После plugin class и members будут открыты для inheritance/proxy according to plugin rules.

Для Spring обычно используют wrapper plugin `kotlin-spring`.

```kotlin
plugins {
    id("org.jetbrains.kotlin.plugin.spring") version "2.4.0"
}
```

## No-arg plugin

No-arg plugin генерирует synthetic zero-argument constructor для classes с заданными annotations.

```kotlin
plugins {
    kotlin("plugin.noarg") version "2.4.0"
}

noArg {
    annotation("com.example.NoArg")
}
```

Annotation и entity:

```kotlin
annotation class NoArg

@NoArg
data class LessonEntity(
    val id: Long,
    val slug: String
)
```

Сгенерированный constructor synthetic: напрямую из Kotlin или Java он обычно не вызывается как обычный public constructor, но reflection-based frameworks могут им воспользоваться.

Для JPA есть wrapper plugin `kotlin-jpa`.

```kotlin
plugins {
    kotlin("plugin.jpa") version "2.4.0"
}
```

`invokeInitializers` управляет тем, будут ли выполняться initializers из synthetic constructor.

```kotlin
noArg {
    invokeInitializers = true
}
```

## Power-assert plugin

Power-assert plugin улучшает diagnostics assertions. Он показывает values sub-expressions в failed assertion.

```kotlin
plugins {
    kotlin("plugin.power-assert") version "2.4.0"
}
```

Настройка:

```kotlin
import org.jetbrains.kotlin.gradle.ExperimentalKotlinGradlePluginApi

@OptIn(ExperimentalKotlinGradlePluginApi::class)
powerAssert {
    // Указываем функции, вызовы которых нужно трансформировать.
    functions = listOf(
        "kotlin.assert",
        "kotlin.test.assertTrue",
        "kotlin.test.assertEquals"
    )
}
```

Пример test:

```kotlin
fun shouldRenderTitle() {
    val expected = "Kotlin"
    val actual = "Kotlingo"

    assert(expected == actual) {
        "Заголовок должен совпасть"
    }
}
```

Plugin помогает понять, какие intermediate values привели к failure.

## Compiler plugin против KSP

Compiler plugin:

- может влиять на compilation pipeline;
- может менять generated bytecode/IR behavior;
- сложнее в разработке и сопровождении;
- сильнее связан с compiler internals;
- подходит для language/framework integration.

KSP:

- читает Kotlin symbols;
- генерирует новые source/resource files;
- не меняет existing code;
- проще и стабильнее для annotation-driven code generation;
- подходит для adapters, registries, factories, mappers.

Если задача — «найти annotations и сгенерировать file», начинай с KSP. Если задача — «изменить, как compiler понимает declarations или calls», нужен compiler plugin.

## Production decision checklist

Перед подключением plugin спроси:

- есть ли официальная документация и поддержка;
- совместим ли plugin с текущей версией Kotlin;
- работает ли он с target platforms проекта;
- поддерживает ли Gradle configuration cache и incremental builds;
- влияет ли на IDE highlighting;
- нужен ли он во всех modules или только в конкретных;
- как обновлять plugin при Kotlin upgrade.

```kotlin
plugins {
    kotlin("jvm") version "2.4.0"
    kotlin("plugin.serialization") version "2.4.0"
}
```

Версии official Kotlin compiler plugins обычно должны совпадать с версией Kotlin plugin.

<MouseTrap>
Не подключай compiler plugin «на всякий случай». Он становится частью toolchain и может влиять на время сборки, IDE diagnostics и стратегию обновления Kotlin.
</MouseTrap>

## Практический пример: backend module

```kotlin
plugins {
    kotlin("jvm") version "2.4.0"
    kotlin("plugin.serialization") version "2.4.0"
    id("org.jetbrains.kotlin.plugin.spring") version "2.4.0"
}

dependencies {
    implementation("org.jetbrains.kotlinx:kotlinx-serialization-json:1.11.0")
}
```

Код:

```kotlin
import kotlinx.serialization.Serializable

@Serializable
data class LessonPayload(
    val slug: String,
    val title: String
)

@org.springframework.stereotype.Service
class LessonService {
    fun save(payload: LessonPayload) {
        // Spring plugin откроет service class для proxy-сценариев.
        println("Сохраняем ${payload.slug}")
    }
}
```

Здесь serialization plugin отвечает за serializers, а spring/all-open integration отвечает за framework-compatible class shape.

<CatTip>
Compiler plugins сильны, когда bridge между idiomatic Kotlin и требованиями framework слишком сложен для обычной библиотеки. Включай их точечно и документируй причину.
</CatTip>
