## Почему обычный `T` исчезает в runtime

На JVM type arguments generic types обычно erased. Runtime различает `List`, но не надёжно `List<String>` и `List<Int>`.

```kotlin
fun <T> describe(value: T): String {
    return value?.let { it::class.simpleName } ?: "null"
}
```

Можно узнать class фактического объекта, но нельзя обратиться к `T::class`:

```kotlin
// fun <T> typeName(): String = T::class.simpleName.orEmpty()
```

Обычная generic function компилируется один раз для всех T, поэтому concrete type argument не встроен в её body.

## Erasure generic instances

```kotlin
val titles: List<String> = listOf("Kotlin")
val scores: List<Int> = listOf(100)
```

В runtime оба являются List-like instances. Проверка запрещена/ненадёжна:

```kotlin
// if (value is List<String>) { ... }
```

Разрешена star projection:

```kotlin
fun inspect(value: Any) {
    if (value is List<*>) {
        value.forEach { element -> println(element) }
    }
}
```

Elements имеют type `Any?`, потому что exact argument неизвестен.

## Reified сохраняет type argument в inline call site

```kotlin
inline fun <reified T> typeName(): String {
    return T::class.simpleName ?: "Без имени"
}

fun main() {
    println(typeName<String>())
    println(typeName<Int>())
}
```

Требуются два модификатора:

- function должна быть `inline`;
- parameter помечается `reified`.

Compiler вставляет body в call site и подставляет concrete type, поэтому operations над `T` становятся возможны.

## `is T`

```kotlin
inline fun <reified T> Any?.isInstanceOf(): Boolean {
    return this is T
}

fun main() {
    println("Kotlin".isInstanceOf<String>())
    println(36.isInstanceOf<String>())
}
```

Обычный `<T>` не разрешает `value is T` из-за erasure.

## Safe cast `as? T`

```kotlin
inline fun <reified T> Any?.castOrNull(): T? {
    return this as? T
}

val title: String? = "Reified".castOrNull<String>()
val missing: Int? = "Reified".castOrNull<Int>()
```

Для classifier-level check это удобно. Но nested generic arguments всё ещё требуют осторожности.

## Nested generic erasure остаётся

Вызов с `T = List<String>` не делает каждый element runtime-reified автоматически.

```kotlin
inline fun <reified T> Any?.unsafeGenericCastOrNull(): T? {
    return this as? T
}
```

Runtime способен проверить, что объект — List, но не гарантирует String для каждого element. Cast может пройти, а failure возникнет позже при чтении.

Безопасная element validation:

```kotlin
inline fun <reified T> Any?.listOfTypeOrNull(): List<T>? {
    val list = this as? List<*> ?: return null
    if (!list.all { element -> element is T }) return null

    return list.map { element ->
        element as T
    }
}
```

После явной проверки каждый element cast обоснован. Nullable `T` требует отдельного понимания: `null is T` true, если T nullable.

<MouseTrap title="Мышеловка: reified не отменяет erasure вложенных аргументов">
`T::class` знает outer classifier. Для `List<String>` runtime List не хранит String argument. Проверяй элементы, используй serializer/type token или schema.
</MouseTrap>

## `T::class`

```kotlin
inline fun <reified T : Any> classifierName(): String {
    return T::class.qualifiedName ?: T::class.simpleName ?: "anonymous"
}
```

Bound `T : Any` удобен для APIs, которым нужен non-null `KClass<T>`.

```kotlin
fun main() {
    println(classifierName<String>())
}
```

`KClass` предоставляет metadata classifier, но полный reflection API зависит от platform/dependencies.

### JVM `Class<T>`

```kotlin
inline fun <reified T : Any> javaClassOf(): Class<T> {
    return T::class.java
}
```

Это JVM-specific API. В multiplatform common code используй `KClass`/platform abstraction.

## `typeOf<T>()`

`typeOf` возвращает `KType`, который может описывать nullability и nested type arguments в metadata token.

```kotlin
import kotlin.reflect.typeOf

inline fun <reified T> capturedType(): String {
    return typeOf<T>().toString()
}

fun main() {
    println(capturedType<List<String?>>())
}
```

`KType` token содержит больше type metadata, чем `KClass`, но runtime values всё равно не доказывают соответствие этому token. Serializer/reflective code должен проверять данные.

## `filterIsInstance`

Стандартная library использует reified:

```kotlin
val mixed: List<Any> = listOf("Kotlin", 36, "Reified", true)
val titles: List<String> = mixed.filterIsInstance<String>()
```

Без reified API пришлось бы принять `KClass<T>`/`Class<T>` или cast strategy.

Собственная версия:

```kotlin
inline fun <reified T> Iterable<*>.instances(): List<T> {
    return buildList {
        for (value in this@instances) {
            if (value is T) add(value)
        }
    }
}
```

## Reified factory

```kotlin
interface LessonComponent

class TextComponent : LessonComponent
class QuizComponent : LessonComponent

inline fun <reified T : LessonComponent> createComponent(): T {
    val instance: LessonComponent = when (T::class) {
        TextComponent::class -> TextComponent()
        QuizComponent::class -> QuizComponent()
        else -> error("Неизвестный component: ${T::class.qualifiedName}")
    }

    return instance as T
}
```

Такой registry должен быть closed/explicit. Reflection constructor `T::class.java.getDeclaredConstructor().newInstance()`:

- JVM-only;
- требует доступный no-arg constructor;
- может ломаться при obfuscation/modules;
- оборачивает exceptions;
- хуже обычной factory dependency.

## Reified и serializers

Serialization APIs часто выглядят так:

```kotlin
inline fun <reified T> decode(
    raw: String,
    decoder: TypedDecoder
): T {
    return decoder.decode(raw, typeOf<T>()) as T
}

interface TypedDecoder {
    fun decode(raw: String, type: kotlin.reflect.KType): Any?
}
```

Production serializer должен вернуть validated T; unchecked cast в adapter допустим только если decoder contract гарантирует type token. Реальные libraries обычно предоставляют `serializer<T>()`, generated serializers или explicit schema objects.

Reified упрощает call site:

```kotlin
// val lesson: Lesson = json.decode("...")
```

Но correctness обеспечивает serializer, не сам `reified`.

## Alternative: явный `KClass`

Обычная non-inline function:

```kotlin
fun <T : Any> isInstance(value: Any?, type: kotlin.reflect.KClass<T>): Boolean {
    return type.isInstance(value)
}

val result = isInstance("Kotlin", String::class)
```

Преимущества explicit token:

- function не inline;
- type выбирается dynamically;
- меньше code duplication;
- token можно хранить в registry.

Reified удобнее, когда type известен compile time на call site.

## Alternative: factory/decoder parameter

```kotlin
fun <T> parse(raw: String, parser: (String) -> T): T {
    return parser(raw)
}

val id = parse("36", String::toInt)
```

Если algorithm нужна operation, а не runtime type, strategy parameter проще reified/reflection.

<CatTip title="Котосовет: передавай способность, если нужна способность">
Не используй `T::class` для огромного `when`, если caller может передать parser/factory. Reified полезен для type-directed API, но dependency injection остаётся яснее.
</CatTip>

## Ограничения reified

Reified parameter:

- разрешён только у inline function/property accessor в допустимой форме;
- не объявляется у class/interface;
- должен получать reifiable type на call site;
- не может получить runtime identity для arbitrary erased nested values;
- увеличивает inlined code at call sites;
- делает body частью public inline ABI.

```kotlin
// class Box<reified T>
// fun <reified T> ordinary(value: T)
```

Обе формы недопустимы.

## Передача обычного T в reified API

```kotlin
inline fun <reified T> checkType(value: Any?): Boolean = value is T

fun <T> cannotForward(value: Any?): Boolean {
    // return checkType<T>(value)
    return false
}
```

Обычный `T` уже erased и не может стать reified. Forwarding function тоже должна быть inline reified или принимать explicit token.

```kotlin
inline fun <reified T> forward(value: Any?): Boolean {
    return checkType<T>(value)
}
```

## Nullable reified type

```kotlin
inline fun <reified T> acceptsNull(): Boolean {
    return null is T
}

println(acceptsNull<String>())
println(acceptsNull<String?>())
```

Nullability является частью reified check. Для API, требующего classifier non-null type, используй `<reified T : Any>`.

## Reified и variance

```kotlin
inline fun <reified T> Iterable<*>.firstOfTypeOrNull(): T? {
    return firstOrNull { value -> value is T } as? T
}
```

Star projection безопасно даёт `Any?`, reified проверяет каждый element. Это не требует unsafe cast container целиком.

## Public inline ABI

Body public inline function компилируется в client call sites. Изменение implementation требует recompilation clients для нового поведения и ограничивает доступ к private/internal declarations.

```kotlin
@PublishedApi
internal fun typeLog(message: String) {
    println(message)
}

inline fun <reified T> loggedTypeName(): String {
    return T::class.qualifiedName.orEmpty().also(::typeLog)
}
```

`@PublishedApi` делает helper частью binary contract. Не используй его как обычную internal detail.

## Performance

Плюсы:

- нет ручной передачи KClass;
- `is T` может компилироваться в прямой check;
- lambda inline API может не аллоцироваться.

Минусы:

- code duplication на call sites;
- большой body раздувает bytecode;
- reflection/serializer всё равно может быть дорогим;
- nested generic validation не появляется автоматически.

Оставляй inline body компактным, тяжёлую работу делегируй `@PublishedApi internal`/public non-inline core с explicit token.

## Практический пример: типизированная шина событий

```kotlin
interface CourseEvent

data class LessonOpened(val lessonId: Int) : CourseEvent
data class LessonCompleted(val lessonId: Int, val score: Int) : CourseEvent

class EventBus {
    private data class Subscription(
        val eventType: kotlin.reflect.KClass<*>,
        val handler: (CourseEvent) -> Unit
    )

    private val subscriptions = mutableListOf<Subscription>()

    inline fun <reified E : CourseEvent> subscribe(
        noinline handler: (E) -> Unit
    ) {
        subscribe(E::class, handler)
    }

    fun <E : CourseEvent> subscribe(
        eventType: kotlin.reflect.KClass<E>,
        handler: (E) -> Unit
    ) {
        val safeHandler: (CourseEvent) -> Unit = { event ->
            if (eventType.isInstance(event)) {
                @Suppress("UNCHECKED_CAST")
                handler(event as E)
            }
        }
        subscriptions += Subscription(eventType, safeHandler)
    }

    fun publish(event: CourseEvent) {
        subscriptions
            .asSequence()
            .filter { subscription -> subscription.eventType.isInstance(event) }
            .forEach { subscription -> subscription.handler(event) }
    }
}

fun main() {
    val bus = EventBus()

    bus.subscribe<LessonOpened> { event ->
        println("Открыт урок ${event.lessonId}")
    }
    bus.subscribe<LessonCompleted> { event ->
        println("Урок ${event.lessonId}: ${event.score} баллов")
    }

    bus.publish(LessonOpened(36))
    bus.publish(LessonCompleted(36, 100))
}
```

Здесь:

- reified overload превращает `E` в `E::class`;
- non-inline core принимает explicit KClass;
- unchecked cast изолирован после runtime `isInstance` check;
- `noinline` нужен, потому что handler сохраняется/передаётся другой function;
- event bus не пытается проверять nested generic payload;
- большой runtime algorithm не дублируется целиком на call sites.

## Итоги

- JVM generic type arguments обычно erased;
- ordinary `T` нельзя использовать в `is T` или `T::class`;
- `inline fun <reified T>` подставляет type в call site;
- reified разрешает `is T`, `as? T`, `T::class` и `typeOf<T>()`;
- nested generic arguments не становятся автоматически проверяемыми;
- collections нужно валидировать element-by-element или serializer schema;
- explicit KClass/KType/factory лучше для dynamic/non-inline API;
- обычный erased T нельзя передать как reified argument;
- nullable T участвует в runtime check;
- public inline body является частью client ABI;
- reified улучшает ergonomics, но не заменяет validation и архитектуру dependencies.
