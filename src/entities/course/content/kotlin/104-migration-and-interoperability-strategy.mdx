## Миграция — это процесс продукта

Миграция — это не только преобразование кода. Он меняет сборку, API, тесты, инструменты и привычки команды.

Examples:

— Java для Котлина;
- Kotlin version upgrade;
- Gradle plugin upgrade;
- Java target upgrade;
- enabling explicit API mode;
- переход на сопрограммы;
- перенос кода в общие исходные наборы KMP;
- изменение формата сериализации;
- deprecating old API.

Safe migration has:

- small steps;
- rollback path;
- тесты перед масштабными переписываниями;
- compatibility rules;
- CI coverage;
- release notes;
- measurable completion criteria.

<CatTip>
Миграция — это не прыжок с шкафа, а серия аккуратных шагов по полке: проверил лапу, перенес вес, пошел дальше.
</CatTip>

## Java в Котлин

Смешанные проекты Java/Kotlin — это нормально. Вам не нужно конвертировать все сразу.

Хорошие первые цели:

- tests;
- small value objects;
- листовые утилиты с тестами;
- new features;
- internal services;
- DTO mappers;
- код с низкой сложностью наследования.

Risky first targets:

- публичный Java API, активно используемый внешними потребителями;
- annotation-processor-heavy code;
- reflection-heavy frameworks;
- generated code;
- примитивы параллелизма с тонким поведением памяти;
- giant inheritance hierarchies.

Java class:

```java
package dev.kotlingo.lesson;

public final class JavaLesson {
    private final String slug;
    private final String title;

    public JavaLesson(String slug, String title) {
        this.slug = slug;
        this.title = title;
    }

    public String getSlug() {
        return slug;
    }

    public String getTitle() {
        return title;
    }
}
```

Kotlin equivalent:

```kotlin
package dev.kotlingo.lesson

data class Lesson(
    val slug: String,
    val title: String
)
```

Но совместимость общедоступных API может отличаться: необходимо учитывать конструктор Java, геттеры, `equals`, `hashCode`, `toString` и аннотации, допускающие значение NULL.

## Nullability migration

Обнуляемость Java часто бывает неявной. Котлин заставляет заключить контракт.

Java:

```java
public interface LessonRepository {
    Lesson find(String slug);
}
```

Kotlin migration choices:

```kotlin
interface LessonRepository {
    fun find(slug: String): Lesson?
}
```

or:

```kotlin
interface StrictLessonRepository {
    fun get(slug: String): Lesson
}
```

Принимайте решения, основываясь на поведении, а не на выходе преобразователя.

Boundary normalization:

```kotlin
class JavaRepositoryAdapter(
    private val delegate: JavaLessonRepository
) {
    fun find(slug: String): Lesson? {
        val lesson = delegate.find(slug)

        return lesson?.let { javaLesson ->
            Lesson(
                slug = javaLesson.slug,
                title = javaLesson.title ?: "Без названия"
            )
        }
    }
}
```

<MouseTrap>
Не принимай platform types как «потом разберемся». Nullability is the migration. Если contract не выбран, Kotlin-код просто перенес Java-неясность в новые файлы.
</MouseTrap>

## Java-facing Kotlin APIs

Когда мигрированный код Kotlin вызывается из Java, сохраняйте эргономику Java.

```kotlin
class LessonFacade {
    @JvmOverloads
    fun title(slug: String, fallback: String = "Без названия"): String {
        return if (slug.isBlank()) fallback else "Урок $slug"
    }

    companion object {
        @JvmStatic
        fun create(): LessonFacade {
            return LessonFacade()
        }
    }
}
```

Для проверенных ожиданий исключений Java:

```kotlin
import java.io.IOException

class LessonFileReader {
    @Throws(IOException::class)
    fun read(path: java.nio.file.Path): String {
        return java.nio.file.Files.readString(path)
    }
}
```

Используйте тесты Java для API, ориентированного на Java.

```java
import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.assertEquals;

final class LessonFacadeJavaTest {
    @Test
    void usesDefaultFallback() {
        LessonFacade facade = LessonFacade.create();

        assertEquals("Без названия", facade.title(""));
    }
}
```

## Kotlin version upgrades

Kotlin upgrade affects:

- compiler;
- Gradle plugin;
- language version;
- API version;
- stdlib;
- compiler plugins;
- KSP/kapt;
- binary compatibility;
- IDE behavior;
- предупреждения и диагностика.

Upgrade process:

- прочитать официальное руководство по совместимости целевой версии;
- обновить плагин Kotlin Gradle в одной ветке;
- update плагин компилятора versions together;
- run clean build;
- запустить тесты для всех целей;
- fix new warnings deliberately;
- проверка сгенерированного кода и обработчиков аннотаций;
- тестовая издательская библиотека;
- развертывать постепенно, если монорепо большое.

Gradle:

```kotlin
plugins {
    kotlin("jvm") version "2.4.0"
}
```

Версию языка/API можно закрепить во время миграции.

```kotlin
kotlin {
    compilerOptions {
        languageVersion.set(org.jetbrains.kotlin.gradle.dsl.KotlinVersion.KOTLIN_2_4)
        apiVersion.set(org.jetbrains.kotlin.gradle.dsl.KotlinVersion.KOTLIN_2_4)
    }
}
```

Не переходите вслепую к версиям библиотеки, используемой множеством сервисов.

## Language feature flags

Некоторые функции Kotlin требуют согласия или флагов компилятора, прежде чем они станут стабильными.

Examples:

- экспериментальные API с `@OptIn`;
- предпросмотр language features;
- плагин компилятора features;
- контекстно-зависимое разрешение перед стабильным выпуском;
- прогрессивный режим для более ранней диагностики.

Localized opt-in:

```kotlin
import kotlinx.coroutines.ExperimentalCoroutinesApi

@OptIn(ExperimentalCoroutinesApi::class)
fun useExperimentalApi() {
    println("Experimental API используется осознанно")
}
```

Предпочитайте локальную подписку, а не глобальную подписку на весь проект, когда использование изолировано.

Gradle opt-in:

```kotlin
kotlin {
    compilerOptions {
        freeCompilerArgs.add("-opt-in=kotlin.RequiresOptIn")
    }
}
```

Задокументируйте, почему существует флаг и когда его удалить.

<CatTip>
Feature flag без даты пересмотра — как коробка на шкафу: все привыкли, что она там, но никто не помнит, что внутри.
</CatTip>

## Deprecation cycles

Удаление общедоступного API должно быть поэтапным.

Typical cycle:

- introduce replacement;
- пометить старый API как `WARNING`;
- update internal callers;
- publish release notes;
- позже поднять до `ERROR`;
- удалить в следующей основной версии или в документально подтвержденном выпуске.

```kotlin
class LessonApi {
    fun open(slug: LessonSlug): Lesson {
        return Lesson(slug, "Урок ${slug.value}")
    }

    @Deprecated(
        message = "Use open(LessonSlug) to avoid invalid slug values.",
        replaceWith = ReplaceWith("open(LessonSlug(slug))"),
        level = DeprecationLevel.WARNING
    )
    fun open(slug: String): Lesson {
        return open(LessonSlug(slug))
    }
}
```

Later:

```kotlin
@Deprecated(
    message = "Use open(LessonSlug). This overload will be removed in the next major release.",
    replaceWith = ReplaceWith("open(LessonSlug(slug))"),
    level = DeprecationLevel.ERROR
)
fun openOld(slug: String): Lesson {
    return open(LessonSlug(slug))
}
```

Используйте `ReplaceWith`, чтобы упростить миграцию IDE.

## Binary compatibility during migration

Миграция может случайно нарушить двоичную совместимость.

Risky changes:

- преобразование класса Java в класс данных Kotlin;
- изменение подписи конструктора;
- замена статического метода Java сопутствующим методом без `@JvmStatic`;
- изменение имени пакета или файла верхнего уровня;
- adding параметры по умолчанию without Java перегрузки where Java callers need them;
- изменение аннотаций обнуления в публичном API;
- изменение типа возвращаемого значения с типа платформы на более строгий тип;
- removing old перегрузка too early.

Используйте проверку двоичной совместимости для опубликованных библиотек.

```kotlin
plugins {
    id("org.jetbrains.kotlinx.binary-compatibility-validator") version "0.18.1"
}

apiValidation {
    ignoredProjects += listOf("sample-app")
}
```

Перед публикацией проверьте общедоступный ABI в CI.

## Compatibility guides

Official compatibility guides tell what changed between Kotlin versions:

- language changes;
- compiler diagnostics;
- standard library behavior;
- Gradle plugin changes;
- multiplatform changes;
- устаревания и удаления.

Проверочный список миграции for a Kotlin upgrade:

- прочитайте руководство перед сменой версии;
- поиск по кодовой базе упомянутых API;
- запустить сборку с видимыми предупреждениями;
- upgrade ecosystem plugins together;
- check generated code;
- check binary compatibility;
- проверить поведение во время выполнения на всех целях;
- обновить документацию и пример кода.

## Interoperability strategy

Взаимодействие должно иметь границы.

Java boundary:

```kotlin
class JavaFriendlyLessonApi {
    @JvmOverloads
    fun search(query: String, limit: Int = 20): List<Lesson> {
        require(limit > 0) { "limit должен быть положительным" }

        return listOf(
            Lesson(
                slug = LessonSlug("104-migration-and-interoperability-strategy"),
                title = "Migration And Interoperability Strategy"
            )
        ).take(limit)
    }
}
```

Kotlin-friendly internal API:

```kotlin
class KotlinLessonSearch {
    fun search(request: SearchRequest): SearchResult {
        return SearchResult.Ready(emptyList())
    }
}

data class SearchRequest(
    val query: String,
    val limit: Int
)

sealed interface SearchResult {
    data class Ready(
        val lessons: List<Lesson>
    ) : SearchResult

    data class Rejected(
        val reason: String
    ) : SearchResult
}
```

Не заставляйте одну поверхность удовлетворять каждого потребителя. Используйте адаптеры, когда вызывающим программам Java, Kotlin, JS или Swift требуется другая эргономика.

## Боевой migration plan

Пример плана миграции сервисов с Java на Kotlin:

- add Kotlin plugin without converting боевой код;
- конвертировать тесты или небольшие листовые классы;
- add `kotlin.test` and JUnit Platform настройка;
- define nullability policy;
- convert internal value objects;
- convert services behind existing interfaces;
- сохранять стабильность фасада, обращенного к Java;
- добавить Java-тесты для фасада;
- introduce deprecations for old перегрузки;
- validate binary compatibility;
- удалять устаревшие API только в запланированном последнем выпуске.

Пример плана обновления версии Kotlin:

- read compatibility guide;
- update version catalog;
- update compiler plugins;
- run clean build;
- run all target tests;
- fix warnings intentionally;
- при необходимости обновить стиль/форматирование кода;
- check public ABI;
- publish release candidate internally;
- отслеживать ошибки времени выполнения после развертывания.

<MouseTrap>
Миграция «большого взрыва» кажется быстрой до первого отката. Хорошая миграция оставляет систему доступной для выпуска после каждого значимого шага.
</MouseTrap>

## Итоговый проверочный список

Для миграции и взаимодействия:

- конвертировать поведение с помощью тестов, а не только синтаксиса;
- choose nullability contracts manually;
- поддерживать стабильность Java API с помощью аннотаций там, где это необходимо;
- прочтите руководства по совместимости перед обновлением версии;
- использовать флаги функций локально и документировать их;
- устаревать перед удалением;
- проверка бинарной совместимости библиотек;
- держать адаптеры для разных языков потребителей;
- сделать возможным откат.

<CatTip>
Самая сильная миграционная стратегия — та, после которой никто не замечает драму: код стал лучше, релизы идут, старые callers не сломались, а команда понимает следующий шаг.
</CatTip>
