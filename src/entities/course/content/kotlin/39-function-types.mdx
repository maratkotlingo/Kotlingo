## Функция как типизированное значение

Function type описывает входные параметры и результат операции.

```kotlin
val formatScore: (Int) -> String = { score -> "$score баллов" }

fun main() {
    println(formatScore(95))
}
```

`(Int) -> String` означает: значение принимает один `Int` и возвращает `String`.

## Полный синтаксис

```kotlin
(A, B) -> C
```

- types слева от стрелки — параметры;
- type справа — результат;
- круглые скобки parameter list обязательны;
- parameter count называется arity.

```kotlin
val noArguments: () -> String = { "Kotlin" }
val oneArgument: (String) -> Int = { text -> text.length }
val twoArguments: (Int, Int) -> Int = { left, right -> left + right }
```

## Имена parameters в function type

```kotlin
val progressFormatter: (
    completed: Int,
    total: Int
) -> String = { completed, total ->
    "$completed из $total"
}
```

Имена улучшают документацию и IDE hints, но не являются частью type identity.

```kotlin
val formatter: (done: Int, all: Int) -> String = progressFormatter
```

Types и порядок совпадают, поэтому присваивание допустимо.

## Вызов через `invoke`

Function value вызывается двумя эквивалентными формами:

```kotlin
val double: (Int) -> Int = { value -> value * 2 }

println(double(21))
println(double.invoke(21))
```

Синтаксис круглых скобок использует operator function `invoke`.

## `Unit` result

Обратный вызов без полезного результата имеет type `() -> Unit` или `(T) -> Unit`.

```kotlin
val lessonOpened: (Int) -> Unit = { id ->
    println("Открыт урок $id")
}
```

`Unit` — настоящий type с единственным value. Когда expected result — Unit, результат подходящей function/лямбда может быть отброшен.

```kotlin
fun score(): Int = 100

val action: () -> Unit = ::score
```

Используй это осознанно: скрытое отбрасывание business result может быть ошибкой дизайна.

## `Nothing` result

```kotlin
val fail: (String) -> Nothing = { message ->
    throw IllegalStateException(message)
}
```

Function никогда не возвращается normally. Благодаря `Nothing` её можно использовать в expression любого expected result:

```kotlin
fun titleOrFail(title: String?): String {
    return title ?: fail("Название отсутствует")
}
```

## Nullable function value

```kotlin
val callback: ((String) -> Unit)? = null
```

Скобки вокруг whole function type обязательны: nullable является сама функция.

```kotlin
callback?.invoke("Событие")
```

Можно задать fallback:

```kotlin
val effectiveCallback = callback ?: { message: String ->
    println("Callback не задан: $message")
}
```

## Nullable result

```kotlin
val finder: (Int) -> String? = { id ->
    if (id == 39) "Function types" else null
}
```

Различие:

- `((Int) -> String)?` — операция может отсутствовать;
- `(Int) -> String?` — операция есть, но результат может отсутствовать;
- `((Int) -> String?)?` — могут отсутствовать и operation, и result.

<MouseTrap title="Мышеловка: nullability зависит от скобок">
`(() -> String)?` и `() -> String?` моделируют разные состояния. Выбирай type по business contract, а не по тому, что проще вызвать.
</MouseTrap>

## Callable references как function values

```kotlin
fun normalizeTitle(raw: String): String = raw.trim()

val normalizer: (String) -> String = ::normalizeTitle
```

Bound member reference:

```kotlin
class Prefixer(private val prefix: String) {
    fun format(value: String): String = "$prefix: $value"
}

val prefixer = Prefixer("Урок")
val formatter: (String) -> String = prefixer::format
```

Unbound reference включает receiver первым parameter:

```kotlin
val unbound: (Prefixer, String) -> String = Prefixer::format
```

## Receiver function type

```kotlin
String.() -> Int
```

Операция вызывается в context receiver String, внутри доступен `this`.

```kotlin
val wordCount: String.() -> Int = {
    trim().split(Regex("""\s+""")).count { it.isNotEmpty() }
}

println("Functional Kotlin".wordCount())
```

Обычный вызов `wordCount("text")` также conceptually передаёт receiver как argument через `invoke` rules, но extension-like syntax лучше показывает intent.

## Receiver type и обычный первый parameter

Kotlin позволяет адаптировать совместимые values между:

```kotlin
(String, Int) -> String
String.(Int) -> String
```

```kotlin
val repeat: (String, Int) -> String = { text, times ->
    text.repeat(times)
}

val receiverRepeat: String.(Int) -> String = repeat
println("кот".receiverRepeat(3))
```

Receiver меняет syntax и implicit member resolution, но не создаёт новый runtime class value принципиально иного назначения.

## DSL receiver

```kotlin
class LessonBuilder {
    var title: String = ""
    var order: Int = 0

    fun build(): Lesson {
        require(title.isNotBlank()) { "Название обязательно" }
        require(order > 0) { "Порядок должен быть положительным" }
        return Lesson(order, title.trim())
    }
}

data class Lesson(val order: Int, val title: String)

fun lesson(block: LessonBuilder.() -> Unit): Lesson {
    return LessonBuilder().apply(block).build()
}
```

Call site:

```kotlin
val item = lesson {
    order = 39
    title = "Function types"
}
```

Receiver function type формирует DSL, но nested receivers могут конфликтовать. Для больших DSL нужен `@DslMarker`.

## Typealias для function type

```kotlin
typealias LessonPredicate = (Lesson) -> Boolean
typealias LessonAction = Lesson.() -> Unit
```

Alias улучшает readability, но не создаёт новый type. Два aliases одинаковой expansion взаимозаменяемы.

Если нужна отдельная domain identity и методы по умолчанию, используй `fun interface`.

## Function type variance

Parameters contravariant, result covariant.

```kotlin
open class Content(val title: String)
class LessonContent(title: String) : Content(title)

val contentTitle: (Content) -> String = { content -> content.title }
val lessonTitle: (LessonContent) -> CharSequence = contentTitle
```

Function, принимающая любой Content, принимает LessonContent. String result подходит как CharSequence.

## Function type с несколькими receivers

Обычный stable function type имеет один extension receiver. Дополнительный context можно передать parameters или использовать context parameters в подходящем language mode.

```kotlin
typealias RenderBlock = LessonBuilder.(prefix: String) -> String
```

Receiver и parameters имеют разные roles.

## Suspend function type

```kotlin
suspend (Int) -> String
```

Suspending function value можно вызвать только из suspend function/coroutine context.

```kotlin
suspend fun loadLesson(id: Int): String {
    return "Урок $id"
}

suspend fun render(
    id: Int,
    loader: suspend (Int) -> String
): String {
    return "Открыто: ${loader(id)}"
}

suspend fun example() {
    val loader: suspend (Int) -> String = ::loadLesson
    println(render(39, loader))
}
```

Suspend type — часть signature. Обычная function не может просто вызвать suspend обратный вызов.

### Suspending лямбда

```kotlin
val loader: suspend (Int) -> String = { id ->
    loadLesson(id)
}
```

Lambda получает suspend capability из expected type.

## Обычный и suspend type не взаимозаменяемы автоматически

Обычная function может быть вызвана из suspend context, но type `(Int) -> String` не всегда автоматически является `suspend (Int) -> String` во всех перегрузка/inference scenarios. Явно объявляй expected type и избегай перегрузки, различающихся только suspend modifier, если место вызова с лямбда становится ambiguous на поддерживаемом compiler.

## Function type или `fun interface`

Function type подходит:

- local обратный вызов/strategy;
- collection transformation;
- composition;
- Kotlin-first API;
- multiple operations не нужны.

`fun interface` подходит:

- domain contract с именем;
- Java SAM interop;
- annotations/методы по умолчанию;
- разграничение двух одинаковых signatures;
- binary API, где interface identity важна.

```kotlin
fun interface LessonValidator {
    fun validate(lesson: Lesson): Boolean
}
```

SAM instance не тождественна raw function value, хотя лямбда conversion удобна.

## Function objects и allocation

Captured лямбда/function reference может быть отдельным object. Non-capturing лямбда compiler иногда кеширует, inline API может устранить object. Точная representation — бэкенд optimization.

Не оптимизируй API вслепую:

- измеряй hot loops;
- не создавай обратный вызов repeatedly без причины;
- используй inline для подходящих higher-order utilities;
- помни о code size.

## Практический пример: middleware pipeline

```kotlin
data class LessonRequest(
    val lessonId: Int,
    val student: String
)

data class LessonResponse(
    val status: Int,
    val body: String
)

typealias LessonHandler = (LessonRequest) -> LessonResponse
typealias Middleware = (LessonHandler) -> LessonHandler

fun loggingMiddleware(log: (String) -> Unit): Middleware {
    return { next ->
        { request ->
            log("Запрос урока ${request.lessonId}")
            val response = next(request)
            log("Ответ ${response.status}")
            response
        }
    }
}

fun validationMiddleware(): Middleware {
    return { next ->
        handler@{ request ->
            if (request.lessonId <= 0) {
                return@handler LessonResponse(400, "Некорректный идентификатор")
            }
            if (request.student.isBlank()) {
                return@handler LessonResponse(400, "Имя студента обязательно")
            }
            next(request)
        }
    }
}

fun compose(
    terminal: LessonHandler,
    middlewares: List<Middleware>
): LessonHandler {
    return middlewares.asReversed().fold(terminal) { next, middleware ->
        middleware(next)
    }
}

fun main() {
    val terminal: LessonHandler = { request ->
        LessonResponse(200, "Открыт урок ${request.lessonId} для ${request.student}")
    }

    val handler = compose(
        terminal = terminal,
        middlewares = listOf(
            loggingMiddleware(::println),
            validationMiddleware()
        )
    )

    println(handler(LessonRequest(39, "Мария")))
    println(handler(LessonRequest(-1, "Мария")))
}
```

Pipeline показывает functions, принимающие и возвращающие functions, aliases сложных signatures, local labeled return и композицию поведения без inheritance.

## Итоги

- function type записывает parameters слева, result справа от `->`;
- имена parameters документируют, но не меняют type identity;
- function value вызывается через `()` или `invoke`;
- `Unit` означает отсутствие полезного result, `Nothing` — отсутствие normal return;
- nullable function и nullable result различаются скобками;
- receiver type `A.() -> B` даёт implicit `this` и DSL syntax;
- callable references совместимы с подходящими function types;
- parameters function type contravariant, result covariant;
- suspend function type требует suspend context;
- typealias сокращает type, `fun interface` создаёт именованный contract;
- runtime allocation зависит от capture, inline и бэкенд.
