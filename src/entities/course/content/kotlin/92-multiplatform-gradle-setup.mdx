## Минимальная Gradle-конфигурация

Kotlin Multiplatform project использует Gradle plugin `org.jetbrains.kotlin.multiplatform`.

```kotlin
// build.gradle.kts
plugins {
    kotlin("multiplatform") version "2.4.0"
}

kotlin {
    jvm()
}
```

Один и тот же плагин может объявлять множество целей.

```kotlin
plugins {
    kotlin("multiplatform") version "2.4.0"
}

kotlin {
    jvm()

    js {
        browser()
        nodejs()
    }

    iosArm64()
    iosSimulatorArm64()
}
```

<CatTip>
настройка Gradle в KMP — это карта кошачьего дома: targets говорят, где будут комнаты, source sets говорят, какие вещи лежат в каждой комнате.
</CatTip>

## `settings.gradle.kts`

Репозитории проектов и репозитории плагинов находятся в настройках.

```kotlin
// settings.gradle.kts
pluginManagement {
    repositories {
        google()
        mavenCentral()
        gradlePluginPortal()
    }
}

dependencyResolutionManagement {
    repositoriesMode.set(RepositoriesMode.FAIL_ON_PROJECT_REPOS)

    repositories {
        google()
        mavenCentral()
    }
}

rootProject.name = "kotlingo-shared"
```

Используй `google()` только если проект использует Android или артефакты из Google Maven.

## Targets

Объявление цели определяет, какие двоичные файлы платформы могут быть созданы.

JVM:

```kotlin
kotlin {
    jvm {
        testRuns["test"].executionTask.configure {
            useJUnitPlatform()
        }
    }
}
```

Android:

```kotlin
plugins {
    kotlin("multiplatform") version "2.4.0"
    id("com.android.library") version "9.0.0"
}

kotlin {
    androidTarget()
}

android {
    namespace = "dev.kotlingo.shared"
    compileSdk = 36

    defaultConfig {
        minSdk = 24
    }
}
```

JS:

```kotlin
kotlin {
    js {
        browser()
        nodejs()
        binaries.executable()
    }
}
```

Wasm JS:

```kotlin
kotlin {
    wasmJs {
        browser()
        binaries.executable()
    }
}
```

Native iOS:

```kotlin
kotlin {
    iosArm64()
    iosSimulatorArm64()
}
```

Выбирайте цели осознанно. Каждая дополнительная цель увеличивает матрицу сборки, ограничения зависимостей и поверхность тестирования.

<MouseTrap>
Не добавляй target «на всякий случай». Если project не тестируется на этой platform, target быстро превращается в красный build после очередного common dependency.
</MouseTrap>

## Source sets block

`sourceSets` настраивает общие наборы исходных кодов и исходные коды платформы.

```kotlin
kotlin {
    jvm()
    js {
        browser()
    }
    iosArm64()
    iosSimulatorArm64()

    sourceSets {
        commonMain {
            dependencies {
                implementation("org.jetbrains.kotlinx:kotlinx-coroutines-core:1.11.0")
            }
        }

        commonTest {
            dependencies {
                implementation(kotlin("test"))
            }
        }

        jvmMain {
            dependencies {
                implementation("org.jetbrains.kotlinx:kotlinx-coroutines-swing:1.11.0")
            }
        }

        jsMain {
            dependencies {
                implementation(npm("uuid", "12.0.0"))
            }
        }
    }
}
```

Общие зависимости видны всем зависимым исходным наборам.

Зависимости платформы видны только в этом исходном наборе платформы и его дочерних элементах.

## Version catalog

Боевой builds should centralize versions in `libs.versions.toml`.

```toml
# gradle/libs.versions.toml
[versions]
kotlin = "2.4.0"
kotlinx-coroutines = "1.11.0"
android-gradle-plugin = "9.0.0"

[libraries]
kotlinx-coroutines-core = { module = "org.jetbrains.kotlinx:kotlinx-coroutines-core", version.ref = "kotlinx-coroutines" }
kotlinx-coroutines-test = { module = "org.jetbrains.kotlinx:kotlinx-coroutines-test", version.ref = "kotlinx-coroutines" }

[plugins]
kotlin-multiplatform = { id = "org.jetbrains.kotlin.multiplatform", version.ref = "kotlin" }
android-library = { id = "com.android.library", version.ref = "android-gradle-plugin" }
```

Build script:

```kotlin
plugins {
    alias(libs.plugins.kotlin.multiplatform)
    alias(libs.plugins.android.library)
}

kotlin {
    androidTarget()
    jvm()

    sourceSets {
        commonMain.dependencies {
            implementation(libs.kotlinx.coroutines.core)
        }

        commonTest.dependencies {
            implementation(kotlin("test"))
            implementation(libs.kotlinx.coroutines.test)
        }
    }
}
```

Это позволяет избежать дрейфа версий между модулями.

## Common dependencies

Общий исходный набор может зависеть только от общесовместимых библиотек.

```kotlin
kotlin {
    sourceSets {
        commonMain.dependencies {
            implementation("org.jetbrains.kotlinx:kotlinx-coroutines-core:1.11.0")
            implementation("org.jetbrains.kotlinx:kotlinx-serialization-json:1.9.0")
            implementation("io.ktor:ktor-client-core:3.3.0")
        }
    }
}
```

Общая зависимость должна публиковать многоплатформенные метаданные. Если библиотека предназначена только для JVM, ее нельзя использовать в `commonMain`.

Wrong:

```kotlin
kotlin {
    sourceSets {
        commonMain.dependencies {
            // Плохо: Guava is JVM library, not common multiplatform API.
            implementation("com.google.guava:guava:33.5.0-jre")
        }
    }
}
```

Correct:

```kotlin
kotlin {
    sourceSets {
        jvmMain.dependencies {
            implementation("com.google.guava:guava:33.5.0-jre")
        }
    }
}
```

## Platform dependencies

JVM dependency:

```kotlin
kotlin {
    sourceSets {
        jvmMain.dependencies {
            implementation("org.slf4j:slf4j-api:2.1.0")
        }

        jvmTest.dependencies {
            implementation("org.junit.jupiter:junit-jupiter:6.0.0")
        }
    }
}
```

Android dependency:

```kotlin
kotlin {
    sourceSets {
        androidMain.dependencies {
            implementation("androidx.datastore:datastore-preferences:1.2.0")
        }
    }
}
```

Зависимость iOS от мультиплатформенной библиотеки:

```kotlin
kotlin {
    sourceSets {
        iosMain.dependencies {
            implementation("app.cash.sqldelight:native-driver:2.3.2")
        }
    }
}
```

JS npm dependency:

```kotlin
kotlin {
    sourceSets {
        jsMain.dependencies {
            implementation(npm("uuid", "12.0.0"))
        }
    }
}
```

Правило: зависимость должна быть объявлена ​​в самом высоком исходном наборе, где она действительна и необходима.

## Intermediate source sets

Sometimes default hierarchy is enough. Sometimes you need пользовательский sharing.

Пример: обмениваться кодом только между JVM и JS.

```kotlin
kotlin {
    jvm()
    js {
        browser()
    }
    linuxX64()

    sourceSets {
        val commonMain by getting
        val jvmMain by getting
        val jsMain by getting

        val webAndJvmMain by creating {
            dependsOn(commonMain)

            dependencies {
                implementation("org.jetbrains.kotlinx:kotlinx-coroutines-core:1.11.0")
            }
        }

        jvmMain.dependsOn(webAndJvmMain)
        jsMain.dependsOn(webAndJvmMain)
    }
}
```

Используй пользовательские intermediate source sets только когда паттерн шаринга реален. Иначе граф сборки становится сложным для понимания.

## Compiler options

современная настройка Gradle should configure compiler options explicitly.

```kotlin
kotlin {
    compilerOptions {
        allWarningsAsErrors.set(false)
        freeCompilerArgs.add("-Xexpect-actual-classes")
    }

    jvm {
        compilerOptions {
            jvmTarget.set(org.jetbrains.kotlin.gradle.dsl.JvmTarget.JVM_21)
        }
    }
}
```

Используйте параметры компилятора для конкретной цели, когда они требуются платформе.

```kotlin
kotlin {
    js {
        compilerOptions {
            sourceMap.set(true)
        }
    }
}
```

Не разбрасывайте флаги компилятора по подпроектам без документации. Флаги являются частью контракта на строительство.

## Android target details

Для библиотеки Android:

```kotlin
plugins {
    alias(libs.plugins.kotlin.multiplatform)
    alias(libs.plugins.android.library)
}

kotlin {
    androidTarget {
        compilerOptions {
            jvmTarget.set(org.jetbrains.kotlin.gradle.dsl.JvmTarget.JVM_21)
        }
    }

    sourceSets {
        androidMain.dependencies {
            implementation("androidx.core:core-ktx:1.18.0")
        }

        androidUnitTest.dependencies {
            implementation(kotlin("test"))
        }
    }
}

android {
    namespace = "dev.kotlingo.shared"
    compileSdk = 36

    defaultConfig {
        minSdk = 24
    }
}
```

Наборы исходных кодов Android связаны с концепциями плагинов Android Gradle. Не допускайте использования платформы Android в `commonMain`.

## Native framework basics

Для интеграции с iOS вы часто создаете двоичные файлы платформы.

```kotlin
kotlin {
    iosArm64()
    iosSimulatorArm64()

    targets.withType<org.jetbrains.kotlin.gradle.plugin.mpp.KotlinNativeTarget>().configureEach {
        binaries.framework {
            baseName = "KotlingoShared"
            isStatic = true
        }
    }
}
```

Это создает framework-артефакты для native targets. Детали интеграции с Apple будут разобраны в последующих платформенных уроках.

## Publication basics

Kotlin Multiplatform publication is not one артефакт.

Обычно оно включает в себя:

- target-specific publications, например JVM артефакт;
- platform-specific Native артефакты;
- metadata артефакты;
- root `kotlinMultiplatform` publication that points consumers to target-specific артефакты.

Minimal Maven publish настройка:

```kotlin
plugins {
    kotlin("multiplatform") version "2.4.0"
    `maven-publish`
}

group = "dev.kotlingo"
version = "1.0.0"

kotlin {
    jvm()
    iosArm64()
    iosSimulatorArm64()
}

publishing {
    repositories {
        maven {
            name = "localKotlingo"
            url = layout.buildDirectory.dir("repo").get().asFile.toURI()
        }
    }
}
```

Consumers should depend on root coordinates from common source set, and Gradle resolves target артефакты.

```kotlin
kotlin {
    sourceSets {
        commonMain.dependencies {
            implementation("dev.kotlingo:kotlingo-shared:1.0.0")
        }
    }
}
```

Публикация требует аккуратных metadata, signing и настройки репозитория для публичных репозиториев. Не публикуй наполовину настроенные наборы target.

<MouseTrap>
В KMP library publishing важно публиковать весь набор артефакты consistently. Если root publication указывает на target артефакт, которого нет в репозиторий, consumers получат странные dependency resolution failures.
</MouseTrap>

## Full в стиле боевого кода sample

```kotlin
// build.gradle.kts
plugins {
    alias(libs.plugins.kotlin.multiplatform)
    alias(libs.plugins.android.library)
    `maven-publish`
}

group = "dev.kotlingo"
version = "0.1.0"

kotlin {
    explicitApi()

    androidTarget {
        compilerOptions {
            jvmTarget.set(org.jetbrains.kotlin.gradle.dsl.JvmTarget.JVM_21)
        }
    }

    jvm {
        testRuns["test"].executionTask.configure {
            useJUnitPlatform()
        }
    }

    js {
        browser()
        nodejs()
    }

    iosArm64()
    iosSimulatorArm64()

    sourceSets {
        commonMain.dependencies {
            implementation(libs.kotlinx.coroutines.core)
        }

        commonTest.dependencies {
            implementation(kotlin("test"))
            implementation(libs.kotlinx.coroutines.test)
        }

        androidMain.dependencies {
            implementation("androidx.datastore:datastore-preferences:1.2.0")
        }

        jvmMain.dependencies {
            implementation("org.slf4j:slf4j-api:2.1.0")
        }

        jsMain.dependencies {
            implementation(npm("uuid", "12.0.0"))
        }
    }
}

android {
    namespace = "dev.kotlingo.shared"
    compileSdk = 36

    defaultConfig {
        minSdk = 24
    }
}

publishing {
    repositories {
        maven {
            name = "localKotlingo"
            url = layout.buildDirectory.dir("repo").get().asFile.toURI()
        }
    }
}
```

Почему это в стиле боевого кода:

- versions centralized через catalog;
- targets declared explicitly;
- зависимости живут в правильных исходных наборах;
- `explicitApi` protects library API;
- tests have common coroutine test dependency;
- Конфигурация Android изолирована в блоке `android`;
- publication configured intentionally.

## Проверочный список настройки

Перед тем как считать KMP настройка Gradle готовым, проверь:

- цели соответствуют реальным поддерживаемым платформам;
- общие зависимости действительно мультиплатформенны;
- зависимости платформы не просачиваются в `commonMain`;
- тесты существуют в `commonTest` и тесты платформы там, где это необходимо;
- source set hierarchy matches real sharing;
- настроено пространство имен Android и SDK;
- JS/Wasm executable/browser/node settings match product;
- Собственные двоичные файлы настраиваются только там, где это необходимо;
- publication includes all target артефакты you promise;
- CI запускает задачи для каждой поддерживаемой цели.

<CatTip>
Хороший KMP build script скучно предсказуем: targets сверху, source sets ясно разделены, dependencies лежат там, где им место, а publication не обещает лишнего.
</CatTip>
