## Когда нужен Kotlin/JS

Kotlin/JS компилирует Kotlin в JavaScript.

Сценарии использования:

- browser applications;
- Node.js tools;
- serverless functions;
- общий validation logic for web and бэкенд;
— Kotlin-обертки над JS-библиотеками;
- TypeScript-facing libraries;
- запасной вариант для старых браузеров, когда Wasm не подходит;
— постепенное повторное использование кода домена Kotlin в веб-стеке.

```kotlin
kotlin {
    js {
        browser()
        nodejs()
        binaries.executable()
    }
}
```

`browser()` configures browser environment. `nodejs()` configures Node.js environment.

<CatTip>
Kotlin/JS — это кот, который умеет ходить по JavaScript-подоконнику: доменная логика остается Kotlin-овой, а runtime говорит на языке browser or Node.
</CatTip>

## Source sets

Typical KMP layout:

```text
src/
  commonMain/
    kotlin/
  jsMain/
    kotlin/
  jsTest/
    kotlin/
```

Common code:

```kotlin
// commonMain/kotlin/lesson/SlugRules.kt
package lesson

object SlugRules {
    fun normalize(value: String): String {
        return value.trim()
            .lowercase()
            .replace(Regex("\\s+"), "-")
    }
}
```

JS-specific code:

```kotlin
// jsMain/kotlin/browser/BrowserTitle.kt
package browser

import kotlinx.browser.document

fun setPageTitle(title: String) {
    // DOM API доступен в browser-oriented JS source set.
    document.title = title
}
```

Не помещайте импорт DOM или Node в `commonMain`.

## Browser target

Цель браузера использует веб-пакет для объединения.

```kotlin
kotlin {
    js {
        browser {
            webpackTask {
                outputFileName = "kotlingo.js"
            }

            testTask {
                useKarma {
                    useChromeHeadless()
                }
            }
        }

        binaries.executable()
    }
}
```

Код браузера может использовать объявления DOM:

```kotlin
// jsMain/kotlin/browser/ProgressDomRenderer.kt
package browser

import kotlinx.browser.document

class ProgressDomRenderer(
    private val elementId: String
) {
    fun render(percent: Int) {
        val element = document.getElementById(elementId)
            ?: error("Element '$elementId' не найден")

        element.textContent = "Прогресс: $percent%"
    }
}
```

Если код манипулирует DOM, он принадлежит исходному набору JS/браузера или модулю пользовательского интерфейса, а не чистому модулю общего домена.

## Node target

Цель Node полезна для CLI, бессерверных приложений и инструментов.

```kotlin
kotlin {
    js {
        nodejs {
            testTask {
                useMocha()
            }
        }

        binaries.executable()
    }
}
```

Node-specific external declaration:

```kotlin
// jsMain/kotlin/node/NodeProcess.kt
package node

external val process: Process

external interface Process {
    val env: dynamic
}

fun readEnvironment(name: String): String? {
    // dynamic здесь ограничен JS adapter layer.
    return process.env[name] as? String
}
```

<MouseTrap>
`dynamic` удобен, но он выключает часть пользы Kotlin type system. Держи `dynamic` рядом с JS boundary и сразу переводь values в typed Kotlin models.
</MouseTrap>

## IR compiler

Современный Kotlin/JS использует конвейер компиляции на основе IR.

Что это дает:

- ленивая инициализация свойств верхнего уровня;
- инкрементальная компиляция двоичных файлов для разработки;
- dead code elimination in боевой builds;
- minification support;
— улучшенная основа для экспорта и объявлений TypeScript;
- Согласованная архитектура компилятора с другими целями Kotlin.

Development and боевой tasks differ:

```bash
./gradlew jsBrowserDevelopmentRun
./gradlew jsBrowserProductionWebpack
```

Top-level property initialization:

```kotlin
package startup

val expensiveIndex: Map<String, Int> = run {
    // Kotlin/JS initializes top-level property lazily when first used.
    mapOf("flow" to 84, "javascript" to 95)
}

fun findLesson(name: String): Int? {
    return expensiveIndex[name]
}
```

DCE removes unused declarations in боевой bundles. If declaration is called from JavaScript, export it explicitly.

## NPM dependencies

Kotlin/JS может зависеть от пакетов npm.

```kotlin
kotlin {
    js {
        browser()

        sourceSets {
            jsMain.dependencies {
                implementation(npm("uuid", "12.0.0"))
            }
        }
    }
}
```

External declaration:

```kotlin
// jsMain/kotlin/external/Uuid.kt
@file:JsModule("uuid")
@file:JsNonModule

package external

external fun v4(): String
```

Typed wrapper:

```kotlin
// jsMain/kotlin/platform/JsUuidGenerator.kt
package platform

import external.v4

class JsUuidGenerator {
    fun randomUuid(): String {
        // JS package isolated behind typed Kotlin API.
        return v4()
    }
}
```

Dependency types:

- `npm` — runtime dependency;
- `devNpm` — development-only dependency;
- `optionalNpm` — optional dependency;
- `peerNpm` — зависимость однорангового узла, ожидаемая от потребителя.

## JavaScript interop

Используйте объявления `external` для описания API JS.

```kotlin
// jsMain/kotlin/interop/ConsoleTimer.kt
package interop

external object console {
    fun time(label: String)

    fun timeEnd(label: String)
}

inline fun <T> measureInConsole(label: String, block: () -> T): T {
    console.time(label)

    return try {
        block()
    } finally {
        console.timeEnd(label)
    }
}
```

Используйте `@JsName` для управления сгенерированным именем JavaScript.

```kotlin
package exports

import kotlin.js.JsName

class LessonApi {
    @JsName("normalizeSlug")
    fun normalize(value: String): String {
        return value.trim().lowercase().replace(" ", "-")
    }
}
```

Without `@JsName`, перегрузки and signatures may be mangled to avoid collisions.

## `@JsExport`

`@JsExport` делает объявления Kotlin доступными для потребителей JavaScript/TypeScript.

```kotlin
package exports

import kotlin.js.JsExport

@JsExport
class ExportedLessonCard(
    val slug: String,
    val title: String
)

@JsExport
fun createLessonCard(slug: String, title: String): ExportedLessonCard {
    return ExportedLessonCard(slug, title)
}
```

Экспортируемый API должен быть разработан для JavaScript:

- stable names;
- simple parameter types;
- clear nullability;
- никаких абстракций, основанных только на Kotlin, в публичной поверхности;
- избегать случайного раскрытия внутренних объектов домена;
- используйте DTO для границы JS.

```kotlin
package exports

import kotlin.js.JsExport

@JsExport
data class LessonProgressDto(
    val completed: Int,
    val total: Int
) {
    val percent: Int
        get() = if (total == 0) 0 else completed * 100 / total
}
```

## TypeScript declarations

Kotlin/JS может генерировать объявления TypeScript для экспортируемого API. Экспортируемые декларации должны быть дружелюбны к потребителям TS.

Хорошо:

```kotlin
package exports

import kotlin.js.JsExport

@JsExport
class SearchResult(
    val slug: String,
    val title: String
)

@JsExport
class SearchApi {
    fun search(query: String): Array<SearchResult> {
        return arrayOf(
            SearchResult(
                slug = "95-javascript-target",
                title = "JavaScript Target"
            )
        )
    }
}
```

Risky:

```kotlin
package exports

class InternalSearchState(
    val cache: MutableMap<String, MutableList<String>>
)
```

Не экспортируйте внутренние изменяемые структуры в TypeScript, если вы можете экспортировать DTO.

## Типы Kotlin в JavaScript

Important mapping concerns:

- Kotlin `Int`, `Double`, `Float` become JS `Number`;
- `Long` может потребовать поддержки BigInt в современной конфигурации JS;
- `String` maps cleanly;
- `Boolean` maps cleanly;
— массивы и коллекции могут использовать оболочки Kotlin;
- значения, допускающие значение NULL, могут принимать вид `null` или `undefined` в зависимости от экспорта;
- исключения отображаются как ошибки JS на границе.

```kotlin
package exports

import kotlin.js.JsExport

@JsExport
class SafeCounter {
    private var value: Int = 0

    fun increment(): Int {
        value += 1
        return value
    }
}
```

Для API-интерфейсов, ориентированных на JS, отдавайте предпочтение `Int`, `Double`, `String`, `Boolean`, массивам и экспортированным классам DTO.

## Testing Kotlin/JS

Браузерные тесты обычно используют Karma; Тесты узлов используют Mocha.

```kotlin
kotlin {
    js {
        browser {
            testTask {
                useKarma {
                    useChromeHeadless()
                }
            }
        }

        nodejs {
            testTask {
                useMocha()
            }
        }
    }
}
```

Test:

```kotlin
package lesson

import kotlin.test.Test
import kotlin.test.assertEquals

class SlugRulesTest {
    @Test
    fun normalizesSlug() {
        val slug = SlugRules.normalize(" JavaScript Target ")

        assertEquals("javascript-target", slug)
    }
}
```

CI должен установить браузеры, если включены тесты браузера. Gradle может настроить средство запуска тестов, но он не устанавливает Chrome волшебным образом для каждого образа CI.

## Проверочный список для боевого кода

Для цели JS:

- намеренно выбирайте браузер, Node или оба;
- исключить API DOM/Node из `commonMain`;
- isolate `dynamic`;
- обертывать пакеты npm типизированными API;
- export only stable JS-facing DTOs;
- добавьте `@JsName`, где имена имеют значение;
- understand DCE and боевой bundles;
- настроить тесты браузера/узла в CI;
- не предполагайте, что числовая модель JS равна числовой модели JVM.

<CatTip>
Kotlin/JS сияет, когда Kotlin владеет правилами и типами, в то время как взаимодействие с JavaScript остается узким, хорошо обозначенным кошачьим клапаном на краю.
</CatTip>
