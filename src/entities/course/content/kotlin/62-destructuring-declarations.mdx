## Destructuring создает несколько переменных сразу

Destructuring declaration позволяет разложить object на несколько variables.

```kotlin
data class User(
    val name: String,
    val age: Int
)

val user = User(name = "Мила", age = 28)

val (name, age) = user

println(name) // Мила
println(age)  // 28
```

Compiler превращает такой код в calls `component1`, `component2` и так далее.

```kotlin
val nameManual = user.component1()
val ageManual = user.component2()
```

<CatTip>
Destructuring — это аккуратно развернуть коробку с данными и разложить содержимое по маленьким мискам с понятными именами.
</CatTip>

## `componentN`

Для destructuring справа должен быть object, у которого доступны operator functions `component1`, `component2`, ..., сколько нужно переменных.

```kotlin
class LessonRef(
    private val slug: String,
    private val title: String
) {
    operator fun component1(): String {
        return slug
    }

    operator fun component2(): String {
        return title
    }
}

val ref = LessonRef("01-introduction", "Введение")
val (slug, title) = ref

println("$slug -> $title")
```

`operator` обязателен для custom component functions.

```kotlin
class BrokenRef(
    private val slug: String
) {
    fun component1(): String {
        // Без operator destructuring использовать нельзя.
        return slug
    }
}
```

## Data classes

Data class автоматически генерирует `componentN` для primary constructor properties в порядке объявления.

```kotlin
data class Lesson(
    val order: Int,
    val slug: String,
    val title: String
)

val lesson = Lesson(1, "01-introduction", "Введение")
val (order, lessonSlug, lessonTitle) = lesson

println("$order. $lessonTitle ($lessonSlug)")
```

Properties outside primary constructor не получают generated component.

```kotlin
data class Course(
    val id: String,
    val title: String
) {
    val normalizedTitle: String = title.trim().lowercase()
}

val course = Course("kotlin", " Kotlin ")
val (id, courseTitle) = course

println(id)
println(courseTitle)
// normalizedTitle не участвует в destructuring.
```

<MouseTrap>
Порядок primary constructor properties становится частью destructuring API. Переставишь поля — destructuring-код начнет получать другие values без изменения имен переменных.
</MouseTrap>

## Pair

`Pair` имеет `component1` и `component2`.

```kotlin
val pair = "language" to "Kotlin"

val (key, value) = pair

println("$key = $value")
```

Pair удобен для локального glue-кода, но в публичных API часто лучше data class.

```kotlin
data class LanguageSetting(
    val key: String,
    val value: String
)
```

У data class поля имеют имена и смысл, а `first`/`second` быстро теряют контекст.

## Triple

`Triple` имеет три components.

```kotlin
val triple = Triple("kotlin", "collections", 56)

val (language, section, lessonCount) = triple

println("$language / $section / $lessonCount")
```

Если triple начинает жить дольше пары строк, почти всегда лучше заменить его data class.

```kotlin
data class CourseSnapshot(
    val language: String,
    val section: String,
    val lessonCount: Int
)
```

## Map entries

`Map.Entry` поддерживает destructuring через `component1` для key и `component2` для value.

```kotlin
val progress = mapOf(
    "01-introduction" to 100,
    "02-types" to 80
)

for ((slug, percent) in progress) {
    println("$slug: $percent%")
}
```

Это читается лучше, чем `entry.key` и `entry.value`, когда оба значения нужны сразу.

```kotlin
val labels = progress.map { (slug, percent) ->
    // Lambda получает одну entry, но destructuring разбирает ее на key и value.
    "$slug -> $percent%"
}
```

## Destructuring в `for` loops

Любая collection элементов с component-functions может destructuring-иться в loop.

```kotlin
val lessons = listOf(
    Lesson(1, "01-introduction", "Введение"),
    Lesson(2, "02-types", "Типы")
)

for ((number, currentSlug, currentTitle) in lessons) {
    println("$number: $currentTitle ($currentSlug)")
}
```

С `withIndex` можно получить index и value.

```kotlin
for ((index, currentLesson) in lessons.withIndex()) {
    // index приходит из IndexedValue.component1, lesson — из component2.
    println("${index + 1}: ${currentLesson.title}")
}
```

## Underscore `_`

Если component не нужен, используй `_`.

```kotlin
val (_, titleOnly) = "01-introduction" to "Введение"

println(titleOnly)
```

Для skipped components соответствующий `componentN` не вызывается.

```kotlin
class ExpensiveComponents {
    operator fun component1(): String {
        println("component1")
        return "first"
    }

    operator fun component2(): String {
        println("component2")
        return "second"
    }
}

val (_, second) = ExpensiveComponents()

println(second)
```

В этом примере `component1` не должен вызываться, потому что первый component пропущен underscore.

## Destructuring в lambda parameters

Lambda может destructuring-ить один parameter.

```kotlin
val rendered = progress.mapValues { (_, percent) ->
    // Key не нужен, поэтому используем underscore.
    "$percent%"
}

println(rendered)
```

Важно различать два parameters и один destructured parameter.

```kotlin
val pairHandler: (Pair<String, Int>) -> String = { (slug, percent) ->
    "$slug: $percent%"
}

val twoArgumentHandler: (String, Int) -> String = { slug, percent ->
    "$slug: $percent%"
}
```

Скобки вокруг destructuring обязательны.

```kotlin
progress.map { (slug, percent) ->
    "$slug -> $percent"
}
```

Можно указать type whole destructured parameter.

```kotlin
val labels = progress.map { (slug, percent): Map.Entry<String, Int> ->
    "$slug: $percent%"
}
```

Или type отдельного component.

```kotlin
val strictLabels = progress.map { (slug: String, percent: Int) ->
    "$slug: $percent%"
}
```

## Destructuring не создает связь с исходным object

Destructured variables получают values на момент declaration.

```kotlin
data class MutableBox(var value: Int)

val box = MutableBox(10)
val (snapshot) = box

box.value = 20

println(snapshot) // 10
println(box.value) // 20
```

Если component itself является mutable object, destructuring копирует reference, а не deep copy.

```kotlin
data class LessonGroup(
    val titles: MutableList<String>
)

val group = LessonGroup(mutableListOf("Введение"))
val (titles) = group

titles.add("Типы")

println(group.titles) // [Введение, Типы]
```

## Custom component functions как API

Custom destructuring стоит использовать там, где порядок components очевиден.

```kotlin
data class Coordinate(
    val x: Int,
    val y: Int
)

// Data class уже генерирует component1/component2.
val (x, y) = Coordinate(10, 20)
```

Если порядок не очевиден, destructuring может навредить.

```kotlin
data class Money(
    val amountCents: Long,
    val currency: String
)

val (amount, currency) = Money(1_500, "RUB")
```

Этот пример еще читается. Но если components одного типа, риск выше.

```kotlin
data class DateParts(
    val day: Int,
    val month: Int,
    val year: Int
)

val (day, month, year) = DateParts(9, 7, 2026)
```

Перепутать `day` и `month` легко, поэтому для public API часто безопаснее явно обращаться к properties.

```kotlin
val parts = DateParts(9, 7, 2026)
println("${parts.day}.${parts.month}.${parts.year}")
```

## Production-пример: parsing результата

```kotlin
data class ParsedLesson(
    val order: Int,
    val slug: String,
    val title: String
)

fun renderLessonList(lessons: List<ParsedLesson>): List<String> {
    return lessons.map { (order, slug, title) ->
        // Для короткого локального форматирования destructuring делает код компактным.
        "$order. $title [$slug]"
    }
}
```

Если logic растет, лучше использовать named access.

```kotlin
fun renderVerboseLesson(lesson: ParsedLesson): String {
    val level = if (lesson.order <= 10) {
        "базовый"
    } else {
        "продвинутый"
    }

    return "${lesson.title}: ${lesson.slug}, уровень: $level"
}
```

## Production-пример: validation map input

```kotlin
data class ValidationError(
    val key: String,
    val reason: String
)

fun validateProgress(progress: Map<String, Int>): List<ValidationError> {
    return progress.mapNotNull { (slug, percent) ->
        when {
            slug.isBlank() -> ValidationError(
                key = slug,
                reason = "Slug не должен быть пустым"
            )

            percent !in 0..100 -> ValidationError(
                key = slug,
                reason = "Процент должен быть от 0 до 100"
            )

            else -> null
        }
    }
}
```

Map destructuring здесь делает правило валидации прямым: работаем сразу с key и value.

## Частые ошибки

Слишком длинный destructuring ухудшает читаемость.

```kotlin
data class BigModel(
    val a: String,
    val b: String,
    val c: String,
    val d: String
)

val (a, b, c, d) = BigModel("a", "b", "c", "d")
```

В таком случае named properties часто яснее.

```kotlin
val model = BigModel("a", "b", "c", "d")
println(model.c)
```

Нельзя destructuring-ить больше components, чем предоставляет type.

```kotlin
val pair = "slug" to 100

// val (slug, percent, extra) = pair // Ошибка компиляции: component3 нет.
```

Не меняй порядок primary constructor properties data class без учета callers, которые используют destructuring.

## Практические правила

Используй destructuring для:

- короткого локального кода;
- `Pair`, `Map.Entry`, `withIndex`;
- data class с очевидным порядком components;
- parsing результата regex через `destructured`;
- loop/lambda, где key и value нужны сразу.

Избегай destructuring для:

- больших моделей;
- public API с неочевидным порядком;
- нескольких соседних values одного типа;
- кода, где имена properties важнее компактности.

<CatTip>
Destructuring хорош, когда читатель сразу понимает, что лежит в каждой переменной. Если приходится вспоминать порядок constructor-а, лучше дать свойствам говорить самим за себя.
</CatTip>
