## Reflection смотрит на программу во время выполнения

Reflection — это набор возможностей, которые позволяют runtime-коду изучать структуру программы: классы, функции, properties, constructors, annotations и иногда вызывать их динамически.

```kotlin
data class Lesson(
    val slug: String,
    val title: String
)

val klass = Lesson::class

println(klass.simpleName) // Lesson
```

Reflection полезна для serialization, DI containers, routing, test frameworks, mappers, admin tooling. Но это не повседневная замена обычным typed calls.

<CatTip>
Reflection — это кошачье зеркало для кода: можно увидеть форму объекта, но если постоянно тыкать лапой в зеркало вместо обычных вызовов, код станет медленнее и хрупче.
</CatTip>

## `KClass`

`KClass` представляет Kotlin class reference.

```kotlin
import kotlin.reflect.KClass

fun describeClass(type: KClass<*>): String {
    return buildString {
        append("simpleName=")
        append(type.simpleName)
        append(", qualifiedName=")
        append(type.qualifiedName)
    }
}

println(describeClass(Lesson::class))
```

`::class` для statically known class:

```kotlin
val lessonClass = Lesson::class
```

`object::class` для runtime class конкретного instance:

```kotlin
val value: Any = Lesson("01-introduction", "Введение")
val runtimeClass = value::class

println(runtimeClass.simpleName) // Lesson
```

На JVM `KClass` и `java.lang.Class` — разные представления. Между ними можно переходить.

```kotlin
val javaClass: Class<Lesson> = Lesson::class.java
val kotlinClass: KClass<Lesson> = javaClass.kotlin
```

## `kotlin-reflect` dependency

Core class references и callable references доступны в стандартной библиотеке. Но расширенная reflection на JVM требует dependency `kotlin-reflect`.

```kotlin
dependencies {
    implementation(kotlin("reflect"))
}
```

Без `kotlin-reflect` многие возможности из `kotlin.reflect.full`, например `memberProperties`, будут недоступны или ограничены.

```kotlin
import kotlin.reflect.full.memberProperties

fun propertyNames(type: KClass<*>): List<String> {
    return type.memberProperties.map { property ->
        property.name
    }
}
```

<MouseTrap>
Не добавляй `kotlin-reflect` автоматически во все модули. Это отдельная runtime dependency, и она увеличивает размер приложения. Добавляй ее там, где reflection действительно нужна.
</MouseTrap>

## Callable references

Callable reference создается через `::`.

```kotlin
fun isPublished(slug: String): Boolean {
    return slug.firstOrNull()?.isDigit() == true
}

val predicate: (String) -> Boolean = ::isPublished

println(predicate("01-introduction")) // true
```

Function reference можно передавать в higher-order functions.

```kotlin
val slugs = listOf("01-introduction", "draft-99", "02-types")

val published = slugs.filter(::isPublished)

println(published) // [01-introduction, 02-types]
```

Если функция overload-нута, expected type помогает compiler выбрать нужную.

```kotlin
fun normalize(value: String): String = value.trim().lowercase()
fun normalize(value: Int): Int = value.coerceIn(0, 100)

val stringNormalizer: (String) -> String = ::normalize
val intNormalizer: (Int) -> Int = ::normalize
```

## Member function references

Unbound member reference требует receiver как первый argument.

```kotlin
val trimFunction: (String) -> String = String::trim

println(trimFunction("  Kotlin  ")) // Kotlin
```

Bound reference уже привязан к конкретному receiver.

```kotlin
val regex = Regex("""\d+""")
val isNumber: (CharSequence) -> Boolean = regex::matches

println(isNumber("42")) // true
```

Unbound reference на тот же метод имеет receiver parameter.

```kotlin
val matches: (Regex, CharSequence) -> Boolean = Regex::matches

println(matches(regex, "42")) // true
```

## Property references

Property reference тоже создается через `::`.

```kotlin
val titleProperty = Lesson::title

val title = titleProperty.get(Lesson("01-introduction", "Введение"))

println(title) // Введение
```

Top-level property reference можно читать через `get`.

```kotlin
val platformName = "kotlingo"

println(::platformName.name) // platformName
println(::platformName.get()) // kotlingo
```

Mutable property reference поддерживает `set`.

```kotlin
class Progress {
    var percent: Int = 0
}

val progress = Progress()
val property = Progress::percent

property.set(progress, 80)

println(progress.percent) // 80
```

Property reference часто полезна в collection operations.

```kotlin
val lessons = listOf(
    Lesson("01-introduction", "Введение"),
    Lesson("02-types", "Типы")
)

val titles = lessons.map(Lesson::title)

println(titles) // [Введение, Типы]
```

## Constructor references

Constructor reference выглядит как `::ClassName`.

```kotlin
data class LessonCard(
    val slug: String,
    val title: String
)

val factory: (String, String) -> LessonCard = ::LessonCard

val card = factory("01-introduction", "Введение")

println(card)
```

Constructor references удобны для factories, mapping и dependency wiring.

```kotlin
val rawPairs = listOf(
    "01-introduction" to "Введение",
    "02-types" to "Типы"
)

val cards = rawPairs.map { (slug, title) ->
    LessonCard(slug, title)
}
```

Если constructor signature совпадает с expected function type, можно передать `::LessonCard` напрямую.

## `KFunction` и `KProperty`

Callable references имеют reflection-типы.

```kotlin
import kotlin.reflect.KFunction1
import kotlin.reflect.KProperty1

val functionReference: KFunction1<String, Boolean> = ::isPublished
val propertyReference: KProperty1<Lesson, String> = Lesson::title

println(functionReference.name) // isPublished
println(propertyReference.name) // title
```

Обычному business-коду чаще достаточно function type. Reflection-type нужен, если ты читаешь metadata вроде `name`, parameters или annotations.

## Динамический вызов через `call`

С `kotlin-reflect` callable можно вызвать динамически через `call`.

```kotlin
import kotlin.reflect.full.memberFunctions

data class Course(val title: String) {
    fun label(prefix: String): String {
        return "$prefix: $title"
    }
}

val course = Course("Kotlin")
val labelFunction = Course::class.memberFunctions.first { function ->
    function.name == "label"
}

val label = labelFunction.call(course, "Курс")

println(label) // Курс: Kotlin
```

Динамический вызов теряет compile-time safety. Ошибки количества аргументов и типов становятся runtime-ошибками.

## Reflection и annotations

Reflection часто используется для чтения annotations.

```kotlin
annotation class Route(val path: String)

class LessonController {
    @Route("/lessons")
    fun listLessons(): List<String> {
        return listOf("01-introduction", "02-types")
    }
}

fun findRoutes(controllerClass: KClass<*>): List<String> {
    return controllerClass.members.mapNotNull { member ->
        val route = member.annotations.filterIsInstance<Route>().firstOrNull()

        route?.path
    }
}
```

Это типичный фундамент простого routing/metadata механизма.

## Java reflection interop

На JVM можно перейти к Java reflection.

```kotlin
import kotlin.reflect.jvm.javaField
import kotlin.reflect.jvm.javaGetter

val propertyRef = Lesson::title

println(propertyRef.javaGetter)
println(propertyRef.javaField)
```

Это нужно, когда библиотека принимает `java.lang.reflect.Method`, `Field` или `Class`.

## Performance and limitations

Reflection обычно медленнее обычных вызовов и хуже проверяется compiler-ом.

```kotlin
val direct = lesson.title
val reflective = Lesson::title.get(lesson)
```

Direct access быстрее, проще и безопаснее. Reflection оправдана, когда structure неизвестна до runtime или framework строит generic mechanism.

Ограничения:

- Kotlin/JS и Kotlin/Native поддерживают reflection не так полно, как JVM;
- `kotlin-reflect` увеличивает dependency footprint;
- private access требует дополнительных правил и может ломать инкапсуляцию;
- obfuscation/minification может влиять на имена;
- dynamic call errors появляются во время выполнения.

## Production-пример: простой mapper в Map

```kotlin
import kotlin.reflect.KClass
import kotlin.reflect.full.memberProperties

fun Any.toStringMap(): Map<String, String?> {
    return this::class.memberProperties.associate { property ->
        val value = property.getter.call(this)

        // Значения приводим к String только для отображения, не для хранения typed data.
        property.name to value?.toString()
    }
}

data class LessonSummary(
    val slug: String,
    val title: String,
    val minutes: Int
)

val summary = LessonSummary("01-introduction", "Введение", 14)

println(summary.toStringMap())
```

Такой mapper удобен для debug/admin tooling, но для core serialization лучше использовать специализированные библиотеки.

## Практические правила

Используй callable references для типобезопасного функционального кода.

```kotlin
val titles = lessons.map(Lesson::title)
```

Используй `KClass`, когда нужно передать type metadata.

```kotlin
fun register(type: KClass<*>) {
    println(type.qualifiedName)
}
```

Используй full reflection только для framework-level задач.

```kotlin
val names = Lesson::class.memberProperties.map { it.name }
```

<CatTip>
Reflection лучше держать на границах системы: framework, adapters, diagnostics. Внутри доменной логики обычные typed calls почти всегда чище и быстрее.
</CatTip>
