## Что такое Kotlin Multiplatform

Kotlin Multiplatform позволяет писать часть кода один раз и компилировать ее под несколько platforms.

Обычно так разделяют:

- domain logic;
- validation;
- algorithms;
- networking contracts;
- serialization models;
- база данных abstractions;
- business rules;
- state machines;
- общий ViewModel-like logic;
- tests for общий behavior.

Platform-specific остаются:

- набор UI-инструментов;
- filesystem details;
- platform permissions;
- sensors;
- platform lifecycle;
- Java, Android, iOS, JS, Wasm, Native APIs;
- dependency injection integration;
- platform-specific logging;
- build packaging.

```kotlin
// commonMain/kotlin/lesson/Lesson.kt
package lesson

data class Lesson(
    val slug: String,
    val title: String,
    val body: String
) {
    fun shortTitle(maxLength: Int): String {
        require(maxLength > 0) { "maxLength должен быть положительным" }

        return if (title.length <= maxLength) {
            title
        } else {
            title.take(maxLength - 1) + "…"
        }
    }
}
```

Этот код не зависит от JVM, Android или iOS, поэтому ему место в `commonMain`.

<CatTip>
Kotlin Multiplatform — это не «один бинарник для всего», а аккуратная кошачья полка: общая логика лежит по центру, platform-specific миски стоят по краям.
</CatTip>

## Target

Target — build destination, под который Kotlin компилирует код.

Примеры targets:

- `jvm`;
- `androidTarget`;
- `js`;
- `wasmJs`;
- `linuxX64`;
- `macosArm64`;
- `iosArm64`;
- `iosSimulatorArm64`.

```kotlin
plugins {
    kotlin("multiplatform") version "2.4.0"
}

kotlin {
    jvm()
    js {
        browser()
    }
    iosArm64()
    iosSimulatorArm64()
}
```

Каждый target создает platform-specific compilation and source sets. Если target не объявлен, code не компилируется под эту platform.

## Source set

Source set — набор source files, resources, dependencies и compiler options, которые компилируются для определенного набора targets.

Типичная структура:

```text
src/
  commonMain/
    kotlin/
      lesson/
        Lesson.kt
  commonTest/
    kotlin/
      lesson/
        LessonTest.kt
  jvmMain/
    kotlin/
      lesson/
        JvmLessonFileReader.kt
  jvmTest/
    kotlin/
      lesson/
        JvmLessonFileReaderTest.kt
  iosMain/
    kotlin/
      lesson/
        IosPlatformClock.kt
```

`commonMain` компилируется во все заявленные цели.

`jvmMain` компилируется только в целевую JVM.

`iosMain` compiles to iOS targets that depend on it, например `iosArm64` and `iosSimulatorArm64`.

```kotlin
// commonMain/kotlin/lesson/Slug.kt
package lesson

@JvmInline
value class Slug(
    val value: String
) {
    init {
        require(value.isNotBlank()) { "Slug не должен быть пустым" }
        require(value == value.lowercase()) { "Slug должен быть в lower case" }
    }
}
```

Common source set не может использовать platform-specific APIs.

```kotlin
// commonMain/kotlin/lesson/WrongCommonCode.kt
package lesson

fun wrong() {
    // Этот код не скомпилируется в commonMain:
    // java.io.File("lesson.md").readText()
    //
    // Причина: java.io доступен на JVM, но не является общим API для всех targets.
}
```

<MouseTrap>
Если код в `commonMain` импортирует `java.*`, `android.*`, `platform.Foundation.*` или browser API напрямую, он уже перестал быть truly common code.
</MouseTrap>

## Platform source set

Platform source set содержит code for one platform family or target.

```kotlin
// jvmMain/kotlin/lesson/JvmLessonFileReader.kt
package lesson

import java.nio.file.Files
import java.nio.file.Path

class JvmLessonFileReader {
    fun read(path: Path): String {
        // JVM-specific filesystem API разрешен в jvmMain.
        return Files.readString(path)
    }
}
```

```kotlin
// iosMain/kotlin/lesson/IosDeviceName.kt
package lesson

import platform.UIKit.UIDevice

fun iosDeviceName(): String {
    // iOS API разрешен в iosMain, но не в commonMain.
    return UIDevice.currentDevice.name
}
```

Наборы исходных кодов платформы могут зависеть от общего кода.

Общий набор исходного кода не может зависеть от набора исходного кода платформы.

```kotlin
// jvmMain/kotlin/lesson/JvmLessonPrinter.kt
package lesson

class JvmLessonPrinter {
    fun print(lesson: Lesson) {
        // Lesson объявлен в commonMain и доступен в jvmMain.
        println("${lesson.slug}: ${lesson.title}")
    }
}
```

## Компиляция под конкретную цель

Когда Kotlin компилирует target, он берет all source sets labeled with that target.

Например, для `jvm` target компилятор берет:

- `commonMain`;
- промежуточные source sets, если они участвуют в hierarchy;
- `jvmMain`.

Для `iosArm64`:

- `commonMain`;
- `nativeMain`, если есть;
- `appleMain`, если есть;
- `iosMain`, если есть;
- `iosArm64Main`.

Это объясняет важное правило: platform source set sees common API, потому что они компилируются together for the target.

## Hierarchical source sets

Hierarchical source sets позволяют делить код не только между all targets or one target, но и между группой targets.

Пример:

```text
commonMain
  nativeMain
    appleMain
      iosMain
        iosArm64Main
        iosSimulatorArm64Main
      macosMain
        macosArm64Main
  jvmMain
  jsMain
```

Зачем нужны intermediate source sets:

- общий Native code;
- общий Apple code;
- общий iOS device and simulator code;
- общий browser and Node JS code;
- общий JVM and Android code, если проект так устроен;
- общий код for targets with similar platform APIs.

```kotlin
// appleMain/kotlin/platform/AppleUuid.kt
package platform

import platform.Foundation.NSUUID

fun randomAppleUuid(): String {
    // Apple-specific API доступен для Apple targets, но не для Android/JVM/JS.
    return NSUUID().UUIDString()
}
```

Этот code нельзя положить в `commonMain`, но можно положить в source set, который компилируется только для Apple targets.

## `dependsOn`

`dependsOn` — relation between source sets.

Если `A.dependsOn(B)`, это значит:

- `A` видит API from `B`;
- `A` наследует зависимости от `B`;
- `A` может предоставлять реализации `actual` для объявлений `expect` из `B`;
- `B` должен компилироваться для каждой цели `A`;
- внутренние объявления из `B` видны `A` внутри модуля.

В большинстве обычных проектов Kotlin Gradle plugin создает стандартную hierarchy сам.

```kotlin
kotlin {
    iosArm64()
    iosSimulatorArm64()

    sourceSets {
        val iosMain by getting {
            dependsOn(commonMain.get())
        }

        val iosArm64Main by getting {
            dependsOn(iosMain)
        }

        val iosSimulatorArm64Main by getting {
            dependsOn(iosMain)
        }
    }
}
```

Этот ручной пример полезен для понимания модели, но в современных проектах default hierarchy обычно избавляет от такой рутины.

## Правила common-кода

В common code доступны:

- Kotlin standard library common APIs;
- общие зависимости, опубликованные в виде мультиплатформенных библиотек;
- declarations from source sets, от которых current source set depends;
- объявления `expect` из общего исходного набора;
— чистый код Kotlin без импорта платформы.

В common code недоступны:

- JDK-specific APIs;
- Android framework APIs;
- UIKit/Foundation directly, если source set не Apple-specific;
- browser DOM APIs, если source set не JS/Wasm browser-specific;
- platform-only dependencies;
- assumptions about filesystem, threads, locale, time zone, entropy source, crypto provider.

```kotlin
// commonMain/kotlin/lesson/LessonValidator.kt
package lesson

class LessonValidator {
    fun validate(lesson: Lesson): List<String> {
        val errors = mutableListOf<String>()

        if (lesson.slug.isBlank()) {
            errors += "Slug не должен быть пустым"
        }

        if (lesson.title.isBlank()) {
            errors += "Title не должен быть пустым"
        }

        return errors
    }
}
```

Такой validator идеально живет в common code.

## How common code touches platform code

Common code не импортирует platform APIs directly, но может работать с ними через boundaries:

- интерфейсы определены совместно и реализованы на платформе;
- Объявления `expect` совпадают с реализациями `actual` для каждой платформы;
- внедрение зависимостей с уровня приложения;
- обратный вызов interfaces;
- abstract repositories;
— мультиплатформенные библиотеки, которые уже скрывают различия платформ.

Interface approach:

```kotlin
// commonMain/kotlin/platform/Clock.kt
package platform

interface Clock {
    fun nowMillis(): Long
}

class LessonTimer(
    private val clock: Clock
) {
    fun elapsedSince(startMillis: Long): Long {
        return clock.nowMillis() - startMillis
    }
}
```

JVM implementation:

```kotlin
// jvmMain/kotlin/platform/JvmClock.kt
package platform

class JvmClock : Clock {
    override fun nowMillis(): Long {
        return System.currentTimeMillis()
    }
}
```

`expect/actual` approach будет следующей темой.

## Тесты в мультиплатформенных проектах

`commonTest` тестирует общее поведение и компилирует для всех целей, где это применимо.

```kotlin
// commonTest/kotlin/lesson/LessonValidatorTest.kt
package lesson

import kotlin.test.Test
import kotlin.test.assertTrue

class LessonValidatorTest {
    @Test
    fun rejectsBlankTitle() {
        val lesson = Lesson(
            slug = "90-multiplatform-mental-model",
            title = "",
            body = "Текст"
        )

        val errors = LessonValidator().validate(lesson)

        assertTrue(errors.any { error -> "Title" in error })
    }
}
```

`jvmTest`, `iosTest`, `jsTest` нужны, когда test depends on platform behavior.

```kotlin
// jvmTest/kotlin/platform/JvmClockTest.kt
package platform

import kotlin.test.Test
import kotlin.test.assertTrue

class JvmClockTest {
    @Test
    fun returnsPositiveTimestamp() {
        val clock = JvmClock()

        assertTrue(clock.nowMillis() > 0)
    }
}
```

## Дизайн API в мультиплатформенности

Общий API должен быть platform-neutral.

Плохие common names:

- `FilePath`, если JS/iOS representation отличается;
- `ThreadConfig`, если targets without threads participate;
- `ActivityProvider`, если common module должен знать Android;
- `NSErrorWrapper`, если JVM/JS не используют NSError.

Лучшие common concepts:

- `LessonSource`;
- `Clock`;
- `UuidGenerator`;
- `SettingsStore`;
- `HttpClient`;
- `Logger`;
- `FileHandle`, если abstraction deliberately hides platform file details.

```kotlin
// commonMain/kotlin/storage/SettingsStore.kt
package storage

interface SettingsStore {
    suspend fun readString(key: String): String?

    suspend fun writeString(key: String, value: String)
}
```

Реализации платформы могут использовать:

- Android DataStore;
- JVM file;
- iOS NSUserDefaults;
- browser localStorage;
- in-memory test фиктивный.

<CatTip>
Лучший common API говорит языком продукта, а не языком platform. `LessonProgressStore` переносится лучше, чем `SharedPreferencesWrapper`.
</CatTip>

## Боевой mental проверочный список

Перед переносом code в `commonMain` спроси:

- использует ли он только общие зависимости;
- можно ли это проверить в `commonTest`;
- избегает ли имен платформ в публичном API;
- абстрагированы ли время, случайные значения, файловая система и локаль;
- можно ли внедрить поведение, специфичное для платформы;
- нужен ли `expect/actual` или достаточно интерфейса;
- будет ли добавление новой цели сломать этот дизайн;
- это иерархия исходного набора, выражающая реальное, а не желаемое разделение.

Kotlin Multiplatform работает лучше всего, когда общий код отвечает за решения предметной области, а код платформы отвечает за интеграцию платформы.
