## API — это обещание

API — это любая граница, используемая другим кодом:

- public library classes;
- module interfaces;
- service methods;
- DTOs;
- extension functions;
- sealed results;
- Gradle module exports;
- Java-facing Kotlin classes;
- multiplatform common declarations.

Хороший Kotlin API усложняет выражение недопустимых состояний и упрощает обработку ожидаемых состояний.

```kotlin
class LessonService {
    fun find(slug: LessonSlug): Lesson? {
        return if (slug.value == "102-api-design-in-kotlin") {
            Lesson(slug, "API Design In Kotlin")
        } else {
            null
        }
    }
}

@JvmInline
value class LessonSlug(
    val value: String
) {
    init {
        require(value.isNotBlank()) { "Slug не должен быть пустым" }
    }
}

data class Lesson(
    val slug: LessonSlug,
    val title: String
)
```

<CatTip>
Хороший Kotlin API похож на аккуратную кошачью дверцу: внутрь проходят только допустимые значения, а вызывающий код сразу понимает, что может случиться.
</CatTip>

## Обнуляемость как контракт

В Котлине возможность обнуления является частью системы типов и контракта API.

```kotlin
interface LessonRepository {
    suspend fun find(slug: LessonSlug): Lesson?

    suspend fun get(slug: LessonSlug): Lesson
}
```

Здесь:

- `find` говорит, что отсутствие — это нормально;
- `get` говорит, что отсутствие является исключительным.

Реализация:

```kotlin
class InMemoryLessonRepository(
    lessons: List<Lesson>
) : LessonRepository {
    private val lessonsBySlug = lessons.associateBy { lesson -> lesson.slug }

    override suspend fun find(slug: LessonSlug): Lesson? {
        return lessonsBySlug[slug]
    }

    override suspend fun get(slug: LessonSlug): Lesson {
        return lessonsBySlug[slug] ?: throw LessonNotFoundException(slug)
    }
}

class LessonNotFoundException(
    slug: LessonSlug
) : RuntimeException("Урок не найден: ${slug.value}")
```

Избегайте возврата с нулевым значением, когда отсутствие не является частью бизнес-модели. Избегайте исключений, когда отсутствие является общей ветвью.

<MouseTrap>
`String?` — это не документация. Если `null` имеет смысл, назови метод и тип так, чтобы вызывающий код понимал: `null` означает «не найдено», «не загружено», «не разрешено» или «неизвестно».
</MouseTrap>

## Не возвращать коллекции, допускающие значение NULL

Предпочитайте пустую коллекцию коллекции, допускающей значение NULL.

```kotlin
interface RecommendationService {
    suspend fun recommendationsFor(slug: LessonSlug): List<LessonSlug>
}
```

Плохо:

```kotlin
interface BadRecommendationService {
    suspend fun recommendationsFor(slug: LessonSlug): List<LessonSlug>?
}
```

Если есть разница между «не загружено» и «загружено пусто», смоделируйте ее явно.

```kotlin
sealed interface RecommendationState {
    data object NotLoaded : RecommendationState

    data class Ready(
        val slugs: List<LessonSlug>
    ) : RecommendationState

    data class Failed(
        val reason: String
    ) : RecommendationState
}
```

## Результаты на sealed-типах

Используй sealed-типы результата, когда вызывающий код должен обработать несколько доменных исходов.

```kotlin
sealed interface OpenLessonResult {
    data class Ready(
        val lesson: Lesson
    ) : OpenLessonResult

    data class NotFound(
        val slug: LessonSlug
    ) : OpenLessonResult

    data class AccessDenied(
        val slug: LessonSlug
    ) : OpenLessonResult
}

class OpenLessonUseCase(
    private val repository: LessonRepository,
    private val accessPolicy: AccessPolicy
) {
    suspend fun open(user: User, slug: LessonSlug): OpenLessonResult {
        val lesson = repository.find(slug)
            ?: return OpenLessonResult.NotFound(slug)

        if (!accessPolicy.canOpen(user, lesson)) {
            return OpenLessonResult.AccessDenied(slug)
        }

        return OpenLessonResult.Ready(lesson)
    }
}

data class User(
    val id: String,
    val premium: Boolean
)

class AccessPolicy {
    fun canOpen(user: User, lesson: Lesson): Boolean {
        return user.premium || !lesson.title.contains("Advanced")
    }
}
```

Caller:

```kotlin
fun render(result: OpenLessonResult): String {
    return when (result) {
        is OpenLessonResult.Ready -> result.lesson.title
        is OpenLessonResult.NotFound -> "Урок не найден"
        is OpenLessonResult.AccessDenied -> "Нет доступа"
    }
}
```

Запечатанный результат делает ветвление исчерпывающим.

## `Result`

Kotlin `Result` полезен для общих конвейеров успеха/неудачи, но запечатанный тип для конкретного домена часто более понятен.

```kotlin
suspend fun loadTitle(slug: LessonSlug): Result<String> {
    return runCatching {
        "Урок ${slug.value}"
    }
}
```

Используй `Result`, когда:

- отказ носит технический и родовой характер;
- вы составляете операции;
- вызывающий код will map/recover/log generically.

Используйте герметичный тип, когда:

- результаты – бизнес-состояния;
- Пользовательский интерфейс требует отдельной обработки;
- результат является частью общедоступного API;
- вам нужен исчерпывающий `when` с именованными случаями.

## Immutability по умолчанию

Предпочитай неизменяемые данные и read-only коллекции в публичном API.

```kotlin
data class Course(
    val slug: String,
    val lessons: List<Lesson>
)
```

Избегайте раскрытия изменяемых коллекций:

```kotlin
class BadCourse {
    val lessons: MutableList<Lesson> = mutableListOf()
}
```

Safer mutable internals:

```kotlin
class CourseBuilder {
    private val lessons = mutableListOf<Lesson>()

    fun add(lesson: Lesson): CourseBuilder {
        lessons += lesson
        return this
    }

    fun build(): Course {
        return Course(
            slug = "kotlin",
            lessons = lessons.toList()
        )
    }
}
```

Только для чтения `List` does not guarantee deep immutability, but it prevents вызывающий код from mutating through that API.

## Классы значений для концепций

Используйте классы ценностей, чтобы предотвратить примитивную одержимость.

```kotlin
@JvmInline
value class UserId(
    val value: String
) {
    init {
        require(value.isNotBlank()) { "UserId не должен быть пустым" }
    }
}

@JvmInline
value class Percent(
    val value: Int
) {
    init {
        require(value in 0..100) { "Percent должен быть в диапазоне 0..100" }
    }
}
```

Без классов значений API может случайно смешать две строки.

```kotlin
fun markCompleted(userId: UserId, slug: LessonSlug) {
    println("${userId.value} completed ${slug.value}")
}
```

## Extension API boundaries

Функции расширения — это статические функции с синтаксисом получателя. На самом деле они не добавляют членов в класс.

Good extension:

```kotlin
fun String.toLessonSlug(): LessonSlug {
    return LessonSlug(
        trim()
            .lowercase()
            .replace(Regex("\\s+"), "-")
    )
}
```

Risky extension:

```kotlin
fun LessonRepository.deleteEverything() {
    // Плохо: опасная operation выглядит как natural member чужого API.
}
```

Используйте расширения для:

- formatting;
- mapping;
- small domain conversions;
- convenience around stable types;
- DSL-подобные API, для которых требуется возможность обнаружения.

Избегайте расширений для:

- operations requiring hidden dependencies;
- побочные эффекты that look harmless;
- overriding-like behavior;
- общедоступный API, где вызывающие абоненты Java являются основными пользователями.

## Binary compatibility

Для опубликованных библиотек JVM изменения, совместимые с исходным кодом, все равно могут привести к нарушению двоичного кода.

Risky changes:

- removing public class/function/property;
- изменение сигнатуры функции;
- изменение типа возврата;
- изменение параметров конструктора;
- добавление абстрактного члена в интерфейс;
- изменение первичного конструктора класса данных;
- изменение тела встроенной публичной функции, которое неправильно ссылается на внутренние детали;
- moving declarations between packages.

Используйте явный режим API:

```kotlin
kotlin {
    explicitApi()
}
```

Тогда публичные объявления нуждаются в явной видимости и возвращаемых типах.

```kotlin
public class LessonApi {
    public fun title(slug: String): String {
        return "Урок $slug"
    }
}
```

<CatTip>
For library APIs, explicit return types are like таблички на кошачьих дверцах: вызывающий код and compiler see exactly what contract is promised.
</CatTip>

## Java-facing API

Если пользователи Java используют Kotlin API, проектируйте и для Java.

```kotlin
class LessonFacade(
    private val service: LessonService
) {
    @JvmOverloads
    fun title(slug: String, fallback: String = "Без названия"): String {
        return service.findTitle(slug) ?: fallback
    }
}

class LessonService {
    fun findTitle(slug: String): String? {
        return if (slug.isBlank()) null else "Урок $slug"
    }
}
```

Consider:

- `@JvmOverloads` для параметров по умолчанию;
- `@JvmStatic` для сопутствующих функций, используемых из Java;
- `@JvmName` для перегрузки и конфликтов имен;
- `@Throws` для проверенных ожиданий исключений;
- избегать типов, предназначенных только для Kotlin, в API-интерфейсах с большим количеством Java.

## Проверочный список для боевого кода

Для проектирования API Kotlin:

- допустимость `null` отражает реальный контракт;
- коллекции пусты и не могут иметь значение NULL, если только состояние не требует явного моделирования;
- результаты домена используют запечатанные типы;
- публичные данные по умолчанию неизменяемы;
- value-классы защищают доменные понятия;
- функции-расширения не скрывают опасные побочные эффекты;
- публичные API имеют явные типы возвращаемых значений;
- API-интерфейсы, ориентированные на Java, тестируются на Java, когда потребители Java имеют значение;
- перед публикацией проверяется бинарная совместимость.

<MouseTrap>
Задолженность по разработке API обходится дорого, потому что вызывающие абоненты формируют вокруг нее привычки. Сбивающий с толку нулевой, изменяемый или общедоступный API случайно становится архитектурой.
</MouseTrap>
