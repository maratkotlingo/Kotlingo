## Контракт сообщает compiler эффект вызова

Contract описывает факты, которые compiler может использовать для control-flow и data-flow analysis.

```kotlin
@OptIn(kotlin.contracts.ExperimentalContracts::class)
fun requireTitle(value: String?) {
    kotlin.contracts.contract {
        returns() implies (value != null)
    }

    require(value != null) { "Название обязательно" }
}
```

После normal return compiler знает, что `value != null`:

```kotlin
fun printTitle(title: String?) {
    requireTitle(title)
    println(title.length)
}
```

Contract не выполняет проверку. Проверку выполняет body `require`. DSL только обещает compiler связь между возвратом и условием.

## Почему обычной функции недостаточно

Без contract compiler анализирует function как отдельную black box:

```kotlin
fun checkTitle(value: String?) {
    require(value != null) { "Название обязательно" }
}

fun example(title: String?) {
    checkTitle(title)
    // title всё ещё имеет static type String?.
}
```

Compiler не выводит межпроцедурно произвольные postconditions из body. Contract делает ограниченный effect частью signature metadata.

## Пользовательский contracts experimental

API `kotlin.contracts` для объявления пользовательский contracts помечен `ExperimentalContracts`. Нужно opt-in:

```kotlin
@file:OptIn(kotlin.contracts.ExperimentalContracts::class)
```

или annotation на declaration/file/module compiler setting.

Experimental означает, что DSL/API/поддерживаемые effects могут меняться. Standard library contracts могут использоваться клиентами без того, чтобы каждый место вызова объявлял пользовательский contract.

Extended contracts, включая новые effects вроде `holdsIn`, имеют отдельный более экспериментальный opt-in и зависят от актуальной compiler version.

<MouseTrap title="Мышеловка: opt-in не доказывает истинность контракта">
Compiler верит автору. Ложный contract может разрешить unsafe smart cast или неверный definite assignment. Покрой body тестами на каждый declared effect.
</MouseTrap>

## Contract должен быть в начале

`contract { ... }` должен быть первым statement body function, до validation, logging и других calls.

```kotlin
@OptIn(kotlin.contracts.ExperimentalContracts::class)
fun requireEnabled(value: Boolean) {
    kotlin.contracts.contract {
        returns() implies value
    }

    require(value) { "Функция должна быть включена" }
}
```

Не все произвольные boolean expressions поддерживаются contract DSL. Compiler разрешает ограниченный набор stable conditions над parameters/receivers.

## Ограничение top-level

Актуальный contract declaration API предназначен для top-level functions. Не проектируй contract как обычный runtime object/member metadata. Проверяй compiler version: contract capabilities развиваются.

## `returns()`

`returns()` означает normal completion с любым result, то есть функция не бросила exception и не завершилась иным образом.

```kotlin
@OptIn(kotlin.contracts.ExperimentalContracts::class)
fun requireLesson(value: Any?) {
    kotlin.contracts.contract {
        returns() implies (value is Lesson)
    }

    require(value is Lesson) { "Ожидался Lesson" }
}

data class Lesson(val title: String)
```

```kotlin
fun render(value: Any?) {
    requireLesson(value)
    println(value.title)
}
```

Smart cast возникает после successful return.

## `returns(value)`

Contract связывает конкретный returned value с condition.

```kotlin
@OptIn(kotlin.contracts.ExperimentalContracts::class)
fun isLesson(value: Any?): Boolean {
    kotlin.contracts.contract {
        returns(true) implies (value is Lesson)
    }

    return value is Lesson
}
```

```kotlin
fun renderIfLesson(value: Any?) {
    if (isLesson(value)) {
        println(value.title)
    }
}
```

Contract не утверждает, что false обязательно означает `value !is Lesson`, если обратный effect не объявлен/не выводится.

Можно описать обе стороны, если body точно эквивалентно condition:

```kotlin
@OptIn(kotlin.contracts.ExperimentalContracts::class)
fun isPresent(value: Any?): Boolean {
    kotlin.contracts.contract {
        returns(true) implies (value != null)
        returns(false) implies (value == null)
    }
    return value != null
}
```

## `returnsNotNull()`

Effect означает: если result non-null, условие истинно.

```kotlin
@OptIn(kotlin.contracts.ExperimentalContracts::class)
fun Any?.asLessonOrNull(): Lesson? {
    kotlin.contracts.contract {
        returnsNotNull() implies (this@asLessonOrNull is Lesson)
    }

    return this as? Lesson
}
```

```kotlin
fun inspect(value: Any?) {
    val lesson = value.asLessonOrNull()
    if (lesson != null) {
        println(lesson.title)
    }
}
```

`returnsNotNull` полезен selector/cast helper, но simple `as?` часто уже достаточно.

## `implies`

`implies` связывает observable effect с condition:

```kotlin
returns() implies (value != null)
returns(true) implies (value is Lesson)
returnsNotNull() implies (value != null)
```

Condition должна быть безопасно анализируемой: null checks, type checks, boolean parameters и поддерживаемые combinations. Нельзя описать arbitrary база данных state, collection size mutation или временную зависимость как надёжный compile-time fact.

## `callsInPlace`

Contract сообщает, что лямбда вызывается только во время owner function и не сохраняется после её завершения.

```kotlin
@OptIn(kotlin.contracts.ExperimentalContracts::class)
inline fun <T> exactlyOnce(block: () -> T): T {
    kotlin.contracts.contract {
        callsInPlace(block, kotlin.contracts.InvocationKind.EXACTLY_ONCE)
    }

    return block()
}
```

Compiler может использовать это для definite assignment:

```kotlin
val title: String
exactlyOnce {
    title = "Contracts"
}
println(title.length)
```

Без guarantee compiler не знает, была ли лямбда вызвана.

## InvocationKind

Доступны четыре основных вида:

- `EXACTLY_ONCE` — ровно один раз;
- `AT_MOST_ONCE` — ноль или один раз;
- `AT_LEAST_ONCE` — один или больше;
- `UNKNOWN` — вызвана in place, количество неизвестно.

## `AT_MOST_ONCE`

```kotlin
@OptIn(kotlin.contracts.ExperimentalContracts::class)
inline fun runIf(
    condition: Boolean,
    block: () -> Unit
) {
    kotlin.contracts.contract {
        callsInPlace(block, kotlin.contracts.InvocationKind.AT_MOST_ONCE)
    }

    if (condition) block()
}
```

Нельзя считать, что variable точно initialized в block: block мог не выполниться.

## `AT_LEAST_ONCE`

```kotlin
@OptIn(kotlin.contracts.ExperimentalContracts::class)
inline fun repeatAtLeastOnce(times: Int, block: () -> Unit) {
    kotlin.contracts.contract {
        callsInPlace(block, kotlin.contracts.InvocationKind.AT_LEAST_ONCE)
    }

    require(times > 0) { "Число повторов должно быть положительным" }
    repeat(times) { block() }
}
```

Contract правдив только потому, что validation гарантирует `times > 0` до loop. Если порядок изменить или exception возникнет до первого block, promise может стать ложным относительно normal/abrupt semantics.

## `UNKNOWN`

```kotlin
@OptIn(kotlin.contracts.ExperimentalContracts::class)
inline fun forEvery(values: List<Int>, block: (Int) -> Unit) {
    kotlin.contracts.contract {
        callsInPlace(block, kotlin.contracts.InvocationKind.UNKNOWN)
    }
    values.forEach(block)
}
```

Для empty list вызовов ноль, для non-empty — несколько. Lambda не сохраняется, но точное количество неизвестно.

## `callsInPlace` и escaping

Следующая implementation не имеет права обещать calls-in-place:

```kotlin
class Queue {
    private val tasks = mutableListOf<() -> Unit>()

    fun schedule(task: () -> Unit) {
        tasks += task
    }
}
```

Task сохраняется после возврата `schedule`. Ложный contract разрушил бы analysis captured variables/lifetimes.

## Standard library contracts

Многие standard inline functions имеют contracts, благодаря которым compiler понимает:

- `run`, `let`, `apply`, `also` вызывают block exactly once;
- `requireNotNull` возвращает non-null;
- `isNullOrEmpty`/type helpers помогают flow analysis в поддерживаемых случаях.

Не каждый smart cast обязан происходить из public contract DSL: compiler также имеет intrinsic knowledge и обычный control-flow analysis.

## Contract и runtime

Contracts:

- не создают runtime check;
- не меняют return value;
- не ловят exception;
- не синхронизируют threads;
- не гарантируют business invariant;
- влияют на compilation/analysis call sites.

```kotlin
@OptIn(kotlin.contracts.ExperimentalContracts::class)
fun broken(value: String?) {
    kotlin.contracts.contract {
        returns() implies (value != null)
    }

    // Ложная реализация возвращается и для null.
}
```

Так писать нельзя. Compiler доверяет promise, а runtime value всё ещё null.

## Ответственность автора

Для каждого effect проверь:

1. все normal return paths;
2. все boolean result paths;
3. exceptions до лямбда invocation;
4. количество лямбда calls;
5. сохранение/передачу лямбда;
6. future changes body;
7. concurrency/reentrancy;
8. tests на null/type/control-flow boundaries.

<CatTip title="Котосовет: contract — часть семантического API">
Review contract так же строго, как unsafe cast. Изменение body без обновления effect может сделать корректно компилирующийся client небезопасным.
</CatTip>

## Contract и inline

`callsInPlace` часто используется с inline functions, но concepts различны:

- inline — code transformation/performance/control flow;
- contract — declarative effect для analysis.

Inline сам по себе не обещает exact call count. Function может не вызвать лямбда или вызвать несколько раз.

## Contract и smart cast stability

Даже правильный contract не отменяет правила стабильности variable. Mutable property с пользовательский getter/open access может не smart-cast:

```kotlin
class Holder(var title: String?)

fun render(holder: Holder) {
    val title = holder.title
    requireTitle(title)
    println(title.length)
}
```

Local `val` фиксирует snapshot. `holder.title` может измениться между calls.

## API versioning

Contracts записываются в metadata и влияют на source analysis clients. При изменении:

- пересобери clients;
- проверь compiler compatibility;
- не ослабляй effect молча;
- учитывай experimental evolution;
- тестируй multiple supported Kotlin versions для library.

## Практический пример: typed validation helpers

```kotlin
data class LessonDraft(
    val title: String?,
    val content: String?
)

@OptIn(kotlin.contracts.ExperimentalContracts::class)
fun requireNonBlank(value: String?, field: String) {
    kotlin.contracts.contract {
        returns() implies (value != null)
    }

    require(!value.isNullOrBlank()) { "$field не должен быть пустым" }
}

@OptIn(kotlin.contracts.ExperimentalContracts::class)
inline fun <T> validateExactlyOnce(block: () -> T): T {
    kotlin.contracts.contract {
        callsInPlace(block, kotlin.contracts.InvocationKind.EXACTLY_ONCE)
    }
    return block()
}

fun publish(draft: LessonDraft): String {
    val normalizedTitle: String
    val normalizedContent: String

    validateExactlyOnce {
        requireNonBlank(draft.title, "Название")
        requireNonBlank(draft.content, "Содержание")

        normalizedTitle = draft.title.trim()
        normalizedContent = draft.content.trim()
    }

    return "$normalizedTitle\n\n$normalizedContent"
}

fun main() {
    val text = publish(
        LessonDraft(
            title = " Contracts ",
            content = " Compiler effects "
        )
    )
    println(text)
}
```

`returns() implies value != null` помогает smart cast properties только в stable context; здесь data class `val` properties stable. `EXACTLY_ONCE` разрешает definite assignment local vals.

## Итоги

- contract описывает compile-time effect вызова;
- пользовательский DSL остаётся experimental и требует opt-in;
- contract statement должен находиться в начале top-level function;
- `returns()` связывает normal return с condition;
- `returns(true/false)` описывает конкретный boolean result;
- `returnsNotNull()` описывает non-null result;
- `implies` поддерживает ограниченные stable conditions;
- `callsInPlace` описывает lifetime и количество лямбда invocations;
- InvocationKind бывает EXACTLY_ONCE, AT_MOST_ONCE, AT_LEAST_ONCE и UNKNOWN;
- contract не выполняет runtime validation;
- ложный effect делает compiler assumptions небезопасными;
- inline и contract связаны часто, но решают разные задачи;
- author обязан поддерживать contract при каждом изменении body.
