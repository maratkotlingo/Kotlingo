## Что такое `select`

`select` позволяет одновременно ожидать несколько suspending alternatives и выполнить ту ветку, которая стала доступна первой.

Типичные alternatives:

- receive из channel;
- send в channel;
- await result `Deferred`;
- timeout через `onTimeout`;
- закрытие channel через `onReceiveCatching`;
- custom selectable clauses from libraries.

```kotlin
import kotlinx.coroutines.channels.ReceiveChannel
import kotlinx.coroutines.selects.select

suspend fun receiveFromFirstReady(
    first: ReceiveChannel<String>,
    second: ReceiveChannel<String>
): String = select {
    first.onReceive { value ->
        "first:$value"
    }

    second.onReceive { value ->
        "second:$value"
    }
}
```

API `select` в `kotlinx.coroutines` остается experimental, поэтому для public library API его стоит изолировать за собственным abstraction.

<CatTip>
`select` — это кошачий слух: coroutine прислушивается к нескольким мискам сразу и бежит к той, где первой зазвенело.
</CatTip>

## Выбор из нескольких channels

Классический случай: есть два `ReceiveChannel`, нужно взять value из того, который готов первым.

```kotlin
import kotlinx.coroutines.CoroutineScope
import kotlinx.coroutines.channels.ReceiveChannel
import kotlinx.coroutines.channels.produce
import kotlinx.coroutines.delay
import kotlinx.coroutines.runBlocking
import kotlinx.coroutines.selects.select

fun CoroutineScope.fastLessons(): ReceiveChannel<String> = produce {
    while (true) {
        delay(200)
        send("fast:87-channels")
    }
}

fun CoroutineScope.slowLessons(): ReceiveChannel<String> = produce {
    while (true) {
        delay(500)
        send("slow:88-select")
    }
}

suspend fun selectLesson(
    fast: ReceiveChannel<String>,
    slow: ReceiveChannel<String>
): String = select {
    fast.onReceive { value ->
        value
    }

    slow.onReceive { value ->
        value
    }
}

fun channelSelectExample() = runBlocking {
    val fast = fastLessons()
    val slow = slowLessons()

    repeat(5) {
        println(selectLesson(fast, slow))
    }

    fast.cancel()
    slow.cancel()
}
```

`select` returns result of selected clause. Поэтому каждая clause должна возвращать compatible type.

## `onReceiveCatching`

`onReceive` throws, если channel closed. Для robust code чаще нужен `onReceiveCatching`.

```kotlin
import kotlinx.coroutines.channels.ReceiveChannel
import kotlinx.coroutines.selects.select

suspend fun receiveSafely(
    first: ReceiveChannel<String>,
    second: ReceiveChannel<String>
): String = select {
    first.onReceiveCatching { result ->
        result.getOrNull()
            ?.let { value -> "first:$value" }
            ?: "first:closed"
    }

    second.onReceiveCatching { result ->
        result.getOrNull()
            ?.let { value -> "second:$value" }
            ?: "second:closed"
    }
}
```

Если closed channel участвует в select, `onReceiveCatching` может стать selectable immediately. Поэтому closed alternatives нужно явно исключать или корректно обрабатывать.

```kotlin
import kotlinx.coroutines.channels.ReceiveChannel
import kotlinx.coroutines.selects.select

suspend fun receiveNullable(
    first: ReceiveChannel<String>,
    second: ReceiveChannel<String>
): String? = select {
    first.onReceiveCatching { result ->
        result.getOrNull()
    }

    second.onReceiveCatching { result ->
        result.getOrNull()
    }
}
```

<MouseTrap>
Closed channel — тоже готовая alternative. Если забыть про `onReceiveCatching`, можно получить exception или бесконечно выбирать уже закрытую ветку.
</MouseTrap>

## Biased selection

`select` biased: если несколько clauses готовы одновременно, earlier clause может выигрывать чаще.

```kotlin
import kotlinx.coroutines.channels.Channel
import kotlinx.coroutines.selects.select

suspend fun biasedExample(
    first: Channel<String>,
    second: Channel<String>
): String = select {
    first.onReceive { value ->
        "first:$value"
    }

    second.onReceive { value ->
        "second:$value"
    }
}
```

Для fairness:

- меняй порядок clauses между iterations;
- используй explicit priority, если bias нужен;
- объединяй sources через channel/Flow operators;
- измеряй поведение под реальной нагрузкой;
- избегай `select`, если обычный `merge` или `combine` выражает intent понятнее.

## Selecting to send

`onSend` позволяет выбрать channel, который готов принять value.

```kotlin
import kotlinx.coroutines.channels.SendChannel
import kotlinx.coroutines.selects.select

suspend fun sendToAvailableWorker(
    first: SendChannel<IndexJob>,
    second: SendChannel<IndexJob>,
    job: IndexJob
): String = select {
    first.onSend(job) {
        "sent:first"
    }

    second.onSend(job) {
        "sent:second"
    }
}

data class IndexJob(
    val slug: String
)
```

Это полезно для routing, но часто проще иметь один `Channel` и несколько workers reading from it.

```kotlin
import kotlinx.coroutines.channels.Channel
import kotlinx.coroutines.launch
import kotlinx.coroutines.runBlocking

fun simplerWorkerQueueExample() = runBlocking {
    val jobs = Channel<IndexJob>()

    repeat(2) { workerId ->
        launch {
            for (job in jobs) {
                println("Worker $workerId индексирует ${job.slug}")
            }
        }
    }

    jobs.send(IndexJob("88-select"))
    jobs.close()
}
```

<CatTip>
Если все workers одинаковые, один общий channel обычно проще и устойчивее, чем ручной `select` по нескольким outbound channels.
</CatTip>

## Selecting deferred values

`Deferred` можно выбирать через `onAwait`.

```kotlin
import kotlinx.coroutines.Deferred
import kotlinx.coroutines.async
import kotlinx.coroutines.coroutineScope
import kotlinx.coroutines.delay
import kotlinx.coroutines.selects.select

suspend fun fastestMirror(slug: String): String = coroutineScope {
    val primary = async {
        delay(300)
        "primary:$slug"
    }

    val secondary = async {
        delay(150)
        "secondary:$slug"
    }

    val winner = select<String> {
        primary.onAwait { value ->
            value
        }

        secondary.onAwait { value ->
            value
        }
    }

    primary.cancel()
    secondary.cancel()

    winner
}
```

Если после winner остальные computations больше не нужны, их нужно отменить. `select` сам не отменяет alternatives, которые проиграли.

Более generic helper:

```kotlin
import kotlinx.coroutines.Deferred
import kotlinx.coroutines.selects.select

suspend fun <T> awaitFirst(deferreds: List<Deferred<T>>): T {
    require(deferreds.isNotEmpty()) { "deferreds не должен быть пустым" }

    return select {
        deferreds.forEach { deferred ->
            deferred.onAwait { value ->
                value
            }
        }
    }
}
```

## Timeouts через `onTimeout`

`onTimeout` добавляет time-based alternative.

```kotlin
import kotlinx.coroutines.channels.ReceiveChannel
import kotlinx.coroutines.selects.onTimeout
import kotlinx.coroutines.selects.select

suspend fun receiveWithTimeout(
    channel: ReceiveChannel<String>,
    timeoutMillis: Long
): String = select {
    channel.onReceiveCatching { result ->
        result.getOrNull() ?: "closed"
    }

    onTimeout(timeoutMillis) {
        "timeout"
    }
}
```

Для simple timeout around one suspending operation часто лучше `withTimeout` or `withTimeoutOrNull`.

```kotlin
import kotlinx.coroutines.withTimeoutOrNull
import kotlin.time.Duration.Companion.milliseconds

suspend fun receiveWithSimpleTimeout(
    channel: kotlinx.coroutines.channels.ReceiveChannel<String>
): String? {
    return withTimeoutOrNull(300.milliseconds) {
        channel.receive()
    }
}
```

Use `select.onTimeout`, когда timeout is one alternative among several real operations.

## Switch latest deferred from channel

Пример, где `select` действительно полезен: есть channel of new requests and current deferred result. Если пришел новый request, старый result больше не нужен.

```kotlin
import kotlinx.coroutines.CoroutineScope
import kotlinx.coroutines.Deferred
import kotlinx.coroutines.async
import kotlinx.coroutines.channels.ReceiveChannel
import kotlinx.coroutines.channels.produce
import kotlinx.coroutines.isActive
import kotlinx.coroutines.selects.select

fun CoroutineScope.switchLatestLessons(
    input: ReceiveChannel<String>,
    loader: suspend (String) -> LessonCard
) = produce<LessonCard> {
    var currentSlug = input.receiveCatching().getOrNull() ?: return@produce
    var current: Deferred<LessonCard> = async {
        loader(currentSlug)
    }

    while (isActive) {
        val nextSlug = select<String?> {
            input.onReceiveCatching { result ->
                result.getOrNull()
            }

            current.onAwait { card ->
                send(card)
                input.receiveCatching().getOrNull()
            }
        }

        if (nextSlug == null) {
            current.cancel()
            break
        }

        current.cancel()
        currentSlug = nextSlug
        current = async {
            loader(currentSlug)
        }
    }
}

data class LessonCard(
    val slug: String,
    val title: String
)
```

В Flow world похожий intent часто выражается через `flatMapLatest`, и это обычно проще.

```kotlin
import kotlinx.coroutines.ExperimentalCoroutinesApi
import kotlinx.coroutines.flow.Flow
import kotlinx.coroutines.flow.flatMapLatest
import kotlinx.coroutines.flow.flow

@OptIn(ExperimentalCoroutinesApi::class)
fun latestLessonCards(
    slugs: Flow<String>,
    loader: suspend (String) -> LessonCard
): Flow<LessonCard> =
    slugs.flatMapLatest { slug ->
        flow {
            emit(loader(slug))
        }
    }
```

## Practical limitations

`select` powerful, но сложность быстро растет.

Риски:

- experimental API может измениться;
- biased selection can surprise;
- loser alternatives are not automatically cancelled;
- closed channels require explicit handling;
- reading such code harder than ordinary `coroutineScope`;
- many use cases are simpler with Flow operators;
- public API with select internals becomes harder to maintain.

<MouseTrap>
Не используй `select` просто потому, что он выглядит продвинуто. Если задача выражается через `withTimeout`, `first`, `merge`, `flatMapLatest`, worker queue или `coroutineScope`, обычно это будет легче поддерживать.
</MouseTrap>

## Production pattern: first healthy source

Ниже service выбирает первый ответ от primary/secondary sources, отменяет проигравшую работу и не проглатывает cancellation.

```kotlin
import kotlinx.coroutines.CancellationException
import kotlinx.coroutines.async
import kotlinx.coroutines.coroutineScope
import kotlinx.coroutines.selects.select

class LessonMirrorClient(
    private val primary: LessonSource,
    private val secondary: LessonSource
) {
    suspend fun load(slug: String): LessonPayload = coroutineScope {
        val primaryRequest = async {
            primary.load(slug)
        }

        val secondaryRequest = async {
            secondary.load(slug)
        }

        try {
            select<LessonPayload> {
                primaryRequest.onAwait { payload ->
                    payload
                }

                secondaryRequest.onAwait { payload ->
                    payload
                }
            }
        } catch (exception: CancellationException) {
            throw exception
        } finally {
            primaryRequest.cancel()
            secondaryRequest.cancel()
        }
    }
}

data class LessonPayload(
    val slug: String,
    val body: String
)

class LessonSource(
    private val name: String
) {
    suspend fun load(slug: String): LessonPayload {
        return LessonPayload(
            slug = slug,
            body = "Payload from $name for $slug"
        )
    }
}
```

Этот pattern подходит, когда:

- sources взаимозаменяемы;
- нужен первый successful response;
- slower request можно cancel;
- caller owns cancellation через surrounding scope;
- exceptions still propagate to caller.
