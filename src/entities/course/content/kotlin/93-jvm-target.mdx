## Зачем нужен JVM target

JVM target компилирует Kotlin code в JVM bytecode. В Kotlin Multiplatform он нужен для:

- серверные сервисы;
- desktop applications;
- CLI tools;
- Плагины Gradle и инструменты сборки;
- server-side общие модули;
- JVM-тесты на общую логику;
- интеграция с Java-библиотеками;
- миграция с Java на Kotlin;
- общий доменный код reused by Android and бэкенд, если границы подходят.

```kotlin
plugins {
    kotlin("multiplatform") version "2.4.0"
}

kotlin {
    jvm()
}
```

После объявления target появляются source sets вроде `jvmMain` and `jvmTest`.

```text
src/
  commonMain/
    kotlin/
  commonTest/
    kotlin/
  jvmMain/
    kotlin/
  jvmTest/
    kotlin/
```

<CatTip>
JVM target — это теплая серверная лежанка Kotlin: common code приносит доменную логику, а `jvmMain` подключает Java, JUnit, JDBC, SLF4J и боевой runtime.
</CatTip>

## `jvmMain`

`jvmMain` содержит JVM-specific code. Здесь можно использовать:

- Java standard library;
- JVM-only dependencies;
- reflection APIs;
- потоки и исполнители;
- JDBC;
- filesystem через `java.nio.file`;
- SLF4J/Logback;
- Spring, Ktor, Micronaut, Quarkus;
— Помощники по тестированию, специфичные для JUnit, в `jvmTest`.

Common code:

```kotlin
// commonMain/kotlin/lesson/LessonSource.kt
package lesson

interface LessonSource {
    suspend fun read(slug: String): String
}
```

JVM implementation:

```kotlin
// jvmMain/kotlin/lesson/JvmFileLessonSource.kt
package lesson

import kotlinx.coroutines.Dispatchers
import kotlinx.coroutines.withContext
import java.nio.file.Files
import java.nio.file.Path

class JvmFileLessonSource(
    private val root: Path
) : LessonSource {
    override suspend fun read(slug: String): String = withContext(Dispatchers.IO) {
        // java.nio.file блокирует текущий thread, поэтому используем Dispatchers.IO.
        Files.readString(root.resolve("$slug.md"))
    }
}
```

`commonMain` не должен импортировать `java.nio.file.Path`; `jvmMain` может.

<MouseTrap>
Если common model начинает принимать `java.io.File`, ты больше не проектируешь multiplatform API. Передавай common abstraction: `String`, `ByteArray`, `PathLike`, `LessonSource` или собственный value class.
</MouseTrap>

## JVM toolchain

Toolchain задает JDK, которым Gradle компилирует JVM code.

```kotlin
kotlin {
    jvmToolchain(21)
}
```

В KMP target-specific form:

```kotlin
import org.jetbrains.kotlin.gradle.dsl.JvmTarget

kotlin {
    jvm {
        compilerOptions {
            jvmTarget.set(JvmTarget.JVM_21)
        }
    }
}
```

Различай:

- JDK использовался для запуска Gradle;
- Набор инструментов Java, используемый для компиляции кода;
- `jvmTarget`, версия байт-кода, созданная компилятором Kotlin;
- среда выполнения JRE/JDK, в которой фактически запускается приложение.

Боевой rule: compile target and runtime target должны быть согласованы. Если bytecode generated for JVM 21, его нельзя запускать на Java 17.

## Java interop

Цель JVM может естественным образом вызывать Java.

```java
// jvmMain/java/dev/kotlingo/legacy/LegacySlugNormalizer.java
package dev.kotlingo.legacy;

public final class LegacySlugNormalizer {
    public String normalize(String value) {
        return value.trim().toLowerCase().replace(' ', '-');
    }
}
```

Kotlin usage:

```kotlin
// jvmMain/kotlin/lesson/JvmSlugService.kt
package lesson

import dev.kotlingo.legacy.LegacySlugNormalizer

class JvmSlugService(
    private val normalizer: LegacySlugNormalizer = LegacySlugNormalizer()
) {
    fun normalize(value: String): String {
        // Java method returns platform type, поэтому boundary лучше нормализовать явно.
        return normalizer.normalize(value)
            .ifBlank { error("Slug не должен быть пустым") }
    }
}
```

API Kotlin, предназначенные для пользователей Java, должны учитывать:

- `@JvmName`;
- `@JvmOverloads`;
- `@JvmStatic`;
- `@JvmField`;
- `@Throws`;
- аннотации обнуляемости, созданные Kotlin;
- избегать чрезмерно специфичных для Kotlin типов функций в общедоступном API, ориентированном на Java;
- двоичная совместимость опубликованных библиотек JVM.

```kotlin
package api

class LessonFacade(
    private val service: LessonService
) {
    @JvmOverloads
    fun title(slug: String, fallback: String = "Без названия"): String {
        return service.findTitle(slug) ?: fallback
    }
}

class LessonService {
    fun findTitle(slug: String): String? {
        return if (slug.isBlank()) null else "Урок $slug"
    }
}
```

## Platform types

Обнуляемость Java не всегда известна Котлину. Значения Java могут отображаться как типы платформы.

```java
package dev.kotlingo.legacy;

public final class UserRecord {
    public String displayName() {
        return null;
    }
}
```

Kotlin boundary:

```kotlin
package user

import dev.kotlingo.legacy.UserRecord

class UserMapper {
    fun displayName(record: UserRecord): String {
        // Java method could return null, even if Kotlin type looks flexible.
        return record.displayName() ?: "Гость"
    }
}
```

Не позволяйте типам платформ просачиваться глубоко в код предметной области. Нормализуйте их вблизи границы Java.

## JVM dependencies

Объявите зависимости только для JVM в `jvmMain`.

```kotlin
kotlin {
    sourceSets {
        commonMain.dependencies {
            implementation("org.jetbrains.kotlinx:kotlinx-coroutines-core:1.11.0")
        }

        jvmMain.dependencies {
            implementation("org.slf4j:slf4j-api:2.1.0")
            implementation("ch.qos.logback:logback-classic:1.6.0")
        }
    }
}
```

Если зависимость предназначена только для JVM, никогда не помещайте ее в `commonMain`.

```kotlin
kotlin {
    sourceSets {
        jvmMain.dependencies {
            implementation("com.zaxxer:HikariCP:8.0.0")
            implementation("org.postgresql:postgresql:43.0.0")
        }
    }
}
```

Общий код должен зависеть от интерфейсов:

```kotlin
// commonMain/kotlin/progress/ProgressStore.kt
package progress

interface ProgressStore {
    suspend fun completedLessons(userId: String): Int

    suspend fun markCompleted(userId: String, slug: String)
}
```

Реализация JVM может использовать JDBC:

```kotlin
// jvmMain/kotlin/progress/JdbcProgressStore.kt
package progress

import kotlinx.coroutines.Dispatchers
import kotlinx.coroutines.withContext
import javax.sql.DataSource

class JdbcProgressStore(
    private val dataSource: DataSource
) : ProgressStore {
    override suspend fun completedLessons(userId: String): Int = withContext(Dispatchers.IO) {
        dataSource.connection.use { connection ->
            connection.prepareStatement(
                "select count(*) from completed_lessons where user_id = ?"
            ).use { statement ->
                statement.setString(1, userId)

                statement.executeQuery().use { resultSet ->
                    if (resultSet.next()) resultSet.getInt(1) else 0
                }
            }
        }
    }

    override suspend fun markCompleted(userId: String, slug: String): Unit = withContext(Dispatchers.IO) {
        dataSource.connection.use { connection ->
            connection.prepareStatement(
                "insert into completed_lessons(user_id, slug) values (?, ?)"
            ).use { statement ->
                statement.setString(1, userId)
                statement.setString(2, slug)
                statement.executeUpdate()
            }
        }
    }
}
```

## JUnit

`jvmTest` может использовать JUnit 5.

```kotlin
kotlin {
    jvm {
        testRuns["test"].executionTask.configure {
            useJUnitPlatform()
        }
    }

    sourceSets {
        jvmTest.dependencies {
            implementation(kotlin("test"))
            implementation("org.junit.jupiter:junit-jupiter:6.0.0")
        }
    }
}
```

Test:

```kotlin
package lesson

import org.junit.jupiter.api.Test
import kotlin.test.assertEquals

class JvmSlugServiceTest {
    @Test
    fun normalizesTitleToSlug() {
        val service = JvmSlugService()

        val slug = service.normalize(" JVM Target ")

        assertEquals("jvm-target", slug)
    }
}
```

Используйте `commonTest` для поведения, независимого от платформы, и `jvmTest` для поведения, специфичного для JVM.

## Server-side Kotlin

JVM target is the natural target for серверные приложения.

Common use-case:

- `commonMain` contains domain models and сценарии использования;
- `jvmMain` contains HTTP server, база данных adapter, logging, configuration;
- `jvmTest` содержит тесты интеграции с инструментами JVM.

```kotlin
// commonMain/kotlin/lesson/OpenLessonUseCase.kt
package lesson

class OpenLessonUseCase(
    private val source: LessonSource
) {
    suspend fun open(slug: String): LessonPage {
        val body = source.read(slug)

        return LessonPage(
            slug = slug,
            title = body.lineSequence().firstOrNull().orEmpty(),
            body = body
        )
    }
}

data class LessonPage(
    val slug: String,
    val title: String,
    val body: String
)
```

JVM server adapter:

```kotlin
// jvmMain/kotlin/server/LessonController.kt
package server

import lesson.OpenLessonUseCase

class LessonController(
    private val useCase: OpenLessonUseCase
) {
    suspend fun handle(slug: String): HttpResponse {
        val page = useCase.open(slug)

        return HttpResponse(
            status = 200,
            body = "${page.title}\n${page.body}"
        )
    }
}

data class HttpResponse(
    val status: Int,
    val body: String
)
```

Это исключает использование HTTP-фреймворка в общем коде.

## Блокировка ввода-вывода и диспетчеров

JVM имеет множество API-интерфейсов блокировки. Функция приостановки не делает блокирующий вызов неблокируемым.

```kotlin
import kotlinx.coroutines.Dispatchers
import kotlinx.coroutines.withContext
import java.nio.file.Files
import java.nio.file.Path

suspend fun readLesson(path: Path): String = withContext(Dispatchers.IO) {
    // Blocking filesystem call isolated on IO dispatcher.
    Files.readString(path)
}
```

Для работы JVM с привязкой к процессору:

```kotlin
import kotlinx.coroutines.Dispatchers
import kotlinx.coroutines.withContext

suspend fun tokenizeLesson(text: String): List<String> = withContext(Dispatchers.Default) {
    // CPU-bound parsing belongs to Default dispatcher.
    text.splitToSequence(Regex("\\W+"))
        .filter { token -> token.isNotBlank() }
        .map { token -> token.lowercase() }
        .toList()
}
```

## Проверочный список для боевого кода

Для проверки цели JVM:

- `jvmToolchain` и `jvmTarget` являются явными;
- Зависимости только для JVM находятся в `jvmMain`;
- Типы платформ Java нормализованы по границам;
- для блокировки вызовов используйте `Dispatchers.IO` или выделенный диспетчер;
- Конфигурация JUnit находится в цели `jvm`;
- общедоступный API JVM совместим с Java, если используется из Java;
— общий код не импортирует пакеты JVM;
- server adapters depend on сценарии использования, not the other way around.

<CatTip>
Хороший JVM target не смешивает котлеты с мисками: common code хранит смысл приложения, `jvmMain` отвечает за Java runtime, сервер, файлы, базы и tooling.
</CatTip>
