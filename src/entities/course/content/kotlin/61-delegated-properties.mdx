## Делегированное свойство передает accessor-логику другому объекту

Delegated property выглядит так:

```kotlin
val title: String by lazy {
    // Значение будет вычислено при первом чтении.
    "Kotlin"
}
```

Синтаксис общий:

```kotlin
val propertyName: Type by delegateExpression
var mutablePropertyName: Type by delegateExpression
```

Expression после `by` называется delegate. Compiler сгенерирует getter, который вызовет `getValue`, а для `var` еще setter, который вызовет `setValue`.

<CatTip>
Делегат — это кошачий помощник у дверцы свойства: когда кто-то читает или меняет property, помощник решает, что именно произойдет.
</CatTip>

## `getValue` и `setValue`

Минимальный только для чтения delegate должен предоставить operator function `getValue`.

```kotlin
import kotlin.reflect.KProperty

class ConstantDelegate(
    private val value: String
) {
    operator fun getValue(thisRef: Any?, property: KProperty<*>): String {
        // property.name позволяет узнать имя свойства, которое использует делегат.
        println("Читаем ${property.name}")
        return value
    }
}

class Course {
    val title: String by ConstantDelegate("kotlingo")
}

val course = Course()
println(course.title)
```

Для mutable property нужен `setValue`.

```kotlin
import kotlin.reflect.KProperty

class LoggedVar<T>(
    initialValue: T
) {
    private var value: T = initialValue

    operator fun getValue(thisRef: Any?, property: KProperty<*>): T {
        return value
    }

    operator fun setValue(thisRef: Any?, property: KProperty<*>, newValue: T) {
        // Логируем изменение перед записью нового значения.
        println("${property.name}: $value -> $newValue")
        value = newValue
    }
}

class Settings {
    var theme: String by LoggedVar("dark")
}
```

`thisRef` — owner object или receiver extension-property. Для top-level/local properties он может быть `null`.

## `lazy`

`lazy { ... }` возвращает delegate для только для чтения property. Lambda выполняется при первом чтении, результат запоминается.

```kotlin
class CourseRepository {
    val expensiveIndex: Map<String, Int> by lazy {
        println("Строим индекс")

        mapOf(
            "01-introduction" to 0,
            "02-types" to 1
        )
    }
}

val repository = CourseRepository()

println(repository.expensiveIndex["01-introduction"])
println(repository.expensiveIndex["02-types"])
```

Сообщение «Строим индекс» появится один раз.

`lazy` thread-safety modes:

- default mode synchronized: один thread вычисляет value, остальные увидят тот же result;
- `LazyThreadSafetyMode.PUBLICATION`: несколько threads могут выполнить initializer, но опубликовано будет одно value;
- `LazyThreadSafetyMode.NONE`: нет thread-safety guarantees и меньше overhead.

```kotlin
val localCache: MutableMap<String, String> by lazy(LazyThreadSafetyMode.NONE) {
    // Подходит только если property используется строго из одного thread.
    mutableMapOf()
}
```

<MouseTrap>
`lazy` подходит для `val`, а не для `var`. Если значение должно меняться, используй другой механизм.
</MouseTrap>

## Local delegated properties

Delegated property можно объявить внутри function.

```kotlin
fun renderLesson(loadMarkdown: () -> String, shouldRender: Boolean): String {
    val markdown by lazy {
        // Markdown загрузится только если реально понадобится.
        loadMarkdown()
    }

    if (!shouldRender) {
        return "Отрисовка отключена"
    }

    return markdown.trim()
}
```

Это удобно для expensive computation, которая нужна только в некоторых branches.

## `Delegates.observable`

`Delegates.observable(initialValue) { property, oldValue, newValue -> ... }` вызывает handler после изменения value.

```kotlin
import kotlin.properties.Delegates

class ProgressTracker {
    var percent: Int by Delegates.observable(0) { property, oldValue, newValue ->
        // Handler вызывается после того, как property уже получила новое значение.
        println("${property.name}: $oldValue -> $newValue")
    }
}

val tracker = ProgressTracker()
tracker.percent = 10
tracker.percent = 50
```

Handler получает:

- `property` — metadata свойства;
- `oldValue` — значение до assignment;
- `newValue` — значение после assignment.

Observable хорошо подходит для logs, metrics, UI invalidation, простых обратные вызовы.

## `Delegates.vetoable`

`Delegates.vetoable(initialValue) { property, oldValue, newValue -> Boolean }` вызывается до изменения. Если handler возвращает `true`, assignment принимается. Если `false`, old value сохраняется.

```kotlin
import kotlin.properties.Delegates

class SafeProgress {
    var percent: Int by Delegates.vetoable(0) { _, _, newValue ->
        // Разрешаем только проценты в корректном диапазоне.
        newValue in 0..100
    }
}

val progress = SafeProgress()

progress.percent = 50
println(progress.percent) // 50

progress.percent = 150
println(progress.percent) // 50
```

`vetoable` полезен, когда assignment должен быть отклонен без exception. Если нарушение invariant является ошибкой вызывающий код-а, часто лучше пользовательский setter с `require`.

## `Delegates.notNull`

`Delegates.notNull` создает delegate для non-null `var`, которое будет инициализировано позже.

```kotlin
import kotlin.properties.Delegates

class Component {
    var port: Int by Delegates.notNull()

    fun start() {
        // Если port не был установлен, чтение бросит IllegalStateException.
        println("Запуск на порту $port")
    }
}

val component = Component()
component.port = 8080
component.start()
```

Отличия от `lateinit`:

- `Delegates.notNull` работает с primitive-like types, например `Int`;
- `lateinit` доступен только для `var` reference types;
- `lateinit` имеет проверку `::property.isInitialized` в доступных контекстах;
- `Delegates.notNull` является delegate object и имеет соответствующий overhead.

Используй оба подхода осторожно: lifecycle должен гарантировать assignment до чтения.

<MouseTrap>
`Delegates.notNull` не делает значение magically initialized. Он только откладывает ошибку до чтения, если ты забыл присвоить value.
</MouseTrap>

## Map-backed properties

Properties можно делегировать map. Key берется из имени property.

```kotlin
class UserView(
    private val data: Map<String, Any?>
) {
    val name: String by data
    val age: Int by data
}

val user = UserView(
    mapOf(
        "name" to "Мила",
        "age" to 28
    )
)

println(user.name) // Мила
println(user.age)  // 28
```

Для `var` нужен `MutableMap`.

```kotlin
class MutableUserView(
    private val data: MutableMap<String, Any?>
) {
    var name: String by data
    var age: Int by data
}

val mutableUser = MutableUserView(mutableMapOf("name" to "Кот", "age" to 3))
mutableUser.age = 4
```

Map-backed properties удобны для dynamic data, JSON-like structures и prototypes, но они менее type-safe, чем normal data class.

```kotlin
data class UserDto(
    val name: String,
    val age: Int
)
```

Если key отсутствует или value имеет неверный type, ошибка появится во время runtime access.

## Делегирование другому ресурсу

Property может делегировать access другой property через callable reference.

```kotlin
class RenamedSettings {
    var newThemeName: String = "cat-dark"

    @Deprecated(
        message = "Используй newThemeName",
        replaceWith = ReplaceWith("newThemeName")
    )
    var oldThemeName: String by this::newThemeName
}

val settings = RenamedSettings()
settings.oldThemeName = "amber-night"

println(settings.newThemeName) // amber-night
```

Это полезно для backward-compatible rename: старое property остается рабочим, но направляет calls в новое.

## `ReadOnlyProperty` и `ReadWriteProperty`

Стандартная библиотека дает interfaces для delegates.

```kotlin
import kotlin.properties.ReadOnlyProperty
import kotlin.reflect.KProperty

class RequiredConfig(
    private val values: Map<String, String>
) : ReadOnlyProperty<Any?, String> {
    override fun getValue(thisRef: Any?, property: KProperty<*>): String {
        return values[property.name]
            ?: error("Не найден config key: ${property.name}")
    }
}

class AppConfig(values: Map<String, String>) {
    private val config = RequiredConfig(values)

    val apiBaseUrl: String by config
    val analyticsKey: String by config
}
```

Для mutable delegate используй `ReadWriteProperty`.

```kotlin
import kotlin.properties.ReadWriteProperty
import kotlin.reflect.KProperty

class ClampedInt(
    initialValue: Int,
    private val range: IntRange
) : ReadWriteProperty<Any?, Int> {
    private var value: Int = initialValue.coerceIn(range)

    override fun getValue(thisRef: Any?, property: KProperty<*>): Int {
        return value
    }

    override fun setValue(thisRef: Any?, property: KProperty<*>, value: Int) {
        // Значение всегда приводится к допустимому диапазону.
        this.value = value.coerceIn(range)
    }
}

class VolumeState {
    var volume: Int by ClampedInt(initialValue = 50, range = 0..100)
}
```

## `provideDelegate`

`provideDelegate` позволяет перехватить binding property к delegate во время создания property.

```kotlin
import kotlin.properties.ReadOnlyProperty
import kotlin.reflect.KProperty

class ConfigSource(
    private val values: Map<String, String>
) {
    operator fun provideDelegate(
        thisRef: Any?,
        property: KProperty<*>
    ): ReadOnlyProperty<Any?, String> {
        require(property.name in values) {
            "Config key '${property.name}' должен существовать"
        }

        return ReadOnlyProperty { _, _ ->
            values.getValue(property.name)
        }
    }
}

class StrictConfig(values: Map<String, String>) {
    private val source = ConfigSource(values)

    val apiBaseUrl: String by source
}
```

Без `provideDelegate` ошибка отсутствующего key могла бы появиться только при чтении property. С `provideDelegate` validation происходит при binding.

## Боевой-пример: delegate для feature flags

```kotlin
import kotlin.properties.ReadOnlyProperty
import kotlin.reflect.KProperty

interface FeatureFlagStore {
    fun isEnabled(name: String): Boolean
}

class FeatureFlagDelegate(
    private val store: FeatureFlagStore,
    private val defaultValue: Boolean
) : ReadOnlyProperty<Any?, Boolean> {
    override fun getValue(thisRef: Any?, property: KProperty<*>): Boolean {
        return try {
            store.isEnabled(property.name)
        } catch (exception: RuntimeException) {
            // Если хранилище временно недоступно, используем безопасный default.
            defaultValue
        }
    }
}

class Features(store: FeatureFlagStore) {
    private fun flag(defaultValue: Boolean): FeatureFlagDelegate {
        return FeatureFlagDelegate(store, defaultValue)
    }

    val newCourseNavigation: Boolean by flag(defaultValue = false)
    val amberCatTheme: Boolean by flag(defaultValue = true)
}
```

Delegate убирает повторяющийся код получения flag по имени property и централизует fallback.

## Когда использовать delegated properties

Используй delegated properties, когда:

- одинаковая accessor-логика повторяется во многих properties;
- value expensive и нужно lazy initialization;
- нужен observable/vetoable assignment;
- property читает данные из map/config/store;
- нужно backward-compatible переименование property;
- пользовательский delegate делает domain intent очевиднее.

Не используй delegate, если обычный property с getter/setter проще.

```kotlin
class SimpleProgress {
    var percent: Int = 0
        set(value) {
            field = value.coerceIn(0, 100)
        }
}
```

<CatTip>
Делегат должен делать property понятнее, а не превращать чтение поля в мини-квест. Если поведение неожиданное, назови delegate явно и задокументируй invariant.
</CatTip>
