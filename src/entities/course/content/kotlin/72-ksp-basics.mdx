## Что такое KSP

KSP, Kotlin Symbol Processing, — framework для анализа Kotlin source symbols и генерации кода или ресурсов во время сборки.

```kotlin
@GenerateLessonAdapter
data class LessonDto(
    val slug: String,
    val title: String
)
```

KSP processor может найти `@GenerateLessonAdapter` и сгенерировать adapter class, registry или mapping code.

KSP не изменяет существующий код. Он читает symbols и добавляет generated output, который затем компилируется вместе с project sources.

<CatTip>
KSP — это кошачий архивариус сборки: он читает карточки с аннотациями, составляет каталог и кладет рядом новые файлы, но не переписывает исходники лапой.
</CatTip>

## KSP в build.gradle.kts

В consumer module подключают KSP plugin и processor dependency.

```kotlin
plugins {
    kotlin("jvm") version "2.4.0"
    id("com.google.devtools.ksp") version "2.3.9"
}

dependencies {
    // Processor запускается во время сборки и генерирует код.
    ksp(project(":lesson-processor"))

    // Annotations нужны исходному коду.
    implementation(project(":lesson-annotations"))
}
```

Версия KSP связана с версией Kotlin. Перед обновлением Kotlin нужно проверить совместимую версию KSP.

## Основные роли

В типичном KSP-решении есть несколько modules:

- annotations module — легкие annotations, которые использует app/library code;
- модуль процессора — реализация KSP-процессора;
- consumer module — проект, где есть annotated source code.

Annotation:

```kotlin
@Target(AnnotationTarget.CLASS)
@Retention(AnnotationRetention.SOURCE)
annotation class GenerateLessonAdapter
```

Consumer:

```kotlin
@GenerateLessonAdapter
data class LessonDto(
    val slug: String,
    val title: String,
    val minutes: Int
)
```

Generated result может выглядеть так:

```kotlin
class LessonDtoAdapter {
    fun toMap(value: LessonDto): Map<String, Any?> {
        return mapOf(
            "slug" to value.slug,
            "title" to value.title,
            "minutes" to value.minutes
        )
    }
}
```

## Symbol processing

KSP работает на уровне symbols, а не строк source code.

Processor видит:

- files: `KSFile`;
- declarations: `KSDeclaration`;
- classes: `KSClassDeclaration`;
- functions: `KSFunctionDeclaration`;
- properties: `KSPropertyDeclaration`;
- parameters: `KSValueParameter`;
- annotations: `KSAnnotation`;
- types: `KSTypeReference`, `KSType`.

Он не анализирует bodies expressions как полноценный плагин компилятора. KSP не предназначен для поиска `if`, `for` или конкретных statements внутри function body.

<MouseTrap>
KSP не может «починить» существующую функцию и заменить ее тело. Если тебе нужно изменять semantics existing code, это уже территория плагин компилятора, а не KSP.
</MouseTrap>

## `SymbolProcessor`

Processor обычно реализует `SymbolProcessor`.

```kotlin
import com.google.devtools.ksp.processing.CodeGenerator
import com.google.devtools.ksp.processing.Resolver
import com.google.devtools.ksp.processing.SymbolProcessor
import com.google.devtools.ksp.processing.SymbolProcessorEnvironment
import com.google.devtools.ksp.processing.SymbolProcessorProvider
import com.google.devtools.ksp.symbol.KSAnnotated

class LessonProcessor(
    private val environment: SymbolProcessorEnvironment
) : SymbolProcessor {
    override fun process(resolver: Resolver): List<KSAnnotated> {
        val symbols = resolver.getSymbolsWithAnnotation(
            "com.example.GenerateLessonAdapter"
        )

        // В реальном processor здесь будет validation и code generation.
        for (symbol in symbols) {
            environment.logger.info("Найден symbol: $symbol")
        }

        return emptyList()
    }
}

class LessonProcessorProvider : SymbolProcessorProvider {
    override fun create(environment: SymbolProcessorEnvironment): SymbolProcessor {
        return LessonProcessor(environment)
    }
}
```

`process` может возвращать deferred symbols, если processor пока не может обработать symbol из-за неполной информации.

## Регистрация processor provider

Processor provider регистрируется как service.

```text
META-INF/services/com.google.devtools.ksp.processing.SymbolProcessorProvider
```

Содержимое файла:

```text
com.example.LessonProcessorProvider
```

Это стандартный Java service loader mechanism.

## `Resolver`

`Resolver` — основной вход в symbols текущего processing round.

```kotlin
val symbols = resolver.getSymbolsWithAnnotation(
    "com.example.GenerateLessonAdapter"
)
```

Частые операции:

- `getSymbolsWithAnnotation` — найти symbols с annotation;
- `getAllFiles` — получить source files;
- `getClassDeclarationByName` — найти class declaration по qualified name;
- `getDeclarationsFromPackage` — получить declarations package.

```kotlin
val stringDeclaration = resolver.getClassDeclarationByName(
    resolver.getKSNameFromString("kotlin.String")
)
```

## Validation symbols

Перед генерацией проверяй, что symbol имеет ожидаемый type.

```kotlin
import com.google.devtools.ksp.symbol.KSClassDeclaration
import com.google.devtools.ksp.symbol.ClassKind

fun validateAnnotatedClass(symbol: KSAnnotated): KSClassDeclaration? {
    val declaration = symbol as? KSClassDeclaration ?: return null

    if (declaration.classKind != ClassKind.CLASS) {
        return null
    }

    return declaration
}
```

Для боевой processor надо выдавать diagnostics.

```kotlin
environment.logger.error(
    message = "@GenerateLessonAdapter можно ставить только на class",
    symbol = symbol
)
```

## Code generation

Generated file создается через `CodeGenerator`.

```kotlin
import com.google.devtools.ksp.processing.Dependencies

fun generateAdapter(
    codeGenerator: CodeGenerator,
    source: KSFile,
    packageName: String,
    className: String
) {
    val file = codeGenerator.createNewFile(
        dependencies = Dependencies(
            aggregating = false,
            sources = arrayOf(source)
        ),
        packageName = packageName,
        fileName = "${className}Adapter"
    )

    file.writer().use { writer ->
        writer.appendLine("package $packageName")
        writer.appendLine()
        writer.appendLine("class ${className}Adapter")
    }
}
```

`Dependencies` важны для incremental processing. Они связывают generated output с source files, от которых он зависит.

## Isolating и aggregating outputs

Isolating output зависит от конкретного source.

```kotlin
Dependencies(
    aggregating = false,
    sources = arrayOf(sourceFile)
)
```

Пример: `LessonDtoAdapter` зависит только от `LessonDto`.

Aggregating output собирает информацию из многих sources.

```kotlin
Dependencies(
    aggregating = true,
    sources = allAnnotatedFiles.toTypedArray()
)
```

Пример: `GeneratedRouteRegistry`, который содержит все classes с `@Route`.

KSP применяет categorization к outputs, а не просто ко всему processor. Это помогает incremental builds.

## Incremental processing

KSP поддерживает incremental processing по умолчанию. Он старается reprocess только affected files.

Чтобы это работало хорошо:

- указывай correct `Dependencies`;
- не делай global scan без необходимости;
- разделяй isolating outputs и aggregating outputs;
- не читай files напрямую из filesystem без регистрации dependencies;
- логируй и валидируй deferred symbols.

Troubleshooting:

```properties
ksp.incremental=false
```

Этот флаг можно временно поставить в `gradle.properties`, чтобы проверить, связана ли ошибка с incremental processing.

## Generated sources location

Generated Kotlin sources обычно попадают в build directory KSP.

```text
build/generated/ksp/main/kotlin
```

Для tests:

```text
build/generated/ksp/test/kotlin
```

Точный path зависит от platform/source set.

## KSP против kapt

kapt — bridge к Java annotation processing API. Он особенно нужен для Java-first processors.

KSP — Kotlin-first. Он понимает Kotlin symbols directly:

- extension functions;
- declaration-site variance;
- nullability;
- Kotlin properties;
- local declarations в пределах KSP model;
- Kotlin type system ближе к исходному коду.

KSP обычно быстрее, потому что не требует генерировать Java stubs так, как kapt.

Но kapt все еще нужен, если library поддерживает только Java annotation processor.

```kotlin
dependencies {
    // Kotlin-first processor.
    ksp("com.example:lesson-processor:1.0")

    // Java annotation processor, если KSP-варианта нет.
    // kapt("com.example:legacy-processor:1.0")
}
```

## Ограничения KSP

KSP не может:

- менять existing source code;
- менять language semantics;
- анализировать arbitrary expressions/statements как плагин компилятора;
- гарантировать runtime behavior без generated code;
- заменить reflection во всех случаях.

KSP может:

- читать declarations и annotations;
- читать types и relationships;
- генерировать code/resources;
- выдавать diagnostics;
- улучшать compile-time safety для repetitive patterns.

## Боевой-пример: route registry generator

Annotation:

```kotlin
@Target(AnnotationTarget.CLASS)
@Retention(AnnotationRetention.SOURCE)
annotation class Route(
    val path: String
)
```

Annotated code:

```kotlin
@Route("/lessons")
class LessonController

@Route("/admin/lessons")
class AdminLessonController
```

Generated registry:

```kotlin
object GeneratedRoutes {
    val paths: Map<String, String> = mapOf(
        "/lessons" to "com.example.LessonController",
        "/admin/lessons" to "com.example.AdminLessonController"
    )
}
```

Такой registry можно создать at compile time и не сканировать classpath reflection-ом на startup.

## Практические правила

Используй KSP, если:

- нужно генерировать boilerplate по annotations;
- processor должен понимать Kotlin types;
- runtime reflection слишком дорогая или недоступна;
- generated code должен быть виден compiler-у.

Не используй KSP, если:

- достаточно обычной function;
- код можно написать явно и он не повторяется;
- нужна runtime discovery без генерации;
- нужно менять semantics existing declarations.

<CatTip>
KSP хорош там, где повторяемый код можно вывести из структуры declarations. Если структура ясная, processor будет мурлыкать и экономить ручной труд.
</CatTip>
