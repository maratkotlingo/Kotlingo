## Что такое Kotlin/Native

Kotlin/Native компилирует Kotlin code в native binaries for platforms where JVM is not the runtime.

Сценарии использования:

- iOS общие модули;
- macOS command-line tools;
- Собственные службы или инструменты Linux;
- Windows native tools;
- библиотеки, экспортированные потребителям C/C++;
- Фреймворки Apple, используемые из Swift/Objective-C;
- code sharing between mobile platforms through KMP.

```kotlin
kotlin {
    iosArm64()
    iosSimulatorArm64()
    macosArm64()
    linuxX64()
}
```

Native target produces `klib` по умолчанию. To produce executable, static/общий library or Apple framework, configure `binaries`.

<CatTip>
Kotlin/Native — это кот без JVM-переноски: Kotlin code идет прямо в native runtime и должен уважать правила C, Swift, Objective-C, linker-а и платформенной памяти.
</CatTip>

## Native source sets

Typical iOS layout:

```text
src/
  commonMain/
    kotlin/
  iosMain/
    kotlin/
  iosArm64Main/
    kotlin/
  iosSimulatorArm64Main/
    kotlin/
```

Общий iOS code goes to `iosMain`.

```kotlin
// iosMain/kotlin/platform/IosDeviceInfo.kt
package platform

import platform.UIKit.UIDevice

class IosDeviceInfo {
    fun modelName(): String {
        return UIDevice.currentDevice.model
    }
}
```

Вставляйте различия, специфичные для цели, в `iosArm64Main` или `iosSimulatorArm64Main` только тогда, когда это действительно необходимо.

## Memory manager

Современный Kotlin/Native использует автоматическое управление памятью с отслеживанием сборщика мусора. Он интегрирован с Swift/Objective-C ARC на границах взаимодействия.

Practical implications:

- ordinary Kotlin object graphs no longer require old freezing discipline;
- разделяемое изменяемое состояние still needs concurrency design;
- циклы сохранения с объектами Objective-C/Swift все еще имеют значение;
- собственные ресурсы по-прежнему требуют явного закрытия или ограниченного владения;
- Необходимо измерение производительности на целевых устройствах.

```kotlin
// commonMain/kotlin/state/LessonCache.kt
package state

class LessonCache {
    private val lessons = mutableMapOf<String, String>()

    fun put(slug: String, title: String) {
        lessons[slug] = title
    }

    fun get(slug: String): String? {
        return lessons[slug]
    }
}
```

Это обычное изменяемое состояние Kotlin. Он по-прежнему требует синхронизации, если к нему обращаются из нескольких сопрограмм/потоков.

<MouseTrap>
«Старого freezing ада больше нет» не означает «гонки данных исчезли». Общий изменяемое состояние still needs ownership, confinement, mutex or immutable snapshots.
</MouseTrap>

## Замораживание истории и современное поведение

Older Kotlin/Native memory model required freezing objects to share them across threads. Frozen objects could not be mutated, and many designs had to avoid common JVM-style изменяемое состояние.

Современный менеджер памяти Kotlin/Native удалил эту ментальную модель из обычного кода приложения.

What still matters:

- do not rely on global изменяемое состояние without synchronization;
- использовать структурированный параллелизм;
- isolate platform обратные вызовы;
- помните о принадлежности Swift/Objective-C;
- close native resources deterministically.

```kotlin
import kotlinx.coroutines.sync.Mutex
import kotlinx.coroutines.sync.withLock

class NativeSafeCounter {
    private val mutex = Mutex()
    private var value = 0

    suspend fun increment(): Int = mutex.withLock {
        value += 1
        value
    }
}
```

Используйте общие инструменты параллелизма; не восстанавливайте старые обходные пути зависания, если вы не поддерживаете устаревший код.

## C interop

Kotlin/Native может вызывать библиотеки C через cinterop.

General рабочий процесс:

- create `.def` file;
- указать Gradle определение cinterop;
- generate Kotlin bindings;
- вызывать сгенерированные API из собственного исходного набора;
- wrap generated low-level API behind safe Kotlin API.

Gradle:

```kotlin
kotlin {
    linuxX64 {
        compilations.getByName("main") {
            cinterops {
                val libcurl by creating {
                    definitionFile.set(project.file("src/nativeInterop/cinterop/libcurl.def"))
                    packageName("curl")
                }
            }
        }
    }
}
```

Definition file:

```properties
headers = curl/curl.h
headerFilter = curl/*
compilerOpts = -I/usr/include
linkerOpts = -lcurl
```

Wrapper:

```kotlin
// linuxX64Main/kotlin/http/CurlVersion.kt
package http

import curl.curl_version
import kotlinx.cinterop.toKString

class CurlInfo {
    fun version(): String {
        // Generated C interop API stays inside native adapter.
        return curl_version()?.toKString().orEmpty()
    }
}
```

<MouseTrap>
C interop bindings are low-level. Не разносите generated pointer-heavy API по common code. Сразу оборачивайте его в маленькие Kotlin abstractions.
</MouseTrap>

## Native resources

API C и платформы часто возвращают ресурсы, которые необходимо закрыть.

```kotlin
package nativeio

import kotlinx.cinterop.CPointer
import platform.posix.FILE
import platform.posix.fclose
import platform.posix.fopen

class NativeFile private constructor(
    private val pointer: CPointer<FILE>
) {
    fun close() {
        fclose(pointer)
    }

    companion object {
        fun open(path: String, mode: String): NativeFile {
            val file = fopen(path, mode)
                ?: error("Не удалось открыть файл $path")

            return NativeFile(file)
        }
    }
}
```

Safer scoped helper:

```kotlin
package nativeio

inline fun <T> withNativeFile(path: String, mode: String, block: (NativeFile) -> T): T {
    val file = NativeFile.open(path, mode)

    return try {
        block(file)
    } finally {
        // Native resource закрывается детерминированно.
        file.close()
    }
}
```

## Взаимодействие Objective-C и Swift

Kotlin/Native взаимодействует со Swift через Objective-C.

You can:

— вызывать фреймворки Objective-C из Kotlin;
- использовать предварительно импортированные платформы Apple, такие как Foundation и UIKit;
- экспортировать модуль Kotlin как фреймворк;
— вызвать Kotlin из Swift/Objective-C;
- доработать экспортируемый API аннотациями.

Calling Apple API:

```kotlin
// iosMain/kotlin/platform/IosUuid.kt
package platform

import platform.Foundation.NSUUID

class IosUuidGenerator {
    fun next(): String {
        return NSUUID().UUIDString()
    }
}
```

Скрытие объявлений Kotlin из Swift/Objective-C:

```kotlin
package api

import kotlin.native.HiddenFromObjC

class LessonApi {
    fun publicTitle(slug: String): String {
        return "Урок $slug"
    }

    @HiddenFromObjC
    fun kotlinOnlyDebugLine(): String {
        return "internal-debug"
    }
}
```

## Исключения на границе Swift

Исключения Котлина требуют тщательного сопоставления.

```kotlin
package api

class LessonNotFoundException(
    slug: String
) : Exception("Урок не найден: $slug")

class NativeLessonApi {
    @Throws(LessonNotFoundException::class)
    fun title(slug: String): String {
        if (slug.isBlank()) {
            throw LessonNotFoundException(slug)
        }

        return "Урок $slug"
    }
}
```

Без `@Throws` неприостанавливаемые исключения Kotlin, пересекающие Swift/Objective-C, не являются обычными ошибками Swift и могут привести к завершению работы программы. Целенаправленно создавайте API, ориентированный на Swift.

## Framework export

Цели Apple могут создавать платформы Objective-C.

```kotlin
kotlin {
    iosArm64()
    iosSimulatorArm64()

    targets.withType<org.jetbrains.kotlin.gradle.plugin.mpp.KotlinNativeTarget>().configureEach {
        binaries.framework {
            baseName = "KotlingoShared"
            isStatic = true
        }
    }
}
```

Для доставки Apple на нескольких архитектурах используйте XCFramework.

```kotlin
import org.jetbrains.kotlin.gradle.plugin.mpp.apple.XCFramework

kotlin {
    val xcf = XCFramework("KotlingoShared")

    listOf(
        iosArm64(),
        iosSimulatorArm64()
    ).forEach { target ->
        target.binaries.framework {
            baseName = "KotlingoShared"
            isStatic = true
            xcf.add(this)
        }
    }
}
```

Задачи сборки включают задачи сборки XCFramework, созданные плагином Gradle.

## Export dependencies carefully

Платформа может экспортировать зависимости, но экспортированные зависимости становятся частью API платформы и двоичных файлов.

```kotlin
kotlin {
    iosArm64 {
        binaries.framework {
            baseName = "KotlingoShared"

            // Export only dependencies that Swift callers must directly see.
            export(project(":shared-api"))
        }
    }

    sourceSets {
        iosMain.dependencies {
            api(project(":shared-api"))
        }
    }
}
```

Избегайте транзитивного экспорта, если он вам действительно не нужен. Это увеличивает размер двоичного файла и поверхность API.

## Проверочный список для боевого кода

Для собственной цели:

- объявляйте только те цели, которые вы тестируете;
- намеренно использовать промежуточные исходные наборы, такие как `iosMain`;
- держать взаимодействие C за обертками;
- close native resources deterministically;
— разрабатывать API Swift/Objective-C отдельно от внутреннего API Kotlin;
- используйте `@Throws` для ошибок Swift;
— скрывать или уточнять объявления, которые не должны быть общедоступными для Swift;
- намеренно настраивать фреймворки или XCFrameworks;
- измерять на реальных устройствах;
- помните, что диспетчер памяти современен, но параллелизм все еще нуждается в разработке.

<CatTip>
Нативная цель вознаграждает чистоту: общий код остается спокойным, нативные адаптеры небольшие, платформы предоставляют дружественный API, и у каждого указателя есть ответственный владелец.
</CatTip>
