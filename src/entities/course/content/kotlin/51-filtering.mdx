## Что такое filtering

Filtering — это выбор subset элементов из collection по условию. Условие задается predicate-функцией, то есть функцией, которая получает element и возвращает `Boolean`.

```kotlin
val ages = listOf(12, 18, 21, 64)

val adults = ages.filter { age ->
    // Оставляем только элементы, которые удовлетворяют бизнес-правилу.
    age >= 18
}

println(adults) // [18, 21, 64]
```

Ключевая идея: filtering не изменяет исходную read-only collection. Большинство filtering-операций возвращают новый `List`, сохраняя relative order элементов, которые прошли проверку.

<CatTip>
Фильтрация — это «кошачий проход через дверцу»: каждый элемент подходит к predicate, получает `true` или `false`, и только элементы с `true` проходят дальше.
</CatTip>

## `filter`

`filter` оставляет элементы, для которых predicate вернул `true`.

```kotlin
data class Lesson(
    val slug: String,
    val title: String,
    val minutes: Int,
    val completed: Boolean
)

val lessons = listOf(
    Lesson(slug = "01-introduction", title = "Введение", minutes = 14, completed = true),
    Lesson(slug = "02-types", title = "Типы", minutes = 28, completed = false),
    Lesson(slug = "03-null-safety", title = "Null-safety", minutes = 35, completed = false)
)

val unfinished = lessons.filter { lesson ->
    // Predicate возвращает true для незавершенных уроков.
    !lesson.completed
}

println(unfinished.map { it.title }) // [Типы, Null-safety]
```

`filter` хорошо подходит, когда условие читается прямо: active users, paid orders, visible tasks, valid tokens.

```kotlin
val paidOrders = orders.filter { order ->
    // В production-коде predicate обычно выражает доменное правило.
    order.status == OrderStatus.PAID && order.totalCents > 0
}
```

Если predicate становится длинным, вынеси правило в именованную функцию или property. Это делает pipeline читаемым.

```kotlin
fun Lesson.requiresAttention(): Boolean {
    // Урок требует внимания, если он еще не завершен и достаточно длинный.
    return !completed && minutes >= 30
}

val priorityLessons = lessons.filter { lesson ->
    lesson.requiresAttention()
}
```

## `filterNot`

`filterNot` делает противоположное: оставляет элементы, для которых predicate вернул `false`.

```kotlin
val published = articles.filterNot { article ->
    // Убираем черновики из публичного списка.
    article.isDraft
}
```

Это не просто синтаксический сахар. `filterNot` помогает читать отрицательные правила без двойного отрицания.

```kotlin
val blockedUsers = users.filter { user ->
    user.isBlocked
}

val allowedUsers = users.filterNot { user ->
    user.isBlocked
}
```

<MouseTrap>
Не смешивай `filterNot` с отрицанием внутри predicate без необходимости. `users.filterNot { !it.isActive }` работает, но читается хуже, чем `users.filter { it.isActive }`.
</MouseTrap>

## Порядок элементов

Filtering не сортирует collection. Порядок результата соответствует порядку исходного traversal.

```kotlin
val numbers = listOf(5, 1, 4, 2, 3)

val even = numbers.filter { number ->
    number % 2 == 0
}

println(even) // [4, 2]
```

Если нужен другой порядок, используй ordering-операции после filtering:

```kotlin
val sortedEven = numbers
    .filter { number -> number % 2 == 0 }
    .sorted()

println(sortedEven) // [2, 4]
```

## `filterIndexed`

`filterIndexed` передает в predicate не только element, но и index.

```kotlin
val checkpoints = listOf("start", "warmup", "practice", "quiz", "finish")

val visibleMilestones = checkpoints.filterIndexed { index, checkpoint ->
    // Показываем первый, последний и каждый второй промежуточный шаг.
    index == 0 || index == checkpoints.lastIndex || index % 2 == 0
}

println(visibleMilestones) // [start, practice, finish]
```

Используй `filterIndexed`, когда index является частью правила:

- отобрать каждый третий element;
- оставить первые N matching-элементов с нестандартной логикой;
- удалить элементы на технических позициях;
- сравнить element с соседним значением по index.

```kotlin
val readings = listOf(21.0, 21.1, 40.0, 21.2, 21.3)

val withoutSuddenSpikes = readings.filterIndexed { index, reading ->
    // Первый элемент не с чем сравнивать, поэтому оставляем его.
    if (index == 0) {
        true
    } else {
        val previous = readings[index - 1]
        val diff = kotlin.math.abs(reading - previous)

        // Убираем измерения, которые слишком резко отличаются от предыдущих.
        diff <= 5.0
    }
}

println(withoutSuddenSpikes) // [21.0, 21.1, 21.2, 21.3]
```

<MouseTrap>
Не используй `filterIndexed`, если index не нужен. Лишний index часто маскирует сложную логику и делает pipeline менее выразительным.
</MouseTrap>

## Filtering nullable values

Если collection содержит nullable values, можно фильтровать их вручную:

```kotlin
val rawScores: List<Int?> = listOf(10, null, 30, null, 50)

val nonNullScores = rawScores.filter { score ->
    score != null
}
```

Но тип результата будет `List<Int?>`, потому что compiler не меняет generic type после обычного `filter`.

```kotlin
val stillNullable: List<Int?> = rawScores.filter { score ->
    score != null
}
```

Для превращения `List<Int?>` в `List<Int>` используй `filterNotNull`.

## `filterNotNull`

`filterNotNull` удаляет `null` и сужает тип элементов до non-null type.

```kotlin
val rawScores: List<Int?> = listOf(10, null, 30, null, 50)

val scores: List<Int> = rawScores.filterNotNull()

val average = scores.average()
println(average) // 30.0
```

После `filterNotNull` можно вызывать members non-null type без safe call.

```kotlin
val nicknames: List<String?> = listOf("Murka", null, "Barsik", "")

val normalized = nicknames
    .filterNotNull()
    .filter { nickname ->
        // Здесь nickname уже String, а не String?.
        nickname.isNotBlank()
    }
    .map { nickname ->
        // Нормализуем имена для отображения.
        nickname.trim().replaceFirstChar { char -> char.uppercase() }
    }

println(normalized) // [Murka, Barsik]
```

<CatTip>
`filterNotNull` особенно полезен после чтения внешних данных: JSON, query parameters, user input, nullable Java API. Он одновременно чистит данные и улучшает тип.
</CatTip>

## `filterIsInstance`

`filterIsInstance` оставляет элементы конкретного runtime type и возвращает collection с уточненным типом.

```kotlin
sealed interface FeedEvent

data class LessonOpened(val slug: String) : FeedEvent
data class QuizPassed(val slug: String, val score: Int) : FeedEvent
data class SubscriptionExpired(val userId: String) : FeedEvent

val events: List<FeedEvent> = listOf(
    LessonOpened("01-introduction"),
    QuizPassed("02-types", score = 92),
    SubscriptionExpired("u-42"),
    QuizPassed("03-null-safety", score = 78)
)

val quizEvents: List<QuizPassed> = events.filterIsInstance<QuizPassed>()

val passedWithHighScore = quizEvents.filter { event ->
    // event уже имеет тип QuizPassed, поэтому score доступен напрямую.
    event.score >= 90
}

println(passedWithHighScore) // [QuizPassed(slug=02-types, score=92)]
```

`filterIsInstance` применяется для mixed collections, event streams, UI models и migration-кода. В чистой доменной модели слишком частое использование может быть сигналом, что collection хранит слишком разные сущности.

```kotlin
val rawValues: List<Any?> = listOf("10", 20, "30", null, 40L)

val ints: List<Int> = rawValues.filterIsInstance<Int>()

println(ints) // [20]
```

<MouseTrap>
`filterIsInstance<Int>()` не превращает строки `"10"` в числа и не конвертирует `Long` в `Int`. Он проверяет именно runtime type.
</MouseTrap>

## Filtering `Map`

Для `Map` есть filtering по entries, keys и values.

```kotlin
val lessonProgress = mapOf(
    "01-introduction" to 100,
    "02-types" to 80,
    "03-null-safety" to 0,
    "04-collections" to 45
)

val startedLessons = lessonProgress.filter { entry ->
    // entry имеет key и value.
    entry.value > 0
}

println(startedLessons) // {01-introduction=100, 02-types=80, 04-collections=45}
```

Можно destructure `Map.Entry`:

```kotlin
val almostDone = lessonProgress.filter { (slug, percent) ->
    // slug здесь используется как часть бизнес-правила.
    slug.startsWith("0") && percent in 80..99
}

println(almostDone) // {02-types=80}
```

`filterKeys` фильтрует только по key:

```kotlin
val basicLessons = lessonProgress.filterKeys { slug ->
    // Отбираем уроки базового блока по соглашению о slug.
    slug.startsWith("01") || slug.startsWith("02") || slug.startsWith("03")
}
```

`filterValues` фильтрует только по value:

```kotlin
val completedLessons = lessonProgress.filterValues { percent ->
    // Значение 100 означает полное прохождение урока.
    percent == 100
}
```

Результат filtering для `Map` тоже является новой map, обычно сохраняющей iteration order исходной map.

## `partition`

`partition` разделяет collection на две части за один проход:

- first list содержит элементы, для которых predicate вернул `true`;
- second list содержит элементы, для которых predicate вернул `false`.

```kotlin
val tasks = listOf(
    "прочитать теорию",
    "решить задачу",
    "пройти квиз",
    "повторить null-safety"
)

val (practice, theory) = tasks.partition { task ->
    // Практическими считаем задачи и квизы.
    task.contains("задачу") || task.contains("квиз")
}

println(practice) // [решить задачу, пройти квиз]
println(theory)   // [прочитать теорию, повторить null-safety]
```

`partition` возвращает `Pair<List<T>, List<T>>`. Destructuring делает результат удобным.

```kotlin
data class Payment(val id: String, val amountCents: Int, val confirmed: Boolean)

val payments = listOf(
    Payment("p-1", amountCents = 2_000, confirmed = true),
    Payment("p-2", amountCents = 0, confirmed = false),
    Payment("p-3", amountCents = 4_500, confirmed = true)
)

val (confirmed, pending) = payments.partition { payment ->
    // Правило подтверждения держим в одном месте.
    payment.confirmed && payment.amountCents > 0
}

println("Подтверждено: ${confirmed.map { it.id }}")
println("Ожидает проверки: ${pending.map { it.id }}")
```

`partition` лучше двух отдельных `filter`/`filterNot`, когда нужны обе стороны условия. Так collection обходится один раз, а правило не дублируется.

## Когда использовать какой метод

Используй `filter`, когда нужен subset по positive condition.

```kotlin
val active = users.filter { user -> user.active }
```

Используй `filterNot`, когда правило естественно выражается как исключение.

```kotlin
val visible = users.filterNot { user -> user.hidden }
```

Используй `filterIndexed`, когда index является частью доменной логики.

```kotlin
val everyFourth = items.filterIndexed { index, _ -> index % 4 == 0 }
```

Используй `filterNotNull`, когда нужно удалить `null` и получить non-null type.

```kotlin
val ids: List<String> = rawIds.filterNotNull()
```

Используй `filterIsInstance`, когда mixed collection нужно сузить по runtime type.

```kotlin
val errors = events.filterIsInstance<ErrorEvent>()
```

Используй `partition`, когда нужны обе группы: matching и non-matching.

```kotlin
val (valid, invalid) = rows.partition { row -> row.isValid() }
```

## Production-пример: фильтрация уроков для dashboard

```kotlin
enum class Difficulty {
    BEGINNER,
    INTERMEDIATE,
    ADVANCED
}

data class CourseLesson(
    val slug: String,
    val title: String,
    val difficulty: Difficulty,
    val estimatedMinutes: Int,
    val completedAtMillis: Long?,
    val tags: Set<String>
)

data class Dashboard(
    val recommended: List<CourseLesson>,
    val completed: List<CourseLesson>,
    val postponed: List<CourseLesson>
)

fun buildDashboard(
    lessons: List<CourseLesson>,
    allowedTags: Set<String>,
    maxMinutes: Int
): Dashboard {
    val visibleLessons = lessons.filter { lesson ->
        // Урок видим, если у него есть хотя бы один тег из разрешенного набора.
        lesson.tags.any { tag -> tag in allowedTags }
    }

    val (completed, notCompleted) = visibleLessons.partition { lesson ->
        // completedAtMillis хранит null, пока урок не завершен.
        lesson.completedAtMillis != null
    }

    val recommended = notCompleted
        .filter { lesson ->
            // Рекомендуем короткие или средние уроки, чтобы пользователь мог быстро продолжить курс.
            lesson.estimatedMinutes <= maxMinutes
        }
        .filterNot { lesson ->
            // Продвинутые темы не показываем в рекомендациях, пока пользователь выбирает быстрый режим.
            lesson.difficulty == Difficulty.ADVANCED
        }

    val postponed = notCompleted.filterNot { lesson ->
        // Все, что не попало в recommended, считается отложенным.
        lesson in recommended
    }

    return Dashboard(
        recommended = recommended,
        completed = completed,
        postponed = postponed
    )
}
```

Здесь filtering используется как цепочка доменных ограничений. Каждый шаг оставляет данные в понятном состоянии: сначала видимые уроки, затем completed/notCompleted, затем recommended/postponed.

## Производительность и side effects

Обычные filtering-операции для `Iterable` создают новую collection. Если pipeline длинный и исходная collection большая, можно рассмотреть `Sequence`, но только после измерений.

```kotlin
val result = lessons
    .asSequence()
    .filter { lesson ->
        // В sequence этот predicate будет выполняться лениво.
        lesson.tags.contains("collections")
    }
    .filterNot { lesson ->
        lesson.completedAtMillis != null
    }
    .take(10)
    .toList()
```

Predicate лучше делать pure: без изменения внешних переменных, записи в файл, сетевых вызовов и мутаций элементов.

```kotlin
var rejected = 0

val valid = lessons.filter { lesson ->
    // Так можно, но side effect усложняет тестирование и повторное использование.
    if (lesson.estimatedMinutes <= 0) {
        rejected += 1
        false
    } else {
        true
    }
}
```

Более надежный вариант — сначала разделить данные, а затем явно посчитать rejected.

```kotlin
val (validLessons, rejectedLessons) = lessons.partition { lesson ->
    lesson.estimatedMinutes > 0
}

val rejectedCount = rejectedLessons.size
```

<CatTip>
Фильтрация лучше всего читается как серия маленьких проверок. Пусть каждый predicate отвечает на один вопрос, тогда pipeline будет мурлыкать даже через полгода сопровождения.
</CatTip>
