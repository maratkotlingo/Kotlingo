## Что такое structured concurrency

Structured concurrency — принцип, по которому concurrent work имеет явную структуру ownership:

- каждая child coroutine принадлежит scope;
- parent scope ждет своих children;
- cancellation parent-а отменяет children;
- failure child-а обычно отменяет siblings и parent;
- lifetime background work привязан к lifetime owner-а;
- вызывающий код не получает «бесхозную» работу, которая продолжает жить после завершения функции.

```kotlin
import kotlinx.coroutines.async
import kotlinx.coroutines.coroutineScope

suspend fun loadLessonPage(slug: String): LessonPage = coroutineScope {
    val lesson = async { LessonRepository().loadLesson(slug) }
    val author = async { AuthorRepository().loadAuthor(slug) }

    LessonPage(
        lesson = lesson.await(),
        author = author.await(),
        widgets = emptyList()
    )
}
```

В этом примере `loadLessonPage` не вернется, пока both children не завершатся. Если вызывающий код отменит `loadLessonPage`, обе child coroutines тоже отменятся.

<CatTip>
Structured concurrency — это правило «все котята возвращаются в корзинку»: запущенная работа не должна теряться за пределами scope.
</CatTip>

## Parent-child hierarchy

Coroutine builders создают hierarchy.

```kotlin
import kotlinx.coroutines.Job
import kotlinx.coroutines.delay
import kotlinx.coroutines.launch
import kotlinx.coroutines.runBlocking

fun hierarchyExample() = runBlocking {
    val parent = launch {
        launch {
            delay(100)
            println("Child A завершен")
        }

        launch {
            delay(150)
            println("Child B завершен")
        }
    }

    parent.join()

    println("Parent завершен только после children")
}
```

Родитель не завершается, пока не завершатся все дочерние элементы.

```kotlin
import kotlinx.coroutines.delay
import kotlinx.coroutines.launch
import kotlinx.coroutines.runBlocking

fun parentWaitsForChildren() = runBlocking {
    val parent = launch {
        launch {
            delay(300)
            println("Долгий child завершен")
        }

        println("Body parent coroutine завершился")
    }

    parent.join()

    println("join parent вернулся после child")
}
```

Cancellation goes down:

```kotlin
import kotlinx.coroutines.cancelAndJoin
import kotlinx.coroutines.delay
import kotlinx.coroutines.launch
import kotlinx.coroutines.runBlocking

fun cancellationGoesDown() = runBlocking {
    val parent = launch {
        launch {
            try {
                delay(10_000)
            } finally {
                println("Child cleanup после parent cancellation")
            }
        }
    }

    delay(100)
    parent.cancelAndJoin()
}
```

Провал идет вверх, а затем вбок:

```kotlin
import kotlinx.coroutines.coroutineScope
import kotlinx.coroutines.delay
import kotlinx.coroutines.launch

suspend fun failurePropagationExample() = coroutineScope {
    launch {
        delay(100)
        error("Metadata service упал")
    }

    launch {
        try {
            delay(10_000)
        } finally {
            // Sibling отменен, потому что первый child упал.
            println("Body loader остановлен")
        }
    }
}
```

<MouseTrap>
Если child failure не должен отменять siblings, это не повод использовать global coroutine. Используй `supervisorScope` или owner-level `SupervisorJob` и явно обработай ожидаемые ошибки.
</MouseTrap>

## Scope ownership

Scope должен принадлежать объекту, который понимает его lifetime.

Примеры owner-ов:

- application process;
- HTTP request;
- UI screen или view model;
- CLI command;
- background worker;
- test case.

Плохой признак: репозиторий, parser или utility function сами создают hidden scope и прячут work от вызывающий код-а.

```kotlin
import kotlinx.coroutines.GlobalScope
import kotlinx.coroutines.launch

class BadProgressRepository {
    fun markCompleted(userId: String, slug: String) {
        GlobalScope.launch {
            // Caller не может отменить, дождаться или надежно обработать ошибку этой работы.
            saveProgress(userId, slug)
        }
    }

    private suspend fun saveProgress(userId: String, slug: String) {
        println("Сохраняем прогресс $userId для $slug")
    }
}
```

Такой код нарушает structured concurrency:

- вызывающий код думает, что operation завершилась, но save еще идет;
- exception может потеряться или уйти в global handler;
- test становится flaky, потому что работа переживает test;
- shutdown application не гарантирует завершение;
- cancellation request-а не отменяет работу.

Лучше сделать API suspending.

```kotlin
class ProgressRepository {
    suspend fun markCompleted(userId: String, slug: String) {
        // Caller сам решает, где и как запускать operation.
        saveProgress(userId, slug)
    }

    private suspend fun saveProgress(userId: String, slug: String) {
        println("Сохраняем прогресс $userId для $slug")
    }
}
```

Caller теперь явно owns work.

```kotlin
import kotlinx.coroutines.launch
import kotlinx.coroutines.runBlocking

fun markCompletedFromCli() = runBlocking {
    val repository = ProgressRepository()

    launch {
        repository.markCompleted(
            userId = "u-42",
            slug = "80-structured-concurrency"
        )
    }

    // runBlocking дождется child launch.
}
```

## Lifecycle alignment

Lifecycle alignment означает: work starts and stops вместе с owner-ом.

```kotlin
import kotlinx.coroutines.CoroutineName
import kotlinx.coroutines.CoroutineScope
import kotlinx.coroutines.Dispatchers
import kotlinx.coroutines.SupervisorJob
import kotlinx.coroutines.cancel
import kotlinx.coroutines.delay
import kotlinx.coroutines.launch

class LessonProgressWorker {
    private val scope = CoroutineScope(
        SupervisorJob() +
            Dispatchers.Default +
            CoroutineName("lesson-progress-worker")
    )

    fun start(): kotlinx.coroutines.Job = scope.launch {
        while (true) {
            flushProgressBatch()
            delay(5_000)
        }
    }

    fun stop() {
        // Все children этого worker-а получают cancellation.
        scope.cancel()
    }

    private suspend fun flushProgressBatch() {
        delay(100)
        println("Batch прогресса отправлен")
    }
}
```

Этот class сам владеет scope, потому что он является lifecycle owner-ом. У него есть `start` и `stop`.

Если owner уже существует, лучше принимать external scope.

```kotlin
import kotlinx.coroutines.CoroutineName
import kotlinx.coroutines.CoroutineScope
import kotlinx.coroutines.Job
import kotlinx.coroutines.delay
import kotlinx.coroutines.launch

class LessonNotificationLoop(
    private val scope: CoroutineScope
) {
    fun start(): Job = scope.launch(CoroutineName("lesson-notifications")) {
        while (true) {
            sendQueuedNotifications()
            delay(10_000)
        }
    }

    private suspend fun sendQueuedNotifications() {
        delay(100)
        println("Notifications отправлены")
    }
}
```

Такой design делает ownership явным: кто передал scope, тот отвечает за cancellation.

<CatTip>
Если не можешь назвать владельца scope, scope почти наверняка расположен не там. Сначала найди хозяина lifecycle, потом запускай coroutines.
</CatTip>

## `coroutineScope` как локальная структура

`coroutineScope` — основной инструмент structured concurrency внутри `suspend fun`.

```kotlin
import kotlinx.coroutines.async
import kotlinx.coroutines.coroutineScope

suspend fun buildDashboard(userId: String): Dashboard = coroutineScope {
    val profile = async { loadProfile(userId) }
    val progress = async { loadProgress(userId) }
    val achievements = async { loadAchievements(userId) }

    Dashboard(
        profile = profile.await(),
        progress = progress.await(),
        achievements = achievements.await()
    )
}

data class Dashboard(
    val profile: Profile,
    val progress: Progress,
    val achievements: List<Achievement>
)

data class Profile(
    val id: String,
    val displayName: String
)

data class Progress(
    val completedLessons: Int
)

data class Achievement(
    val name: String
)

suspend fun loadProfile(userId: String): Profile {
    return Profile(userId, "Kotlin ученик")
}

suspend fun loadProgress(userId: String): Progress {
    return Progress(completedLessons = userId.length)
}

suspend fun loadAchievements(userId: String): List<Achievement> {
    return listOf(Achievement("Первые корутины"))
}
```

Если `loadProfile` fails, `loadProgress` and `loadAchievements` are cancelled. Это хорошо для mandatory dashboard data: incomplete mandatory data means whole dashboard invalid.

## `supervisorScope` для независимых children

Structured concurrency не требует fail-fast всегда. Она требует явного ownership. Для независимых children используется supervision.

```kotlin
import kotlinx.coroutines.async
import kotlinx.coroutines.supervisorScope

suspend fun loadDashboardExtras(userId: String): List<DashboardExtra> = supervisorScope {
    val streak = async { loadStreak(userId) }
    val promo = async { loadPromo(userId) }
    val survey = async { loadSurvey(userId) }

    listOfNotNull(
        runCatching { streak.await() }.getOrNull(),
        runCatching { promo.await() }.getOrNull(),
        runCatching { survey.await() }.getOrNull()
    )
}

data class DashboardExtra(
    val key: String,
    val text: String
)

suspend fun loadStreak(userId: String): DashboardExtra {
    return DashboardExtra("streak", "Серия пользователя $userId")
}

suspend fun loadPromo(userId: String): DashboardExtra {
    error("Promo service временно недоступен")
}

suspend fun loadSurvey(userId: String): DashboardExtra {
    return DashboardExtra("survey", "Опрос недели")
}
```

Здесь failure `promo` не отменяет `streak` and `survey`, но работа все равно scoped. Function не вернется, пока supervised children не завершатся или не будут обработаны.

## Избегай global work

Global work — работа без владельца, часто через `GlobalScope`, manually created root `CoroutineScope` или detached `Job`.

Плохой example:

```kotlin
import kotlinx.coroutines.GlobalScope
import kotlinx.coroutines.async

fun loadLessonAsync(slug: String) = GlobalScope.async {
    LessonRepository().loadLesson(slug)
}
```

Проблемы:

- вызывающий код получил `Deferred`, но lifetime coroutine не привязан к вызывающий код scope;
- распространение исключения отделено от запроса;
- test должен вручную чистить global jobs;
- application shutdown может прервать operation в случайном месте;
- API поддерживает функции асинхронного типа вместо приостановки функций.

Лучше:

```kotlin
suspend fun loadLesson(slug: String): Lesson {
    return LessonRepository().loadLesson(slug)
}
```

Если вызывающий код хочет concurrency, он сам использует `async` внутри своего scope.

```kotlin
import kotlinx.coroutines.async
import kotlinx.coroutines.coroutineScope

suspend fun loadTwoLessons(
    firstSlug: String,
    secondSlug: String
): Pair<Lesson, Lesson> = coroutineScope {
    val first = async { loadLesson(firstSlug) }
    val second = async { loadLesson(secondSlug) }

    first.await() to second.await()
}
```

<MouseTrap>
Функции с суффиксом `Async`, которые сами запускают global coroutine и возвращают `Deferred`, часто являются запахом дизайна. В Kotlin coroutines idiomatic API обычно остается `suspend`.
</MouseTrap>

## Request-scope пример

Представим бэкенд handler. Request cancellation должна остановить all work for request.

```kotlin
import kotlinx.coroutines.async
import kotlinx.coroutines.coroutineScope

class LessonHttpHandler(
    private val lessons: LessonRepository,
    private val progress: ProgressRepository,
    private val recommendations: RecommendationService
) {
    suspend fun handle(userId: String, slug: String): LessonResponse = coroutineScope {
        val lesson = async {
            lessons.loadLesson(slug)
        }

        val userProgress = async {
            progress.loadProgress(userId)
        }

        val related = async {
            recommendations.relatedTo(slug)
        }

        LessonResponse(
            title = lesson.await().title,
            body = lesson.await().body,
            completedLessons = userProgress.await().completedLessons,
            relatedSlugs = related.await()
        )
    }
}

data class LessonResponse(
    val title: String,
    val body: String,
    val completedLessons: Int,
    val relatedSlugs: List<String>
)

class RecommendationService {
    suspend fun relatedTo(slug: String): List<String> {
        return listOf(slug, "81-cancellation")
    }
}
```

Есть маленькая inefficiency: `lesson.await()` вызван дважды. `Deferred.await()` после completion returns same value, но читать лучше так:

```kotlin
import kotlinx.coroutines.async
import kotlinx.coroutines.coroutineScope

suspend fun LessonHttpHandler.handleReadable(
    userId: String,
    slug: String
): LessonResponse = coroutineScope {
    val lessonDeferred = async { LessonRepository().loadLesson(slug) }
    val progressDeferred = async { ProgressRepository().loadProgress(userId) }
    val relatedDeferred = async { RecommendationService().relatedTo(slug) }

    val lesson = lessonDeferred.await()
    val progress = progressDeferred.await()
    val related = relatedDeferred.await()

    LessonResponse(
        title = lesson.title,
        body = lesson.body,
        completedLessons = progress.completedLessons,
        relatedSlugs = related
    )
}
```

Request-scope property: если HTTP client disconnected and framework cancels handler coroutine, all three child operations should stop.

## Fire-and-forget: когда все-таки нужно

Иногда business действительно требует work after request returns: отправить email, обработать audit event, обновить search index. Это не значит, что work должна быть global. Значит, ей нужен другой owner.

```kotlin
import kotlinx.coroutines.CoroutineName
import kotlinx.coroutines.CoroutineScope
import kotlinx.coroutines.launch

class AuditService(
    private val applicationScope: CoroutineScope
) {
    fun recordLessonOpened(userId: String, slug: String) {
        applicationScope.launch(CoroutineName("audit-lesson-opened")) {
            persistAuditEvent(userId, slug)
        }
    }

    private suspend fun persistAuditEvent(userId: String, slug: String) {
        println("Audit: $userId открыл $slug")
    }
}
```

Это fire-and-forget relative to request, но не relative to application. Application owns `applicationScope` and cancels it on shutdown.

Более надежный боевой design часто использует queue:

```kotlin
class AuditQueue(
    private val sender: AuditSender
) {
    suspend fun enqueue(event: AuditEvent) {
        // В настоящем приложении здесь может быть durable queue или transactional outbox.
        sender.send(event)
    }
}

data class AuditEvent(
    val userId: String,
    val slug: String
)

class AuditSender {
    suspend fun send(event: AuditEvent) {
        println("Отправляем audit event ${event.userId}:${event.slug}")
    }
}
```

## Проверочный список для structured concurrency

Перед запуском coroutine спроси:

- кто владеет scope;
- кто отменит work;
- кто дождется completion;
- где будет обработана exception;
- должна ли sibling failure отменять остальных;
- что произойдет при shutdown, screen close, request cancellation или test finish;
- нужен ли вообще concurrent execution, или sequential suspend code проще.

<CatTip>
Самый надежный coroutine design обычно выглядит скучно: `suspend fun`, `coroutineScope`, явный owner scope и минимум detached jobs. Это именно та скука, которая делает боевой спокойным.
</CatTip>
