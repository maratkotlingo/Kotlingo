## Альтернативное имя существующего типа

`typealias` создаёт новое имя, но не новый тип.

```kotlin
typealias LessonId = Int

fun openLesson(id: LessonId): String {
    return "Открыт урок $id"
}

fun main() {
    val raw: Int = 33
    println(openLesson(raw))
}
```

`LessonId` полностью раскрывается в `Int`. Значения assignment-compatible в обе стороны, runtime wrapper не создаётся.

## Зачем нужны aliases

Aliases полезны для:

- длинных generic types;
- function types;
- nested/inner class names;
- локального domain vocabulary;
- миграции source names;
- сокрытия технического шума без изменения semantics.

```kotlin
typealias LessonsByCategory = Map<String, List<String>>

fun lessonIndex(): LessonsByCategory {
    return mapOf(
        "Классы" to listOf("Data classes", "Value classes")
    )
}
```

Alias сообщает роль структуры лучше, чем повторяющийся `Map<String, List<String>>`.

## Function type aliases

```kotlin
typealias LessonFilter = (String) -> Boolean
typealias LessonMapper = (String) -> String
typealias CompletionCallback = (lessonId: Int, score: Int) -> Unit
```

Имена parameters внутри function type служат документацией и IDE hints, но не входят в type identity.

```kotlin
fun processTitles(
    titles: List<String>,
    filter: LessonFilter,
    mapper: LessonMapper
): List<String> {
    return titles.filter(filter).map(mapper)
}

fun main() {
    val nonBlank: LessonFilter = { title -> title.isNotBlank() }
    val normalize: LessonMapper = String::trim

    println(processTitles(listOf(" Kotlin ", ""), nonBlank, normalize))
}
```

Обычная `(String) -> Boolean` совместима с `LessonFilter`, потому что alias раскрывается.

## Receiver function type aliases

```kotlin
typealias LessonBuilderBlock = LessonBuilder.() -> Unit

class LessonBuilder {
    var title: String = ""
    var order: Int = 0
}

fun lesson(block: LessonBuilderBlock): LessonBuilder {
    return LessonBuilder().apply(block)
}
```

Receiver type делает DSL signature короче. Alias не меняет правила receiver resolution.

## Suspend function aliases

```kotlin
typealias LessonLoader = suspend (Int) -> String

suspend fun renderLesson(id: Int, loader: LessonLoader): String {
    return "Урок: ${loader(id)}"
}
```

`suspend` является частью functional type. Обычная `(Int) -> String` не тождественна `suspend (Int) -> String`.

## Nullable function type и приоритет скобок

```kotlin
typealias OptionalCallback = ((String) -> Unit)?
```

Nullable является сама function value. Скобки важны: nullable result function имеет другой type.

```kotlin
typealias NullableResultProvider = () -> String?
```

- `((String) -> Unit)?` — обратный вызов может отсутствовать;
- `() -> String?` — обратный вызов обязателен, result может быть null.

```kotlin
fun notify(callback: OptionalCallback, message: String) {
    callback?.invoke(message)
}
```

## Generic aliases

```kotlin
typealias LookupResult<T> = Result<T>
typealias LessonCache<K> = MutableMap<K, List<String>>
typealias Validator<T> = (T) -> List<String>
```

Type parameter alias подставляется в underlying type:

```kotlin
fun <T> validate(value: T, validator: Validator<T>): List<String> {
    return validator(value)
}
```

Alias может менять порядок/фиксировать часть arguments:

```kotlin
typealias StringKeyedMap<V> = Map<String, V>

val titles: StringKeyedMap<String> = mapOf("33" to "Type aliases")
```

## Aliases для nested и inner classes

```kotlin
class Course {
    class Lesson(val title: String)
}

typealias CourseLesson = Course.Lesson

val lesson = CourseLesson("Type aliases")
```

Alias можно использовать в constructor-like call, но создаётся исходный `Course.Lesson`.

Для inner type alias также не устраняет необходимость outer instance:

```kotlin
class Session {
    inner class Step(val name: String)
}

typealias SessionStep = Session.Step

fun createStep(session: Session): SessionStep {
    return session.Step("Изучить aliases")
}
```

## Nested type aliases

Актуальный Kotlin позволяет объявлять aliases внутри classes/objects, если они не захватывают type parameters внешнего declaration.

```kotlin
class LessonGraph {
    typealias NodeId = Int
    typealias Edges = Map<NodeId, Set<NodeId>>

    fun outgoing(graph: Edges, node: NodeId): Set<NodeId> {
        return graph[node].orEmpty()
    }
}
```

Scope соответствует nested class: `LessonGraph.NodeId` доступен согласно visibility.

### Запрет захвата outer type parameter

```kotlin
class Graph<Node> {
    // typealias Path = List<Node>
}
```

Alias не может неявно захватить `Node` внешнего class. Объяви собственный parameter:

```kotlin
class Graph<Node> {
    typealias Path<T> = List<T>

    fun singleton(node: Node): Path<Node> = listOf(node)
}
```

Nested aliases не поддерживаются как `expect`/`actual` declarations в Kotlin Multiplatform. Проверяй language/toolchain version проекта: это более новая возможность, чем top-level aliases.

## Visibility alias

```kotlin
private typealias MutableLessonIndex = MutableMap<Int, String>
internal typealias ModuleLessonHandler = (Int) -> Unit
```

Alias не может эффективно раскрыть менее видимый underlying type наружу.

```kotlin
private class InternalRecord(val title: String)

private typealias RecordList = List<InternalRecord>
```

Public alias на private type создал бы unusable public contract.

## Alias не создаёт новый тип

```kotlin
typealias StudentId = Long
typealias CourseId = Long

fun enroll(studentId: StudentId, courseId: CourseId) {
    println("$studentId -> $courseId")
}

fun main() {
    val student: StudentId = 10L
    val course: CourseId = 20L

    enroll(course, student)
}
```

Ошибка смысла компилируется: оба aliases — `Long`.

Для type safety нужен value class:

```kotlin
@JvmInline
value class SafeStudentId(val value: Long)

@JvmInline
value class SafeCourseId(val value: Long)
```

<MouseTrap title="Мышеловка: alias не защищает от перепутанных аргументов">
Он меняет имя только для читателя и compiler diagnostics. Для compile-time разделения ID, единиц измерения и валидированных строк используй value/class.
</MouseTrap>

## Alias и перегрузки

Нельзя перегрузить functions по alias и underlying type:

```kotlin
typealias LessonNumber = Int

// Эти signatures одинаковы после раскрытия alias.
// fun render(value: Int): String = value.toString()
// fun render(value: LessonNumber): String = "Урок $value"
```

То же относится к JVM signature, reflection и generics. Alias не даёт дополнительного runtime classifier.

## Reflection и runtime

```kotlin
typealias LessonTitle = String

val title: LessonTitle = "Type aliases"
println(title::class)
```

Runtime class — `String`. Нельзя получить `LessonTitle::class`, потому что alias не является classifier с собственным KClass.

Serializer, DI container и reflection framework видят underlying type. Annotation на alias может иметь ограниченную use-site semantics, но не превращает его в runtime wrapper.

## Alias против import alias

Import alias переименовывает declaration только в текущем файле:

```kotlin
import company.api.Lesson as ApiLesson
import company.domain.Lesson as DomainLesson
```

`typealias` создаёт declaration, доступное другим files согласно visibility:

```kotlin
typealias ApiLessonRecord = company.api.Lesson
```

Import alias решает collision имён. Type alias документирует переиспользуемый type expression/API vocabulary.

## Public API и читаемость

Плюсы public alias:

- короткие signatures;
- domain term;
- source migration path;
- реализация общего шума, скрытого от читателей.

Минусы:

- clients могут не видеть реальный type;
- IDE/docs показывают alias и expansion по-разному;
- несколько aliases одного type создают illusion safety;
- изменение expansion может ломать source/binary/serialization assumptions;
- Java обычно не видит Kotlin alias как отдельный declaration.

<CatTip title="Котосовет: alias должен сокращать форму, не скрывать смысл">
`LessonValidator = (Lesson) -> List<Issue>` полезен. `Data = Map<String, Any?>` прячет weakly typed schema и переносит ошибки в runtime.
</CatTip>

## Alias и binary compatibility

Alias существует главным образом в Kotlin metadata/source model и раскрывается для generated signatures. Изменение underlying type может изменить JVM descriptor, generic signature или source compatibility.

```kotlin
typealias LessonId = Int
```

Замена на `Long` — не безопасная internal refactoring для public functions. Клиенты нужно пересобрать, а binary API изменится.

Переименование public alias ломает source imports, даже если expansion прежнее. Старый alias можно временно оставить deprecated:

```kotlin
typealias LessonIdentifier = Int

@Deprecated(
    message = "Используй LessonIdentifier",
    replaceWith = ReplaceWith("LessonIdentifier")
)
typealias OldLessonId = LessonIdentifier
```

## Aliases и Java interop

Java source видит expanded type:

```kotlin
typealias LessonCallback = (Int) -> Unit

fun register(callback: LessonCallback) {
    callback(33)
}
```

В JVM API это Kotlin `Function1`-подобная signature, а не Java type `LessonCallback`. Для Java-friendly API лучше `fun interface`:

```kotlin
fun interface LessonListener {
    fun onLesson(id: Int)
}
```

## Type alias или named wrapper

Выбирай alias, если нужны:

- shorter name;
- identical operations и compatibility underlying type;
- zero semantic distinction;
- function signature vocabulary.

Выбирай value/data/regular class, если нужны:

- type safety;
- validation;
- methods/invariants;
- runtime identity/classifier;
- пользовательский serialization;
- independent API evolution.

## Практический пример: pipeline обработки контента

```kotlin
data class LessonDraft(
    val id: Int,
    val title: String,
    val content: String
)

data class ValidationIssue(
    val field: String,
    val message: String
)

typealias DraftNormalizer = (LessonDraft) -> LessonDraft
typealias DraftValidator = (LessonDraft) -> List<ValidationIssue>
typealias DraftConsumer = (LessonDraft) -> Unit
typealias Validators = List<DraftValidator>
typealias IssuesByField = Map<String, List<ValidationIssue>>

fun normalizeDraft(draft: LessonDraft): LessonDraft {
    return draft.copy(
        title = draft.title.trim().replace(Regex("""\s+"""), " "),
        content = draft.content.trim()
    )
}

val titleValidator: DraftValidator = { draft ->
    if (draft.title.length in 3..120) {
        emptyList()
    } else {
        listOf(
            ValidationIssue(
                field = "title",
                message = "Название должно содержать от 3 до 120 символов"
            )
        )
    }
}

val contentValidator: DraftValidator = { draft ->
    if (draft.content.isNotEmpty()) {
        emptyList()
    } else {
        listOf(
            ValidationIssue(
                field = "content",
                message = "Содержание обязательно"
            )
        )
    }
}

fun processDraft(
    draft: LessonDraft,
    normalizer: DraftNormalizer,
    validators: Validators,
    onValid: DraftConsumer
): IssuesByField {
    val normalized = normalizer(draft)
    val issues = validators.flatMap { validator -> validator(normalized) }

    if (issues.isEmpty()) {
        onValid(normalized)
    }

    return issues.groupBy(ValidationIssue::field)
}

fun main() {
    val issues = processDraft(
        draft = LessonDraft(33, "  Type aliases  ", "Function aliases"),
        normalizer = ::normalizeDraft,
        validators = listOf(titleValidator, contentValidator),
        onValid = { lesson -> println("Готов урок ${lesson.id}: ${lesson.title}") }
    )

    println(issues)
}
```

Aliases улучшают signatures, но не скрывают domain models и не создают ложные ID types. Любая `(LessonDraft) -> List<ValidationIssue>` совместима с `DraftValidator`, что здесь является намеренным свойством.

## Итоги

- `typealias` даёт второе имя существующему типу;
- alias полностью раскрывается и не создаёт runtime wrapper;
- function aliases документируют обратные вызовы, receiver и suspend types;
- nullable function и nullable result требуют разных скобок;
- generic alias может фиксировать/переставлять type arguments;
- nested aliases не могут захватывать outer type parameters;
- aliases одного underlying type взаимозаменяемы;
- alias не подходит type-safe IDs и перегрузка distinction;
- reflection/Java видят underlying representation;
- import alias локален файлу, typealias является declaration;
- public expansion — часть source/binary contract;
- используй alias для читаемости формы, wrapper — для новой semantics.
