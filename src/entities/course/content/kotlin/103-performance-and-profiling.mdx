## Производительность начинается с измерения

Оптимизация без измерений часто ускоряет не то место.

Правильный порядок:

- define user-visible problem;
- measure baseline;
- identify hotspot;
- change one thing;
- measure again;
- keep benchmark or регрессия test where appropriate;
- document tradeoff.

```kotlin
class LessonSearch(
    private val lessons: List<Lesson>
) {
    fun search(query: String): List<Lesson> {
        val normalized = query.lowercase()

        return lessons.filter { lesson ->
            lesson.title.lowercase().contains(normalized)
        }
    }
}

data class Lesson(
    val slug: String,
    val title: String
)
```

Возможно, это нормально. Возможно, внутренний цикл `lowercase()` является горячей точкой. Вы узнаете это только после профилирования с реалистичными данными.

<CatTip>
Performance work без профиля — это кот, который охотится под диваном на шум из холодильника. Сначала найди настоящую мышку.
</CatTip>

## Макро- и микроизмерения

Macro measurement answers product-level questions:

- request latency;
- throughput;
- startup time;
- memory footprint;
- UI frame time;
- import duration;
- Использование процессора при реальной нагрузке.

Micro measurement answers narrow questions:

- быстрее ли этот парсер;
- имеет ли значение это распределение;
- дешевле ли конвейер последовательности;
- does inline remove лямбда allocation;
- появляется ли бокс в хот лупе.

Сначала используйте макропрофилирование. Используйте микротесты для изолированных, повторяемых решений.

## JMH

В JVM JMH — это стандартная программа для микробенчмарка.

Gradle plugin настройка varies by project, but benchmark code usually looks like this:

```kotlin
import org.openjdk.jmh.annotations.Benchmark
import org.openjdk.jmh.annotations.Fork
import org.openjdk.jmh.annotations.Measurement
import org.openjdk.jmh.annotations.Scope
import org.openjdk.jmh.annotations.State
import org.openjdk.jmh.annotations.Warmup

@State(Scope.Benchmark)
@Warmup(iterations = 3)
@Measurement(iterations = 5)
@Fork(2)
open class SlugBenchmark {
    private val titles = List(1_000) { index ->
        "Kotlin Performance Lesson $index"
    }

    @Benchmark
    fun normalizeAll(): List<String> {
        return titles.map { title ->
            title.trim()
                .lowercase()
                .replace(" ", "-")
        }
    }
}
```

JMH решает проблемы прогрева, разветвлений и оптимизации JVM, которые делают наивное время ненадежным.

<MouseTrap>
`val started = System.nanoTime()` вокруг одного вызова функции обычно не является надежным эталоном. Прогрев JVM, устранение мертвого кода и сбор мусора могут доминировать в результате.
</MouseTrap>

## Профилирование с помощью JFR

Java Flight Recorder помогает идентифицировать настоящие горячие точки JVM:

- CPU samples;
- allocation pressure;
- lock contention;
- exceptions;
- thread activity;
- GC pauses;
- file/socket I/O;
- method profiling.

Typical рабочий процесс:

```bash
java -XX:StartFlightRecording=filename=kotlingo.jfr,duration=60s -jar app.jar
```

Затем проверьте запись в JDK Mission Control или другом средстве просмотра JFR.

Используйте профилирующие вопросы:

- куда тратится процессор;
- which methods allocate most;
- which allocations survive GC;
- где потоки блокируются;
- являются ли исключения в горячих путях;
- дорогие ли журналы или сериализация.

## Allocation hotspots

Возможности Kotlin выразительны, но могут выделять:

- лямбды captured in non-inline calls;
- intermediate collections;
- boxing primitives;
- конкатенация строк в циклах;
- data class copies;
- последовательности с множеством мелких оберток;
- coroutine state machines;
- regex creation inside hot paths.

Плохо:

```kotlin
fun countTagsBad(lessons: List<LessonWithTags>): Int {
    return lessons
        .flatMap { lesson -> lesson.tags }
        .filter { tag -> tag.startsWith("kotlin") }
        .map { tag -> tag.lowercase() }
        .distinct()
        .count()
}

data class LessonWithTags(
    val slug: String,
    val tags: List<String>
)
```

Это читаемо, но может выделять промежуточные списки.

Manual hot-path version:

```kotlin
fun countTagsHotPath(lessons: List<LessonWithTags>): Int {
    val unique = HashSet<String>()

    for (lesson in lessons) {
        for (tag in lesson.tags) {
            if (tag.startsWith("kotlin")) {
                unique += tag.lowercase()
            }
        }
    }

    return unique.size
}
```

Не переписывайте все вручную. Используйте этот стиль только там, где профилирование доказывает его важность.

## Sequences versus collections

Коллекции жаждут. Каждый оператор может создать промежуточную коллекцию.

```kotlin
fun eagerPipeline(values: List<String>): List<String> {
    return values
        .filter { value -> value.isNotBlank() }
        .map { value -> value.trim() }
        .take(10)
}
```

Последовательность ленивая.

```kotlin
fun lazyPipeline(values: List<String>): List<String> {
    return values.asSequence()
        .filter { value -> value.isNotBlank() }
        .map { value -> value.trim() }
        .take(10)
        .toList()
}
```

Последовательность может помочь, когда:

- pipeline has many operators;
- вход большой;
- `take`/`first` stops early;
- avoiding intermediate collections matters.

Последовательность может навредить, когда:

- ввод небольшой;
- трубопровод короткий;
- виртуальные вызовы/обертки стоят дороже, чем сэкономленные аллокации;
- код становится труднее читать.

Измеряйте реальные данные.

## Inline tradeoffs

`inline` can remove лямбда allocation and call overhead for higher-order functions.

```kotlin
inline fun <T> measureBlock(name: String, block: () -> T): T {
    val started = System.nanoTime()

    return try {
        block()
    } finally {
        val durationMs = (System.nanoTime() - started) / 1_000_000
        println("$name took $durationMs ms")
    }
}
```

Tradeoffs:

- larger bytecode;
- публичные inline-функции раскрывают детали реализации;
- binary compatibility risks;
- трассировки стека могут быть менее прямыми;
- не каждая функция приносит пользу.

Используй `inline` для:

- small higher-order helpers;
- reified type parameters;
- критичные к производительности библиотечные функции после измерения;
- DSL functions where лямбды are central.

Избегайте `inline` для больших функций.

## Boxing

Общие примитивы блоков кода на JVM.

```kotlin
fun sumGeneric(values: List<Int>): Int {
    var sum = 0

    for (value in values) {
        sum += value
    }

    return sum
}
```

`List<Int>` хранит упакованный `Integer` на JVM. Примитивные массивы избегают упаковки.

```kotlin
fun sumArray(values: IntArray): Int {
    var sum = 0

    for (value in values) {
        sum += value
    }

    return sum
}
```

Используй примитивные массивы в горячем числовом коде. Не используй их везде по умолчанию.

## Coroutine overhead

Сопрограммы легкие, но не бесплатные.

Costs:

- state machine allocation;
- dispatcher scheduling;
- context switching;
- cancellation checks;
- `async`/`Deferred` objects;
- Flow operator overhead;
- канальные буферы и синхронизация.

Плохое разветвление мелких задач:

```kotlin
import kotlinx.coroutines.async
import kotlinx.coroutines.awaitAll
import kotlinx.coroutines.coroutineScope

suspend fun normalizeBad(values: List<String>): List<String> = coroutineScope {
    values.map { value ->
        async {
            value.trim().lowercase()
        }
    }.awaitAll()
}
```

Better:

```kotlin
fun normalize(values: List<String>): List<String> {
    return values.map { value ->
        value.trim().lowercase()
    }
}
```

Используйте сопрограммы для параллелизма, асинхронного ввода-вывода и структурированной отмены, а не для каждой небольшой операции ЦП.

## Logging cost

Ведение журнала может стать проблемой производительности:

- eager string interpolation;
- huge payload rendering;
- high-cardinality structured fields;
- sync appenders;
- бревна в тугих петлях;
- трассировка стека исключений в ожидаемых ветвях.

Better:

```kotlin
logger.debug("Parsed lesson count={}", count)
```

Guard expensive rendering:

```kotlin
if (logger.isDebugEnabled) {
    logger.debug("Parsed lesson tree={}", renderDebugTree(tree))
}
```

## Проверочный список для боевого кода

Для исполнительской работы:

- воспроизвести проблему с реалистичной рабочей нагрузкой;
- профиль перед оптимизацией;
- отдельные макро- и микроизмерения;
- использовать JMH для микротестов JVM;
- проверять хотспоты распределения, а не только ЦП;
- сравнивать конвейеры последовательности и сбора с реальными данными;
- намеренно используйте встроенные функции;
- избегать разветвления сопрограмм для небольших задач ЦП;
- preserve readability unless performance evidence justifies complexity;
- keep benchmark or performance регрессия check for critical path.

<CatTip>
Хорошая optimization leaves code no more clever than necessary. Если измерение не доказало пользу, не заставляй будущего разработчика распутывать клубок ради воображаемой скорости.
</CatTip>
