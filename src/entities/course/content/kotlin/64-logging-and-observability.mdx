## Observability шире, чем logs

Observability — способность понять состояние системы по внешним сигналам.

Основные сигналы:

- logs — события и диагностические сообщения;
- metrics — числовые показатели over time;
- traces — путь request-а через services;
- errors — исключения and crash reports;
- профили — ЦП, поведение распределения и блокировки;
- audits — бизнес-события, важные для продукта.

Logging is one part. Хорошая система наблюдаемости отвечает:

- что произошло;
- где произошло;
- с каким request/user/entity;
- как долго длилось;
- почему упало;
- какие downstream calls участвовали;
- можно ли найти все события одного request-а.

<CatTip>
Logs — это кошачьи следы, metrics — счетчик шагов, traces — карта маршрута. Боевая отладка требует все три, а не только громкое «мяу» в консоль.
</CatTip>

## JVM logging facades

На JVM обычно используют facade, а не конкретный logger напрямую.

Common choice:

- SLF4J в качестве фасада;
- Logback или Log4j2 в качестве реализации;
- JUL-мост, если существует устаревшая версия Java Util Logging;
— структурированный кодер для логов JSON, когда логи попадают в централизованное хранилище.

Gradle:

```kotlin
dependencies {
    implementation("org.slf4j:slf4j-api:2.1.0")
    runtimeOnly("ch.qos.logback:logback-classic:1.6.0")
}
```

Logger:

```kotlin
package dev.kotlingo.lesson

import org.slf4j.LoggerFactory

class LessonService {
    private val logger = LoggerFactory.getLogger(LessonService::class.java)

    fun open(slug: String): Lesson {
        logger.info("Opening lesson {}", slug)

        return Lesson(
            slug = slug,
            title = "Урок $slug"
        )
    }
}

data class Lesson(
    val slug: String,
    val title: String
)
```

При использовании API-интерфейсов в стиле SLF4J используйте параметризованное ведение журнала, а не интерполяцию строк.

```kotlin
logger.info("Lesson {} opened by user {}", slug, userId)
```

Это позволяет избежать быстрого построения строк, когда уровень журнала отключен.

<MouseTrap>
`logger.debug("Payload: ${expensiveToRender()}")` вычисляет строку до проверки уровня. В hot paths используй parameterized logging or explicit level check.
</MouseTrap>

## Kotlin-friendly logger helper

У Kotlin нет встроенного API ведения журналов. Маленький помощник может уменьшить шаблонность.

```kotlin
package dev.kotlingo.logging

import org.slf4j.Logger
import org.slf4j.LoggerFactory

inline fun <reified T : Any> loggerFor(): Logger {
    return LoggerFactory.getLogger(T::class.java)
}
```

Использование:

```kotlin
package dev.kotlingo.progress

import dev.kotlingo.logging.loggerFor

class ProgressService {
    private val logger = loggerFor<ProgressService>()

    fun complete(userId: String, slug: String) {
        logger.info("Marking lesson {} completed for user {}", slug, userId)
    }
}
```

Держите помощника крошечным. Не создавайте собственную среду ведения журналов, если у вас нет реальных требований к кроссплатформенности.

## Structured logs

Структурированный журнал хранит поля отдельно, а не только текст сообщения.

Text log:

```text
User u-42 opened lesson 101-logging-and-observability
```

Structured log concept:

```json
{
  "event": "lesson.opened",
  "user_id": "u-42",
  "slug": "101-logging-and-observability",
  "duration_ms": 24
}
```

SLF4J 2 поддерживает свободный API с парами ключ-значение.

```kotlin
import org.slf4j.LoggerFactory

class StructuredLessonLogger {
    private val logger = LoggerFactory.getLogger(StructuredLessonLogger::class.java)

    fun lessonOpened(userId: String, slug: String, durationMs: Long) {
        logger.atInfo()
            .setMessage("lesson opened")
            .addKeyValue("event", "lesson.opened")
            .addKeyValue("user_id", userId)
            .addKeyValue("slug", slug)
            .addKeyValue("duration_ms", durationMs)
            .log()
    }
}
```

Структурированные поля должны быть стабильными и доступными для запросов. Избегайте помещения всех данных в сообщение произвольной формы.

## Log levels

Используйте уровни последовательно:

- `trace` — very detailed flow, usually disabled;
- `debug` — developer diagnostics;
- `info` — important lifecycle/business events;
- `warn` — unexpected but recoverable condition;
- `error` — failed operation needing attention.

```kotlin
class LessonImportService(
    private val logger: org.slf4j.Logger
) {
    fun import(slugs: List<String>) {
        logger.info("Starting lesson import: count={}", slugs.size)

        for (slug in slugs) {
            try {
                importOne(slug)
            } catch (exception: RuntimeException) {
                logger.warn("Skipping lesson {} because import failed", slug, exception)
            }
        }

        logger.info("Lesson import finished")
    }

    private fun importOne(slug: String) {
        require(slug.isNotBlank()) { "slug не должен быть пустым" }
    }
}
```

Не входите в систему по адресу `error` для ожидаемой проверки пользователя. `error` должен означать сбой в работе.

## Correlation IDs

Correlation ID ties logs for one request, message or рабочий процесс.

```kotlin
data class RequestContext(
    val correlationId: String,
    val userId: String?
)
```

Passing explicitly:

```kotlin
class LessonHandler(
    private val service: LessonOpenService,
    private val logger: org.slf4j.Logger
) {
    suspend fun handle(context: RequestContext, slug: String): LessonResponse {
        logger.info(
            "Handling open lesson: correlation_id={} slug={}",
            context.correlationId,
            slug
        )

        val lesson = service.open(context, slug)

        return LessonResponse(lesson.title)
    }
}

data class LessonResponse(
    val title: String
)

class LessonOpenService {
    suspend fun open(context: RequestContext, slug: String): Lesson {
        return Lesson(slug, "Урок $slug")
    }
}
```

Явное прохождение скучное, но очень понятное. Для фреймворков осторожно используйте распространение контекста запроса.

## MDC

Сопоставленный диагностический контекст хранит контекстные поля для журналов в текущем контексте выполнения.

```kotlin
import org.slf4j.MDC

inline fun <T> withCorrelationId(correlationId: String, block: () -> T): T {
    MDC.put("correlation_id", correlationId)

    return try {
        block()
    } finally {
        MDC.remove("correlation_id")
    }
}
```

В коде сопрограммы локальный поток MDC не следует автоматически за сопрограммой между диспетчерами, если не настроена интеграция.

```kotlin
import kotlinx.coroutines.slf4j.MDCContext
import kotlinx.coroutines.withContext
import org.slf4j.MDC

suspend fun <T> withCoroutineMdc(correlationId: String, block: suspend () -> T): T {
    MDC.put("correlation_id", correlationId)

    return try {
        withContext(MDCContext()) {
            block()
        }
    } finally {
        MDC.remove("correlation_id")
    }
}
```

Dependency:

```kotlin
dependencies {
    implementation("org.jetbrains.kotlinx:kotlinx-coroutines-slf4j:1.11.0")
}
```

<MouseTrap>
MDC по своей природе является локальным для потоков. Сопрограммы могут возобновиться в другом потоке, поэтому MDC без распространения контекста сопрограммы дает отсутствующие или неправильные идентификаторы корреляции.
</MouseTrap>

## Metrics

Metrics answer numeric questions:

- request count;
- error rate;
- latency percentiles;
- queue depth;
- cache hit rate;
- coroutine worker backlog;
- import duration;
- payload size.

Используйте абстракцию на границе домена/приложения.

```kotlin
interface Metrics {
    fun incrementCounter(name: String, tags: Map<String, String> = emptyMap())

    fun recordDurationMillis(name: String, durationMs: Long, tags: Map<String, String> = emptyMap())
}
```

Использование:

```kotlin
class MeasuredLessonService(
    private val delegate: LessonOpenService,
    private val metrics: Metrics
) {
    suspend fun open(context: RequestContext, slug: String): Lesson {
        val started = System.nanoTime()

        return try {
            delegate.open(context, slug)
        } finally {
            val durationMs = (System.nanoTime() - started) / 1_000_000

            metrics.recordDurationMillis(
                name = "lesson.open.duration",
                durationMs = durationMs,
                tags = mapOf("lesson_slug" to slug)
            )
        }
    }
}
```

Be careful with tag cardinality. User IDs, raw URLs, exception messages and unique slugs can create too many time series depending on metrics бэкенд.

## Tracing

Распределенная трассировка отслеживает запросы между службами.

OpenTelemetry concepts:

- trace — whole request journey;
- span — one operation inside trace;
- атрибуты — структурированные поля по диапазону;
— распространение контекста — передача контекста трассировки через асинхронные вызовы и границы службы.

Conceptual wrapper:

```kotlin
interface Tracer {
    suspend fun <T> span(name: String, attributes: Map<String, String>, block: suspend () -> T): T
}
```

Использование:

```kotlin
class TracedLessonRepository(
    private val tracer: Tracer,
    private val source: LessonSource
) {
    suspend fun load(slug: String): Lesson {
        return tracer.span(
            name = "lesson.repository.load",
            attributes = mapOf("lesson.slug" to slug)
        ) {
            source.load(slug)
        }
    }
}

interface LessonSource {
    suspend fun load(slug: String): Lesson
}
```

Продолжайте отслеживать границы и важные операции. Не оборачивайте каждую крошечную приватную функцию промежутками.

## Avoiding hidden побочные эффекты

Ведение журнала не должно менять поведение бизнеса.

Плохо:

```kotlin
class BadLessonService(
    private val logger: org.slf4j.Logger
) {
    fun open(slug: String): Lesson {
        logger.info("Opening {}", slug)

        if (logger.isDebugEnabled) {
            // Плохо: debug branch changes behavior by mutating state.
            warmCache(slug)
        }

        return Lesson(slug, "Урок $slug")
    }

    private fun warmCache(slug: String) {
        println("Warming cache for $slug")
    }
}
```

Хорошо:

```kotlin
class GoodLessonService(
    private val logger: org.slf4j.Logger,
    private val cache: LessonCache
) {
    fun open(slug: String): Lesson {
        logger.info("Opening {}", slug)

        cache.warm(slug)

        return Lesson(slug, "Урок $slug")
    }
}

class LessonCache {
    fun warm(slug: String) {
        println("Прогреваем cache для $slug")
    }
}
```

Код наблюдаемости должен наблюдать и сообщать. Он не должен тайно определять поведение продукта.

## Sensitive data

Never log secrets:

- passwords;
- tokens;
- refresh tokens;
- session cookies;
- private keys;
- full payment data;
- raw personal documents;
- sensitive user input.

```kotlin
data class LoginRequest(
    val email: String,
    val password: String
) {
    fun safeLogFields(): Map<String, String> {
        return mapOf(
            "email_domain" to email.substringAfter("@", missingDelimiterValue = "unknown")
        )
    }
}
```

Зарегистрируйте безопасные производные поля, а не необработанные секреты.

## Multiplatform logging boundary

Для KMP определите общий интерфейс регистратора.

```kotlin
interface AppLogger {
    fun info(message: String, fields: Map<String, String> = emptyMap())

    fun warn(message: String, fields: Map<String, String> = emptyMap(), throwable: Throwable? = null)

    fun error(message: String, fields: Map<String, String> = emptyMap(), throwable: Throwable? = null)
}
```

JVM adapter:

```kotlin
class Slf4jAppLogger(
    private val logger: org.slf4j.Logger
) : AppLogger {
    override fun info(message: String, fields: Map<String, String>) {
        logger.atInfo()
            .setMessage(message)
            .apply {
                fields.forEach { (key, value) ->
                    addKeyValue(key, value)
                }
            }
            .log()
    }

    override fun warn(message: String, fields: Map<String, String>, throwable: Throwable?) {
        logger.warn("$message fields=$fields", throwable)
    }

    override fun error(message: String, fields: Map<String, String>, throwable: Throwable?) {
        logger.error("$message fields=$fields", throwable)
    }
}
```

Common code depends on `AppLogger`; platform code decides actual бэкенд.

## Проверочный список для боевого кода

Наблюдение перед отправкой:

- логи структурированы там, где важен централизованный поиск;
- идентификатор корреляции присутствует в журналах запросов;
- настроено распространение сопрограммы MDC, если используется MDC;
- конфиденциальные значения редактируются или не регистрируются;
- метрики избегают неограниченной мощности;
— трассировки покрывают важные границы, а не каждый приватный метод;
- log levels have team-wide meaning;
- logging does not hide побочные эффекты;
- общий код использует абстракции, когда цель является мультиплатформенной.

<CatTip>
Хорошая observability не шумит ради шума. Она оставляет ровно столько следов, чтобы ночью найти проблему, не разбудив весь дом ложными `error`-ами.
</CatTip>
