## Зачем нужны context parameters

Context parameters позволяют функции или property объявить зависимость, которая должна быть доступна в surrounding context, но не передается как обычный argument в каждый вызов.

```kotlin
interface LessonLogger {
    fun log(message: String)
}

context(logger: LessonLogger)
fun publishLesson(slug: String) {
    // logger не обычный параметр функции, но он доступен из context.
    logger.log("Публикуем урок: $slug")
}
```

Вызов выглядит так:

```kotlin
val logger = object : LessonLogger {
    override fun log(message: String) {
        println(message)
    }
}

context(logger) {
    publishLesson("69-context-parameters")
}
```

Главная идея: функция явно пишет в signature, какие contextual capabilities ей нужны, а вызывающий код предоставляет их один раз на scope.

<CatTip>
Context parameter — это как тихий кошачий помощник в комнате: его не передают в каждую миску руками, но все функции в комнате знают, что он рядом и может помочь.
</CatTip>

## Современный синтаксис `context(name: Type)`

Актуальный синтаксис использует `context` с именованными параметрами.

```kotlin
interface Clock {
    fun nowMillis(): Long
}

context(logger: LessonLogger, clock: Clock)
fun logLessonView(slug: String) {
    val timestamp = clock.nowMillis()

    // Оба context parameters доступны по именам.
    logger.log("[$timestamp] Открыт урок $slug")
}
```

Context parameters находятся перед declaration.

```kotlin
context(logger: LessonLogger)
val welcomeMessage: String
    get() {
        // Property с context parameter не имеет backing field.
        logger.log("Создаем приветствие")
        return "Добро пожаловать в kotlingo"
    }
```

Можно использовать `_`, если значение нужно только для resolution, но не нужно обращаться к нему по имени.

```kotlin
context(_: LessonLogger)
fun greetWithLogging() {
    // Здесь нельзя написать _.log(...), но функции, которым нужен LessonLogger, могут быть вызваны.
    publishLesson("intro")
}
```

## Чем context parameter отличается от обычного параметра

Обычный parameter передается явно в каждый вызов.

```kotlin
fun publishLessonExplicit(logger: LessonLogger, slug: String) {
    logger.log("Публикуем урок: $slug")
}

publishLessonExplicit(logger, "01-introduction")
```

Context parameter объявляется в signature, но подставляется compiler-ом из context scope.

```kotlin
context(logger: LessonLogger)
fun publishLessonContextual(slug: String) {
    logger.log("Публикуем урок: $slug")
}

context(logger) {
    publishLessonContextual("01-introduction")
}
```

Это уменьшает boilerplate в глубоких call chains, но dependency остается видимой в declaration.

## Разрешение по типу

Kotlin ищет matching context value по type at место вызова.

```kotlin
interface PermissionChecker {
    fun canEdit(slug: String): Boolean
}

context(checker: PermissionChecker)
fun requireCanEdit(slug: String) {
    require(checker.canEdit(slug)) {
        "Нет права редактировать урок $slug"
    }
}
```

Caller предоставляет value compatible type.

```kotlin
val checker = object : PermissionChecker {
    override fun canEdit(slug: String): Boolean {
        // В учебном примере разрешаем редактировать все опубликованные slug.
        return slug.firstOrNull()?.isDigit() == true
    }
}

context(checker) {
    requireCanEdit("01-introduction")
}
```

Name context parameter в declaration важен внутри function body и для explicit context arguments, но resolution primarily происходит по type.

## Неоднозначности

Если в одном scope доступны два compatible values одного типа, compiler не сможет выбрать.

```kotlin
val auditLogger = object : LessonLogger {
    override fun log(message: String) {
        println("audit: $message")
    }
}

val consoleLogger = object : LessonLogger {
    override fun log(message: String) {
        println("console: $message")
    }
}

context(auditLogger, consoleLogger) {
    // publishLesson("01-introduction")
    // Ошибка: два LessonLogger подходят одинаково хорошо.
}
```

Неоднозначность лучше решать архитектурно: разделить types, сузить scope или передать context argument явно.

```kotlin
interface AuditLogger : LessonLogger
interface UiLogger : LessonLogger
```

Разные interfaces делают dependency точнее.

## Explicit context arguments

Explicit context arguments позволяют указать, какой context value использовать в конкретном вызове.

```kotlin
context(logger: LessonLogger)
fun renderLogLine(slug: String): String {
    logger.log("Готовим строку для $slug")
    return "Урок: $slug"
}

context(audit: LessonLogger, ui: LessonLogger)
fun renderWithSelectedLogger(slug: String): String {
    return renderLogLine(logger = ui, slug)
}
```

Этот синтаксис особенно полезен, когда перегрузки отличаются context parameters.

```kotlin
class EmailSender {
    fun send(message: String) {
        println("email: $message")
    }
}

class SmsSender {
    fun send(message: String) {
        println("sms: $message")
    }
}

context(emailSender: EmailSender)
fun sendNotification(message: String) {
    emailSender.send(message)
}

context(smsSender: SmsSender)
fun sendNotification(message: String) {
    smsSender.send(message)
}

context(defaultEmail: EmailSender, defaultSms: SmsSender)
fun notifyByBothChannels() {
    sendNotification(emailSender = defaultEmail, message = "Новый урок")
    sendNotification(smsSender = defaultSms, message = "Новый урок")
}
```

<MouseTrap>
Explicit context arguments являются отдельной experimental-возможностью. По официальной документации для нее нужен compiler argument `-Xexplicit-context-arguments`.
</MouseTrap>

Gradle:

```kotlin
kotlin {
    compilerOptions {
        // Включает синтаксис explicit context arguments.
        freeCompilerArgs.add("-Xexplicit-context-arguments")
    }
}
```

Если несколько вызовов используют одни и те же contextual values, обычно чище создать `context(...) { ... }` block.

```kotlin
context(uiLogger) {
    renderLogLine("01-introduction")
    renderLogLine("02-types")
}
```

## Отличие от старых context receivers

Старые context receivers были experimental и использовали receiver-like syntax.

```kotlin
// Старый стиль context receivers.
context(LessonLogger)
fun oldPublishLesson(slug: String) {
    // Зависимость выглядит как receiver, а не как именованный параметр.
    log("Публикуем урок: $slug")
}
```

Современный context parameter именуется явно.

```kotlin
context(logger: LessonLogger)
fun newPublishLesson(slug: String) {
    logger.log("Публикуем урок: $slug")
}
```

Практические отличия:

- context parameters имеют names;
- dependency читается как parameter, а не как дополнительный receiver;
- меньше конфликтов с extension receivers и DSL receivers;
- проще явно обратиться к нужному context value;
- старые context receivers заменяются context parameters и требуют миграции.

## Context parameter плюс extension receiver

Context parameter может сочетаться с extension receiver.

```kotlin
context(logger: LessonLogger)
fun String.asPublishedSlug(): String {
    val normalized = trim().lowercase()

    logger.log("Нормализуем slug: $normalized")

    return normalized
}

context(logger) {
    println("  01-Introduction  ".asPublishedSlug())
}
```

Здесь `String` — receiver, а `logger` — contextual dependency.

## Ограничения

По текущей документации context parameters находятся в развитии, и важные ограничения такие:

- constructors не могут объявлять context parameters;
- properties с context parameters не могут иметь backing fields;
- properties с context parameters не могут иметь initializers;
- properties с context parameters не могут использовать delegation;
- explicit context arguments требуют отдельного experimental flag.

```kotlin
// Такой constructor сейчас не является допустимой формой:
// class Lesson context(logger: LessonLogger) constructor(val slug: String)
```

Для класса dependency передавай явно или создавай factory function с context parameter.

```kotlin
data class PublishedLesson(
    val slug: String,
    val createdBy: String
)

interface CurrentUser {
    val username: String
}

context(user: CurrentUser)
fun publishedLesson(slug: String): PublishedLesson {
    return PublishedLesson(
        slug = slug,
        createdBy = user.username
    )
}
```

## Боевой-пример: scoped validation

```kotlin
interface SlugPolicy {
    fun isValid(slug: String): Boolean
}

interface ValidationReporter {
    fun report(message: String)
}

data class ValidatedLesson(
    val slug: String,
    val title: String
)

context(policy: SlugPolicy, reporter: ValidationReporter)
fun validateLesson(slug: String, title: String): Result<ValidatedLesson> {
    val normalizedSlug = slug.trim().lowercase()
    val normalizedTitle = title.trim()

    if (!policy.isValid(normalizedSlug)) {
        reporter.report("Некорректный slug: $normalizedSlug")
        return Result.failure(IllegalArgumentException("Некорректный slug"))
    }

    if (normalizedTitle.isEmpty()) {
        reporter.report("Пустой title для $normalizedSlug")
        return Result.failure(IllegalArgumentException("Пустой title"))
    }

    return Result.success(
        ValidatedLesson(
            slug = normalizedSlug,
            title = normalizedTitle
        )
    )
}
```

Использование:

```kotlin
val policy = object : SlugPolicy {
    override fun isValid(slug: String): Boolean {
        return slug.matches(Regex("""[a-z0-9]+(?:-[a-z0-9]+)*"""))
    }
}

val reporter = object : ValidationReporter {
    override fun report(message: String) {
        println("validation: $message")
    }
}

context(policy, reporter) {
    val result = validateLesson("69-context-parameters", "Context parameters")
    println(result)
}
```

## Когда использовать

Context parameters уместны, когда dependency:

- нужна многим functions в одном scope;
- редко меняется внутри call chain;
- является capability, service, policy, logger, clock, locale или transaction;
- не должна становиться global singleton;
- должна быть видна в function signature.

Не используй context parameters для значений, которые являются core input конкретной операции.

```kotlin
context(logger: LessonLogger)
fun renameLesson(slug: String, newTitle: String) {
    // slug и newTitle остаются обычными параметрами, потому что это данные операции.
    logger.log("Переименовываем $slug в $newTitle")
}
```

<CatTip>
Если значение отвечает на вопрос «что обрабатываем?», передавай обычным параметром. Если отвечает «в какой среде выполняем?», context parameter может быть хорошим решением.
</CatTip>
