## Type inference из действий внутри builder

Обычный type inference анализирует arguments и expected result вокруг вызова. Builder inference дополнительно собирает constraints из вызовов внутри lambda with receiver.

```kotlin
val lessonIds = buildList {
    add(36)
    add(37)
    add(38)
}
```

До анализа body неизвестно, какой `E` нужен `buildList<E>`. Вызов `add(Int)` сообщает constraint, и result становится `List<Int>`.

```kotlin
val titles = buildList {
    add("Reified")
    add("Builder inference")
}
```

Result — `List<String>`.

## Зачем это нужно DSL

Без builder inference caller пришлось бы писать type arguments/expected type:

```kotlin
val titles = buildList<String> {
    add("Kotlin")
}
```

В выразительном DSL type становится очевиден из operations:

```kotlin
val lessons = buildMap {
    put(37, "Definitely non-nullable types")
    put(38, "Builder inference")
}
```

Compiler выводит `Map<Int, String>` по `put`.

## Regular inference имеет приоритет

Compiler использует builder inference, только если обычный inference не зафиксировал type arguments.

```kotlin
val explicitResult: List<Number> = buildList {
    add(1)
    add(2.5)
}
```

Expected type снаружи уже задаёт `E = Number`. Body проверяется относительно него.

Это важно для diagnostics: внешний context может сделать inner call ошибочным, хотя отдельно builder вывел бы другой type.

## Lambda with receiver

Собственный builder обычно принимает `Builder<T>.() -> Unit`.

```kotlin
class ItemHolder<T> {
    private val items = mutableListOf<T>()

    fun add(item: T) {
        items += item
    }

    fun snapshot(): List<T> = items.toList()
}

fun <T> buildItems(block: ItemHolder<T>.() -> Unit): List<T> {
    return ItemHolder<T>().apply(block).snapshot()
}
```

Usage:

```kotlin
val titles = buildItems {
    add("Generics")
    add("Builder inference")
}
```

`T = String` выводится из receiver method `add(item: T)`.

## Требования к receiver type

Чтобы собственный builder участвовал в inference:

1. lambda имеет receiver type;
2. receiver использует type parameters, которые нужно вывести;
3. receiver предоставляет доступные members/extensions с этими parameters в signatures;
4. body создаёт constraints через arguments, expected types или связанные calls.

```kotlin
class LessonCollector<T> {
    private val values = mutableListOf<T>()

    fun emit(value: T) {
        values += value
    }

    fun lastOrNull(): T? = values.lastOrNull()

    fun result(): List<T> = values.toList()
}

fun <T> collectLessons(
    block: LessonCollector<T>.() -> Unit
): List<T> {
    return LessonCollector<T>().apply(block).result()
}
```

`emit` даёт input constraint, `lastOrNull` — output, который можно связать с expected type.

## Postponed type variable

Во время анализа body точный `T` временно неизвестен. Compiler рассматривает его как postponed type variable и накапливает constraints.

```kotlin
val titles = buildList {
    val first = getOrNull(0)
    val expected: String? = first
}
```

Assignment сообщает, что element type должен соответствовать String. В конце compiler решает constraints и получает `List<String>`.

Пока type postponed, гарантированно доступны root operations `equals`, `hashCode`, `toString`; specialized extensions становятся доступны, когда receiver constraint связывает type.

## Constraint из input method

```kotlin
val result = buildList {
    add("value")
}
```

`add` принимает E, передан String: E должен быть String или подходящим supertype. Compiler выбирает наиболее specific solution с учётом всех constraints.

## Constraint из output expected type

```kotlin
val result = buildList {
    val first: Float = get(0)
}
```

`get` возвращает E, assignment требует Float, поэтому E выводится Float.

Код не выполняется в примере без element и упадёт runtime при вызове builder, поэтому production-вариант использует безопасный `getOrNull` или предварительный `add`:

```kotlin
val safeResult = buildList {
    add(1.5f)
    val first: Float = first()
    check(first > 0f)
}
```

## Constraint через extension

```kotlin
fun String.lessonSlug(): String {
    return trim().lowercase().replace(' ', '-')
}

val titles = buildList {
    add("Builder inference")
    val slug = first().lessonSlug()
    println(slug)
}
```

`lessonSlug` требует String receiver и помогает подтвердить E = String.

## Несколько type parameters

```kotlin
class PairBuilder<K, V> {
    private val values = linkedMapOf<K, V>()

    fun put(key: K, value: V) {
        values[key] = value
    }

    fun result(): Map<K, V> = values.toMap()
}

fun <K, V> buildPairs(
    block: PairBuilder<K, V>.() -> Unit
): Map<K, V> {
    return PairBuilder<K, V>().apply(block).result()
}

val topics = buildPairs {
    put(37, "Definitely non-nullable types")
    put(38, "Builder inference")
}
```

Constraints independently выводят K = Int, V = String.

## Несколько builder lambdas

Inference может связывать parameters между несколькими receiver lambdas одного вызова.

```kotlin
fun <K, V> buildIndex(
    valuesBlock: MutableList<V>.() -> Unit,
    indexBlock: MutableMap<K, V>.() -> Unit
): Pair<List<V>, Map<K, V>> {
    val values = mutableListOf<V>().apply(valuesBlock)
    val index = mutableMapOf<K, V>().apply(indexBlock)
    return values.toList() to index.toMap()
}

val result = buildIndex(
    valuesBlock = {
        add("Builder inference")
    },
    indexBlock = {
        put(38, "Builder inference")
    }
)
```

V constraints из обеих lambdas согласуются как String, K выводится Int.

## Противоречивые constraints

```kotlin
val mixed = buildList {
    add(1)
    add("Kotlin")
}
```

Compiler может вывести общий supertype, например `Any`, в зависимости от context и inference rules. Это не всегда желаемо: mixed list теряет specialized operations.

Явный expected type документирует намерение:

```kotlin
val mixed: List<Any> = buildList {
    add(1)
    add("Kotlin")
}
```

Если constraints несовместимы с bound или expected type, compiler сообщает inference/type mismatch error.

```kotlin
fun <T : CharSequence> buildText(
    block: MutableList<T>.() -> Unit
): List<T> = buildList(block)

// buildText { add(42) } не удовлетворяет T : CharSequence.
```

## Пустой builder и отсутствие constraints

```kotlin
// val empty = buildItems { }
```

Compiler может не вывести T. Дай type argument или expected type:

```kotlin
val emptyTitles = buildItems<String> { }
val emptyScores: List<Int> = buildItems { }
```

<MouseTrap title="Мышеловка: пустое тело не сообщает тип">
Builder inference не угадывает domain intention. Если ни body, ни expected context не создают constraint, укажи type явно.
</MouseTrap>

## Unsupported receiver shape

Форма, где receiver — сам голый type parameter, не является обычным supported builder pattern:

```kotlin
// fun <T> buildUnknown(block: T.() -> Unit): T
```

Compiler не знает, какой объект создать и какие public members T должны участвовать в constraints. Используй named generic builder type:

```kotlin
class ValueBuilder<T> {
    private var value: T? = null

    fun set(value: T) {
        this.value = value
    }

    fun requireValue(): T {
        return value ?: error("Значение не задано")
    }
}
```

Для nullable T sentinel `null` неоднозначен; production builder может хранить separate boolean/sealed slot.

## Public members влияют на inference

Receiver должен expose operations с T. Private member builder недоступен lambda и не создаёт call-site constraint.

```kotlin
class PublicCollector<T> {
    private val values = mutableListOf<T>()

    fun add(value: T) {
        values += value
    }
}
```

Extension тоже может внести constraint:

```kotlin
fun <T> PublicCollector<T>.addAll(values: Iterable<T>) {
    values.forEach(::add)
}

val lessons = PublicCollector<String>().apply {
    addAll(listOf("Generics", "Inference"))
}
```

## `@BuilderInference`: историческая роль и deprecation

В старых Kotlin versions annotation явно включала builder inference для lambda parameter:

```text
@OptIn(kotlin.experimental.ExperimentalTypeInference::class)
fun <T> legacyBuilder(
    @kotlin.BuilderInference block: ItemHolder<T>.() -> Unit
): List<T> {
    return ItemHolder<T>().apply(block).snapshot()
}
```

Это архивная форма для чтения старого source, а не пример для нового проекта. В актуальном Kotlin builder inference включается автоматически, если обычного inference недостаточно. `@BuilderInference` deprecated: новые API не должны её добавлять; в новых compiler lineages deprecation усиливается до ошибки/скрытия.

Если поддерживается старый compiler, annotation может встречаться в legacy source. Migration:

1. обновить Kotlin toolchain;
2. удалить annotation и experimental opt-in;
3. собрать call sites;
4. добавить explicit types только там, где inference объективно неоднозначен.

<CatTip title="Котосовет: не копируй устаревшую BuilderInference-аннотацию">
Проектируй receiver API так, чтобы type parameters участвовали в public method signatures. Современный compiler активирует builder inference автоматически.
</CatTip>

## Builder inference не равен `@DslMarker`

Builder inference решает types. `@DslMarker` ограничивает неявные receivers во вложенных DSL.

```kotlin
@DslMarker
annotation class CourseDsl

@CourseDsl
class CourseBuilder
```

Они решают разные проблемы и часто используются вместе.

## Вложенные builders и receiver ambiguity

```kotlin
@DslMarker
annotation class LessonDsl

@LessonDsl
class CourseBuilder {
    private val lessons = mutableListOf<Lesson>()

    fun lesson(block: LessonBuilder.() -> Unit) {
        lessons += LessonBuilder().apply(block).build()
    }

    fun build(): List<Lesson> = lessons.toList()
}

@LessonDsl
class LessonBuilder {
    var title: String = ""

    fun build(): Lesson {
        require(title.isNotBlank()) { "Название обязательно" }
        return Lesson(title.trim())
    }
}

data class Lesson(val title: String)
```

`@DslMarker` предотвращает случайное обращение к outer receiver, inference определяет generic types там, где builders generic.

## Overloads и diagnostics

Overloaded receiver methods могут дать неоднозначные constraints:

```kotlin
class AmbiguousBuilder<T> {
    fun add(value: T) = Unit
    fun add(values: List<T>) = Unit
}
```

Вызов `add(emptyList())` не сообщает element type без context. Укажи:

```kotlin
val builder = AmbiguousBuilder<String>()
builder.add(emptyList<String>())
```

При сложной ошибке inference:

1. добавь expected type result;
2. укажи type arguments builder call;
3. типизируй промежуточную variable внутри lambda;
4. проверь conflicting add/put calls;
5. сократи overload set;
6. убедись, что receiver methods expose T;
7. минимизируй nested lambdas для диагностики.

## Common supertype и widening

```kotlin
open class Content
class Lesson : Content()
class Quiz : Content()

val contents: List<Content> = buildList {
    add(Lesson())
    add(Quiz())
}
```

Expected `List<Content>` предотвращает вывод неудобного intersection/common supertype. Для public code явный abstraction часто лучше сложного inferred type.

## API design builders

Хороший generic builder:

- возвращает immutable snapshot;
- не отдаёт mutable receiver наружу;
- проверяет invariants в `build`;
- имеет operations с T в signatures;
- избегает ambiguous overloads;
- использует DSL marker для nested receivers;
- не прячет I/O в setters;
- даёт explicit type fallback.

## Практический пример: generic validation DSL

```kotlin
data class ValidationIssue(
    val rule: String,
    val message: String
)

fun interface Rule<T> {
    fun validate(value: T): ValidationIssue?
}

class RulesBuilder<T> {
    private val rules = mutableListOf<Rule<T>>()

    fun rule(name: String, predicate: (T) -> Boolean, message: String) {
        require(name.isNotBlank()) { "Имя правила обязательно" }
        require(message.isNotBlank()) { "Сообщение обязательно" }

        rules += Rule { value ->
            if (predicate(value)) {
                null
            } else {
                ValidationIssue(name, message)
            }
        }
    }

    fun include(rule: Rule<T>) {
        rules += rule
    }

    fun build(): List<Rule<T>> = rules.toList()
}

fun <T> rules(block: RulesBuilder<T>.() -> Unit): List<Rule<T>> {
    return RulesBuilder<T>().apply(block).build()
}

fun <T> validate(value: T, rules: List<Rule<T>>): List<ValidationIssue> {
    return rules.mapNotNull { rule -> rule.validate(value) }
}

data class LessonDraft(
    val title: String,
    val content: String
)

fun main() {
    val lessonRules = rules {
        rule(
            name = "title-required",
            predicate = { draft: LessonDraft -> draft.title.isNotBlank() },
            message = "Название обязательно"
        )
        rule(
            name = "content-required",
            predicate = { draft -> draft.content.isNotBlank() },
            message = "Содержание обязательно"
        )
    }

    val issues = validate(
        value = LessonDraft("Builder inference", ""),
        rules = lessonRules
    )

    issues.forEach(::println)
}
```

Как выводится T:

- `rules` сначала имеет postponed T;
- explicit parameter type первой predicate задаёт `T = LessonDraft`;
- вторая predicate получает уже известный LessonDraft;
- result становится `List<Rule<LessonDraft>>`;
- `validate` связывает value и rules тем же T;
- deprecated annotation не нужна.

Можно задать type снаружи и не аннотировать первую lambda:

```kotlin
val lessonRules: List<Rule<LessonDraft>> = rules {
    rule("title-required", { it.title.isNotBlank() }, "Название обязательно")
}
```

## Итоги

- builder inference собирает type constraints внутри receiver lambda;
- regular inference и expected types используются первыми;
- неизвестный type временно является postponed variable;
- input methods, output expected types и extensions дают constraints;
- один builder может выводить несколько parameters;
- несколько lambdas могут связывать общие parameters;
- пустой builder требует explicit context;
- contradictory calls приводят к common supertype или ошибке;
- receiver должен быть named generic type с доступными T-members;
- `@BuilderInference` исторически включала механизм, но теперь deprecated;
- современный Kotlin активирует builder inference автоматически;
- `@DslMarker` управляет receivers, а не type inference;
- явный type остаётся правильным инструментом для API clarity и diagnostics.
