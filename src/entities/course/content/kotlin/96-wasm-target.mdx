## Что такое Kotlin/Wasm

Kotlin/Wasm компилирует код Kotlin в WebAssembly.

Wasm — это переносимый формат двоичных инструкций для виртуальной машины на основе стека. Kotlin/Wasm позволяет Kotlin работать:

- в браузерах через `wasmJs`;
- вне браузера через `wasmWasi`;
- как создание многоплатформенных веб-приложений;
- во средах выполнения, которые поддерживают необходимые предложения Wasm.

```kotlin
kotlin {
    wasmJs {
        browser()
        binaries.executable()
    }
}
```

Kotlin/Wasm is Beta, so it is practical for many experiments and growing боевой scenarios, but you should still track compatibility notes and browser/runtime requirements.

<CatTip>
Wasm target — это быстрый кошачий лифт: Kotlin code поднимается в browser runtime не через обычный JavaScript, а через WebAssembly.
</CatTip>

## `wasmJs`

`wasmJs` targets JavaScript-compatible Wasm runtime, usually browser.

```kotlin
kotlin {
    wasmJs {
        browser {
            testTask {
                useKarma {
                    useChromeHeadless()
                }
            }
        }

        binaries.executable()
    }
}
```

Source set:

```text
src/
  commonMain/
    kotlin/
  wasmJsMain/
    kotlin/
  wasmJsTest/
    kotlin/
```

Пример взаимодействия с браузером:

```kotlin
// wasmJsMain/kotlin/browser/WasmDomTitle.kt
package browser

import kotlinx.browser.document

fun updateTitle(title: String) {
    // Browser declarations can be used from wasmJs browser source set.
    document.title = title
}
```

## Browser runtime requirements

Приложениям браузера Kotlin/Wasm требуется поддержка браузером современных функций WebAssembly, включая предложения по сбору мусора и обработке исключений, необходимые Kotlin.

Practical consequences:

- test real target browsers;
- document minimum browser versions;
- сохраните запасной вариант JS, если продукт должен поддерживать старые браузеры;
- не предполагайте, что все встроенные веб-просмотры поддерживают одни и те же функции Wasm;
- Следите за примечаниями к выпуску Kotlin/Wasm при обновлении Kotlin.

```kotlin
// commonMain/kotlin/web/RuntimeChoice.kt
package web

sealed interface WebRuntime {
    data object Wasm : WebRuntime

    data object JavaScriptFallback : WebRuntime
}
```

<MouseTrap>
Wasm support is not just «browser supports WebAssembly». Kotlin/Wasm needs specific Wasm proposals. Старые webviews can be the real боевой trap.
</MouseTrap>

## Создание мультиплатформы на Wasm

Compose Multiplatform uses Kotlin/Wasm as primary web target for общий UI.

Conceptual настройка:

```kotlin
kotlin {
    wasmJs {
        browser()
        binaries.executable()
    }

    sourceSets {
        commonMain.dependencies {
            implementation(compose.runtime)
            implementation(compose.foundation)
            implementation(compose.material3)
        }
    }
}
```

Общий UI:

```kotlin
// commonMain/kotlin/ui/LessonBadge.kt
package ui

import androidx.compose.material3.Text
import androidx.compose.runtime.Composable

@Composable
fun LessonBadge(completed: Boolean) {
    val text = if (completed) {
        "Урок завершен"
    } else {
        "Урок в процессе"
    }

    Text(text)
}
```

Even with общий UI, platform boundaries remain:

- browser storage;
- routing integration;
- permissions;
- file picking;
- аналитика SDKs;
- push notifications;
- platform-specific performance tuning.

## `wasmWasi`

`wasmWasi` предназначен для WASI, системного интерфейса WebAssembly, для сред выполнения без браузера.

```kotlin
kotlin {
    wasmWasi {
        nodejs()
        binaries.executable()
    }
}
```

Сценарии использования:

- sandboxed command-line tools;
- server-side Wasm experiments;
- edge runtime experiments;
- portable computation modules;
- среды, предоставляющие API-интерфейсы WASI.

WASI предоставляет стандартизированный доступ к платформенным возможностям без прямой привязки кода к API конкретной ОС.

## Взаимодействие с JavaScript из Wasm

Kotlin/Wasm поддерживает взаимодействие с JavaScript, но подписи более ограничены, чем в обычном Kotlin.

External declaration:

```kotlin
// wasmJsMain/kotlin/interop/BrowserConsole.kt
package interop

external object console {
    fun log(message: String)
}

fun logFromWasm(message: String) {
    console.log(message)
}
```

Inline JS expression:

```kotlin
// wasmJsMain/kotlin/interop/Now.kt
package interop

fun nowMillis(): Double = js("Date.now()")
```

Сохраняйте взаимодействие JS узким и типизированным. Модуль Wasm с большой динамической поверхностью JS становится трудно поддерживать.

## Type limitations

Взаимодействие Kotlin/Wasm JS поддерживает ограниченный набор типов сигнатур.

Practical friendly types:

- `Int`;
- `Double`;
- `Boolean`;
- `String`;
- `Unit` как тип возвращаемого значения;
- типы взаимодействия `JsAny` и JS, где это необходимо;
— экспортированные классы, специально разработанные для границ JS.

Избегайте экспорта произвольных графов домена Kotlin напрямую.

```kotlin
package exports

import kotlin.js.JsExport

@JsExport
class WasmLessonCard(
    val slug: String,
    val title: String
)

@JsExport
fun makeCard(slug: String, title: String): WasmLessonCard {
    return WasmLessonCard(slug, title)
}
```

## Browser API support

Kotlin/Wasm standard library provides browser API declarations, including DOM-related APIs.

```kotlin
// wasmJsMain/kotlin/browser/ProgressElement.kt
package browser

import kotlinx.browser.document

fun renderProgress(elementId: String, completed: Int, total: Int) {
    val percent = if (total == 0) 0 else completed * 100 / total
    val element = document.getElementById(elementId)
        ?: error("Element '$elementId' не найден")

    element.textContent = "Прогресс: $percent%"
}
```

Используйте это из исходного набора браузера, а не из модуля общего домена.

## NPM dependencies

В Kotlin/Wasm улучшена поддержка зависимостей npm, но совместимость библиотек по-прежнему зависит от формы взаимодействия и предположений времени выполнения.

```kotlin
kotlin {
    wasmJs {
        browser()

        sourceSets {
            wasmJsMain.dependencies {
                implementation(npm("nanoid", "6.0.0"))
            }
        }
    }
}
```

Оберни использование npm:

```kotlin
// wasmJsMain/kotlin/external/NanoId.kt
@file:JsModule("nanoid")

package external

external fun nanoid(): String
```

```kotlin
// wasmJsMain/kotlin/platform/WasmIdGenerator.kt
package platform

import external.nanoid

class WasmIdGenerator {
    fun nextId(): String {
        // npm dependency hidden behind typed platform adapter.
        return nanoid()
    }
}
```

<MouseTrap>
Не ожидай, что любой npm package автоматически удобен для Wasm. Если package relies on Node globals, dynamic runtime tricks or unsupported types, wrapper can become fragile.
</MouseTrap>

## Wasm и общие исходные наборы

Проект KMP может совместно использовать код JS и Wasm через общие наборы исходных кодов.

```text
commonMain
  webMain
    jsMain
    wasmJsMain
  jvmMain
```

Gradle:

```kotlin
kotlin {
    js {
        browser()
    }

    wasmJs {
        browser()
    }

    sourceSets {
        val commonMain by getting
        val jsMain by getting
        val wasmJsMain by getting

        val webMain by creating {
            dependsOn(commonMain)
        }

        jsMain.dependsOn(webMain)
        wasmJsMain.dependsOn(webMain)
    }
}
```

Поместите независимую от браузера веб-логику в `webMain`, взаимодействие, специфичное для JS, в `jsMain`, взаимодействие, специфичное для Wasm, в `wasmJsMain`.

## Current constraints

Отслеживайте эти ограничения:

- Kotlin/Wasm является бета-версией;
- browser/runtime support matters;
- Сигнатуры типов взаимодействия JS ограничены;
- отладка улучшается, но отличается от JVM;
— библиотеки экосистемы могут пока не поддерживать Wasm;
- размер двоичного файла и запуск требуют реального измерения;
- собственные API браузера по-прежнему нуждаются в обертках;
- старым браузерам может потребоваться резервный вариант Kotlin/JS.

Боевой readiness is not a checkbox. It depends on your browser matrix, performance budget, library dependencies and release process.

## Проверочный список для боевого кода

Для цели Wasm:

- намеренно объявить `wasmJs` или `wasmWasi`;
- протестируйте реальные браузеры или рантайм-версии;
- isolate JS interop;
- сохранять экспортируемый API простым;
- измерить размер запуска и пакета;
— решить, нужен ли резервный вариант JS;
- не предполагайте, что API-интерфейсы Node и браузера взаимозаменяемы;
- документировать текущие ограничения для продукта и обеспечения качества.

<CatTip>
Wasm эффективен, когда рассматривается как реальная цель, а не как флаг новинки. Дайте ему тесты, матрицу браузера, типизированные границы и бюджеты производительности, и он будет мурлыкать гораздо надежнее.
</CatTip>
