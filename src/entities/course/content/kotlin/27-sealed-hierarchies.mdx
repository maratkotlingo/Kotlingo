## Иерархия с контролируемым набором вариантов

`sealed class` и `sealed interface` ограничивают множество прямых subclasses. Compiler знает все разрешённые прямые варианты и может проверить исчерпывающий `when`.

```kotlin
sealed interface LoadState {
    data object Idle : LoadState
    data object Loading : LoadState
    data class Success(val title: String) : LoadState
    data class Failure(val message: String) : LoadState
}
```

В отличие от enum, каждый subtype может иметь собственную структуру:

- `Idle` и `Loading` не несут payload;
- `Success` хранит title;
- `Failure` хранит message;
- instances data subclasses создаются для конкретных данных.

## Closed hierarchy не означает один instance

Sealed ограничивает **типы**, а не количество объектов.

```kotlin
val first = LoadState.Success("Data classes")
val second = LoadState.Success("Enum classes")
```

Оба объекта имеют subtype `Success`, но разные payload.

Singleton-варианты объявляются `object`/`data object`:

```kotlin
val loading = LoadState.Loading
```

## `sealed class`

Sealed class может иметь constructor, state и concrete members.

```kotlin
sealed class ValidationResult(val valid: Boolean) {
    data object Valid : ValidationResult(valid = true)

    data class Invalid(
        val errors: List<String>
    ) : ValidationResult(valid = false)

    fun requireValid() {
        check(valid) { "Результат содержит ошибки" }
    }
}
```

Constructor sealed class по умолчанию имеет protected visibility. Его можно сделать private, но `public`/`internal` constructor sealed class недопустим: произвольные клиенты не должны создавать base instance.

```kotlin
sealed class Token private constructor() {
    data object Start : Token()
    data object End : Token()
}
```

Sealed class полезен, когда вариантам нужны общие state, protected helpers или единый constructor contract.

## `sealed interface`

Sealed interface не имеет constructor/state и позволяет implementations наследовать другой class.

```kotlin
sealed interface CourseEvent

data class LessonOpened(val id: Int) : CourseEvent

data class LessonCompleted(val id: Int, val score: Int) : CourseEvent
```

Один class может реализовать несколько interfaces, поэтому sealed interface легче комбинировать с capabilities.

```kotlin
interface Auditable {
    val auditMessage: String
}

data class CoursePublished(val courseId: Int) : CourseEvent, Auditable {
    override val auditMessage: String = "Опубликован курс $courseId"
}
```

## Правила прямых наследников

Прямые subclasses sealed type должны:

- быть объявлены в том же module;
- находиться в том же package;
- иметь proper qualified name;
- не быть local или anonymous classes;
- соблюдать обычную visibility base type.

Они могут находиться в другом файле того же package/module, а не обязательно внутри base declaration.

```kotlin
// Файл CourseCommand.kt, package kotlingo.course
sealed interface CourseCommand

// Файл PublishCourse.kt, тот же package и module
data class PublishCourse(val id: Int) : CourseCommand
```

Для multiplatform direct subclasses также связаны source set rules: ожидаемые/фактические declarations и hierarchy должны учитывать source-set boundaries.

<MouseTrap title="Мышеловка: sealed не ограничен одним файлом">
Современный Kotlin разрешает прямых наследников в разных файлах, если package и module совпадают. Поэтому при анализе hierarchy ищи по всему package, а не только рядом с declaration.
</MouseTrap>

## Косвенные наследники

Ограничение относится к direct subclasses. Direct subclass может быть `open`, после чего его subclasses подчиняются обычным правилам.

```kotlin
sealed interface ApiError

open class ServerError(
    val status: Int,
    val message: String
) : ApiError

class GatewayError(message: String) : ServerError(502, message)
```

Exhaustive `when` по `ApiError` обязан обработать direct branch `ServerError`, но не перечислять каждый будущий subtype `ServerError`.

Если нужен полностью закрытый leaf set, direct branches тоже делай final/data/object/sealed appropriately.

## Exhaustive `when`

```kotlin
fun render(state: LoadState): String = when (state) {
    LoadState.Idle -> "Ожидание"
    LoadState.Loading -> "Загрузка"
    is LoadState.Success -> "Урок: ${state.title}"
    is LoadState.Failure -> "Ошибка: ${state.message}"
}
```

`else` не нужен. Если добавить subtype `Cancelled`, compiler потребует обновить `when`.

Внутри type branch работает smart cast: у `Success` доступен `title`, у `Failure` — `message`.

### Nullable sealed type

`LoadState?` добавляет отдельный вариант `null`:

```kotlin
fun renderNullable(state: LoadState?): String = when (state) {
    null -> "Состояние не задано"
    LoadState.Idle -> "Ожидание"
    LoadState.Loading -> "Загрузка"
    is LoadState.Success -> state.title
    is LoadState.Failure -> state.message
}
```

Nullability не является subtype sealed hierarchy, поэтому обрабатывается явно.

## Почему не стоит писать `else`

```kotlin
fun unsafeRender(state: LoadState): String = when (state) {
    LoadState.Loading -> "Загрузка"
    else -> "Другое"
}
```

При новом subtype compile-time reminder не появится: он попадёт в `else`. Для closed domain model перечисляй все branches.

`else` оправдан, если business semantics сознательно объединяет все остальные варианты и добавление нового варианта действительно должно автоматически следовать той же ветке. Такое решение стоит документировать тестом.

<CatTip title="Котосовет: exhaustive when — бесплатная миграционная карта">
Не скрывай варианты catch-all веткой. При расширении модели compiler покажет every сценарий использования, renderer и mapper, где нужно принять осознанное решение.
</CatTip>

## Sealed и `data object`

`data object` удобен для stateless variants:

```kotlin
sealed interface SaveResult {
    data object Saved : SaveResult
    data object NoChanges : SaveResult
    data class Rejected(val reason: String) : SaveResult
}
```

Почему не обычный `object`:

- generated `toString` возвращает stable declaration name без `@hash`;
- equality/hash semantics согласованы с data-ориентированной hierarchy;
- симметрия с data classes делает вывод отладки понятнее.

Для singleton identity обычный `object` тоже корректен; `data object` подчёркивает участие в value model.

У `data object` нет `copy` и `componentN`: единственный instance нечего копировать/деструктурировать.

## Sealed class против enum

Enum:

```kotlin
enum class SimpleState {
    IDLE,
    LOADING,
    SUCCESS,
    FAILURE
}
```

Не хранит данные конкретного success/error. Приходится создавать nullable side fields:

```kotlin
data class UnsafeUiState(
    val state: SimpleState,
    val title: String?,
    val error: String?
)
```

Так возникают недопустимые combinations: `SUCCESS` с `title = null`, `LOADING` с error.

Sealed model делает illegal states unrepresentable:

```kotlin
sealed interface UiState {
    data object Idle : UiState
    data object Loading : UiState
    data class Content(val title: String) : UiState
    data class Error(val message: String) : UiState
}
```

Enum лучше, если все variants действительно одинаковы и singleton. Enum class может реализовать sealed interface как один direct subtype family:

```kotlin
sealed interface AccessRule

enum class Role : AccessRule {
    STUDENT,
    MENTOR,
    ADMIN
}

data class FeatureFlag(val name: String) : AccessRule
```

Enum не может наследовать sealed class, потому что enum уже наследует `Enum`.

## Error model вместо exception-only API

Sealed result делает ожидаемые failures частью type contract.

```kotlin
sealed interface LessonLookupResult {
    data class Found(val lesson: Lesson) : LessonLookupResult
    data class NotFound(val id: Int) : LessonLookupResult
    data object Offline : LessonLookupResult
    data class InvalidId(val raw: String) : LessonLookupResult
}

data class Lesson(val id: Int, val title: String)
```

Client обязан рассмотреть variants:

```kotlin
fun message(result: LessonLookupResult): String = when (result) {
    is LessonLookupResult.Found -> result.lesson.title
    is LessonLookupResult.NotFound -> "Урок ${result.id} не найден"
    LessonLookupResult.Offline -> "Нет соединения"
    is LessonLookupResult.InvalidId -> "Некорректный id: ${result.raw}"
}
```

Expected failures лучше typed result; programming errors/invariant violations по-прежнему могут быть exceptions.

## Generic sealed result

```kotlin
sealed interface Result<out T> {
    data class Success<T>(val value: T) : Result<T>
    data class Failure(val error: DomainError) : Result<Nothing>
}

sealed interface DomainError {
    data object Offline : DomainError
    data class NotFound(val id: Int) : DomainError
    data class Validation(val messages: List<String>) : DomainError
}
```

Covariance `out T` и `Nothing` позволяют `Failure` быть результатом любого success type.

```kotlin
fun loadTitle(found: Boolean): Result<String> {
    return if (found) {
        Result.Success("Sealed classes")
    } else {
        Result.Failure(DomainError.NotFound(29))
    }
}
```

Generic variance подробно рассматривается позже.

## Domain states и state machine

```kotlin
sealed interface StudySessionState {
    data object NotStarted : StudySessionState
    data class Active(val lessonId: Int, val completed: Set<Int>) : StudySessionState
    data class Paused(val lessonId: Int, val completed: Set<Int>) : StudySessionState
    data class Finished(val completed: Set<Int>) : StudySessionState
}

sealed interface StudySessionEvent {
    data class Start(val lessonId: Int) : StudySessionEvent
    data class CompleteLesson(val nextLessonId: Int?) : StudySessionEvent
    data object Pause : StudySessionEvent
    data object Resume : StudySessionEvent
}
```

Transition function принимает state и event. Не все Cartesian combinations допустимы, поэтому forbidden transitions нужно возвращать typed error или exception according to contract.

```kotlin
sealed interface TransitionResult {
    data class Changed(val state: StudySessionState) : TransitionResult
    data class Rejected(val reason: String) : TransitionResult
}
```

Sealed models отделяют возможные данные от logic transitions.

## Command и event models

```kotlin
sealed interface LessonCommand {
    data class Rename(val title: String) : LessonCommand
    data object Publish : LessonCommand
    data class AddTag(val tag: String) : LessonCommand
}

fun commandName(command: LessonCommand): String = when (command) {
    is LessonCommand.Rename -> "Переименование"
    LessonCommand.Publish -> "Публикация"
    is LessonCommand.AddTag -> "Добавление метки"
}
```

Каждый command содержит только необходимые данные. Нет generic map/string action с runtime casts.

## Visibility sealed variants

Base public sealed type может иметь variants разной visibility, но public API и exhaustive handling должны оставаться применимыми для clients. Скрытый direct subtype в public sealed hierarchy способен усложнить внешний exhaustive `when`, поскольку client не может назвать branch.

Для library design либо делай все relevant variants public, либо не обещай внешнему клиенту exhaustive decomposition и предоставь methods/visitor-like API.

## Эволюция public sealed API

Добавление нового direct subtype source-compatible не для клиентов с exhaustive `when`: после recompilation их code перестанет компилироваться, пока не добавлена branch. Уже скомпилированный client может столкнуться с runtime behaviour, зависящим от generated `when` mapping/versioning.

Поэтому расширение public sealed hierarchy — изменение API, требующее release notes и compatibility анализа.

Если third parties должны добавлять implementations, sealed type не подходит. Используй обычный interface и fallback strategy.

## Serialization sealed hierarchy

Serializer должен записывать discriminator, например `type`, чтобы восстановить subtype.

Conceptual JSON:

```json
{
  "type": "not_found",
  "id": 29
}
```

Нужно продумать:

- стабильные serial names независимо от class rename;
- unknown subtype от более новой версии producer;
- значения по умолчанию и missing fields;
- migration старых variants;
- security: нельзя без whitelist создавать произвольный class по имени.

Sealed помогает serializer построить whitelist, но format определяется framework annotations/configuration.

## Nested и top-level variants

Nested style группирует namespace:

```kotlin
sealed interface AuthenticationResult {
    data class Success(val userId: String) : AuthenticationResult
    data object InvalidCredentials : AuthenticationResult
}
```

Top-level style уменьшает длинные names и может удобнее разделяться по файлам:

```kotlin
sealed interface SyncResult
data object SyncComplete : SyncResult
data class SyncFailed(val reason: String) : SyncResult
```

Выбор организационный. Для небольшого tightly coupled result nested variants читаются ясно. Большая domain hierarchy может жить в нескольких files одного package.

## Практический пример: загрузка урока с typed errors

```kotlin
data class Lesson(
    val id: Int,
    val title: String,
    val content: String
)

sealed interface LoadLessonError {
    data class InvalidId(val raw: String) : LoadLessonError
    data class NotFound(val id: Int) : LoadLessonError
    data object Offline : LoadLessonError
    data class StorageFailure(val diagnosticId: String) : LoadLessonError
}

sealed interface LoadLessonResult {
    data class Success(val lesson: Lesson) : LoadLessonResult
    data class Failure(val error: LoadLessonError) : LoadLessonResult
}

interface LessonSource {
    fun find(id: Int): Lesson?
    fun isAvailable(): Boolean
}

class InMemoryLessonSource(
    private val lessons: Map<Int, Lesson>,
    private val available: Boolean = true
) : LessonSource {
    override fun find(id: Int): Lesson? = lessons[id]
    override fun isAvailable(): Boolean = available
}

fun loadLesson(rawId: String, source: LessonSource): LoadLessonResult {
    val id = rawId.trim().toIntOrNull()
        ?: return LoadLessonResult.Failure(LoadLessonError.InvalidId(rawId))

    if (id <= 0) {
        return LoadLessonResult.Failure(LoadLessonError.InvalidId(rawId))
    }

    if (!source.isAvailable()) {
        return LoadLessonResult.Failure(LoadLessonError.Offline)
    }

    return try {
        val lesson = source.find(id)
            ?: return LoadLessonResult.Failure(LoadLessonError.NotFound(id))
        LoadLessonResult.Success(lesson)
    } catch (error: RuntimeException) {
        val diagnosticId = "lesson-$id-${error::class.simpleName}"
        LoadLessonResult.Failure(
            LoadLessonError.StorageFailure(diagnosticId)
        )
    }
}

fun userMessage(result: LoadLessonResult): String = when (result) {
    is LoadLessonResult.Success -> "Открыт урок: ${result.lesson.title}"
    is LoadLessonResult.Failure -> when (val error = result.error) {
        is LoadLessonError.InvalidId -> "Некорректный id: ${error.raw}"
        is LoadLessonError.NotFound -> "Урок ${error.id} не найден"
        LoadLessonError.Offline -> "Нет соединения. Повтори позже"
        is LoadLessonError.StorageFailure -> {
            "Не удалось прочитать данные. Код: ${error.diagnosticId}"
        }
    }
}

fun main() {
    val source = InMemoryLessonSource(
        lessons = mapOf(
            29 to Lesson(
                id = 29,
                title = "Sealed classes и sealed interfaces",
                content = "Закрытые иерархии и exhaustive when"
            )
        )
    )

    val success = loadLesson("29", source)
    val missing = loadLesson("999", source)
    val invalid = loadLesson("cat", source)

    listOf(success, missing, invalid)
        .map(::userMessage)
        .forEach(::println)
}
```

Design properties:

- parsing, absence, connectivity и infrastructure failure различены типами;
- success всегда содержит fully formed Lesson;
- failure всегда содержит конкретный error;
- `userMessage` exhaustive на обоих уровнях;
- добавление error заставит обновить rendering;
- sensitive exception text не отдаётся user, используется diagnostic id;
- expected failures не управляются catch в client code.

## Итоги

- sealed class/interface контролирует direct subtype set;
- sealed class может иметь constructor/state, interface легче комбинировать;
- прямые наследники находятся в том же package и module;
- indirect branch может оставаться open, если это задумано;
- exhaustive `when` не требует `else` и даёт smart casts;
- nullable sealed value требует branch `null`;
- `data object` подходит stateless value variant;
- enum хорош для однотипных singletons, sealed — для разных payload;
- sealed results делают expected failures частью type contract;
- state models исключают некорректные combinations;
- public sealed hierarchy остаётся API, добавление subtype влияет на clients;
- serialization требует стабильного discriminator и unknown-variant strategy.
