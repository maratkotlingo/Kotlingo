## Функция как значение

Kotlin позволяет не только вызвать функцию, но и передать саму операцию в другую функцию, сохранить её в переменной или вернуть как результат. Callable reference создаётся оператором `::`.

```kotlin
fun normalizeTitle(raw: String): String {
    return raw.trim().replaceFirstChar { char -> char.uppercase() }
}

fun main() {
    val normalizer: (String) -> String = ::normalizeTitle
    println(normalizer("  callable references  "))
}
```

`::normalizeTitle` не вызывает `normalizeTitle`. Он создаёт ссылку, которую можно вызвать позже через `normalizer(...)` или `normalizer.invoke(...)`.

Callable references охватывают:

- top-level и локальные функции;
- member-функции;
- extension-функции;
- properties;
- constructors;
- перегруженные функции при наличии достаточного контекста.

## Ссылка и вызов

Различие видно по синтаксису:

```kotlin
fun currentLesson(): String = "Callable references"

fun main() {
    val value: String = currentLesson()
    val provider: () -> String = ::currentLesson

    println(value)
    println(provider())
}
```

- `currentLesson()` сразу выполняет функцию;
- `::currentLesson` создаёт объект, совместимый с `() -> String`;
- `provider()` вызывает сохранённую операцию.

## Top-level function reference

```kotlin
fun isPassing(score: Int): Boolean {
    return score >= 60
}

fun main() {
    val scores = listOf(45, 60, 72, 100)
    val passingScores = scores.filter(::isPassing)

    println(passingScores)
}
```

`filter` ожидает `(Int) -> Boolean`, и сигнатура `isPassing` ей соответствует.

Обычная лямбда эквивалентна по поведению:

```kotlin
val passingScores = scores.filter { score -> isPassing(score) }
```

Reference короче, когда существующая функция уже точно выражает нужную операцию. Лямбда лучше, когда нужно адаптировать аргументы, добавить условие или дать локальному параметру содержательное имя.

<CatTip title="Котосовет: ссылка должна упрощать чтение">
`items.map(::normalizeTitle)` хорошо читается, если имя функции раскрывает действие. Если читателю приходится переходить к объявлению ради понимания одного простого преобразования, короткая лямбда может быть яснее.
</CatTip>

## Ссылки на локальные функции

Локальная функция может быть передана по ссылке в пределах своей области видимости.

```kotlin
fun validateScores(scores: List<Int>): List<Int> {
    fun isValid(score: Int): Boolean {
        return score in 0..100
    }

    return scores.filter(::isValid)
}
```

Локальная функция может захватывать переменные окружающего блока:

```kotlin
fun scoresAtLeast(scores: List<Int>, threshold: Int): List<Int> {
    fun accepted(score: Int): Boolean {
        return score >= threshold
    }

    return scores.filter(::accepted)
}
```

Захват имеет те же соображения по lifecycle и изменяемое состояние, что и лямбда.

## Bound member reference

Bound reference привязана к конкретному объекту.

```kotlin
class Lesson(private val title: String) {
    fun label(prefix: String): String {
        return "$prefix: $title"
    }
}

fun main() {
    val lesson = Lesson("Callable references")
    val labeler: (String) -> String = lesson::label

    println(labeler("Урок"))
}
```

Receiver `lesson` уже сохранён в reference, поэтому функциональный тип содержит только обычный параметр `prefix`.

Bound reference удерживает ссылку на receiver. Если сохранить её в долгоживущем объекте, receiver также останется достижимым для garbage collector.

<MouseTrap title="Мышеловка: bound reference удерживает объект">
Не сохраняй `screen::render` или `activity::handle` в глобальном registry без управления жизненным циклом. Такая ссылка может удерживать весь объектный граф. Для подписок предусматривай явную отписку.
</MouseTrap>

## Unbound member reference

Unbound reference не привязана к экземпляру. Receiver становится первым параметром функционального типа.

```kotlin
data class Lesson(val title: String, val order: Int) {
    fun displayName(): String {
        return "$order. $title"
    }
}

fun main() {
    val display: (Lesson) -> String = Lesson::displayName
    val lesson = Lesson("Callable references", 21)

    println(display(lesson))
}
```

Для member с параметрами receiver идёт первым:

```kotlin
class Score(private val value: Int) {
    fun isAtLeast(threshold: Int): Boolean {
        return value >= threshold
    }
}

val predicate: (Score, Int) -> Boolean = Score::isAtLeast
```

Вызов: `predicate(Score(80), 60)`.

## Статический тип receiver

Как и обычный member lookup, unbound reference строится от указанного типа.

```kotlin
open class Content {
    open fun kind(): String = "материал"
}

class VideoLesson : Content() {
    override fun kind(): String = "видеоурок"
}

fun main() {
    val kindOf: (Content) -> String = Content::kind
    println(kindOf(VideoLesson()))
}
```

Сигнатура выбрана как `Content::kind`, но `kind` виртуальна, поэтому при вызове происходит обычный runtime dispatch и выполняется override `VideoLesson.kind`.

## Ссылки на extension-функции

Для top-level extension receiver также становится параметром unbound reference.

```kotlin
fun String.toLessonSlug(): String {
    return trim().lowercase().replace(' ', '-')
}

fun main() {
    val slugger: (String) -> String = String::toLessonSlug
    println(slugger("Callable References"))
}
```

Extension выбирается статически, как при обычном вызове. Reference не превращает extension в виртуальный member.

Bound extension reference доступна через конкретный receiver:

```kotlin
val rawTitle = "  Kotlin Functions  "
val createSlug: () -> String = rawTitle::toLessonSlug
println(createSlug())
```

Доступность конкретной формы может зависеть от версии языка и вида callable; ожидаемый функциональный тип помогает компилятору.

## Ссылки на overloaded-функции

Имя перегруженной функции обозначает перегрузка set. Компилятору нужен контекст, чтобы выбрать одну сигнатуру.

```kotlin
fun parseLessonId(raw: String): Int {
    return raw.trim().toInt()
}

fun parseLessonId(raw: String, radix: Int): Int {
    return raw.trim().toInt(radix)
}

fun main() {
    val decimalParser: (String) -> Int = ::parseLessonId
    val radixParser: (String, Int) -> Int = ::parseLessonId

    println(decimalParser("21"))
    println(radixParser("10101", 2))
}
```

Без ожидаемого типа выражение `val parser = ::parseLessonId` неоднозначно.

Контекст может прийти от аргумента функции:

```kotlin
fun consumeParser(parser: (String) -> Int): Int {
    return parser("21")
}

val parsed = consumeParser(::parseLessonId)
```

### Перегруженные members

```kotlin
class Formatter {
    fun format(value: Int): String = "$value баллов"
    fun format(value: String): String = value.trim()
}

fun main() {
    val formatter = Formatter()
    val numberFormatter: (Int) -> String = formatter::format
    val textFormatter: (String) -> String = formatter::format

    println(numberFormatter(95))
    println(textFormatter("  Kotlin  "))
}
```

Receiver уже bound, поэтому различаются только обычные параметры.

<MouseTrap title="Мышеловка: reference требует больше типового контекста, чем вызов">
В прямом вызове типы аргументов выбирают перегрузка. У `::name` аргументов ещё нет, поэтому явно укажи функциональный тип переменной или параметра.
</MouseTrap>

## Адаптация callable reference

Kotlin может адаптировать некоторые ссылки к ожидаемому функциональному типу: учитывать значения по умолчанию, `Unit`-результат и suspend-контекст согласно правилам языка.

### Отбрасывание результата для `Unit`

```kotlin
fun calculateScore(): Int = 100

fun execute(action: () -> Unit) {
    action()
}

fun main() {
    execute(::calculateScore)
}
```

Результат `Int` сознательно игнорируется, потому что ожидается `() -> Unit`.

### Default parameters

```kotlin
fun greeting(name: String, punctuation: String = "!"): String {
    return "Привет, $name$punctuation"
}

fun main() {
    val shortGreeting: (String) -> String = ::greeting
    val fullGreeting: (String, String) -> String = ::greeting

    println(shortGreeting("Котлин-разработчик"))
    println(fullGreeting("Котлин-разработчик", "!!!"))
}
```

Reference может адаптироваться к форме, где default-аргумент опущен, если контекст однозначен.

### `vararg`

`vararg` в function reference представлен соответствующим array-параметром, а не произвольным числом параметров функционального типа.

```kotlin
fun joinTags(vararg tags: String): String = tags.joinToString()

fun main() {
    val joiner: (Array<out String>) -> String = ::joinTags
    println(joiner(arrayOf("functions", "references")))
}
```

На место вызова functional value получает один массив.

## Constructor references

Ссылка на constructor записывается как `::ClassName` и имеет параметры primary constructor.

```kotlin
data class Lesson(
    val title: String,
    val order: Int
)

fun main() {
    val factory: (String, Int) -> Lesson = ::Lesson
    val lesson = factory("Callable references", 21)

    println(lesson)
}
```

У constructor нет имени функции, поэтому имя класса после `::` обозначает создание экземпляра.

### Secondary constructors

Если класс имеет несколько применимых constructors, ожидаемый функциональный тип выбирает нужный.

```kotlin
class Student(val name: String, val level: Int) {
    constructor(name: String) : this(name = name, level = 1)
}

fun main() {
    val beginnerFactory: (String) -> Student = ::Student
    val exactFactory: (String, Int) -> Student = ::Student

    println(beginnerFactory("Анна").level)
    println(exactFactory("Борис", 3).level)
}
```

### Constructor reference в коллекциях

```kotlin
data class Tag(val value: String)

fun main() {
    val tags = listOf("kotlin", "functions").map(::Tag)
    println(tags)
}
```

Это удобно для простых value objects. Если нужна валидация, normalization или nullable failure, отдельная named factory лучше объясняет контракт.

## Property references

Properties тоже являются callables.

```kotlin
data class Lesson(val title: String, val order: Int)

fun main() {
    val titleProperty: (Lesson) -> String = Lesson::title
    val lessons = listOf(
        Lesson("Свойства", 23),
        Lesson("Callable references", 21)
    )

    println(lessons.map(titleProperty))
}
```

`Lesson::title` — unbound property reference. Receiver передаётся при вызове.

Property reference имеет более специализированный reflection-тип:

```kotlin
import kotlin.reflect.KProperty1

val titleProperty: KProperty1<Lesson, String> = Lesson::title
```

Через `get` можно прочитать значение:

```kotlin
val lesson = Lesson("Callable references", 21)
println(titleProperty.get(lesson))
println(titleProperty.name)
```

### Mutable property references

```kotlin
import kotlin.reflect.KMutableProperty1

class Progress(var completed: Int)

fun main() {
    val completedProperty: KMutableProperty1<Progress, Int> = Progress::completed
    val progress = Progress(5)

    completedProperty.set(progress, 6)
    println(completedProperty.get(progress))
}
```

`KMutableProperty1` предоставляет `set`. Не обходи через reflection инварианты, которые должны контролироваться методами объекта.

### Bound property reference

```kotlin
import kotlin.reflect.KProperty0

val lesson = Lesson("Callable references", 21)
val boundTitle: KProperty0<String> = lesson::title

println(boundTitle.get())
```

Receiver сохранён, поэтому property не требует параметров.

## Top-level property references

```kotlin
import kotlin.reflect.KMutableProperty0

var activeLesson: Int = 21

fun main() {
    val active: KMutableProperty0<Int> = ::activeLesson
    active.set(22)

    println(active.get())
}
```

Глобальное изменяемое состояние обычно нежелательно; пример показывает форму reference, а не рекомендуемую архитектуру.

## Function type и `KFunction`

Function reference совместима с функциональным типом и одновременно может предоставлять reflection metadata.

```kotlin
import kotlin.reflect.KFunction1

fun square(value: Int): Int = value * value

fun main() {
    val function: KFunction1<Int, Int> = ::square

    println(function.name)
    println(function.call(5))
    println(function(5))
}
```

`KFunction1` отражает один value parameter. Для полноценной JVM reflection может потребоваться зависимость `kotlin-reflect`; базовые callable references и вызов через functional type работают без использования сложного reflection API.

Reflection медленнее и менее типобезопасна при динамическом `call`, чем прямой вызов. Используй её для frameworks, tooling, serialization metadata и инфраструктуры, а не вместо обычных interfaces.

## References и generic-функции

Generic function reference требует контекста, который фиксирует type arguments.

```kotlin
fun <T> identity(value: T): T = value

fun main() {
    val stringIdentity: (String) -> String = ::identity
    val intIdentity: (Int) -> Int = ::identity

    println(stringIdentity("Kotlin"))
    println(intIdentity(21))
}
```

Нельзя записать type arguments непосредственно как `::identity<String>`. Они выводятся из ожидаемого типа или места применения.

## Suspend function references

Ссылка на `suspend`-функцию имеет suspend functional type.

```kotlin
suspend fun loadLesson(id: Int): String {
    return "Урок $id"
}

suspend fun useLoader(loader: suspend (Int) -> String): String {
    return loader(21)
}

suspend fun runExample() {
    val loader: suspend (Int) -> String = ::loadLesson
    println(useLoader(loader))
}
```

Обычная функция не может напрямую вызвать suspend reference вне coroutine/suspend-контекста.

## Function references и SAM

Java SAM interface или Kotlin `fun interface` можно создавать из callable reference, если сигнатуры совместимы.

```kotlin
fun interface ScoreValidator {
    fun isValid(score: Int): Boolean
}

fun scoreInRange(score: Int): Boolean = score in 0..100

fun main() {
    val validator = ScoreValidator(::scoreInRange)
    println(validator.isValid(95))
}
```

SAM-объект и function value — разные типы. Явное создание `ScoreValidator(...)` фиксирует интерфейсный контракт.

## Практический пример: конвейер обработки уроков

```kotlin
data class LessonDraft(
    val title: String,
    val order: Int,
    val published: Boolean
)

data class LessonCard(
    val title: String,
    val order: Int
)

fun normalizeDraft(draft: LessonDraft): LessonDraft {
    return draft.copy(title = draft.title.trim())
}

fun isPublishable(draft: LessonDraft): Boolean {
    return draft.published && draft.title.isNotBlank() && draft.order > 0
}

fun toCard(draft: LessonDraft): LessonCard {
    return LessonCard(title = draft.title, order = draft.order)
}

fun processDrafts(
    drafts: List<LessonDraft>,
    normalize: (LessonDraft) -> LessonDraft,
    predicate: (LessonDraft) -> Boolean,
    transform: (LessonDraft) -> LessonCard
): List<LessonCard> {
    return drafts
        .map(normalize)
        .filter(predicate)
        .sortedBy(LessonDraft::order)
        .map(transform)
}

fun main() {
    val drafts = listOf(
        LessonDraft("  Классы  ", 22, published = true),
        LessonDraft("Callable references", 21, published = true),
        LessonDraft("Черновик", 0, published = false)
    )

    val cards = processDrafts(
        drafts = drafts,
        normalize = ::normalizeDraft,
        predicate = ::isPublishable,
        transform = ::toCard
    )

    cards.forEach(::println)
}
```

Пример показывает:

- top-level function references как стратегии обработки;
- property reference `LessonDraft::order` как selector для sorting;
- перегрузка `println` разрешается из ожидаемого типа `forEach`;
- функции легко подменить в тесте без сложного framework;
- этапы остаются именованными и независимо проверяемыми.

## Итоги

- `::function` создаёт значение функции, а `function()` выполняет вызов;
- bound reference хранит конкретный receiver;
- unbound member reference получает receiver первым параметром;
- virtual member сохраняет runtime dispatch после выбора сигнатуры;
- extension reference выбирается статически;
- перегрузка set требует ожидаемого функционального типа;
- `::ClassName` создаёт constructor reference;
- `ClassName::property` создаёт unbound property reference;
- mutable property reference предоставляет `set`;
- callable references совместимы с функциональными типами и reflection API;
- generic type arguments выводятся из контекста;
- suspend reference требует suspend functional type;
- reference полезна, когда существующее имя точнее и короче адаптирующей лямбды.
