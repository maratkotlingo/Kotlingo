## Новый тип поверх одного значения

Value class создаёт отдельный Kotlin-тип, который при возможности представляется underlying value без отдельной wrapper allocation.

```kotlin
@JvmInline
value class LessonId(val value: Int)
```

`LessonId` и `Int` — разные типы:

```kotlin
fun loadLesson(id: LessonId): String {
    return "Загружен урок ${id.value}"
}

fun main() {
    val id = LessonId(32)
    println(loadLesson(id))

    // Прямой Int не является LessonId и не подходит функции loadLesson.
}
```

На JVM `@JvmInline` включает inline/value representation rules. В common Kotlin declaration `value class` отражает language concept, а platform representation зависит от target.

## Зачем не оставлять primitive или String

```kotlin
fun enroll(studentId: Int, courseId: Int, lessonId: Int) {
    println("$studentId -> $courseId -> $lessonId")
}
```

Перестановка аргументов скомпилируется. Value classes создают type boundaries:

```kotlin
@JvmInline
value class StudentId(val value: Int)

@JvmInline
value class CourseId(val value: Int)

@JvmInline
value class LessonId(val value: Int)

fun enroll(studentId: StudentId, courseId: CourseId, lessonId: LessonId) {
    println("${studentId.value} -> ${courseId.value} -> ${lessonId.value}")
}
```

Compiler не позволит передать `CourseId` вместо `StudentId`, хотя underlying type одинаков.

## Ограничения declaration

Стабильная single-field value class:

- имеет ровно одну property в primary constructor;
- underlying property объявляется `val`;
- не имеет дополнительных backing fields;
- является final и не может быть base class;
- не наследует обычный class;
- может реализовывать interfaces;
- может содержать functions, computed properties, `init`, secondary constructors и companion object;
- не поддерживает `lateinit` и delegated properties, потому что им нужен дополнительный storage;
- не имеет identity, на которую можно надёжно опираться.

```kotlin
@JvmInline
value class Percentage(val value: Int) {
    init {
        require(value in 0..100) { "Процент должен быть от 0 до 100" }
    }

    val ratio: Double
        get() = value / 100.0

    fun format(): String = "$value%"
}
```

Computed property `ratio` не создаёт backing field.

## Проверка invariants

`init` выполняется при обычном construction:

```kotlin
val valid = Percentage(75)
```

Для parsing без exception можно закрыть constructor и предоставить factory:

```kotlin
@JvmInline
value class SafeLessonId private constructor(val value: Int) {
    companion object {
        fun of(value: Int): SafeLessonId {
            require(value > 0) { "Идентификатор должен быть положительным" }
            return SafeLessonId(value)
        }

        fun parse(raw: String): SafeLessonId? {
            val value = raw.trim().toIntOrNull() ?: return null
            return value.takeIf { it > 0 }?.let(::SafeLessonId)
        }
    }
}
```

Не каждый serialization/reflection framework гарантированно вызывает factory; serializer и validation boundary настраиваются явно.

Secondary constructor обязан делегировать primary и не добавляет второе хранимое поле:

```kotlin
@JvmInline
value class PersonName private constructor(val fullName: String) {
    init {
        require(fullName.isNotBlank()) { "Имя не должно быть пустым" }
    }

    constructor(firstName: String, lastName: String) : this(
        fullName = "${firstName.trim()} ${lastName.trim()}"
    ) {
        require(firstName.isNotBlank()) { "Имя обязательно" }
        require(lastName.isNotBlank()) { "Фамилия обязательна" }
    }
}
```

После construction runtime representation по-прежнему основана только на `fullName`.

<CatTip title="Котосовет: value class хорош для проверенного скаляра">
ID, percentage, email, slug и единица измерения получают отдельный тип и centralized validation. Если model имеет несколько независимых полей и сложный lifecycle, обычный data/class точнее.
</CatTip>

## Functions и computed properties

```kotlin
@JvmInline
value class LessonSlug private constructor(val value: String) {
    val route: String
        get() = "/kotlin/$value"

    fun startsWithOrder(order: Int): Boolean {
        return value.startsWith("$order-")
    }

    override fun toString(): String = value

    companion object {
        fun fromTitle(order: Int, title: String): LessonSlug {
            require(order > 0) { "Порядок должен быть положительным" }
            val normalizedTitle = title
                .trim()
                .lowercase()
                .replace(Regex("""[^a-zа-яё0-9]+"""), "-")
                .trim('-')
            require(normalizedTitle.isNotEmpty()) { "Название не образует slug" }
            return LessonSlug("$order-$normalizedTitle")
        }
    }
}
```

Methods вызываются как у обычного type, но compiler может преобразовать их в static-like functions underlying representation.

Value class не может хранить второе cached field. Для производного значения нужен getter:

```kotlin
@JvmInline
value class Minutes(val value: Int) {
    val seconds: Int
        get() = Math.multiplyExact(value, 60)
}
```

## Реализация interface

```kotlin
interface Displayable {
    fun display(): String
}

@JvmInline
value class Score(val value: Int) : Displayable {
    init {
        require(value in 0..100) { "Балл должен быть от 0 до 100" }
    }

    override fun display(): String = "$value баллов"
}
```

При использовании как concrete `Score` representation может быть unboxed. При приведении к `Displayable` обычно требуется boxed wrapper, потому что interface reference указывает на объект.

```kotlin
val concrete: Score = Score(95)
val abstract: Displayable = concrete

println(abstract.display())
```

Interface implementation расширяет polymorphic use, но может устранить allocation advantage на этой boundary.

## Boxing и unboxing

Compiler выбирает boxed wrapper, когда value должен вести себя как object/reference. Типичные boxing boundaries:

- nullable value class;
- generic type/function;
- interface или `Any`;
- generic collection;
- reflection;
- representation должна различать wrapper/null и underlying states;
- platform interop требует object.

```kotlin
@JvmInline
value class LessonId(val value: Int)

fun direct(id: LessonId): Int = id.value

fun nullable(id: LessonId?): Int? = id?.value

fun <T> generic(value: T): T = value

fun asAny(id: LessonId): Any = id
```

`direct` наиболее вероятно использует raw `int` в JVM bytecode. Остальные формы могут boxing.

Representation — compiler/backend detail. Source semantics не должны зависеть от allocation.

<MouseTrap title="Мышеловка: value class не гарантирует ноль allocations">
Generic collections, nullable values, interfaces и `Any` часто требуют boxing. Проверяй конкретный hot path benchmark и bytecode, а не делай performance вывод только по слову `value`.
</MouseTrap>

## Nullable value class

```kotlin
val selectedLesson: LessonId? = null
```

Underlying `Int` не может представить `null`, поэтому JVM обычно нуждается в boxed form для `LessonId?`.

Если underlying type сам nullable, различаются два состояния:

```kotlin
@JvmInline
value class OptionalCode(val value: String?)

val noWrapper: OptionalCode? = null
val wrapperWithNull: OptionalCode = OptionalCode(null)
```

Это разные semantic values, требующие representation capable of distinction. Nullable underlying value class стоит вводить только при реальном domain различии.

## Generic context

```kotlin
fun <T> singleton(value: T): List<T> = listOf(value)

val ids: List<LessonId> = singleton(LessonId(32))
```

JVM generics работают с reference types, поэтому elements обычно boxed. `List<LessonId>` type-safe, но не равен специализированному `IntArray` по memory layout.

Для миллионов IDs измеряй:

- `List<LessonId>`;
- `IntArray` внутри specialized collection;
- обычные arrays;
- inline processing без materialization.

API clarity обычно важнее небольшой allocation cost, но data-intensive path требует benchmark.

## Generic value class

Underlying property может использовать type parameter в поддерживаемой форме:

```kotlin
@JvmInline
value class Identifier<T>(val value: String)

class LessonEntity
class StudentEntity

typealias LessonIdentifier = Identifier<LessonEntity>
typealias StudentIdentifier = Identifier<StudentEntity>
```

Phantom type parameter различает IDs compile time:

```kotlin
fun lesson(id: LessonIdentifier): String = id.value
```

На JVM representation всё равно String, generic boundaries могут boxing/erasure. Алиасы не создают новые types сами; различие дают type arguments `Identifier<T>`.

## Equality и hash code

Value class equality определяется underlying value semantics.

```kotlin
val first = LessonId(32)
val second = LessonId(32)
val third = LessonId(31)

println(first == second)
println(first == third)
println(first.hashCode())
```

Referential identity `===` для value classes запрещена или не имеет устойчивого смысла: значение может быть unboxed или boxed. Проектируй structural equality.

Если underlying value mutable, wrapper не делает его immutable:

```kotlin
@JvmInline
value class UnsafeBytes(val value: ByteArray)
```

Array equality остаётся reference-based, а content может меняться. Value class не исправляет semantics underlying type автоматически.

## `toString`

Compiler предоставляет representation wrapper, но она не является stable wire format. Можно override:

```kotlin
@JvmInline
value class EmailAddress(val value: String) {
    override fun toString(): String {
        val at = value.indexOf('@')
        return if (at <= 1) "***" else "${value.first()}***${value.substring(at)}"
    }
}
```

Public `value` всё равно раскрывает email. Для secrets нужна более строгая model/API, не только masked `toString`.

## Name mangling на JVM

Functions с value class parameters могут иметь тот же raw JVM shape, что functions с underlying type. Compiler mangles JVM names, добавляя suffix, чтобы избежать clashes.

```kotlin
@JvmInline
value class LessonId(val value: Int)

fun findLesson(id: LessonId): String = "Урок ${id.value}"
fun findLesson(id: Int): String = "Число $id"
```

На Kotlin source overloads различимы, а JVM names compiler разводит mangling.

Следствия:

- Java-вызов может иметь неудобное mangled имя;
- reflection lookup по source name сложнее;
- изменение underlying type ломает ABI;
- `@JvmName` может задать Java-friendly имя в применимых declarations.

```kotlin
@JvmName("findLessonByTypedId")
fun findLessonForJava(id: LessonId): String {
    return "Урок ${id.value}"
}
```

Interop shape проверяй по generated bytecode и Java compile test.

## Java API design

Value classes оптимизированы прежде всего для Kotlin type safety. Для Java consumers:

- expose underlying primitive/String отдельным Java-facing method;
- добавь явно названную bridge function;
- используй `@JvmExposeBoxed` или module option `-Xjvm-expose-boxed`, если текущий compiler и opt-in policy проекта допускают boxed exposure;
- используй обычный class wrapper в public Java API;
- скрой value class внутри Kotlin implementation;
- проверь возможности текущего compiler для boxed exposure.

`@JvmExposeBoxed` можно применять к value class, constructor или function; на актуальном toolchain annotation требует opt-in `ExperimentalStdlibApi`. Она генерирует Java-доступные boxed формы, но увеличивает API surface и не исправляет автоматически inherited functions. Не обещай ergonomic Java API без Java compilation test.

## Overloads с разными value classes

```kotlin
@JvmInline
value class UserId(val value: Long)

@JvmInline
value class CourseId(val value: Long)

fun route(id: UserId): String = "/users/${id.value}"
fun route(id: CourseId): String = "/courses/${id.value}"
```

Kotlin различает overloads. JVM mangling разводит одинаковое underlying `long`. Для Java-facing API разные names `userRoute`/`courseRoute` проще.

## Value class против typealias

```kotlin
typealias RawLessonId = Int

fun loadRaw(id: RawLessonId): Int = id
val rawResult = loadRaw(32)
```

Alias не создаёт type. Value class создаёт distinct type:

```kotlin
fun loadTyped(id: LessonId): Int = id.value
val typedResult = loadTyped(LessonId(32))
```

Typealias улучшает читаемость длинного типа, value class обеспечивает separation и members/validation.

## Value class против data class

Value class:

- один underlying value;
- может быть unboxed;
- ограничена в state/inheritance;
- оптимальна для typed scalar.

Data class:

- один или много values;
- обычное object representation;
- генерирует `copy` и `componentN`;
- удобна для records/state snapshots.

```kotlin
@JvmInline
value class LessonId(val value: Int)

data class LessonSummary(
    val id: LessonId,
    val title: String,
    val published: Boolean
)
```

Они хорошо работают вместе.

## Unit-of-measure types

```kotlin
@JvmInline
value class Minutes(val value: Int) {
    init {
        require(value >= 0) { "Минуты не могут быть отрицательными" }
    }

    operator fun plus(other: Minutes): Minutes {
        return Minutes(Math.addExact(value, other.value))
    }
}

@JvmInline
value class Seconds(val value: Int)

fun estimate(theory: Minutes, practice: Minutes): Minutes {
    return theory + practice
}
```

Compiler не позволит передать `Seconds` вместо `Minutes`. Conversion должна быть явной:

```kotlin
fun Seconds.toWholeMinutes(): Minutes {
    return Minutes(value / 60)
}
```

Имя раскрывает rounding policy.

## Unsigned types как пример

`UInt`, `ULong`, `UByte`, `UShort` используют value class mechanism поверх signed primitives на соответствующих targets. Они дают новый API и arithmetic semantics при efficient representation.

Это не означает автоматическую безопасность domain arithmetic: overflow и conversion rules всё равно нужно понимать.

## ABI и эволюция

Public value class тесно связывает ABI с underlying type и compiler representation.

Breaking changes:

- `Int` → `Long`;
- изменение package/name;
- изменение function signatures;
- изменение visibility constructor;
- compiler mode/toolchain incompatibility;
- validation, отклоняющая ранее допустимые values.

Для опубликованной library:

- фиксируй compiler/target policy;
- используй binary compatibility validator;
- тестируй Kotlin и Java clients;
- не считай underlying type внутренней деталью;
- документируй wire/storage representation отдельно.

## Serialization и databases

Framework должен сохранять underlying `value`, а не compiler-specific wrapper shape.

```kotlin
@JvmInline
value class DatabaseLessonId(val value: Long)
```

Настрой explicit serializer/converter:

- JSON field `lessonId` как число или строка по schema;
- database column `BIGINT`;
- parsing/validation на boundary;
- migration при изменении format.

Не сериализуй reflection-based `toString`.

## Производительность: измерять границы

Value class может убрать allocation в direct concrete calls, но итог зависит от:

- boxing frequency;
- collection/generic use;
- JIT escape analysis;
- inlining;
- nullable representation;
- backend;
- serialization/reflection;
- data size и cache locality.

JVM benchmark должен:

- использовать JMH;
- прогревать JIT;
- предотвращать dead-code elimination;
- сравнивать end-to-end use case;
- измерять allocations profiler;
- использовать production compiler flags.

Type safety — гарантированная причина выбрать value class. Performance — потенциальное преимущество, подтверждаемое измерением.

## Практический пример: типизированные идентификаторы и прогресс

```kotlin
@JvmInline
value class StudentId private constructor(val value: Long) {
    init {
        require(value > 0L) { "Идентификатор студента должен быть положительным" }
    }

    companion object {
        fun of(value: Long): StudentId = StudentId(value)
    }
}

@JvmInline
value class LessonId private constructor(val value: Int) {
    init {
        require(value > 0) { "Идентификатор урока должен быть положительным" }
    }

    companion object {
        fun parse(raw: String): LessonId? {
            val value = raw.trim().toIntOrNull() ?: return null
            return value.takeIf { it > 0 }?.let(::LessonId)
        }
    }
}

@JvmInline
value class CompletionPercent(val value: Int) {
    init {
        require(value in 0..100) { "Процент должен быть от 0 до 100" }
    }

    val complete: Boolean
        get() = value == 100

    override fun toString(): String = "$value%"
}

data class ProgressRecord(
    val studentId: StudentId,
    val lessonId: LessonId,
    val percent: CompletionPercent
)

interface ProgressRepository {
    fun save(record: ProgressRecord)
    fun find(studentId: StudentId, lessonId: LessonId): ProgressRecord?
}

class InMemoryProgressRepository : ProgressRepository {
    private data class Key(
        val studentId: StudentId,
        val lessonId: LessonId
    )

    private val records = mutableMapOf<Key, ProgressRecord>()

    override fun save(record: ProgressRecord) {
        val key = Key(record.studentId, record.lessonId)
        val previous = records[key]
        require(previous == null || record.percent.value >= previous.percent.value) {
            "Прогресс не должен уменьшаться"
        }
        records[key] = record
    }

    override fun find(
        studentId: StudentId,
        lessonId: LessonId
    ): ProgressRecord? {
        return records[Key(studentId, lessonId)]
    }
}

fun main() {
    val studentId = StudentId.of(42L)
    val lessonId = requireNotNull(LessonId.parse("32"))
    val repository = InMemoryProgressRepository()

    repository.save(
        ProgressRecord(
            studentId = studentId,
            lessonId = lessonId,
            percent = CompletionPercent(75)
        )
    )

    val record = repository.find(studentId, lessonId)
    println(record)
    println("Завершён: ${record?.percent?.complete}")
}
```

Design properties:

- scalar values нельзя перепутать по типам;
- validation сосредоточена в value classes;
- data class объединяет domain scalars в record;
- repository использует domain types, а не raw numbers;
- nullable означает реальное отсутствие record/parser failure;
- generic map может boxing IDs, но type safety сохраняется;
- persistence adapter явно конвертирует `.value` в columns.

## Итоги

- value class создаёт distinct type поверх одного underlying value;
- `@JvmInline` задаёт JVM inline representation contract;
- primary constructor содержит одну `val` property;
- value class final, без второго backing state, но с methods/computed properties;
- validation доступна через `init` и factories;
- concrete use может быть unboxed;
- nullable, generic, interface и `Any` boundaries часто требуют boxing;
- value class не имеет стабильной referential identity;
- underlying mutable/equality semantics не исправляются автоматически;
- JVM name mangling разводит signatures, но усложняет Java interop;
- value class сильнее typealias и легче single-field data class;
- public underlying type является частью ABI;
- type safety гарантирована, performance advantage нужно измерять.
