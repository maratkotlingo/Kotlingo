## Три модификатора с разными задачами

`infix`, `operator` и `tailrec` меняют не предметную логику, а способ объявления, вызова или компиляции функции:

- `infix` разрешает вызов без точки и круглых скобок;
- `operator` связывает функцию с операторным синтаксисом Kotlin;
- `tailrec` просит компилятор оптимизировать корректную хвостовую рекурсию в цикл.

Каждый модификатор имеет строгие требования. Добавлять их ради необычного синтаксиса не стоит: вызов должен оставаться очевидным для читателя.

## Infix-функции

Infix-вызов читается как простая фраза:

```kotlin
infix fun Int.percentOf(total: Int): Int {
    require(total > 0) { "Общее значение должно быть положительным" }
    return this * 100 / total
}

fun main() {
    val progress = 18 percentOf 20
    println("$progress%")
}
```

Обычный вызов также остаётся доступен:

```kotlin
val progress = 18.percentOf(20)
```

### Требования к `infix`

Infix-функция обязана:

- быть member-функцией или extension-функцией;
- иметь ровно один value parameter;
- не использовать `vararg` для этого параметра;
- не задавать ему значение по умолчанию;
- иметь модификатор `infix` в объявлении.

```kotlin
data class Tag(val value: String)

infix fun Tag.matches(other: Tag): Boolean {
    return value.equals(other.value, ignoreCase = true)
}
```

Обычная top-level функция без receiver не может быть infix.

### Имена infix-функций

Хорошее имя образует однозначную связь или действие:

- `x to y` создаёт пару;
- `value until end` создаёт диапазон без правой границы;
- `set union other` выражает объединение;
- `permission implies requirement` может выразить правило DSL.

```kotlin
enum class Skill {
    SYNTAX,
    FUNCTIONS,
    COROUTINES
}

data class Prerequisite(
    val skill: Skill,
    val required: Skill
)

infix fun Skill.requires(required: Skill): Prerequisite {
    return Prerequisite(skill = this, required = required)
}

val rule = Skill.COROUTINES requires Skill.FUNCTIONS
```

`requires` читается естественно и не скрывает сложное вычисление.

## Приоритет infix-вызовов

Infix-вызовы имеют более низкий приоритет, чем арифметические операторы, range operator и `is`, но более высокий, чем `&&`, `||`, checks `in` и некоторые другие конструкции. Запоминать полную шкалу необязательно: при смешивании разных операторов ставь скобки.

```kotlin
val result = (2 + 3) percentOf 10
```

Без скобок арифметика всё равно выполнится раньше infix-вызова, но явная группировка быстрее читается.

Infix-вызовы ассоциативны слева:

```kotlin
// a operation b operation c
// читается как (a operation b) operation c,
// если тип результата допускает второй вызов.
```

Цепочка из нескольких словесных операторов быстро становится похожа на новый язык. Для сложной логики обычные методы яснее.

<MouseTrap title="Мышеловка: infix скрывает только пунктуацию, не сложность">
Не используй infix для сетевого запроса, записи в базу или операции с неожиданной стоимостью. Выражение `user sync server` выглядит лёгким, хотя может блокировать поток и падать. Обычное имя `syncWithServer()` честнее показывает действие.
</MouseTrap>

## Infix и наследование

Модификатор `infix` не наследуется автоматически в смысле доступности infix-синтаксиса для override: переопределение должно явно повторить его, если планируется такой вызов.

```kotlin
open class Matcher {
    open infix fun matches(value: String): Boolean = value.isNotEmpty()
}

class KotlinMatcher : Matcher() {
    override infix fun matches(value: String): Boolean {
        return value.contains("kotlin", ignoreCase = true)
    }
}
```

Явный модификатор делает синтаксический контракт видимым в каждом объявлении.

## Operator overloading

Kotlin не разрешает изобретать новые символы операторов. Можно реализовать только заранее определённые соглашения: конкретный синтаксис транслируется в функцию с установленным именем.

```kotlin
data class Score(val points: Int) {
    init {
        require(points >= 0) { "Балл не может быть отрицательным" }
    }

    operator fun plus(other: Score): Score {
        return Score(points + other.points)
    }
}

fun main() {
    val total = Score(40) + Score(55)
    println(total)
}
```

Выражение `left + right` вызывает `left.plus(right)`. Функцию также можно вызвать явно: `left.plus(right)`.

### Требования к operator-функции

Функция должна:

- быть member или extension;
- иметь имя из набора operator conventions;
- соответствовать требуемому числу параметров и ограничениям;
- быть помечена `operator`, если это не Java-метод, распознанный специальными правилами interop.

Компилятор проверяет форму объявления.

<CatTip title="Котосовет: оператор должен сохранять знакомый смысл">
`+` ожидается как сложение или объединение без неожиданной мутации исходных operands. `[]` ожидается как доступ по ключу/индексу. Если доменная операция не похожа на привычный оператор, используй именованную функцию.
</CatTip>

## Унарные операторы

Унарные соглашения не принимают value parameters:

- `+a` вызывает `a.unaryPlus()`;
- `-a` вызывает `a.unaryMinus()`;
- `!a` вызывает `a.not()`.

```kotlin
data class ProgressDelta(val lessons: Int) {
    operator fun unaryMinus(): ProgressDelta {
        return ProgressDelta(-lessons)
    }

    operator fun unaryPlus(): ProgressDelta {
        return this
    }
}

fun main() {
    val added = ProgressDelta(3)
    println(-added)
    println(+added)
}
```

Для `not()` результат не обязан быть `Boolean` по общей форме соглашения, но возвращать другой тип почти всегда разрушает интуицию. В булевом домене ожидается логическое отрицание.

## Инкремент и декремент

- `a++` и `++a` используют `a.inc()`;
- `a--` и `--a` используют `a.dec()`.

`inc` и `dec` не должны мутировать receiver и возвращать его самого. Они возвращают новое значение, которое компилятор присваивает изменяемой переменной.

```kotlin
data class LessonNumber(val value: Int) {
    init {
        require(value > 0) { "Номер урока должен быть положительным" }
    }

    operator fun inc(): LessonNumber = LessonNumber(value + 1)

    operator fun dec(): LessonNumber {
        require(value > 1) { "Перед первым уроком нет урока с положительным номером" }
        return LessonNumber(value - 1)
    }
}

fun main() {
    var current = LessonNumber(19)
    val previousValue = current++

    println("До postfix: $previousValue")
    println("После postfix: $current")
    println("После prefix: ${++current}")
}
```

Для postfix сначала сохраняется старое значение выражения, затем вызывается `inc` и переменной присваивается новое. Для prefix сначала выполняется присваивание нового значения, затем оно становится результатом выражения.

## Бинарная арифметика

Основные соглашения:

- `a + b` → `a.plus(b)`;
- `a - b` → `a.minus(b)`;
- `a * b` → `a.times(b)`;
- `a / b` → `a.div(b)`;
- `a % b` → `a.rem(b)`;
- `a..b` → `a.rangeTo(b)`;
- `a..<b` → `a.rangeUntil(b)`.

Тип правого operand и тип результата могут отличаться от receiver.

```kotlin
data class StudyDuration(val minutes: Int) {
    init {
        require(minutes >= 0) { "Длительность не может быть отрицательной" }
    }

    operator fun plus(other: StudyDuration): StudyDuration {
        return StudyDuration(minutes + other.minutes)
    }

    operator fun times(multiplier: Int): StudyDuration {
        require(multiplier >= 0) { "Множитель не может быть отрицательным" }
        return StudyDuration(minutes * multiplier)
    }

    operator fun div(parts: Int): StudyDuration {
        require(parts > 0) { "Число частей должно быть положительным" }
        return StudyDuration(minutes / parts)
    }
}

fun main() {
    val theory = StudyDuration(30)
    val practice = StudyDuration(45)

    println(theory + practice)
    println(practice * 2)
    println(practice / 3)
}
```

Если нужен симметричный вызов `2 * practice`, понадобится отдельное extension:

```kotlin
operator fun Int.times(duration: StudyDuration): StudyDuration {
    return duration * this
}
```

Kotlin не переставляет operands автоматически.

## Составные присваивания

Для `a += b` компилятор рассматривает соглашения `plusAssign` и `plus`:

- `plusAssign` изменяет receiver и возвращает `Unit`;
- если подходящего `plusAssign` нет, `a = a.plus(b)` может использоваться для mutable-переменной;
- наличие одновременно подходящих `plusAssign` и `plus` способно создать неоднозначность.

Аналогично работают `minusAssign`, `timesAssign`, `divAssign`, `remAssign`.

Immutable value object обычно реализует `plus`:

```kotlin
data class TagSet(val values: Set<String>) {
    operator fun plus(tag: String): TagSet {
        return TagSet(values + tag.trim().lowercase())
    }
}

fun main() {
    var tags = TagSet(setOf("kotlin"))
    tags += "Functions"
    println(tags)
}
```

Mutable container может реализовать `plusAssign`:

```kotlin
class MutableTags {
    private val values = linkedSetOf<String>()

    operator fun plusAssign(tag: String) {
        values += tag.trim().lowercase()
    }

    fun snapshot(): Set<String> = values.toSet()
}

fun main() {
    val tags = MutableTags()
    tags += "Kotlin"
    tags += "Functions"
    println(tags.snapshot())
}
```

`val tags` допустим, потому что ссылка не переназначается: `plusAssign` мутирует объект. Этот эффект должен быть ожидаемым из mutability типа.

<MouseTrap title="Мышеловка: не объявляй одновременно plus и plusAssign без необходимости">
Для одного типа аргумента компилятор может увидеть две допустимые трактовки `+=`. Выбери модель: immutable value с `plus` или явно mutable container с `plusAssign`.
</MouseTrap>

## Равенство: `equals`

`a == b` и `a != b` используют `equals`, объявленный в `Any`, с безопасной обработкой `null`. Для собственного класса обычно достаточно `data class`.

```kotlin
data class LessonId(val value: String)

fun main() {
    println(LessonId("20") == LessonId("20"))
}
```

Нельзя создать произвольную extension-реализацию `equals`, которая изменит `==`: соглашение требует member override с точной сигнатурой `equals(other: Any?): Boolean`.

При ручной реализации `equals` обязательно согласуй `hashCode`. Иначе hash-based collections будут работать некорректно.

`===` и `!==` не перегружаются: они проверяют идентичность ссылок.

## Сравнение: `compareTo`

`a < b`, `a > b`, `a <= b`, `a >= b` вызывают `a.compareTo(b)` и сравнивают результат с нулём.

```kotlin
data class Difficulty(val rank: Int) : Comparable<Difficulty> {
    init {
        require(rank >= 0) { "Ранг не может быть отрицательным" }
    }

    override operator fun compareTo(other: Difficulty): Int {
        return rank.compareTo(other.rank)
    }
}

fun main() {
    println(Difficulty(3) > Difficulty(1))
}
```

Не вычисляй `rank - other.rank`: при больших значениях возможное переполнение изменит знак. Используй `compareTo` встроенного типа или последовательное сравнение полей.

Порядок должен быть транзитивным и согласованным. Согласованность с `equals` желательна, особенно для sorted collections, хотя типовая система не может её доказать.

## `in` и `!in`: `contains`

`element in container` вызывает `container.contains(element)`. Receiver и аргумент меняются местами относительно записи.

```kotlin
data class ScoreBand(val minimum: Int, val maximum: Int) {
    init {
        require(minimum <= maximum) { "Минимум не может превышать максимум" }
    }

    operator fun contains(score: Int): Boolean {
        return score in minimum..maximum
    }
}

fun main() {
    val passing = ScoreBand(60, 100)
    println(85 in passing)
    println(40 !in passing)
}
```

`contains` должен быть быстрым и не иметь побочных эффектов: проверка членства воспринимается как вопрос.

## Индексный доступ: `get` и `set`

- `container[index]` вызывает `container.get(index)`;
- `container[index] = value` вызывает `container.set(index, value)`;
- индексов может быть несколько.

```kotlin
class LessonCatalog(private val titles: MutableMap<Int, String>) {
    operator fun get(id: Int): String? {
        return titles[id]
    }

    operator fun set(id: Int, title: String) {
        require(id > 0) { "Идентификатор должен быть положительным" }
        require(title.isNotBlank()) { "Название не должно быть пустым" }
        titles[id] = title.trim()
    }
}

fun main() {
    val catalog = LessonCatalog(mutableMapOf(19 to "Extensions"))
    catalog[20] = "Infix, operator и tailrec"

    println(catalog[19])
    println(catalog[20])
}
```

Многомерный пример:

```kotlin
class ScoreGrid(
    private val rows: Int,
    private val columns: Int
) {
    private val values = IntArray(rows * columns)

    init {
        require(rows > 0 && columns > 0) { "Размеры сетки должны быть положительными" }
    }

    private fun index(row: Int, column: Int): Int {
        require(row in 0 until rows) { "Строка вне диапазона: $row" }
        require(column in 0 until columns) { "Столбец вне диапазона: $column" }
        return row * columns + column
    }

    operator fun get(row: Int, column: Int): Int {
        return values[index(row, column)]
    }

    operator fun set(row: Int, column: Int, score: Int) {
        require(score in 0..100) { "Балл должен быть от 0 до 100" }
        values[index(row, column)] = score
    }
}
```

## Вызов объекта как функции: `invoke`

`object(arguments)` вызывает `object.invoke(arguments)`.

```kotlin
class ScoreNormalizer(private val minimumPassing: Int) {
    init {
        require(minimumPassing in 1..100) { "Порог должен быть от 1 до 100" }
    }

    operator fun invoke(score: Int): Double {
        require(score in 0..100) { "Балл должен быть от 0 до 100" }
        return score.toDouble() / minimumPassing
    }
}

fun main() {
    val normalize = ScoreNormalizer(minimumPassing = 60)
    println(normalize(90))
}
```

`invoke` подходит объектам, чья основная роль — одна операция: сценарий использования, validator, parser, policy. Если у объекта много равноправных действий, именованные методы яснее.

Можно перегружать `invoke` по обычным правилам, но перегрузка set должен оставаться предсказуемым.

## Итерация в `for`: `iterator`

Для `for (item in value)` нужны соглашения:

- `value.iterator()`;
- у iterator — `hasNext()`;
- у iterator — `next()`.

Соответствующие функции помечаются `operator` там, где это требуется Kotlin-декларациям.

```kotlin
class LessonSequence(private val first: Int, private val last: Int) {
    init {
        require(first <= last) { "Начало не может быть больше конца" }
    }

    operator fun iterator(): Iterator<Int> {
        return (first..last).iterator()
    }
}

fun main() {
    for (lesson in LessonSequence(18, 20)) {
        println("Урок $lesson")
    }
}
```

Для боевой collection чаще реализуют `Iterable<T>`, чтобы получить совместимость со стандартными API:

```kotlin
class Lessons(private val ids: List<Int>) : Iterable<Int> {
    override fun iterator(): Iterator<Int> = ids.iterator()
}
```

`Iterable.iterator` уже является operator-контрактом.

## Деструктуризация: `componentN`

```kotlin
data class LessonSummary(
    val title: String,
    val score: Int,
    val completed: Boolean
)

fun main() {
    val summary = LessonSummary("Операторы", 95, true)
    val (title, score, completed) = summary

    println("$title | $score | $completed")
}
```

Деструктуризация вызывает `component1()`, `component2()`, `component3()` по позиции. Data class генерирует их для properties primary constructor.

Можно объявить вручную:

```kotlin
class Coordinate(private val row: Int, private val column: Int) {
    operator fun component1(): Int = row
    operator fun component2(): Int = column
}
```

Позиционный контракт хрупок: перестановка свойств меняет смысл без изменения имён переменных на место вызова. Для больших объектов явный доступ к properties обычно понятнее.

Underscore пропускает компонент, и соответствующий `componentN` может не вызываться:

```kotlin
val (title, _, completed) = LessonSummary("Операторы", 95, true)
println("$title | $completed")
```

## Property delegation: `getValue`, `setValue`, `provideDelegate`

Синтаксис `val property by delegate` основан на operator conventions.

```kotlin
import kotlin.reflect.KProperty

class TrimmedText(initial: String) {
    private var value: String = initial.trim()

    operator fun getValue(thisRef: Any?, property: KProperty<*>): String {
        return value
    }

    operator fun setValue(thisRef: Any?, property: KProperty<*>, newValue: String) {
        value = newValue.trim()
    }
}

class LessonDraft(title: String) {
    var title: String by TrimmedText(title)
}

fun main() {
    val draft = LessonDraft("  Операторы  ")
    println(draft.title)

    draft.title = "  Tail recursion  "
    println(draft.title)
}
```

- чтение `draft.title` вызывает `getValue`;
- присваивание вызывает `setValue`;
- `thisRef` — владелец property;
- `property` содержит metadata свойства;
- `provideDelegate` может перехватить создание delegate при инициализации и проверить property.

Делегаты подробно рассматриваются в отдельном разделе силлабуса; здесь важно увидеть, что синтаксис основан на operator-функциях.

## Range conventions

`a..b` использует `rangeTo`, а `a..<b` — `rangeUntil`. На практике пользовательский range должен также поддерживать meaningful iteration или проверки членства, иначе знакомый синтаксис вводит в заблуждение.

```kotlin
data class LessonIndex(val value: Int) : Comparable<LessonIndex> {
    init {
        require(value > 0) { "Индекс должен быть положительным" }
    }

    override fun compareTo(other: LessonIndex): Int {
        return value.compareTo(other.value)
    }
}

class LessonIndexRange(
    override val start: LessonIndex,
    override val endInclusive: LessonIndex
) : ClosedRange<LessonIndex>

operator fun LessonIndex.rangeTo(other: LessonIndex): LessonIndexRange {
    return LessonIndexRange(this, other)
}
```

Теперь `LessonIndex(3) in LessonIndex(1)..LessonIndex(5)` использует контракт closed range.

## Полезные правила operator-дизайна

- сохраняй ожидаемые алгебраические свойства, если домен их допускает;
- не меняй operands в `plus`, `minus`, `times` для immutable типов;
- явно проверяй переполнение и недопустимые значения, если это часть домена;
- делай `compareTo` транзитивным;
- согласуй `equals` и `hashCode`;
- используй `get` только для естественного доступа по ключу/индексу;
- используй `invoke` для объекта с одной основной ролью;
- не применяй оператор только ради сокращения нескольких букв.

## `tailrec`: оптимизируемая хвостовая рекурсия

Обычная рекурсивная функция создаёт новый stack frame для каждого вызова. При большой глубине это может привести к `StackOverflowError`.

```kotlin
fun sumDownUnsafe(value: Long): Long {
    return if (value <= 0L) {
        0L
    } else {
        value + sumDownUnsafe(value - 1L)
    }
}
```

Рекурсивный вызов не последний: после него ещё нужно выполнить сложение `value + result`. Это **не хвостовая рекурсия**.

Хвостовая форма переносит промежуточный результат в accumulator:

```kotlin
tailrec fun sumDown(value: Long, accumulator: Long = 0L): Long {
    return if (value <= 0L) {
        accumulator
    } else {
        sumDown(
            value = value - 1L,
            accumulator = accumulator + value
        )
    }
}
```

Рекурсивный вызов — последнее вычисление ветки. Компилятор может превратить его в цикл и повторно использовать frame.

## Условия хвостовой позиции

Вызов считается хвостовым, когда его результат немедленно становится результатом текущей функции. После него не должно быть:

- арифметики;
- преобразования результата;
- второго рекурсивного вызова;
- логики в `finally`;
- оборачивания в объект;
- ожидания другого вычисления.

Хвостовой пример:

```kotlin
tailrec fun gcd(first: Long, second: Long): Long {
    return if (second == 0L) {
        kotlin.math.abs(first)
    } else {
        gcd(second, first % second)
    }
}
```

Нехвостовой пример:

```kotlin
fun factorialUnsafe(value: Long): Long {
    return if (value <= 1L) {
        1L
    } else {
        value * factorialUnsafe(value - 1L)
    }
}
```

Хвостовая версия:

```kotlin
tailrec fun factorial(value: Long, accumulator: Long = 1L): Long {
    require(value >= 0L) { "Факториал определён только для неотрицательных чисел" }

    return if (value <= 1L) {
        accumulator
    } else {
        factorial(
            value = value - 1L,
            accumulator = Math.multiplyExact(accumulator, value)
        )
    }
}
```

`Math.multiplyExact` превращает silent overflow в `ArithmeticException`. Для больших факториалов нужен `BigInteger`.

## Что делает компилятор

При корректной `tailrec`-функции бэкенд заменяет само-рекурсию циклом там, где поддерживается. Это оптимизация только прямого вызова функции самой себя.

Не оптимизируются автоматически:

- взаимная рекурсия `first()` → `second()` → `first()`;
- рекурсия через function reference или лямбда;
- открытый виртуальный dispatch;
- рекурсивный вызов не в tail position.

Если пометить функцию `tailrec`, но подходящих хвостовых вызовов нет, компилятор выдаёт предупреждение. Если часть вызовов хвостовая, оптимизируются только подходящие места; остальные сохраняют риск роста стека.

<MouseTrap title="Мышеловка: tailrec не делает любой рекурсивный алгоритм безопасным">
Модификатор не является магией. Проверь, что каждый путь рекурсии заканчивается прямым self-call. Для обхода ветвящегося дерева с двумя recursive calls обычно нужен явный stack/queue или другой итеративный алгоритм.
</MouseTrap>

## `tailrec` и `try`/`finally`

Рекурсивный вызов внутри `try`, за которым должен выполняться `finally`, не находится в простой хвостовой позиции: после возврата требуется выполнить cleanup.

```kotlin
fun processWithCleanup(value: Int): Int {
    try {
        return if (value <= 0) 0 else processWithCleanup(value - 1)
    } finally {
        println("Завершаем уровень $value")
    }
}
```

Эта функция сознательно не помечена `tailrec`. Для глубокой обработки вынеси cleanup в итеративную структуру.

## Когда цикл проще

Хвостовая рекурсия полезна, когда рекурсивная формулировка естественна: конечные автоматы, Euclidean algorithm, повторные переходы по linked structure. Обычный `while` часто яснее для линейного счётчика.

```kotlin
fun sumDownIterative(value: Long): Long {
    var current = value
    var result = 0L

    while (current > 0L) {
        result = Math.addExact(result, current)
        current -= 1L
    }

    return result
}
```

Не выбирай recursion только ради демонстрации `tailrec`.

## Практический пример: переходы учебной сессии

```kotlin
sealed interface SessionState {
    data class Loading(
        val remainingAttempts: Int
    ) : SessionState

    data class Studying(
        val lessonIndex: Int,
        val totalLessons: Int
    ) : SessionState

    data class Completed(
        val totalLessons: Int
    ) : SessionState

    data class Failed(
        val reason: String
    ) : SessionState
}

sealed interface SessionEvent {
    data object Loaded : SessionEvent
    data object LoadFailed : SessionEvent
    data object LessonCompleted : SessionEvent
}

data class Transition(
    val state: SessionState,
    val event: SessionEvent
)

infix fun SessionState.on(event: SessionEvent): Transition {
    return Transition(this, event)
}

operator fun Transition.invoke(): SessionState {
    return when (val current = state) {
        is SessionState.Loading -> when (event) {
            SessionEvent.Loaded -> SessionState.Studying(
                lessonIndex = 1,
                totalLessons = 3
            )

            SessionEvent.LoadFailed -> if (current.remainingAttempts > 1) {
                current.copy(remainingAttempts = current.remainingAttempts - 1)
            } else {
                SessionState.Failed("Не удалось загрузить курс")
            }

            SessionEvent.LessonCompleted -> current
        }

        is SessionState.Studying -> when (event) {
            SessionEvent.LessonCompleted -> {
                val nextIndex = current.lessonIndex + 1
                if (nextIndex > current.totalLessons) {
                    SessionState.Completed(current.totalLessons)
                } else {
                    current.copy(lessonIndex = nextIndex)
                }
            }

            SessionEvent.Loaded,
            SessionEvent.LoadFailed -> current
        }

        is SessionState.Completed,
        is SessionState.Failed -> current
    }
}

tailrec fun applyEvents(
    state: SessionState,
    events: List<SessionEvent>,
    index: Int = 0
): SessionState {
    if (index >= events.size) return state

    val nextState = (state on events[index])()
    return applyEvents(
        state = nextState,
        events = events,
        index = index + 1
    )
}

fun main() {
    val events = listOf(
        SessionEvent.Loaded,
        SessionEvent.LessonCompleted,
        SessionEvent.LessonCompleted,
        SessionEvent.LessonCompleted
    )

    val initial = SessionState.Loading(remainingAttempts = 3)
    val finalState = applyEvents(initial, events)

    println(finalState)
}
```

Пример использует модификаторы без подмены смысла:

- `state on event` — простой infix-конструктор перехода;
- `transition()` через `invoke` вычисляет следующий state;
- sealed-типы сохраняют исчерпывающие `when`;
- `applyEvents` имеет один прямой self-call в хвостовой позиции;
- index не создаёт `drop(1)` и промежуточные списки на каждом шаге;
- terminal states игнорируют последующие события;
- все ветки возвращают явное состояние без логики-заглушки.

## Итоги

- `infix` разрешён только member/extension-функции с одним обычным параметром без default;
- infix-синтаксис полезен для коротких естественных отношений и DSL;
- смешанные infix/операторные выражения лучше группировать скобками;
- `operator` реализует только фиксированные соглашения Kotlin;
- унарные операции используют `unaryPlus`, `unaryMinus`, `not`, а `++`/`--` — `inc`/`dec`;
- арифметические symbols вызывают `plus`, `minus`, `times`, `div`, `rem`;
- `+=` может использовать `plusAssign` или присваивание результата `plus`;
- `==` основан на member `equals`, а `===` не перегружается;
- сравнения используют `compareTo`, членство — `contains`;
- `[]` использует `get`/`set`, вызов объекта — `invoke`;
- iteration, destructuring и property delegation тоже построены на conventions;
- `tailrec` оптимизирует только прямые self-calls в хвостовой позиции;
- accumulator превращает многие линейные рекурсивные алгоритмы в хвостовую форму;
- ветвящуюся, взаимную и post-processing рекурсию `tailrec` не исправляет;
- знакомый синтаксис обязан сохранять знакомую семантику и стоимость.
