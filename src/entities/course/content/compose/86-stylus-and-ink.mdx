## Где stylus отличается от обычного touch

Stylus input выглядит как еще один pointer, но пользовательские ожидания другие. Палец хорош для нажатия, scroll и broad gestures. Stylus ожидают использовать для handwriting, drawing, annotation, signatures, precise selection, hover preview и pressure-sensitive strokes.

Compose уже умеет получать pointer events, но хороший stylus UX состоит из нескольких слоев:

- text fields принимают handwriting;
- placeholder search field может делегировать handwriting настоящему input;
- pointer input различает stylus, touch и mouse;
- drawing surface учитывает pressure, tilt, orientation и hover;
- wet ink рисуется с минимальной задержкой;
- dry strokes сохраняются как данные;
- brushes являются частью состояния документа;
- keyboard, touch и accessibility остаются альтернативными путями.

```kotlin
import androidx.compose.foundation.background
import androidx.compose.foundation.layout.Box
import androidx.compose.foundation.layout.size
import androidx.compose.runtime.Composable
import androidx.compose.ui.Modifier
import androidx.compose.ui.graphics.Color
import androidx.compose.ui.input.pointer.PointerType
import androidx.compose.ui.input.pointer.pointerInput
import androidx.compose.ui.unit.dp

@Composable
fun StylusAwareSurface(
    onStylusDown: () -> Unit,
    onTouchDown: () -> Unit
) {
    Box(
        modifier = Modifier
            .size(160.dp)
            .background(Color(0xFFE3F2FD))
            .pointerInput(Unit) {
                awaitPointerEventScope {
                    while (true) {
                        val event = awaitPointerEvent()
                        val firstPressed = event.changes.firstOrNull {
                            it.pressed
                        } ?: continue

                        when (firstPressed.type) {
                            PointerType.Stylus,
                            PointerType.Eraser -> onStylusDown()
                            PointerType.Touch -> onTouchDown()
                            else -> Unit
                        }
                    }
                }
            }
    )
}
```

На низком уровне stylus приходит через pointer input. Для text handwriting и ink drawing лучше использовать более специализированные APIs.

<CatTip>
Stylus — это не "палец потоньше". У него есть точность, hover, pressure, кнопки, наклон и сценарии, где задержка в несколько кадров уже ощущается как поломка.
</CatTip>

## Handwriting in text fields

Современный Compose поддерживает stylus handwriting для Material `TextField`. Пользователь может писать stylus рядом с полем, а IME преобразует рукописный ввод в текст.

Минимальный код выглядит как обычный text field:

```kotlin
import androidx.compose.material3.OutlinedTextField
import androidx.compose.material3.Text
import androidx.compose.runtime.Composable
import androidx.compose.ui.Modifier

@Composable
fun HandwritingNameField(
    value: String,
    onValueChange: (String) -> Unit,
    modifier: Modifier = Modifier
) {
    OutlinedTextField(
        value = value,
        onValueChange = onValueChange,
        label = {
            Text(text = "Имя")
        },
        singleLine = true,
        modifier = modifier
    )
}
```

Важно понимать границы:

- handwriting зависит от Android версии, IME и устройства со stylus;
- password keyboard type не должен поддерживать обычный handwriting path;
- поле должно быть настоящим text input, а не просто строкой с иконкой;
- focus и InputConnection должны создаваться корректно;
- тестирование требует реального устройства, совместимого stylus или специальных adb-настроек.

## Handwriting bounds

Handwriting обычно запускается не только строго внутри рамки поля, а в расширенной области вокруг него. Это нужно, чтобы пользователь мог писать естественно, не попадая идеально в маленький прямоугольник.

Из этого следуют практические правила:

- не размещайте два handwriting-enabled поля слишком близко;
- не ставьте destructive action прямо рядом с text field;
- учитывайте плотные формы на tablet;
- проверяйте hit targets в landscape;
- не полагайтесь на pixel-perfect попадание stylus.

<MouseTrap>
Если экран плотный и рядом с полем стоит маленькая кнопка удаления, stylus handwriting может стать конфликтным. Пространство вокруг поля — тоже часть input design.
</MouseTrap>

## Input delegation

Частый паттерн: на экране виден search box, но настоящий input появляется только после перехода на search screen. Для touch это нормально: пользователь нажал, открылся экран, поле получило focus. Для stylus handwriting нужно дать системе понять, что видимый placeholder умеет делегировать handwriting настоящему полю.

Compose предоставляет `handwritingDetector` и `handwritingHandler`. Идея такая:

- placeholder получает stylus handwriting gesture;
- приложение открывает реальный input;
- настоящий `TextField` получает focus;
- handwriting session продолжается уже в functional input.

```kotlin
import androidx.compose.foundation.clickable
import androidx.compose.foundation.layout.Box
import androidx.compose.foundation.layout.fillMaxWidth
import androidx.compose.foundation.layout.padding
import androidx.compose.material3.Text
import androidx.compose.runtime.Composable
import androidx.compose.ui.Modifier
import androidx.compose.ui.input.pointer.pointerHoverIcon
import androidx.compose.ui.platform.LocalFocusManager
import androidx.compose.ui.unit.dp

@Composable
fun SearchPlaceholder(
    onOpenSearch: () -> Unit,
    modifier: Modifier = Modifier
) {
    Box(
        modifier = modifier
            .fillMaxWidth()
            .clickable(onClick = onOpenSearch)
            .padding(16.dp)
    ) {
        Text(text = "Поиск")
    }
}
```

Этот пример намеренно показывает только UI-shell. В реальном stylus delegation добавляются handwriting-specific modifiers и focus handoff. Главное правило: placeholder не должен быть тупиком. Если пользователь начал писать stylus по search surface, приложение должно быстро передать ввод настоящему полю.

## Pointer type, pressure и historical points

На низком уровне Compose pointer input может отличать `PointerType.Stylus` и `PointerType.Eraser`. Для рисования часто важно не только текущее положение, но и поток точек.

```kotlin
import androidx.compose.foundation.Canvas
import androidx.compose.foundation.gestures.awaitEachGesture
import androidx.compose.foundation.layout.fillMaxSize
import androidx.compose.runtime.Composable
import androidx.compose.runtime.getValue
import androidx.compose.runtime.mutableStateOf
import androidx.compose.runtime.remember
import androidx.compose.runtime.setValue
import androidx.compose.ui.Modifier
import androidx.compose.ui.geometry.Offset
import androidx.compose.ui.graphics.Color
import androidx.compose.ui.graphics.Path
import androidx.compose.ui.graphics.drawscope.Stroke
import androidx.compose.ui.input.pointer.PointerType
import androidx.compose.ui.input.pointer.changedToDown
import androidx.compose.ui.input.pointer.pointerInput

data class SimpleStrokePoint(
    val position: Offset,
    val pressure: Float
)

@Composable
fun SimpleStylusSketchPad(
    modifier: Modifier = Modifier
) {
    var points by remember {
        mutableStateOf(emptyList<SimpleStrokePoint>())
    }

    Canvas(
        modifier = modifier
            .fillMaxSize()
            .pointerInput(Unit) {
                awaitEachGesture {
                    while (true) {
                        val event = awaitPointerEvent()
                        val change = event.changes.firstOrNull() ?: continue

                        if (change.changedToDown() && change.type != PointerType.Stylus) {
                            break
                        }

                        if (change.pressed && change.type == PointerType.Stylus) {
                            points = points + SimpleStrokePoint(
                                position = change.position,
                                pressure = change.pressure
                            )
                            change.consume()
                        }

                        if (!change.pressed) {
                            break
                        }
                    }
                }
            }
    ) {
        if (points.size > 1) {
            val path = Path().apply {
                moveTo(points.first().position.x, points.first().position.y)
                points.drop(1).forEach { point ->
                    lineTo(point.position.x, point.position.y)
                }
            }

            drawPath(
                path = path,
                color = Color.Black,
                style = Stroke(
                    width = 4f + points.last().pressure * 8f
                )
            )
        }
    }
}
```

Этот пример учебный. Он не делает сглаживание, не разбивает strokes, не хранит документ и не решает latency. Для production drawing лучше смотреть в сторону Ink API.

## Почему Canvas недостаточно для serious ink

`Canvas` отлично подходит для custom drawing, но handwriting/drawing app требует большего:

- низкая задержка между движением пера и пикселем;
- сглаживание и предсказание движения;
- pressure-sensitive geometry;
- high-quality anti-aliased rendering;
- eraser и selection tools;
- сохранение stroke input, а не только bitmap;
- масштабирование без потери качества;
- экспорт документа;
- undo/redo на уровне strokes;
- стабильная модель координат.

Если просто рисовать линии в bitmap, вы быстро потеряете редактируемость. Пользователь нарисовал stroke, а приложение сохранило только пиксели. Потом нельзя качественно стереть отдельный stroke, поменять brush, пересчитать масштаб или экспортировать в высоком разрешении.

## Ink API

Ink API — это набор Jetpack-библиотек для ink-сценариев. Он разделяет задачу на модули:

- strokes: данные stroke и input batches;
- brush: стиль stroke;
- geometry: пересечения, трансформации, selection, eraser;
- rendering: отрисовка stroke;
- authoring compose: сбор in-progress strokes из Compose input;
- storage: serialization stroke input;
- native loader: загрузка native компонентов.

```kotlin
dependencies {
    val inkVersion = "1.0.0"

    implementation("androidx.ink:ink-nativeloader:$inkVersion")
    implementation("androidx.ink:ink-rendering:$inkVersion")
    implementation("androidx.ink:ink-strokes:$inkVersion")
    implementation("androidx.ink:ink-authoring-compose:$inkVersion")
    implementation("androidx.ink:ink-brush-compose:$inkVersion")
    implementation("androidx.ink:ink-geometry-compose:$inkVersion")
    implementation("androidx.ink:ink-storage:$inkVersion")
}
```

В production проекте версию лучше держать в version catalog:

```toml
[versions]
ink = "1.0.0"

[libraries]
androidx-ink-rendering = { module = "androidx.ink:ink-rendering", version.ref = "ink" }
androidx-ink-strokes = { module = "androidx.ink:ink-strokes", version.ref = "ink" }
androidx-ink-authoring-compose = { module = "androidx.ink:ink-authoring-compose", version.ref = "ink" }
```

## Brush model

Brush описывает, как stroke должен выглядеть. Это не Compose `Brush` из graphics gradients. У Ink API есть свой brush domain:

- brush family;
- color;
- base size;
- epsilon;
- копирование brush с измененными параметрами.

```kotlin
import androidx.compose.ui.graphics.Color
import androidx.ink.brush.Brush
import androidx.ink.brush.StockBrushes
import androidx.ink.brush.compose.createWithComposeColor

fun defaultPressurePen(): Brush {
    return Brush.createWithComposeColor(
        family = StockBrushes.pressurePen(),
        color = Color(0xFF1A1A1A),
        size = 6f,
        epsilon = 0.1f
    )
}
```

`epsilon` влияет на геометрию stroke: чем меньше значение, тем больше точности и потенциально больше памяти/CPU. Значение не должно случайно меняться между сессиями, иначе документ будет вести себя непредсказуемо при zoom и rendering.

<CatTip>
В ink-приложении brush — часть модели документа, а не просто цвет кнопки в toolbar. Если stroke сохранен без brush, вы сохранили половину смысла.
</CatTip>

## Wet strokes и dry strokes

В inking обычно различают:

- wet stroke: stroke, который пользователь рисует прямо сейчас;
- dry stroke: завершенный stroke, который уже committed в документ.

Wet stroke должен быть быстрым. Dry stroke должен быть надежным, редактируемым и сохраняемым.

```kotlin
import androidx.compose.foundation.Canvas
import androidx.compose.foundation.layout.Box
import androidx.compose.foundation.layout.fillMaxSize
import androidx.compose.runtime.Composable
import androidx.compose.runtime.remember
import androidx.compose.ui.Modifier
import androidx.compose.ui.graphics.Matrix
import androidx.ink.authoring.compose.InProgressStrokes
import androidx.ink.brush.Brush
import androidx.ink.rendering.android.canvas.CanvasStrokeRenderer
import androidx.ink.strokes.Stroke

@Composable
fun InkDrawingSurface(
    finishedStrokes: List<Stroke>,
    currentBrush: Brush,
    onGetNextBrush: () -> Brush,
    onStrokesFinished: (List<Stroke>) -> Unit,
    modifier: Modifier = Modifier
) {
    val renderer = remember {
        CanvasStrokeRenderer.create()
    }

    Box(modifier = modifier.fillMaxSize()) {
        Canvas(modifier = Modifier.fillMaxSize()) {
            finishedStrokes.forEach { stroke ->
                renderer.draw(
                    stroke = stroke,
                    canvas = drawContext.canvas.nativeCanvas,
                    strokeToScreenTransform = Matrix()
                )
            }
        }

        InProgressStrokes(
            defaultBrush = currentBrush,
            nextBrush = onGetNextBrush,
            onStrokesFinished = onStrokesFinished,
            modifier = Modifier.fillMaxSize()
        )
    }
}
```

Имена типов и transform API могут отличаться по версии Ink. Архитектурная идея стабильна: in-progress layer собирает новые strokes, callback передает finished strokes, permanent layer рисует сохраненный список.

## Hoisting stroke state

`InProgressStrokes` не должен быть владельцем документа. Документ лучше хранить выше:

```kotlin
import androidx.compose.runtime.Composable
import androidx.compose.runtime.getValue
import androidx.compose.runtime.mutableStateOf
import androidx.compose.runtime.remember
import androidx.compose.runtime.setValue
import androidx.ink.brush.Brush
import androidx.ink.strokes.Stroke

data class InkDocumentState(
    val strokes: List<Stroke> = emptyList(),
    val selectedBrush: Brush
)

@Composable
fun RememberedInkDocument(
    initialBrush: Brush
) {
    var document by remember {
        mutableStateOf(
            InkDocumentState(
                selectedBrush = initialBrush
            )
        )
    }

    InkDrawingSurface(
        finishedStrokes = document.strokes,
        currentBrush = document.selectedBrush,
        onGetNextBrush = {
            document.selectedBrush
        },
        onStrokesFinished = { newStrokes ->
            document = document.copy(
                strokes = document.strokes + newStrokes
            )
        }
    )
}
```

Так вы получаете основу для undo/redo, persistence и collaboration. Если stroke остается внутренним mutable state внутри drawing surface, приложение трудно расширять.

## Undo и redo

Undo в ink-документе лучше делать на уровне операций, а не bitmap snapshots.

```kotlin
import androidx.ink.strokes.Stroke

data class InkHistory(
    val strokes: List<Stroke> = emptyList(),
    val undone: List<Stroke> = emptyList()
) {
    fun add(newStrokes: List<Stroke>): InkHistory {
        return copy(
            strokes = strokes + newStrokes,
            undone = emptyList()
        )
    }

    fun undo(): InkHistory {
        if (strokes.isEmpty()) {
            return this
        }

        val removed = strokes.last()
        return copy(
            strokes = strokes.dropLast(1),
            undone = undone + removed
        )
    }

    fun redo(): InkHistory {
        if (undone.isEmpty()) {
            return this
        }

        val restored = undone.last()
        return copy(
            strokes = strokes + restored,
            undone = undone.dropLast(1)
        )
    }
}
```

В реальном редакторе операция может быть не только "добавить stroke", но и "стереть stroke", "переместить selection", "изменить brush", "вставить изображение".

## Eraser и geometry

Eraser можно реализовать грубо: удалять stroke, который пересекся с областью ластика. Более сложный вариант — разрезать stroke и оставить непересеченные сегменты.

```kotlin
import androidx.ink.geometry.AffineTransform
import androidx.ink.geometry.MutableParallelogram
import androidx.ink.geometry.MutableSegment
import androidx.ink.geometry.MutableVec
import androidx.ink.strokes.Stroke

fun eraseIntersectingStrokes(
    previousPoint: MutableVec,
    currentPoint: MutableVec,
    eraserPadding: Float,
    strokes: List<Stroke>
): List<Stroke> {
    val segment = MutableSegment(
        previousPoint,
        currentPoint
    )
    val eraserArea = MutableParallelogram()
        .populateFromSegmentAndPadding(
            segment,
            eraserPadding
        )

    return strokes.filterNot { stroke ->
        stroke.shape.intersects(
            eraserArea,
            AffineTransform.IDENTITY
        )
    }
}
```

Этот подход прост и понятен, но может быть слишком грубым для художественного редактора. Для заметок, где stroke является единицей смысла, он часто приемлем.

## Coordinate system и zoom

Ink-приложение почти всегда приходит к нескольким системам координат:

- screen coordinates: пиксели текущего viewport;
- world coordinates: координаты документа;
- stroke coordinates: координаты конкретного stroke;
- camera transform: pan/zoom/rotation между world и screen.

Если приложение поддерживает zoom, нельзя сохранять stroke только в screen coordinates. Иначе при изменении масштаба новые strokes окажутся в другой системе.

```kotlin
data class DrawingViewport(
    val scale: Float = 1f,
    val offsetX: Float = 0f,
    val offsetY: Float = 0f
) {
    fun screenToWorld(x: Float, y: Float): Pair<Float, Float> {
        return Pair(
            (x - offsetX) / scale,
            (y - offsetY) / scale
        )
    }

    fun worldToScreen(x: Float, y: Float): Pair<Float, Float> {
        return Pair(
            x * scale + offsetX,
            y * scale + offsetY
        )
    }
}
```

Для Ink API важно выбрать стабильную world unit model и `epsilon`. Обычно удобно думать в density-independent единицах и не менять смысл "1 world unit" в середине жизни документа.

## Сохранение strokes

`rememberSaveable` не умеет магически сохранять сложные Ink objects. Для конфигурационных изменений и persistent storage нужны DTO и converters.

```kotlin
import kotlinx.serialization.Serializable

@Serializable
data class SerializedInkDocument(
    val strokes: List<SerializedStroke>,
    val schemaVersion: Int = 1
)

@Serializable
data class SerializedStroke(
    val inputBatch: ByteArray,
    val brush: SerializedBrush
)

@Serializable
data class SerializedBrush(
    val familyKey: String,
    val color: Long,
    val size: Float,
    val epsilon: Float
)
```

Практический подход:

- сохранять raw stroke input через storage module;
- сохранять brush отдельно как компактный DTO;
- версионировать schema;
- не хранить только bitmap;
- делать migration для старых документов;
- покрыть converters unit-тестами.

<MouseTrap>
Скриншот рисунка не является сохраненным ink-документом. Это export preview. Документ должен хранить strokes, brush и координатную модель.
</MouseTrap>

## Accessibility для drawing surfaces

Canvas или Ink surface сами по себе не становятся доступными. Нужны альтернативные действия и описания.

Что можно сделать:

- дать surface понятное `contentDescription`;
- добавить custom actions для undo/redo;
- продублировать critical actions в toolbar;
- сделать tools доступными через keyboard;
- не требовать stylus как единственный input;
- дать текстовую summary для результата, если это annotation workflow;
- позволить импорт/экспорт через обычные действия.

```kotlin
import androidx.compose.foundation.Canvas
import androidx.compose.foundation.layout.fillMaxSize
import androidx.compose.runtime.Composable
import androidx.compose.ui.Modifier
import androidx.compose.ui.semantics.CustomAccessibilityAction
import androidx.compose.ui.semantics.contentDescription
import androidx.compose.ui.semantics.customActions
import androidx.compose.ui.semantics.semantics

@Composable
fun AccessibleDrawingCanvas(
    canUndo: Boolean,
    onUndo: () -> Unit,
    modifier: Modifier = Modifier
) {
    Canvas(
        modifier = modifier
            .fillMaxSize()
            .semantics {
                contentDescription = "Область рисования"
                if (canUndo) {
                    customActions = listOf(
                        CustomAccessibilityAction(
                            label = "Отменить последний штрих",
                            action = {
                                onUndo()
                                true
                            }
                        )
                    )
                }
            }
    ) {
        // Здесь рисуются сохраненные штрихи документа.
    }
}
```

Если приложение для подписей требует только stylus и не дает keyboard/touch alternative, часть пользователей не сможет выполнить сценарий.

## Testing stylus

Stylus testing сложнее обычного click testing:

- handwriting зависит от IME и Android версии;
- pressure/tilt требуют совместимого hardware;
- latency нельзя оценить обычным unit-тестом;
- storage и converters можно тестировать без UI;
- undo/redo state machine нужно тестировать отдельно;
- accessibility alternatives можно тестировать через semantics;
- drawing performance нужно проверять macrobenchmark/profiling;
- ручная проверка на tablet/ChromeOS остается обязательной.

Пример unit-теста для history:

```kotlin
import com.google.common.truth.Truth.assertThat
import org.junit.Test

class InkHistoryTest {

    @Test
    fun undo_movesLastStrokeToRedoStack() {
        val first = fakeStroke("first")
        val second = fakeStroke("second")

        val history = InkHistory(
            strokes = listOf(first, second)
        ).undo()

        assertThat(history.strokes).containsExactly(first)
        assertThat(history.undone).containsExactly(second)
    }
}
```

`fakeStroke` здесь обозначает test fixture. В настоящем проекте лучше иметь builder для минимального валидного stroke.

## Типичные ошибки

### Считать stylus обычным touch

Так теряются pressure, hover, handwriting и точность.

### Рисовать только в bitmap

Bitmap удобен для export, но плох как основной формат документа.

### Не разделять wet и dry strokes

Live drawing требует другого performance path, чем сохраненные strokes.

### Менять epsilon случайно

Документ становится нестабильным при zoom и восстановлении.

### Хранить brush только в UI state

После восстановления stroke невозможно корректно перерисовать.

### Делать eraser destructive без undo

Пользователь ожидает, что удаление strokes можно отменить.

### Не давать альтернативу stylus

Stylus может быть удобным input, но не должен быть единственным способом выполнить обязательное действие.

## Итоги

Stylus и Ink в Compose требуют отдельной модели мышления:

- Material `TextField` поддерживает stylus handwriting в совместимых условиях;
- placeholder input должен делегировать handwriting настоящему text field;
- pointer input позволяет отличать stylus, eraser, touch и mouse;
- pressure, hover и tilt важны для качественного pen UX;
- `Canvas` подходит для простых demos, но serious ink лучше строить на Ink API;
- Ink API разделяет brush, strokes, geometry, rendering, authoring и storage;
- wet strokes рисуются как live layer, dry strokes хранятся в документе;
- brush является частью persisted state;
- coordinate system и epsilon нужно выбрать осознанно;
- undo/redo лучше строить на уровне операций со strokes;
- drawing surface нуждается в semantics и альтернативных actions.
