## Зачем нужны nested graphs

Nested graph — это навигационный граф внутри другого графа. Он нужен, когда часть приложения является самостоятельным flow: onboarding, login, checkout, создание заказа, регистрация, настройки профиля, игра, мастер заполнения анкеты.

Без вложенных графов root graph быстро превращается в длинный плоский список destinations. В нем сложно понять, какие экраны относятся к одному flow, какой у flow стартовый экран, какие destinations являются внутренними, а какие действительно top-level.

Nested graphs решают несколько задач:

- группируют связанные destinations;
- задают отдельный `startDestination` для flow;
- скрывают внутреннюю структуру flow от root graph;
- позволяют менять внутренние шаги без изменения внешней навигации;
- помогают scope-ить ViewModel и state к конкретному graph;
- упрощают feature-модульность;
- делают back stack понятнее.

```kotlin
import kotlinx.serialization.Serializable

@Serializable
data object CheckoutGraph

@Serializable
data object CartRoute

@Serializable
data object AddressRoute

@Serializable
data object PaymentRoute

@Serializable
data object ConfirmationRoute
```

`CheckoutGraph` — route самого nested graph. `CartRoute`, `AddressRoute`, `PaymentRoute`, `ConfirmationRoute` — destinations внутри checkout flow.

<CatTip>
Вложенный граф полезен, когда у группы экранов есть собственная логика входа, выхода и внутреннего порядка. Если у вас просто два независимых экрана, nested graph может быть лишней церемонией.
</CatTip>

## `navigation<T>`

В type-safe Navigation Compose вложенный граф создается через `navigation<T>(startDestination = StartRoute)`.

```kotlin
import androidx.compose.material3.Text
import androidx.compose.runtime.Composable
import androidx.navigation.compose.NavHost
import androidx.navigation.compose.composable
import androidx.navigation.compose.navigation
import androidx.navigation.compose.rememberNavController

@Composable
fun NestedGraphBasicExample() {
    val navController = rememberNavController()

    NavHost(
        navController = navController,
        startDestination = CheckoutGraph
    ) {
        navigation<CheckoutGraph>(
            startDestination = CartRoute
        ) {
            composable<CartRoute> {
                CartScreen(
                    onContinueClick = {
                        navController.navigate(AddressRoute)
                    }
                )
            }

            composable<AddressRoute> {
                AddressScreen(
                    onContinueClick = {
                        navController.navigate(PaymentRoute)
                    },
                    onBackClick = {
                        navController.popBackStack()
                    }
                )
            }

            composable<PaymentRoute> {
                PaymentScreen(
                    onPaymentComplete = {
                        navController.navigate(ConfirmationRoute) {
                            // После успешной оплаты убираем шаг оплаты из возврата.
                            popUpTo(PaymentRoute) {
                                inclusive = true
                            }
                        }
                    },
                    onBackClick = {
                        navController.popBackStack()
                    }
                )
            }

            composable<ConfirmationRoute> {
                ConfirmationScreen(
                    onFinishClick = {
                        navController.popBackStack(
                            route = CheckoutGraph,
                            inclusive = true
                        )
                    }
                )
            }
        }
    }
}

@Composable
fun CartScreen(
    onContinueClick: () -> Unit
) {
    Text(text = "Корзина")
}

@Composable
fun AddressScreen(
    onContinueClick: () -> Unit,
    onBackClick: () -> Unit
) {
    Text(text = "Адрес доставки")
}

@Composable
fun PaymentScreen(
    onPaymentComplete: () -> Unit,
    onBackClick: () -> Unit
) {
    Text(text = "Оплата")
}

@Composable
fun ConfirmationScreen(
    onFinishClick: () -> Unit
) {
    Text(text = "Заказ оформлен")
}
```

`navigation<CheckoutGraph>` сам по себе не рисует экран. Он создает container graph. Когда пользователь переходит на `CheckoutGraph`, Navigation открывает его `startDestination`, то есть `CartRoute`.

## Root graph и feature graph

В боевой коде вложенный graph лучше вынести в extension function на `NavGraphBuilder`.

```kotlin
import androidx.navigation.NavGraphBuilder
import androidx.navigation.NavHostController
import androidx.navigation.compose.composable
import androidx.navigation.compose.navigation

fun NavGraphBuilder.checkoutGraph(
    navController: NavHostController,
    onCheckoutFinished: () -> Unit
) {
    navigation<CheckoutGraph>(
        startDestination = CartRoute
    ) {
        composable<CartRoute> {
            CartScreen(
                onContinueClick = {
                    navController.navigate(AddressRoute)
                }
            )
        }

        composable<AddressRoute> {
            AddressScreen(
                onContinueClick = {
                    navController.navigate(PaymentRoute)
                },
                onBackClick = {
                    navController.popBackStack()
                }
            )
        }

        composable<PaymentRoute> {
            PaymentScreen(
                onPaymentComplete = {
                    navController.navigate(ConfirmationRoute) {
                        popUpTo(PaymentRoute) {
                            inclusive = true
                        }
                    }
                },
                onBackClick = {
                    navController.popBackStack()
                }
            )
        }

        composable<ConfirmationRoute> {
            ConfirmationScreen(
                onFinishClick = onCheckoutFinished
            )
        }
    }
}
```

Root graph становится гораздо проще:

```kotlin
import androidx.compose.runtime.Composable
import androidx.navigation.compose.NavHost
import androidx.navigation.compose.composable
import androidx.navigation.compose.rememberNavController
import kotlinx.serialization.Serializable

@Serializable
data object HomeRoute

@Composable
fun ShopAppNavHost() {
    val navController = rememberNavController()

    NavHost(
        navController = navController,
        startDestination = HomeRoute
    ) {
        composable<HomeRoute> {
            HomeScreen(
                onCheckoutClick = {
                    navController.navigate(CheckoutGraph)
                }
            )
        }

        checkoutGraph(
            navController = navController,
            onCheckoutFinished = {
                navController.popBackStack(
                    route = CheckoutGraph,
                    inclusive = true
                )
            }
        )
    }
}

@Composable
fun HomeScreen(
    onCheckoutClick: () -> Unit
) {
    androidx.compose.material3.Text(text = "Главный экран магазина")
}
```

Такой стиль превращает feature graph в отдельный строительный блок. Root graph знает только, что существует checkout flow, но не обязан держать всю его внутреннюю разметку в одном файле.

<MouseTrap>
Не тащите все destinations всех feature в один огромный `NavHost`. Навигационный граф должен читаться как карта продукта, а не как список всех composable-функций проекта.
</MouseTrap>

## Encapsulation вложенного графа

Вложенный graph должен иметь внешний API:

- route самого graph;
- функции навигации в graph;
- функцию `NavGraphBuilder.featureGraph`;
- обратные вызовы наружу, если flow завершился или запросил глобальную навигацию.

```kotlin
import androidx.navigation.NavController
import kotlinx.serialization.Serializable

@Serializable
data object CheckoutGraph

fun NavController.navigateToCheckout() {
    navigate(CheckoutGraph) {
        // Повторное открытие checkout не создает новую копию flow сверху.
        launchSingleTop = true
    }
}
```

Внутренние шаги checkout можно оставить internal для feature module:

```kotlin
import kotlinx.serialization.Serializable

@Serializable
internal data object CartRoute

@Serializable
internal data object AddressRoute

@Serializable
internal data object PaymentRoute

@Serializable
internal data object ConfirmationRoute
```

Если модульная структура позволяет, наружу лучше не экспортировать внутренние route. Тогда остальное приложение не сможет прыгать прямо в середину checkout flow без согласованного сценария.

## Graph-scoped state

Иногда нескольким destinations внутри nested graph нужен общий state. Например, checkout хранит корзину, адрес, способ доставки и платежный метод. Можно использовать ViewModel, scoped к graph.

В type-safe API для graph-scoped ViewModel проектные команды часто создают небольшой helper вокруг route graph. В примере ниже показана идея через строковый route, потому что этот перегрузка стабильно понятен и явно демонстрирует lifecycle graph entry.

```kotlin
const val CheckoutGraphRoute = "checkout"
const val CartStringRoute = "checkout/cart"
const val AddressStringRoute = "checkout/address"
```

```kotlin
import androidx.compose.runtime.Composable
import androidx.compose.runtime.remember
import androidx.lifecycle.ViewModel
import androidx.lifecycle.viewmodel.compose.viewModel
import androidx.navigation.NavBackStackEntry
import androidx.navigation.NavHostController
import androidx.navigation.compose.composable
import androidx.navigation.compose.navigation

data class CheckoutDraft(
    val address: String = "",
    val paymentMethodId: String? = null
)

class CheckoutViewModel : ViewModel() {
    var draft = CheckoutDraft()
        private set

    fun updateAddress(address: String) {
        draft = draft.copy(address = address)
    }

    fun selectPaymentMethod(paymentMethodId: String) {
        draft = draft.copy(paymentMethodId = paymentMethodId)
    }
}

@Composable
fun rememberCheckoutViewModel(
    navController: NavHostController,
    backStackEntry: NavBackStackEntry
): CheckoutViewModel {
    val parentEntry = remember(backStackEntry) {
        navController.getBackStackEntry(CheckoutGraphRoute)
    }

    return viewModel(
        viewModelStoreOwner = parentEntry
    )
}

fun NavGraphBuilder.checkoutStringGraph(
    navController: NavHostController
) {
    navigation(
        route = CheckoutGraphRoute,
        startDestination = CartStringRoute
    ) {
        composable(CartStringRoute) { backStackEntry ->
            val checkoutViewModel = rememberCheckoutViewModel(
                navController = navController,
                backStackEntry = backStackEntry
            )

            CartStepScreen(
                draft = checkoutViewModel.draft,
                onContinueClick = {
                    navController.navigate(AddressStringRoute)
                }
            )
        }

        composable(AddressStringRoute) { backStackEntry ->
            val checkoutViewModel = rememberCheckoutViewModel(
                navController = navController,
                backStackEntry = backStackEntry
            )

            AddressStepScreen(
                address = checkoutViewModel.draft.address,
                onAddressChange = checkoutViewModel::updateAddress
            )
        }
    }
}

@Composable
fun CartStepScreen(
    draft: CheckoutDraft,
    onContinueClick: () -> Unit
) {
    androidx.compose.material3.Text(text = "Адрес: ${draft.address}")
}

@Composable
fun AddressStepScreen(
    address: String,
    onAddressChange: (String) -> Unit
) {
    androidx.compose.material3.Text(text = "Текущий адрес: $address")
}
```

Graph-scoped ViewModel живет, пока graph находится в back stack. Когда checkout graph удаляется через `popBackStack(CheckoutGraphRoute, inclusive = true)`, ViewModel очищается.

<CatTip>
Общий state вложенного flow не обязан быть глобальным singleton. Graph-scoped ViewModel хорошо подходит для временного сценария: wizard, checkout, onboarding, multi-step form.
</CatTip>

## Multiple back stacks

Multiple back stacks нужны, когда приложение имеет несколько top-level flows, между которыми пользователь свободно переключается: bottom navigation, navigation rail, drawer. Вкладка "Лента" может быть глубоко в деталях статьи, вкладка "Поиск" — в результатах, вкладка "Профиль" — в настройках. При переключении между вкладками пользователь ожидает вернуться туда, где остановился.

Navigation поддерживает это через сохранение и восстановление состояния back stack:

- `popUpTo(startDestination) { saveState = true }` сохраняет состояние stack, который уходит с экрана;
- `restoreState = true` восстанавливает ранее сохраненное состояние destination;
- `launchSingleTop = true` не создает duplicate destination сверху;
- `findStartDestination()` помогает найти стартовый destination root graph.

```kotlin
import androidx.navigation.NavController
import androidx.navigation.NavGraph.Companion.findStartDestination

fun NavController.navigateToTopLevel(
    route: Any
) {
    navigate(route) {
        popUpTo(graph.findStartDestination().id) {
            // Сохраняем state текущего top-level stack.
            saveState = true
        }

        // Не добавляем вторую копию той же top-level destination сверху.
        launchSingleTop = true

        // Возвращаем сохраненный stack выбранной вкладки, если он есть.
        restoreState = true
    }
}
```

Этот helper работает как идея для typed routes, но в реальном коде часто делают перегрузки или extension functions для каждого top-level destination, чтобы API был точнее.

## Bottom navigation с несколькими stacks

Соберем пример с тремя top-level destinations.

```kotlin
import androidx.compose.material.icons.Icons
import androidx.compose.material.icons.filled.Bookmark
import androidx.compose.material.icons.filled.Home
import androidx.compose.material.icons.filled.Person
import androidx.compose.material.icons.outlined.BookmarkBorder
import androidx.compose.material.icons.outlined.Home
import androidx.compose.material.icons.outlined.Person
import androidx.compose.ui.graphics.vector.ImageVector
import kotlinx.serialization.Serializable

@Serializable
data object FeedGraph

@Serializable
data object SearchGraph

@Serializable
data object ProfileGraph

@Serializable
data object FeedRoute

@Serializable
data object SearchHomeRoute

@Serializable
data object ProfileRoute

data class TopLevelDestination(
    val graphRoute: Any,
    val label: String,
    val selectedIcon: ImageVector,
    val unselectedIcon: ImageVector
)

val topLevelDestinations = listOf(
    TopLevelDestination(
        graphRoute = FeedGraph,
        label = "Лента",
        selectedIcon = Icons.Filled.Home,
        unselectedIcon = Icons.Outlined.Home
    ),
    TopLevelDestination(
        graphRoute = SearchGraph,
        label = "Поиск",
        selectedIcon = Icons.Filled.Bookmark,
        unselectedIcon = Icons.Outlined.BookmarkBorder
    ),
    TopLevelDestination(
        graphRoute = ProfileGraph,
        label = "Профиль",
        selectedIcon = Icons.Filled.Person,
        unselectedIcon = Icons.Outlined.Person
    )

```

```kotlin
import androidx.compose.foundation.layout.padding
import androidx.compose.material3.Icon
import androidx.compose.material3.NavigationBar
import androidx.compose.material3.NavigationBarItem
import androidx.compose.material3.Scaffold
import androidx.compose.material3.Text
import androidx.compose.runtime.Composable
import androidx.compose.runtime.getValue
import androidx.compose.ui.Modifier
import androidx.navigation.NavDestination
import androidx.navigation.NavDestination.Companion.hierarchy
import androidx.navigation.NavGraph.Companion.findStartDestination
import androidx.navigation.compose.NavHost
import androidx.navigation.compose.composable
import androidx.navigation.compose.currentBackStackEntryAsState
import androidx.navigation.compose.navigation
import androidx.navigation.compose.rememberNavController
import androidx.navigation.hasRoute

@Composable
fun MultiBackStackApp() {
    val navController = rememberNavController()
    val backStackEntry by navController.currentBackStackEntryAsState()
    val currentDestination = backStackEntry?.destination

    Scaffold(
        bottomBar = {
            NavigationBar {
                topLevelDestinations.forEach { destination ->
                    val selected = currentDestination.isInTopLevelGraph(
                        graphRoute = destination.graphRoute
                    )

                    NavigationBarItem(
                        selected = selected,
                        onClick = {
                            navController.navigate(destination.graphRoute) {
                                popUpTo(navController.graph.findStartDestination().id) {
                                    saveState = true
                                }
                                launchSingleTop = true
                                restoreState = true
                            }
                        },
                        icon = {
                            Icon(
                                imageVector = if (selected) {
                                    destination.selectedIcon
                                } else {
                                    destination.unselectedIcon
                                },
                                contentDescription = destination.label
                            )
                        },
                        label = {
                            Text(text = destination.label)
                        }
                    )
                }
            }
        }
    ) { innerPadding ->
        NavHost(
            navController = navController,
            startDestination = FeedGraph,
            modifier = Modifier.padding(innerPadding)
        ) {
            navigation<FeedGraph>(
                startDestination = FeedRoute
            ) {
                composable<FeedRoute> {
                    Text(text = "Лента")
                }
            }

            navigation<SearchGraph>(
                startDestination = SearchHomeRoute
            ) {
                composable<SearchHomeRoute> {
                    Text(text = "Поиск")
                }
            }

            navigation<ProfileGraph>(
                startDestination = ProfileRoute
            ) {
                composable<ProfileRoute> {
                    Text(text = "Профиль")
                }
            }
        }
    }
}

private fun NavDestination?.isInTopLevelGraph(
    graphRoute: Any
): Boolean {
    if (this == null) {
        return false
    }

    return hierarchy.any { destination ->
        when (graphRoute) {
            FeedGraph -> destination.hasRoute<FeedGraph>()
            SearchGraph -> destination.hasRoute<SearchGraph>()
            ProfileGraph -> destination.hasRoute<ProfileGraph>()
            else -> false
        }
    }
}
```

`hierarchy` важен: текущий destination может быть внутренним экраном внутри `FeedGraph`, но bottom bar должен подсвечивать всю вкладку "Лента".

## Почему одного back stack мало

Представьте сценарий:

1. Пользователь открывает "Лента".
2. Переходит в детали статьи.
3. Открывает "Поиск".
4. Вводит запрос и открывает результат.
5. Возвращается в "Лента".

Если у приложения один плоский stack без save/restore, возврат в "Лента" может открыть стартовый экран, а не детали статьи. Multiple back stacks сохраняет место пользователя в каждом top-level flow.

## popUpTo, saveState и restoreState подробно

`popUpTo(graph.findStartDestination().id)` говорит: при переключении top-level destination очисти back stack до стартового destination root graph.

`saveState = true` говорит: все destinations, которые удаляются этим pop, нужно сохранить.

`restoreState = true` говорит: если мы уже сохраняли destination, на который сейчас переходим, восстанови его состояние.

`launchSingleTop = true` говорит: если выбранная destination уже сверху, не добавляй ее повторно.

```kotlin
import androidx.navigation.NavController
import androidx.navigation.NavGraph.Companion.findStartDestination

fun NavController.switchMainSection(
    section: Any
) {
    navigate(section) {
        popUpTo(graph.findStartDestination().id) {
            saveState = true
        }
        restoreState = true
        launchSingleTop = true
    }
}
```

Порядок этих options не меняет смысл, но в команде полезно держать единый стиль записи.

## Conditional navigation внутри graph

Частый сценарий: app graph стартует с main graph, но внутри main graph первый экран проверяет session и отправляет пользователя в auth graph.

```kotlin
import androidx.compose.runtime.Composable
import androidx.compose.runtime.LaunchedEffect
import androidx.compose.material3.Text
import androidx.navigation.NavHostController
import kotlinx.serialization.Serializable

@Serializable
data object AuthGraph

@Serializable
data object LoginRoute

@Composable
fun SessionGateRoute(
    authenticated: Boolean,
    navController: NavHostController
) {
    LaunchedEffect(authenticated) {
        if (!authenticated) {
            navController.navigate(AuthGraph) {
                // Убираем protected flow из возврата, если пользователь не авторизован.
                popUpTo(FeedGraph) {
                    inclusive = true
                }
            }
        }
    }

    Text(text = "Проверяем сессию")
}
```

Для таких сценариев важно не выполнять navigation прямо в теле composable. Используйте `LaunchedEffect`, потому что navigation является побочный эффект.

<MouseTrap>
Не вызывайте `navController.navigate()` непосредственно во время composition на основании `if (state)`. Такой код может выполниться повторно при recomposition. Навигация должна происходить из event handler или effect.
</MouseTrap>

## Возврат из вложенного flow

У nested flow должен быть явный способ завершения. Например, после checkout нужно убрать весь checkout graph из stack.

```kotlin
import androidx.navigation.NavController

fun NavController.finishCheckout() {
    popBackStack(
        route = CheckoutGraph,
        inclusive = true
    )
}
```

Если пользователь отменяет checkout, поведение может отличаться:

```kotlin
fun NavController.cancelCheckout() {
    popBackStack(
        route = CheckoutGraph,
        inclusive = true
    )
}
```

Технически функции одинаковые, но отдельные имена улучшают читаемость. В будущем `finishCheckout` может отправлять на success screen, а `cancelCheckout` — возвращать в корзину.

## Навигация прямо во внутренний destination

Navigation позволяет перейти прямо к route внутри nested graph, потому что route является глобальным идентификатором destination. Но делать это нужно осторожно.

```kotlin
navController.navigate(PaymentRoute)
```

Проблема в том, что `PaymentRoute` может ожидать уже заполненный адрес и корзину. Если внешний экран прыгнет прямо на оплату, flow окажется в неконсистентном состоянии.

Лучшие варианты:

- сделать внутренние routes `internal`;
- открывать graph через `navigateToCheckout`;
- передавать стартовый режим в graph route;
- валидировать state на входе каждого шага;
- перенаправлять пользователя на первый недостающий шаг.

## Боевой-пример: checkout route с режимом входа

Иногда flow можно открыть с разным намерением: обычная покупка, повтор заказа, восстановление незавершенной оплаты.

```kotlin
import kotlinx.serialization.Serializable

@Serializable
data class CheckoutGraph(
    val mode: CheckoutMode
)

@Serializable
enum class CheckoutMode {
    NewOrder,
    RepeatOrder,
    ResumePayment
}
```

```kotlin
import androidx.navigation.NavController

fun NavController.navigateToNewCheckout() {
    navigate(
        CheckoutGraph(
            mode = CheckoutMode.NewOrder
        )
    )
}

fun NavController.navigateToRepeatCheckout() {
    navigate(
        CheckoutGraph(
            mode = CheckoutMode.RepeatOrder
        )
    )
}
```

Тогда `CartRoute` или graph-scoped ViewModel может прочитать mode и подготовить state flow. Это лучше, чем открывать конкретный внутренний шаг из внешнего экрана.

## Типичные ошибки

### Делать nested graph для каждого экрана

Graph нужен для flow, а не для одиночного destination. Если вложенность не добавляет смысла, она только усложняет чтение.

### Экспортировать все внутренние routes

Если внешний код может открыть любой шаг flow, encapsulation теряется. Для flow с важным порядком шагов внутренние routes лучше скрывать.

### Забывать hierarchy в bottom navigation

Если сравнивать только текущий route, bottom bar перестанет подсвечивать вкладку на внутренних экранах. Проверяйте hierarchy destination.

### Не использовать saveState и restoreState

Без этих options top-level tabs будут терять состояние при переключении.

### Навигировать из composition

Навигация является побочный эффект. Вызывайте ее из обратные вызовы или `LaunchedEffect` с аккуратными keys.

### Scope-ить общий flow state глобально

Checkout draft, onboarding прогресс и wizard state обычно должны жить столько же, сколько graph. Global singleton переживет flow дольше, чем нужно.

## Итоги

Nested graphs и multiple back stacks делают навигацию масштабируемой:

- `navigation<T>` создает вложенный graph;
- nested graph имеет свой `startDestination`;
- route graph не рисует UI, а группирует destinations;
- feature graph удобно выносить в `NavGraphBuilder` extension;
- внутренние routes flow лучше не экспортировать без необходимости;
- graph-scoped ViewModel подходит для общего state внутри flow;
- bottom navigation и navigation drawer требуют нескольких back stacks;
- `popUpTo`, `saveState`, `restoreState` и `launchSingleTop` сохраняют состояние top-level flows;
- `destination.hierarchy` помогает определить текущую top-level вкладку;
- navigation побочные эффекты должны выполняться из event handlers или effects, а не из тела composable.
