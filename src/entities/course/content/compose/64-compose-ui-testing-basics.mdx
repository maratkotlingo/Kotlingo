## Что проверяют Compose UI tests

Compose UI tests проверяют поведение интерфейса через semantics tree. Это важное отличие от View-тестов: в Compose не каждый composable становится отдельным node, а тесты взаимодействуют не с "виджетами по id", а со смысловыми узлами интерфейса.

Тесты обычно отвечают на вопросы:

- виден ли нужный текст;
- есть ли кнопка с нужным label;
- находится ли элемент в enabled/отключенное состояние;
- выполняется ли обратный вызов при click;
- появляется ли error state после invalid input;
- обновляется ли UI после изменения state;
- можно ли найти элемент по semantics;
- соответствует ли экран accessibility contract.

```kotlin
data class LoginUiState(
    val email: String = "",
    val password: String = "",
    val loading: Boolean = false,
    val errorMessage: String? = null
)
```

Тестировать удобнее composable, который принимает state и обратные вызовы, а не сам создает ViewModel и ходит в сеть.

<CatTip>
Хороший Compose UI test не знает, из какой базы пришли данные. Он знает, какой state получил экран и какое поведение пользователь должен увидеть.
</CatTip>

## Dependencies

Обычно нужны:

```kotlin
dependencies {
    androidTestImplementation("androidx.compose.ui:ui-test-junit4")
    debugImplementation("androidx.compose.ui:ui-test-manifest")
}
```

Если используете новые testing v2 APIs, imports будут из `androidx.compose.ui.test.junit4.v2`. В существующих проектах часто еще встречаются v1 APIs. Смысл правил тот же: создать Compose test environment, установить content, найти nodes, выполнить actions/assertions.

```kotlin
import androidx.compose.ui.test.junit4.createComposeRule
import org.junit.Rule

class ExampleTest {

    @get:Rule
    val composeRule = createComposeRule()
}
```

## `createComposeRule`

`createComposeRule()` используется, когда тестируем composable изолированно, без реальной Activity.

```kotlin
import androidx.compose.ui.test.assertIsDisplayed
import androidx.compose.ui.test.junit4.createComposeRule
import androidx.compose.ui.test.onNodeWithText
import org.junit.Rule
import org.junit.Test

class GreetingTest {

    @get:Rule
    val composeRule = createComposeRule()

    @Test
    fun greeting_isDisplayed() {
        composeRule.setContent {
            Greeting(
                name = "Compose"
            )
        }

        composeRule
            .onNodeWithText("Привет, Compose")
            .assertIsDisplayed()
    }
}
```

`setContent` запускает маленькую composition внутри test rule. Это лучший путь для component-level tests.

## `createAndroidComposeRule`

`createAndroidComposeRule<Activity>()` используется, когда нужен Activity context, real lifecycle, navigation, resources или mixed View/Compose behavior.

```kotlin
import androidx.compose.ui.test.junit4.createAndroidComposeRule
import androidx.compose.ui.test.onNodeWithText
import androidx.compose.ui.test.performClick
import org.junit.Rule
import org.junit.Test

class MainActivityTest {

    @get:Rule
    val composeRule = createAndroidComposeRule<MainActivity>()

    @Test
    fun openComposeCourse() {
        composeRule
            .onNodeWithText("Compose")
            .performClick()

        composeRule
            .onNodeWithText("Jetpack Compose")
            .assertExists()
    }
}
```

Используйте Activity test, когда он действительно нужен. Для большинства UI states быстрее и надежнее isolated test через `createComposeRule`.

## `setContent`

В тесте `setContent` должен создавать боевой-like оболочку:

- theme;
- CompositionLocals;
- фиктивные обратные вызовы;
- fixed test state;
- нужные providers;
- predictable clocks/data.

```kotlin
private fun ComposeContentTestRule.setLoginContent(
    state: LoginUiState,
    onEmailChange: (String) -> Unit = {},
    onPasswordChange: (String) -> Unit = {},
    onSubmit: () -> Unit = {}
) {
    setContent {
        AppTheme {
            LoginScreen(
                state = state,
                onEmailChange = onEmailChange,
                onPasswordChange = onPasswordChange,
                onSubmit = onSubmit
            )
        }
    }
}
```

Так тесты не повторяют boilerplate и не забывают theme.

<MouseTrap>
Тест без темы может проходить, но не проверять реальную ветку стандартные значения Material, typography, colors и semantics ваших компонентов.
</MouseTrap>

## Semantics tree

Compose tests ищут nodes в semantics tree. Semantics создают:

- `Text`;
- `Button`;
- `TextField`;
- `Icon` с `contentDescription`;
- компоненты Material;
- modifiers вроде `semantics`, `testTag`, `clickable`;
- пользовательский semantics properties.

Пример:

```kotlin
import androidx.compose.foundation.clickable
import androidx.compose.material3.Text
import androidx.compose.runtime.Composable
import androidx.compose.ui.Modifier
import androidx.compose.ui.semantics.Role

@Composable
fun ProfileRow(
    name: String,
    onClick: () -> Unit
) {
    Text(
        text = name,
        modifier = Modifier.clickable(
            role = Role.Button,
            onClick = onClick
        )
    )
}
```

Тест:

```kotlin
composeRule
    .onNodeWithText("Мария")
    .assertExists()
```

Если node не находится, сначала печатайте semantics tree, а не угадывайте.

## Assertions

Базовые assertions:

- `assertExists`;
- `assertDoesNotExist`;
- `assertIsDisplayed`;
- `assertIsEnabled`;
- `assertIsNotEnabled`;
- `assertIsSelected`;
- `assertIsOn`;
- `assertIsOff`;
- `assertTextEquals`;
- `assertContentDescriptionEquals`.

```kotlin
import androidx.compose.ui.test.assertIsEnabled
import androidx.compose.ui.test.assertIsNotEnabled
import androidx.compose.ui.test.onNodeWithText

@Test
fun submitButton_disabledWhileLoading() {
    composeRule.setLoginContent(
        state = LoginUiState(
            loading = true
        )
    )

    composeRule
        .onNodeWithText("Войти")
        .assertIsNotEnabled()
}
```

Проверяйте видимое пользователю поведение. Не тестируйте внутреннюю деталь реализации, если пользователь его не видит и не использует.

## Matchers

Matchers описывают, какой node нужен.

```kotlin
import androidx.compose.ui.test.hasClickAction
import androidx.compose.ui.test.hasText
import androidx.compose.ui.test.onNode

composeRule
    .onNode(
        hasText("Войти") and hasClickAction()
    )
    .assertExists()
```

Чем точнее matcher, тем устойчивее тест. Но слишком точный matcher может стать хрупким, если проверяет несущественную деталь.

## Test structure

Хорошая структура:

- Arrange: подготовить state/fakes;
- Act: выполнить действие пользователя;
- Assert: проверить observable result.

```kotlin
@Test
fun invalidEmail_showsError() {
    var submitted = false

    composeRule.setLoginContent(
        state = LoginUiState(
            email = "broken",
            password = "123456"
        ),
        onSubmit = {
            submitted = true
        }
    )

    composeRule
        .onNodeWithText("Войти")
        .performClick()

    assertThat(submitted).isTrue()
}
```

Если UI сам должен показать error после state update, тест должен обновить state через test держатель состояния или фиктивный ViewModel.

## Test держатель состояния

Для интерактивных component tests удобно создать маленький держатель состояния прямо в test content.

```kotlin
@Test
fun emailInput_updatesText() {
    composeRule.setContent {
        var state by remember {
            mutableStateOf(LoginUiState())
        }

        AppTheme {
            LoginScreen(
                state = state,
                onEmailChange = { email ->
                    state = state.copy(email = email)
                },
                onPasswordChange = {},
                onSubmit = {}
            )
        }
    }

    composeRule
        .onNodeWithText("Email")
        .performTextInput("test@example.com")

    composeRule
        .onNodeWithText("test@example.com")
        .assertExists()
}
```

Такой тест проверяет связку `TextField -> callback -> state -> UI`.

## Fakes вместо mocks

Для UI tests часто лучше фиктивный, чем mock:

```kotlin
class FakeLoginEvents {
    val submittedEmails = mutableListOf<String>()

    fun submit(email: String) {
        submittedEmails.add(email)
    }
}
```

Тест:

```kotlin
@Test
fun submit_callsSubmitWithEmail() {
    val events = FakeLoginEvents()

    composeRule.setLoginContent(
        state = LoginUiState(
            email = "test@example.com",
            password = "123456"
        ),
        onSubmit = {
            events.submit("test@example.com")
        }
    )

    composeRule
        .onNodeWithText("Войти")
        .performClick()

    assertThat(events.submittedEmails)
        .containsExactly("test@example.com")
}
```

Фиктивный проще читать и отлаживать, чем мок с длинной настройкой.

## Unit, UI и integration tests

Разделяйте уровни:

- pure reducer/mapper tests: без Compose;
- component UI tests: `createComposeRule`;
- screen tests с фиктивный держатель состояния;
- Activity/navigation tests: `createAndroidComposeRule`;
- сквозные/системные тесты: UIAutomator/Espresso/Macrobenchmark.

Не пытайтесь проверить всю business logic через UI tests. UI tests дороже и медленнее.

## Debugging tests

Если тест не находит node:

```kotlin
composeRule
    .onRoot()
    .printToLog("LoginTree")
```

Проверьте:

- node есть в merged tree или только в unmerged;
- text отличается из-за localization;
- contentDescription null;
- компонент disabled/hidden;
- asynchronous state еще не загрузился;
- test ищет до idle;
- matcher слишком общий или слишком строгий.

## Типичные ошибки

### Тестировать implementation details

Тест по internal тестовый тег вместо видимого текста может пропустить сломанную accessibility.

### Не подключать theme

Компонент тестируется не в тех условиях, где работает в приложении.

### Использовать Activity rule для всего

Тесты становятся медленнее и хрупче.

### Делать real network/база данных в UI test

Тест перестает быть deterministic.

### Не печатать semantics tree при падении

Без дерева вы гадаете, а не диагностируете.

### Проверять только `assertExists`

Элемент может существовать, но быть disabled, invisible или иметь неправильную role.

## Итоги

Основы Compose UI testing:

- тесты работают через semantics tree;
- `createComposeRule` подходит для isolated composable tests;
- `createAndroidComposeRule` нужен для Activity/lifecycle/integration cases;
- `setContent` должен включать theme и фиктивный state;
- assertions проверяют observable behavior;
- matchers должны быть точными, но не хрупкими;
- state лучше передавать явно;
- фиктивный держатели состояния делают интерактивные тесты понятными;
- при падении печатайте semantics tree;
- UI tests дополняют unit/integration tests, а не заменяют все уровни.
