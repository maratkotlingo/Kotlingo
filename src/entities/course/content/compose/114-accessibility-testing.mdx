## Что такое accessibility testing в Compose

Accessibility testing проверяет, что UI понятен и управляем не только визуально. Compose делает semantics центральным слоем для:

- screen readers;
- UI tests;
- autofill;
- accessibility services;
- custom actions;
- traversal.

Автоматические тесты не заменяют ручную проверку TalkBack, keyboard, font scale и real device behavior. Но они хорошо ловят regressions в semantics contract.

```kotlin
data class AccessibilityCheckItem(
    val element: String,
    val expectedRole: String?,
    val expectedLabel: String?,
    val expectedState: String?
)
```

Accessibility — это не отдельная "последняя проверка". Это часть API вашего UI.

<CatTip>
Semantics — это публичный контракт компонента. Если вы меняете label, role или action, вы меняете поведение для screen reader и тестов.
</CatTip>

## Content descriptions

Icon-only actions должны иметь понятный label.

```kotlin
@Composable
fun CloseButton(
    onClose: () -> Unit
) {
    IconButton(onClick = onClose) {
        Icon(
            imageVector = Icons.Default.Close,
            contentDescription = "Закрыть"
        )
    }
}
```

Тест:

```kotlin
import androidx.compose.ui.test.assertHasClickAction
import androidx.compose.ui.test.junit4.createComposeRule
import androidx.compose.ui.test.onNodeWithContentDescription
import org.junit.Rule
import org.junit.Test

class CloseButtonAccessibilityTest {

    @get:Rule
    val composeRule = createComposeRule()

    @Test
    fun closeButton_hasLabelAndClickAction() {
        composeRule.setContent {
            CloseButton(onClose = {})
        }

        composeRule
            .onNodeWithContentDescription("Закрыть")
            .assertHasClickAction()
    }
}
```

Декоративная icon должна иметь `contentDescription = null`, чтобы screen reader не читал шум.

## Не дублировать text

Плохо:

```kotlin
Row {
    Icon(
        imageVector = Icons.Default.Settings,
        contentDescription = "Настройки"
    )
    Text(text = "Настройки")
}
```

Screen reader может прочитать "Настройки" дважды. Лучше:

```kotlin
Row {
    Icon(
        imageVector = Icons.Default.Settings,
        contentDescription = null
    )
    Text(text = "Настройки")
}
```

Тест может проверить отсутствие отдельного icon label, но чаще достаточно проверять общий row/button semantics.

## Roles

Custom interactive UI должен иметь role.

```kotlin
import androidx.compose.foundation.clickable
import androidx.compose.ui.semantics.Role

@Composable
fun CustomCardButton(
    title: String,
    onClick: () -> Unit
) {
    Box(
        modifier = Modifier.clickable(
            role = Role.Button,
            onClick = onClick
        )
    ) {
        Text(text = title)
    }
}
```

Тест:

```kotlin
import androidx.compose.ui.semantics.Role
import androidx.compose.ui.semantics.SemanticsProperties
import androidx.compose.ui.test.SemanticsMatcher
import androidx.compose.ui.test.onNode
import androidx.compose.ui.test.hasText

composeRule
    .onNode(
        hasText("Открыть") and
            SemanticsMatcher.expectValue(
                SemanticsProperties.Role,
                Role.Button
            )
    )
    .assertExists()
```

Material components часто задают role сами. Custom components требуют внимания.

## State descriptions

Если состояние неочевидно из текста, задайте `stateDescription`.

```kotlin
import androidx.compose.ui.semantics.stateDescription
import androidx.compose.ui.semantics.semantics

@Composable
fun FavoriteToggle(
    checked: Boolean,
    onCheckedChange: (Boolean) -> Unit
) {
    Switch(
        checked = checked,
        onCheckedChange = onCheckedChange,
        modifier = Modifier.semantics {
            stateDescription = if (checked) {
                "Добавлено в избранное"
            } else {
                "Не в избранном"
            }
        }
    )
}
```

Тест:

```kotlin
fun hasStateDescription(
    value: String
): SemanticsMatcher {
    return SemanticsMatcher.expectValue(
        SemanticsProperties.StateDescription,
        value
    )
}

composeRule
    .onNode(hasStateDescription("Добавлено в избранное"))
    .assertExists()
```

`stateDescription` должен быть локализован и понятен без визуального контекста.

## Error semantics

Text fields с ошибками должны сообщать error.

```kotlin
import androidx.compose.ui.semantics.error
import androidx.compose.ui.semantics.semantics

@Composable
fun EmailField(
    value: String,
    errorMessage: String?,
    onValueChange: (String) -> Unit
) {
    OutlinedTextField(
        value = value,
        onValueChange = onValueChange,
        label = {
            Text(text = "Email")
        },
        isError = errorMessage != null,
        supportingText = {
            errorMessage?.let {
                Text(text = it)
            }
        },
        modifier = Modifier.semantics {
            errorMessage?.let {
                error(it)
            }
        }
    )
}
```

Тест:

```kotlin
composeRule.setContent {
    EmailField(
        value = "broken",
        errorMessage = "Введите корректный email",
        onValueChange = {}
    )
}

composeRule
    .onNodeWithText("Введите корректный email")
    .assertIsDisplayed()

composeRule
    .onNode(hasError("Введите корректный email"))
    .assertExists()
```

Helper:

```kotlin
fun hasError(
    message: String
): SemanticsMatcher {
    return SemanticsMatcher("имеет ошибку $message") { node ->
        node.config
            .getOrNull(SemanticsProperties.Error)
            ?.contains(message) == true
    }
}
```

## Custom actions

Если gesture сложный, добавьте accessibility alternative через custom action.

```kotlin
import androidx.compose.ui.semantics.CustomAccessibilityAction
import androidx.compose.ui.semantics.customActions

@Composable
fun SwipeableMessageRow(
    message: MessageUi,
    onArchive: () -> Unit
) {
    MessageRow(
        message = message,
        modifier = Modifier.semantics {
            customActions = listOf(
                CustomAccessibilityAction(
                    label = "Архивировать",
                    action = {
                        onArchive()
                        true
                    }
                )
            )
        }
    )
}
```

Тест:

```kotlin
fun hasCustomAction(
    label: String
): SemanticsMatcher {
    return SemanticsMatcher("имеет custom action $label") { node ->
        node.config
            .getOrNull(SemanticsProperties.CustomActions)
            ?.any { action ->
                action.label == label
            } == true
    }
}

composeRule
    .onNode(hasCustomAction("Архивировать"))
    .assertExists()
```

Жест не должен быть единственным способом выполнить важное действие.

<MouseTrap>
Swipe-to-delete без меню, кнопки или custom action недоступен части пользователей. Красивый жест не равен доступному действию.
</MouseTrap>

## Traversal and order

Screen reader order обычно следует layout order. Если визуальный layout сложный, проверяйте traversal вручную и через semantics.

Автоматически проверить полный порядок сложно, но можно:

- избегать неожиданных `clearAndSetSemantics`;
- проверять pane titles;
- проверять headings;
- проверять existence ключевых nodes;
- вручную пройти TalkBack.

Для section headings:

```kotlin
import androidx.compose.ui.semantics.heading

Text(
    text = "Настройки уведомлений",
    modifier = Modifier.semantics {
        heading()
    }
)
```

Тест:

```kotlin
fun isHeading(): SemanticsMatcher {
    return SemanticsMatcher.keyIsDefined(
        SemanticsProperties.Heading
    )
}

composeRule
    .onNode(hasText("Настройки уведомлений") and isHeading())
    .assertExists()
```

## Touch targets

Touch target size лучше проверять через design system tests и screenshot/layout tests. Semantics test может проверить, что action есть, но не всегда надежно проверит physical size.

Пример layout assertion:

```kotlin
composeRule
    .onNodeWithContentDescription("Закрыть")
    .assertWidthIsAtLeast(48.dp)
    .assertHeightIsAtLeast(48.dp)
```

Не все targets обязаны быть ровно 48.dp visual bounds, но interactive area должна быть достаточной.

## Font scale

Accessibility testing включает large font.

Что проверять:

- текст не обрезается;
- buttons не теряют label;
- dialogs scrollable;
- primary action доступен;
- list items не overlap;
- error text виден.

Screenshot tests хорошо подходят для font scale variants. UI tests могут проверить, что ключевые nodes все еще displayed.

## Accessibility scanner and manual checks

Автоматические Compose tests не ловят все. Добавляйте ручной checklist:

- TalkBack linear navigation;
- TalkBack local context menu/custom actions;
- keyboard navigation;
- switch access, если важно;
- font scale 1.3/2.0;
- display size;
- color contrast;
- dark theme;
- landscape;
- large screens;
- real device.

Accessibility Scanner полезен, но не заменяет human review.

## Testing merged vs unmerged semantics

Accessibility обычно работает с merged semantics. Если тест проходит только в unmerged tree, спросите: пользователь тоже видит этот смысл?

```kotlin
composeRule
    .onRoot(useUnmergedTree = true)
    .printToLog("Unmerged")
```

Используйте unmerged для диагностики, а assertions лучше писать к user-facing semantics.

## Accessibility regression checklist

Для PR с новым component:

- есть label у icon-only actions;
- decorative images скрыты;
- custom clickables имеют role;
- state changes имеют stateDescription;
- errors visible and semantic;
- gestures имеют alternative action;
- focus/traversal логичны;
- font scale проверен;
- semantics tests добавлены для critical behavior.

```kotlin
data class AccessibilityRegressionCheck(
    val component: String,
    val labelChecked: Boolean,
    val roleChecked: Boolean,
    val stateChecked: Boolean,
    val manualTalkBackChecked: Boolean
)
```

## Типичные ошибки

### Content description на декоративной иконке

Screen reader читает лишний шум.

### Нет content description на icon button

Пользователь слышит generic unlabeled action.

### Custom clickable без role

Элемент звучит не как кнопка/checkbox/switch.

### Ошибка только цветом

Цвет не объясняет, что исправить.

### Gesture без alternative

Swipe/drag недоступен части пользователей.

### Тестировать только unmerged tree

Можно проверить internal structure и пропустить user-facing semantics.

## Итоги

Accessibility testing в Compose строится на semantics и ручной проверке:

- content descriptions нужны meaningful non-text actions;
- decorative images должны быть скрыты;
- custom interactive components должны иметь role/action;
- `stateDescription` описывает состояние;
- error semantics дополняет visible error text;
- custom actions дают альтернативу сложным gestures;
- headings и pane titles помогают navigation;
- touch target и font scale нужно проверять отдельно;
- merged semantics лучше отражает user-facing contract;
- TalkBack/manual testing остается обязательным для важных flows;
- accessibility tests защищают публичный UI contract от regressions.
