## Что такое adaptive UI

Adaptive UI — это интерфейс, который принимает решения по текущему окну приложения, input devices, posture и available space, а не по предположению "это телефон 360x800".

Android-приложение может работать:

- на телефоне;
- на планшете;
- на foldable в сложенном и разложенном состоянии;
- в split-screen;
- в freeform/desktop window;
- на ChromeOS;
- на внешнем дисплее;
- в portrait, landscape и квадратных aspect ratios;
- с touch, mouse, trackpad, stylus и keyboard;
- с разным font scale и display size.

Compose хорошо подходит для adaptive UI, потому что layout строится из state и constraints. Но это не происходит автоматически. Нужно перестать думать "какой device?" и начать думать "какое окно и какой рабочий процесс?".

```kotlin
data class AdaptiveWindowState(
    val widthClass: WindowWidthClass,
    val heightClass: WindowHeightClass,
    val posture: DevicePosture = DevicePosture.Normal
)

enum class WindowWidthClass {
    Compact,
    Medium,
    Expanded
}

enum class WindowHeightClass {
    Compact,
    Medium,
    Expanded
}

enum class DevicePosture {
    Normal,
    Tabletop,
    Book
}
```

Такая domain-friendly модель часто удобнее, чем протаскивать platform classes во все composables.

<CatTip>
Adaptive UI начинается не с tablet layout, а с честного признания: окно приложения может меняться в любой момент, и state должен пережить это без сюрпризов.
</CatTip>

## Window size classes

Window size class группирует текущий размер окна в крупные категории:

- compact;
- medium;
- expanded.

Обычно width важнее height, потому что вертикальный scroll естественнее, чем горизонтальный. Но height тоже важен для landscape, tabletop, desktop windows и экранов с keyboard/IME.

В Compose Material 3 Adaptive можно получить adaptive info:

```kotlin
import androidx.compose.material3.adaptive.currentWindowAdaptiveInfo
import androidx.compose.runtime.Composable

@Composable
fun AdaptiveAppRoot() {
    val adaptiveInfo = currentWindowAdaptiveInfo()
    val windowSizeClass = adaptiveInfo.windowSizeClass

    AppNavigationFrame(
        windowSizeClass = windowSizeClass
    )
}
```

Для учебного кода ниже используем простую свою модель:

```kotlin
import androidx.compose.runtime.Composable

@Composable
fun rememberAdaptiveWindowState(): AdaptiveWindowState {
    val adaptiveInfo = currentWindowAdaptiveInfo()
    val sizeClass = adaptiveInfo.windowSizeClass

    return AdaptiveWindowState(
        widthClass = when {
            sizeClass.isWidthAtLeastBreakpoint(840) -> WindowWidthClass.Expanded
            sizeClass.isWidthAtLeastBreakpoint(600) -> WindowWidthClass.Medium
            else -> WindowWidthClass.Compact
        },
        heightClass = when {
            sizeClass.isHeightAtLeastBreakpoint(900) -> WindowHeightClass.Expanded
            sizeClass.isHeightAtLeastBreakpoint(480) -> WindowHeightClass.Medium
            else -> WindowHeightClass.Compact
        }
    )
}
```

Этот пример показывает идею. В боевой лучше использовать API и helpers из актуальной Material 3 Adaptive / Window Size Class библиотеки, а не хардкодить magic breakpoints в каждом экране.

## App window, не device screen

Главная ошибка: принимать решения по размеру физического экрана.

На планшете приложение может быть:

- full screen expanded;
- половиной split-screen medium;
- маленьким floating window compact;
- почти квадратным desktop window;
- letterboxed из-за неправильных manifest settings.

Поэтому layout должен реагировать на размер окна приложения.

```kotlin
@Composable
fun AdaptiveHomeScreen(
    windowState: AdaptiveWindowState,
    state: HomeState,
    onOpenItem: (String) -> Unit
) {
    when (windowState.widthClass) {
        WindowWidthClass.Compact -> {
            SinglePaneHome(
                state = state,
                onOpenItem = onOpenItem
            )
        }
        WindowWidthClass.Medium -> {
            ComfortableHome(
                state = state,
                onOpenItem = onOpenItem
            )
        }
        WindowWidthClass.Expanded -> {
            MultiPaneHome(
                state = state,
                onOpenItem = onOpenItem
            )
        }
    }
}
```

Не спрашивайте "это tablet?". Спрашивайте "сколько panes помещается в этом окне без потери usability?".

## Orientation

Orientation полезна, но не должна быть главным decision point.

Плохо:

```kotlin
if (isLandscape) {
    TwoColumnScreen()
} else {
    SingleColumnScreen()
}
```

Почему плохо:

- tablet portrait может быть шире phone landscape;
- desktop window может быть landscape, но narrow;
- foldable tabletop имеет horizontal hinge, но рабочий процесс другой;
- split-screen ломает связь orientation и available width.

Лучше:

```kotlin
val showSecondPane = windowState.widthClass == WindowWidthClass.Expanded
```

Orientation можно использовать как дополнительный signal:

```kotlin
data class AdaptiveLayoutDecision(
    val showNavigationRail: Boolean,
    val showSupportingPane: Boolean,
    val compactHeight: Boolean
)

fun decideLayout(
    windowState: AdaptiveWindowState
): AdaptiveLayoutDecision {
    return AdaptiveLayoutDecision(
        showNavigationRail = windowState.widthClass != WindowWidthClass.Compact,
        showSupportingPane = windowState.widthClass == WindowWidthClass.Expanded,
        compactHeight = windowState.heightClass == WindowHeightClass.Compact
    )
}
```

Так compact landscape phone не получает огромную desktop-композицию только из-за orientation.

<MouseTrap>
`screenOrientation="portrait"` не делает приложение проще. Оно делает приложение хуже на больших экранах, foldables, ChromeOS и desktop windowing.
</MouseTrap>

## Resizability

Adaptive app должна быть resizable. Пользователь может менять размер окна, складывать device, поворачивать экран, открывать split-screen.

Manifest antipatterns:

```xml
<!-- Не фиксируйте ориентацию без крайней необходимости -->
<activity
    android:name=".MainActivity"
    android:screenOrientation="portrait" />
```

```xml
<!-- Не отключайте изменение размера для современного Android-интерфейса -->
<activity
    android:name=".MainActivity"
    android:resizeableActivity="false" />
```

Вместо этого проектируйте state continuity:

- UI-состояние в ViewModel;
- transient field state через `rememberSaveable`;
- selected item хранится отдельно от layout branch;
- scroll states имеют stable keys;
- navigation destination не теряется при resize;
- multi-pane не сбрасывает выбранный detail.

## State continuity

Adaptive layout меняет форму UI, но не должен менять пользовательский context.

```kotlin
data class MailUiState(
    val folders: List<MailFolder>,
    val messages: List<MailSummary>,
    val selectedMessageId: String?,
    val openedFolderId: String,
    val searchQuery: String
)
```

Compact layout:

```kotlin
@Composable
fun CompactMailScreen(
    state: MailUiState,
    onMessageClick: (String) -> Unit
) {
    if (state.selectedMessageId == null) {
        MessageList(
            messages = state.messages,
            onMessageClick = onMessageClick
        )
    } else {
        MessageDetails(
            messageId = state.selectedMessageId
        )
    }
}
```

Expanded layout:

```kotlin
@Composable
fun ExpandedMailScreen(
    state: MailUiState,
    onMessageClick: (String) -> Unit
) {
    MailThreePaneLayout(
        folders = {
            FolderList(
                folders = state.folders,
                selectedFolderId = state.openedFolderId
            )
        },
        messages = {
            MessageList(
                messages = state.messages,
                selectedMessageId = state.selectedMessageId,
                onMessageClick = onMessageClick
            )
        },
        details = {
            state.selectedMessageId?.let { id ->
                MessageDetails(messageId = id)
            } ?: EmptyMessageDetails()
        }
    )
}
```

Один state поддерживает две формы UI. При расширении окна detail появляется рядом, а не создается новый состояние экрана.

## Pane thinking

Adaptive UI часто мыслится panes:

- list pane;
- detail pane;
- supporting pane;
- navigation pane;
- inspector pane;
- editor pane;
- предпросмотр pane.

Pane — это не обязательно Fragment или route. Это смысловая область, которую можно:

- показать рядом;
- скрыть;
- заменить navigation destination;
- превратить в modal;
- свернуть в bottom sheet;
- вынести в secondary window в будущем.

```kotlin
@Composable
fun TwoPaneLayout(
    listPane: @Composable () -> Unit,
    detailPane: @Composable () -> Unit,
    modifier: Modifier = Modifier
) {
    Row(modifier = modifier) {
        Box(
            modifier = Modifier.weight(0.38f)
        ) {
            listPane()
        }
        Box(
            modifier = Modifier.weight(0.62f)
        ) {
            detailPane()
        }
    }
}
```

Собственный pane layout полезен для простых cases. Для canonical layouts лучше рассмотреть Material 3 Adaptive scaffolds.

## Adaptive navigation

Navigation surface зависит от width:

- compact: bottom панель навигации;
- medium: navigation rail;
- expanded: navigation rail или permanent drawer;
- desktop: rail/drawer + keyboard shortcuts + hover states.

```kotlin
import androidx.compose.material3.NavigationBar
import androidx.compose.material3.NavigationRail
import androidx.compose.runtime.Composable

@Composable
fun AdaptiveNavigationFrame(
    windowState: AdaptiveWindowState,
    destinations: List<AppDestination>,
    current: AppDestination,
    onDestinationClick: (AppDestination) -> Unit,
    content: @Composable () -> Unit
) {
    when (windowState.widthClass) {
        WindowWidthClass.Compact -> {
            BottomNavigationFrame(
                navigation = {
                    NavigationBar {
                        destinations.forEach { destination ->
                            AppNavigationBarItem(
                                destination = destination,
                                selected = destination == current,
                                onClick = {
                                    onDestinationClick(destination)
                                }
                            )
                        }
                    }
                },
                content = content
            )
        }
        else -> {
            RailNavigationFrame(
                navigation = {
                    NavigationRail {
                        destinations.forEach { destination ->
                            AppNavigationRailItem(
                                destination = destination,
                                selected = destination == current,
                                onClick = {
                                    onDestinationClick(destination)
                                }
                            )
                        }
                    }
                },
                content = content
            )
        }
    }
}
```

Material 3 Adaptive также предоставляет `NavigationSuiteScaffold`, который автоматизирует часть этого выбора.

## Multi-window

Multi-window означает, что ваше приложение не контролирует весь экран. Оно получает окно, которое может быть narrow, wide, short, square или постоянно resized.

Особенности:

- configuration changes могут происходить часто;
- keyboard и mouse более вероятны;
- hover/focus становятся важнее;
- app может быть рядом с другим app;
- drag-and-drop между windows становится полезным;
- layout не должен зависеть от physical device posture напрямую.

Практический проверочный список:

- не фиксировать orientation;
- не отключать resize;
- не хардкодить минимальную ширину всего экрана;
- использовать constraints и size classes;
- сохранять state;
- проверять small-height windows;
- поддерживать keyboard navigation;
- избегать диалогах, которые шире окна.

## Ограничения на глобальные переменные

Иногда решение нужно принимать локально, не на уровне всего окна. Для этого используйте `BoxWithConstraints`.

```kotlin
import androidx.compose.foundation.layout.BoxWithConstraints
import androidx.compose.foundation.layout.Column
import androidx.compose.foundation.layout.Row
import androidx.compose.material3.Text
import androidx.compose.runtime.Composable
import androidx.compose.ui.Modifier
import androidx.compose.ui.unit.dp

@Composable
fun ResponsiveStatsPanel(
    stats: List<StatItem>,
    modifier: Modifier = Modifier
) {
    BoxWithConstraints(modifier = modifier) {
        if (maxWidth < 360.dp) {
            Column {
                stats.forEach { item ->
                    StatRow(item)
                }
            }
        } else {
            Row {
                stats.forEach { item ->
                    StatCard(
                        item = item,
                        modifier = Modifier.weight(1f)
                    )
                }
            }
        }
    }
}
```

Window size class говорит о root window. `BoxWithConstraints` говорит о конкретном component slot.

## Предпросмотр adaptive states

Adaptive UI нужно предпросмотр-ить в нескольких размерах.

```kotlin
import androidx.compose.ui.tooling.preview.Preview

@Preview(
    name = "compact",
    widthDp = 360,
    heightDp = 800
)
@Preview(
    name = "medium",
    widthDp = 700,
    heightDp = 900
)
@Preview(
    name = "expanded",
    widthDp = 1200,
    heightDp = 900
)
annotation class AdaptivePreviews
```

Использование:

```kotlin
@AdaptivePreviews
@Composable
fun HomeScreenAdaptivePreview() {
    AppTheme {
        AdaptiveHomeScreen(
            windowState = AdaptiveWindowState(
                widthClass = WindowWidthClass.Expanded,
                heightClass = WindowHeightClass.Medium
            ),
            state = HomeState.preview(),
            onOpenItem = {}
        )
    }
}
```

Для корректных предпросмотры лучше отделять screen UI от route, чтобы не создавать ViewModel в предпросмотр.

## Testing adaptive layouts

Что тестировать:

- compact показывает один pane;
- expanded показывает два или три panes;
- выбранный item сохраняется при смене layout;
- back behavior в compact возвращает list;
- resize не сбрасывает search query;
- navigation surface меняется без потери current destination;
- focus order остается логичным;
- font scale не ломает controls;
- small height не скрывает primary actions.

Чистую layout decision logic можно тестировать без Compose:

```kotlin
import com.google.common.truth.Truth.assertThat
import org.junit.Test

class AdaptiveLayoutDecisionTest {

    @Test
    fun expandedWidth_showsSupportingPane() {
        val decision = decideLayout(
            AdaptiveWindowState(
                widthClass = WindowWidthClass.Expanded,
                heightClass = WindowHeightClass.Medium
            )
        )

        assertThat(decision.showSupportingPane).isTrue()
    }
}
```

Так вы фиксируете правила адаптации отдельно от rendering.

## Типичные ошибки

### Проверять только device type

Tablet может быть compact window. Phone landscape может быть wider than expected.

### Делать orientation главным условием

Orientation не равна доступной ширине и не описывает рабочий процесс.

### Сбрасывать state при смене layout

Adaptive UI должен менять форму, а не начинать сценарий заново.

### Блокировать resize

Это ломает large screens, foldables, ChromeOS и desktop windowing.

### Хардкодить размеры pane

Используйте weights, constraints, min/max и adaptive scaffolds.

### Забывать keyboard/mouse

Большие экраны часто используются не только через touch.

## Итоги

Adaptive foundations в Compose строятся вокруг окна и state:

- decisions принимаются по app window, не physical screen;
- window size classes дают compact/medium/expanded модель;
- width обычно важнее height, но height важен для compact-height cases;
- orientation является дополнительным signal, а не главным правилом;
- resizability и multi-window нужно поддерживать, а не отключать;
- state continuity важнее визуального switch;
- pane thinking помогает строить list-detail/supporting layouts;
- navigation surface должна адаптироваться к width;
- local components могут использовать `BoxWithConstraints`;
- adaptive предпросмотры и tests должны покрывать несколько размеров и font scales.
