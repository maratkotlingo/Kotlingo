## Зачем нужен рабочий процесс с инструментами

Compose дает много инструментов, но хаотичное использование превращает диагностику в гадание. Хороший рабочий процесс отвечает на вопросы:

- что именно сломалось;
- где это видно;
- какой инструмент подтверждает проблему;
- какая гипотеза проверяется;
- как измерить до/после;
- какие артефакты сохранить;
- как не повторить регрессию.

Инструменты:

- предпросмотры Android Studio;
- Multipreview;
- параметры предпросмотра;
- Layout Inspector;
- Semantics inspector;
- System Trace;
- Composition tracing;
- Macrobenchmark;
- Baseline Profiles;
- Memory Profiler;
- Screenshot testing;
- артефакты benchmark в CI.

```kotlin
data class ToolingQuestion(
    val problem: String,
    val firstTool: String,
    val followUpTool: String
)
```

Например: "scroll jank" → первый инструмент System Trace/Macrobenchmark, следующий шаг: Layout Inspector для анализа recomposition.

<CatTip>
Рабочий процесс с инструментами — это не список окон в Android Studio. Это порядок мышления: сначала симптом, потом инструмент, потом гипотеза, потом измерение.
</CatTip>

## Разработка через предпросмотр

Предпросмотры помогают быстро проверять состояния:

- загрузка;
- пустое состояние;
- ошибка;
- содержимое;
- long text;
- RTL;
- large font;
- compact/expanded widths;
- dark theme.

```kotlin
import androidx.compose.ui.tooling.preview.Preview

@Preview(
    name = "content",
    widthDp = 360,
    heightDp = 720
)
@Preview(
    name = "expanded",
    widthDp = 900,
    heightDp = 720
)
annotation class ScreenPreviews
```

Использование:

```kotlin
@ScreenPreviews
@Composable
fun LessonCardPreview() {
    AppTheme {
        LessonCard(
            state = LessonCardState.preview(),
            onClick = {}
        )
    }
}
```

Предпросмотр должен использовать фиктивное состояние, а не настоящие ViewModel, сеть или базу данных.

## Параметры предпросмотра

Для набора состояний используйте параметры предпросмотра.

```kotlin
import androidx.compose.ui.tooling.preview.PreviewParameter
import androidx.compose.ui.tooling.preview.PreviewParameterProvider

class LessonCardPreviewProvider :
    PreviewParameterProvider<LessonCardState> {

    override val values: Sequence<LessonCardState> = sequenceOf(
        LessonCardState.preview(completed = false),
        LessonCardState.preview(completed = true),
        LessonCardState.previewLongTitle(),
        LessonCardState.previewError()
    )
}

@Preview
@Composable
fun LessonCardStatesPreview(
    @PreviewParameter(LessonCardPreviewProvider::class)
    state: LessonCardState
) {
    AppTheme {
        LessonCard(
            state = state,
            onClick = {}
        )
    }
}
```

Provider предпросмотра не должен создавать тяжелые объекты или читать настоящие ресурсы вне области, безопасной для предпросмотра.

## Layout Inspector рабочий процесс

Используйте Layout Inspector, когда нужно понять:

- фактическое дерево UI;
- размеры layout и модификаторов;
- recomposition/skipped counters;
- semantics;
- параметры выбранного composable;
- смешанная иерархия View/Compose.

Рабочий процесс:

- запустить debuggable app;
- открыть screen;
- подключить Layout Inspector;
- включить recomposition counts;
- reset counters;
- выполнить конкретное interaction;
- записать suspicious nodes;
- перейти к source double-click;
- проверить report/trace, если есть performance impact.

```kotlin
data class LayoutInspectorNote(
    val interaction: String,
    val hotComposable: String,
    val recompositions: Int,
    val skips: Int,
    val nextStep: String
)
```

Layout Inspector хорош для локализации, но не для финального timing.

## Semantics inspection

Semantics tree важен для:

- accessibility;
- Compose UI tests;
- autofill;
- пользовательский actions;
- отладка merged/unmerged-дерева.

Что проверять:

- role;
- contentDescription;
- stateDescription;
- customActions;
- collection info;
- error semantics;
- hidden/decorative nodes;
- traversal order.

```kotlin
import androidx.compose.ui.test.junit4.createComposeRule
import androidx.compose.ui.test.onRoot
import androidx.compose.ui.test.printToLog

class SemanticsDebugTest {

    @get:Rule
    val composeRule = createComposeRule()

    @Test
    fun printSemanticsTree() {
        composeRule.setContent {
            SettingsScreenPreviewContent()
        }

        composeRule
            .onRoot()
            .printToLog("Semantics")
    }
}
```

Не оставляйте debug prints в обычных tests, если они шумят CI logs.

## System Trace

System Trace нужен, когда есть performance symptom:

- jank;
- slow startup;
- animation stutter;
- input latency;
- GC pauses;
- main thread stalls.

Что смотреть:

- long frames;
- main thread slices;
- RenderThread;
- Choreographer;
- GC;
- binder calls;
- image decode;
- composition/layout/draw;
- пользовательский trace sections.

Добавьте trace sections для app code:

```kotlin
import androidx.tracing.trace

fun loadInitialDashboard() {
    trace("Dashboard/loadInitial") {
        repository.refresh()
        cache.warmUp()
    }
}
```

Не делайте trace section слишком мелкими. Trace должен помогать читать сценарий, а не превращаться в шум.

## Composition tracing

Composition tracing добавляет composable names в system trace.

Сценарии использования:

- понять, какие composables участвуют в long frame;
- связать Layout Inspector hot nodes с trace cost;
- увидеть composition work в flame chart;
- проверить hypothesis "дорогая composition".

```kotlin
dependencies {
    debugImplementation("androidx.compose.runtime:runtime-tracing")
}
```

Включайте tracing в diagnostic build. Не измеряйте финальные боевой numbers с лишним overhead без понимания.

## Macrobenchmark рабочий процесс

Macrobenchmark нужен для повторяемого измерения:

- startup;
- scroll;
- navigation;
- animation;
- critical journey.

```kotlin
@RunWith(AndroidJUnit4::class)
class FeedScrollBenchmark {

    @get:Rule
    val benchmarkRule = MacrobenchmarkRule()

    @Test
    fun scrollFeed() {
        benchmarkRule.measureRepeated(
            packageName = "com.example.app",
            metrics = listOf(
                FrameTimingMetric()
            ),
            iterations = 5,
            startupMode = StartupMode.WARM,
            compilationMode = CompilationMode.Partial()
        ) {
            startActivityAndWait()
            openFeed()
            scrollFeedList()
        }
    }
}
```

Benchmark должен быть:

- deterministic enough;
- representative;
- release-like;
- run на стабильном device;
- сохранять trace артефакты.

## Screenshot testing

Screenshot-тесты ловят визуальные регрессии:

- spacing;
- typography;
- dark theme;
- RTL;
- adaptive widths;
- загрузка/error/content states;
- font scale;
- screenshot-only layout bugs.

Fixture-first UI упрощает screenshot testing:

```kotlin
@Composable
fun LessonCardFixture(
    state: LessonCardState = LessonCardState.preview()
) {
    AppTheme {
        LessonCard(
            state = state,
            onClick = {}
        )
    }
}
```

Screenshot tests не заменяют semantics tests. Они отвечают на другой вопрос: "выглядит ли UI так же?".

## Baseline Profiles in рабочий процесс

Baseline Profiles должны быть частью performance loop:

- генератор покрывает критические пользовательские сценарии;
- macrobenchmark measures до/после;
- профиль сохраняется и просматривается;
- обновлено после изменений навигации/интерфейса;
- CI артефакт includes generated profile and benchmark JSON.

```kotlin
data class ProfileMaintenanceRule(
    val trigger: String,
    val requiredAction: String
)

val rules = listOf(
    ProfileMaintenanceRule(
        trigger = "Переписан startup screen",
        requiredAction = "Обновить baseline profile generator"
    ),
    ProfileMaintenanceRule(
        trigger = "Добавлен новый главный tab",
        requiredAction = "Добавить journey в Macrobenchmark"
    )
)
```

Profile устаревает вместе с app flow.

## Выбор правильного инструмента

### UI не выглядит правильно

Начните с:

- Предпросмотр;
- Layout Inspector;
- screenshot test.

### Accessibility/test не видит node

Начните с:

- Semantics inspector;
- `printToLog`;
- merged/unmerged tree.

### Scroll jank

Начните с:

- Macrobenchmark FrameTimingMetric;
- System Trace;
- Layout Inspector counters;
- Memory Profiler, если есть GC.

### Slow startup

Начните с:

- Macrobenchmark StartupTimingMetric;
- System Trace;
- Baseline Profile check;
- app initialization audit.

### Stale UI

Начните с:

- state mutation audit;
- compiler stability reports;
- Layout Inspector;
- targeted UI test.

## CI артефакты

Для performance/visual CI сохраняйте:

- benchmark JSON;
- trace files;
- generated baseline profiles;
- screenshot diffs;
- device configuration;
- build variant;
- commit SHA;
- test logs.

```kotlin
data class CiPerformanceArtifact(
    val benchmarkName: String,
    val jsonPath: String,
    val tracePath: String,
    val deviceName: String,
    val buildVariant: String
)
```

Без артефактов регрессию невозможно нормально расследовать.

## Гигиена инструментов

Правила:

- инструменты только для debug держать в debug source set;
- trace/log probes удалять после расследования;
- benchmark selectors делать стабильными;
- предпросмотры не должны зависеть от бэкенда;
- фиктивные данные держать рядом с фичей;
- screenshot baselines обновлять осознанно;
- пороги benchmark вводить постепенно;
- отчеты не коммитить случайно, если проект так не договорился.

## Пример рабочего процесса: scroll jank

1. Видимый пользователю симптом: лента дергается при первой прокрутке.
2. Macrobenchmark: p95 времени кадра высокий.
3. System Trace: долгие кадры содержат GC и композицию элементов.
4. Layout Inspector: все видимые `ArticleRow` рекомпозируются при прокрутке.
5. Отчет компилятора: `ArticleUi` стабилен, но родитель читает offset прокрутки.
6. Исправление: перенести чтение scroll state в header через `derivedStateOf`, добавить `contentType`.
7. Повторный запуск Macrobenchmark: p95 улучшился.
8. Добавить заметку в PR с до/после.

```text
Before: p95 31 ms, jank 18%
After: p95 18 ms, jank 4%
Device: Pixel 8, release benchmark, Android 15
```

## Пример рабочего процесса: визуальная регрессия

1. Разница скриншотов показывает сломанный отступ.
2. Предпросмотр воспроизводит широкий вариант.
3. Layout Inspector показывает `Scaffold` innerPadding не применен.
4. Исправление: добавить `Modifier.padding(innerPadding)`.
5. Screenshot baseline обновлять не нужно, тест стал зеленым.

Такой рабочий процесс быстрее, чем гадать по CSS-подобным модификаторам на глаз.

## Типичные ошибки

### Использовать один инструмент для всего

Layout Inspector не заменяет Macrobenchmark. Screenshot-тест не заменяет проверку semantics.

### Не сохранять trace-артефакты

Без trace-артефакта регрессию сложно обсудить в команде.

### Делать предпросмотры с реальным ViewModel

Предпросмотр становится медленным и нестабильным.

### Верить одному запуску benchmark

Нужны итерации, стабильное устройство и тренд.

### Обновлять screenshot baseline автоматически

Так можно принять регрессию как новую норму.

### Оставлять debug-инструментацию в боевой среде

Логи, подсветки и трассировка могут влиять на производительность.

## Итоги

Рабочий процесс с инструментами делает Compose-разработку измеримой:

- предпросмотры ускоряют разработку состояний и адаптивных вариантов;
- Layout Inspector показывает дерево, параметры, recomposition/skips и semantics;
- System Trace нужен для производительности на уровне кадров;
- Composition tracing связывает composable-функции с trace;
- Macrobenchmark дает повторяемые измерения запуска, прокрутки и runtime;
- Baseline Profiles должны поддерживаться вместе с критическими путями;
- screenshot tests ловят визуальные регрессии;
- инструменты semantics нужны для accessibility и тестов;
- CI должен сохранять артефакты benchmark, trace и screenshot;
- хороший рабочий процесс начинается с симптома и гипотезы, а заканчивается измерением до/после.
