## Что значит unnecessary recomposition

Unnecessary recomposition — это recomposition, которая:

- не меняет результат UI;
- пересчитывает слишком большой subtree;
- происходит из-за неудачной state ownership;
- вызвана unstable parameters без причины;
- выполняет expensive work;
- происходит на hot path: scroll, drag, animation, typing.

Важно: не каждая recomposition лишняя. Если пользователь ввел символ в `TextField`, поле должно обновиться. Если progress animation меняется каждый frame, часть UI будет invalid-иться каждый frame.

```kotlin
data class RecompositionDecision(
    val expectedByStateChange: Boolean,
    val changesVisibleOutput: Boolean,
    val happensInHotPath: Boolean
) {
    val suspicious: Boolean
        get() = !changesVisibleOutput && happensInHotPath
}
```

Оптимизация должна начинаться с вопроса: какая recomposition действительно мешает?

<CatTip>
Цель не в том, чтобы Compose "ничего не пересчитывал". Цель в том, чтобы часто меняющийся state пересчитывал ровно тот UI, которому он нужен.
</CatTip>

## Stable parameters

Первый уровень — передавать в UI stable/immutable параметры.

Хорошо:

```kotlin
import androidx.compose.runtime.Immutable

@Immutable
data class ArticleUi(
    val id: String,
    val title: String,
    val author: String,
    val readTimeText: String
)
```

Composable:

```kotlin
@Composable
fun ArticleCard(
    article: ArticleUi,
    onOpen: (String) -> Unit
) {
    Card(
        onClick = {
            onOpen(article.id)
        }
    ) {
        Text(text = article.title)
        Text(text = article.author)
        Text(text = article.readTimeText)
    }
}
```

Плохо:

```kotlin
@Composable
fun ArticleCard(
    article: MutableArticleEntity,
    repository: ArticleRepository
) {
    // Интерфейс получил изменяемый объект предметной области и зависимость от сервиса.
}
```

UI component должен получать rendering data и events, а не domain/service object graph.

## Immutable UI models

Используйте UI DTO:

```kotlin
@Immutable
data class FeedUiState(
    val title: String,
    val articles: ImmutableList<ArticleUi>,
    val loading: Boolean,
    val errorMessage: String?
)
```

Mapping:

```kotlin
fun Article.toUi(): ArticleUi {
    return ArticleUi(
        id = id.value,
        title = title,
        author = author.displayName,
        readTimeText = "$readMinutes мин"
    )
}
```

Так UI получает stable snapshot, а domain остается свободным от Compose annotations.

## Не читать global state слишком высоко

Плохо:

```kotlin
@Composable
fun DashboardScreen(
    clockText: String,
    state: DashboardState
) {
    Column {
        Text(text = clockText)
        HeavyDashboardContent(state = state)
    }
}
```

Если `clockText` обновляется каждую секунду, `DashboardScreen` может стать hot recomposition root.

Лучше:

```kotlin
@Composable
fun DashboardScreen(
    state: DashboardState,
    clockTextProvider: () -> String
) {
    Column {
        ClockLabel(clockTextProvider = clockTextProvider)
        HeavyDashboardContent(state = state)
    }
}

@Composable
fun ClockLabel(
    clockTextProvider: () -> String
) {
    Text(text = clockTextProvider())
}
```

Еще лучше: clock state должен быть scoped к `ClockLabel` или отдельному state holder, если он не нужен всему screen.

## Deferring state reads

State read можно отложить в lambda, чтобы она произошла ниже или в другой фазе.

Плохо:

```kotlin
@Composable
fun ParallaxHeader(
    scrollOffset: Int
) {
    HeaderImage(
        modifier = Modifier.offset(
            y = (-scrollOffset / 2).dp
        )
    )
}
```

Здесь offset вычисляется в composition.

Лучше:

```kotlin
import androidx.compose.foundation.layout.offset
import androidx.compose.runtime.Composable
import androidx.compose.ui.Modifier
import androidx.compose.ui.unit.IntOffset

@Composable
fun ParallaxHeader(
    scrollOffsetProvider: () -> Int
) {
    HeaderImage(
        modifier = Modifier.offset {
            IntOffset(
                x = 0,
                y = -scrollOffsetProvider() / 2
            )
        }
    )
}
```

Lambda-based `offset` читает значение в layout phase. Это может избежать recomposition при каждом scroll pixel.

## Lambda modifiers

Для часто меняющихся значений выбирайте lambda modifiers:

- `Modifier.offset { ... }`;
- `Modifier.drawBehind { ... }`;
- `Modifier.graphicsLayer { ... }`;
- custom layout placement lambdas.

Пример alpha/translation:

```kotlin
import androidx.compose.runtime.Composable
import androidx.compose.ui.Modifier
import androidx.compose.ui.graphics.graphicsLayer

@Composable
fun CollapsingToolbar(
    collapseProgressProvider: () -> Float
) {
    ToolbarContent(
        modifier = Modifier.graphicsLayer {
            val progress = collapseProgressProvider()
            alpha = 1f - progress
            translationY = -48f * progress
        }
    )
}
```

Это не значит, что все animations должны уходить в `graphicsLayer`. Но для frame-by-frame transform state это часто правильная фаза.

## `remember` для expensive objects

`remember` сохраняет результат между recompositions.

```kotlin
import androidx.compose.runtime.Composable
import androidx.compose.runtime.remember
import androidx.compose.ui.graphics.Brush
import androidx.compose.ui.graphics.Color

@Composable
fun GradientPanel() {
    val brush = remember {
        Brush.verticalGradient(
            colors = listOf(
                Color(0xFF0B1D2A),
                Color(0xFF22577A)
            )
        )
    }

    PanelBackground(brush = brush)
}
```

`remember` нужен для:

- expensive calculation;
- object identity;
- avoiding allocations in hot recomposition;
- stateful helpers;
- painters/formatters, если они stable.

Не используйте `remember` как кэш для неправильного state ownership.

## Correct keys

Неполные keys дают stale values.

Плохо:

```kotlin
@Composable
fun FilteredArticles(
    articles: List<ArticleUi>,
    query: String
) {
    val filtered = remember {
        articles.filter { article ->
            article.title.contains(query, ignoreCase = true)
        }
    }

    ArticleList(filtered)
}
```

`filtered` не обновится при смене `articles` или `query`.

Правильно:

```kotlin
@Composable
fun FilteredArticles(
    articles: List<ArticleUi>,
    query: String
) {
    val filtered = remember(articles, query) {
        articles.filter { article ->
            article.title.contains(query, ignoreCase = true)
        }
    }

    ArticleList(filtered)
}
```

Если computation heavy и business-related, вынесите ее из composable.

## `derivedStateOf`

`derivedStateOf` полезен, когда исходный state меняется часто, а derived result меняется редко.

```kotlin
import androidx.compose.foundation.lazy.LazyListState
import androidx.compose.runtime.Composable
import androidx.compose.runtime.derivedStateOf
import androidx.compose.runtime.getValue
import androidx.compose.runtime.remember

@Composable
fun ScrollToTopButton(
    listState: LazyListState,
    onClick: () -> Unit
) {
    val showButton by remember {
        derivedStateOf {
            listState.firstVisibleItemIndex > 0
        }
    }

    if (showButton) {
        FloatingScrollButton(onClick = onClick)
    }
}
```

`firstVisibleItemScrollOffset` меняется постоянно, но `showButton` меняется редко. Это хороший use case.

Плохой use case:

```kotlin
val fullName by remember {
    derivedStateOf {
        "$firstName $lastName"
    }
}
```

Если derived value должен меняться каждый раз при изменении inputs и computation дешевый, `derivedStateOf` лишний.

<MouseTrap>
`derivedStateOf` сам не бесплатен. Используйте его для reduction частых изменений, а не как привычную замену обычному выражению.
</MouseTrap>

## Lambda stability

Callbacks часто являются параметрами. В большинстве cases Compose compiler и strong skipping справляются хорошо. Но problems возникают, если:

- lambda создается высоко и захватывает часто меняющийся state;
- callback object пересоздается и ломает skipping hot child;
- event model заставляет recomposition всего tree.

Обычный нормальный код:

```kotlin
ProductRow(
    product = product,
    onClick = {
        onProductClick(product.id)
    }
)
```

Если Layout Inspector показывает hot recomposition из-за callback, можно стабилизировать событие:

```kotlin
@Immutable
data class ProductRowActions(
    val onOpen: (String) -> Unit,
    val onBookmark: (String) -> Unit
)
```

Но не вводите action classes без измерения. Иногда проще передать две lambdas и оставить код читаемым.

## Event design

Event callbacks должны быть scoped.

Плохо:

```kotlin
@Composable
fun AppRoot(
    state: AppState,
    onEvent: (AppEvent) -> Unit
) {
    CatalogScreen(
        state = state.catalog,
        onEvent = onEvent
    )
}
```

Каждый child получает огромный generic event pipe.

Лучше:

```kotlin
@Composable
fun CatalogScreen(
    state: CatalogState,
    onProductClick: (String) -> Unit,
    onRetry: () -> Unit,
    onQueryChange: (String) -> Unit
) {
    CatalogContent(
        state = state,
        onProductClick = onProductClick,
        onRetry = onRetry,
        onQueryChange = onQueryChange
    )
}
```

Scoped callbacks делают dependencies явными и часто уменьшают accidental recomposition.

## Split composables by state

Разделяйте UI так, чтобы часто меняющийся state не затрагивал статический content.

Плохо:

```kotlin
@Composable
fun PlayerScreen(
    state: PlayerState
) {
    Column {
        AlbumArt(url = state.albumArtUrl)
        Text(text = state.title)
        Text(text = state.positionText)
        ProgressBar(progress = state.progress)
    }
}
```

Если `positionText` и `progress` меняются каждую секунду, можно отделить:

```kotlin
@Composable
fun PlayerScreen(
    state: PlayerState
) {
    Column {
        TrackHeader(
            albumArtUrl = state.albumArtUrl,
            title = state.title
        )
        PlaybackProgress(
            positionText = state.positionText,
            progress = state.progress
        )
    }
}
```

Теперь static header и dynamic progress имеют разные recomposition scopes.

## Avoid backwards writes

Backwards write — запись в state после чтения того же state в composition. Это может создать infinite recomposition или странное поведение.

Плохо:

```kotlin
@Composable
fun BadCounter() {
    var count by remember {
        mutableStateOf(0)
    }

    Text(text = "Count: $count")
    count += 1
}
```

Правильно: state меняется по event/effect, а не прямо в composition.

```kotlin
@Composable
fun GoodCounter(
    count: Int,
    onIncrement: () -> Unit
) {
    Button(onClick = onIncrement) {
        Text(text = "Count: $count")
    }
}
```

Composition должна описывать UI, а не выполнять бесконечную state machine.

## Stable modifiers

Modifier chains тоже создают objects. Обычно это не проблема, но в hot lists можно вынести reusable modifier parts.

```kotlin
val rowModifier = Modifier
    .fillMaxWidth()
    .padding(horizontal = 16.dp, vertical = 8.dp)

LazyColumn {
    items(
        items = messages,
        key = { it.id }
    ) { message ->
        MessageRow(
            message = message,
            modifier = rowModifier
        )
    }
}
```

Если modifier зависит от item state, оставьте его внутри item:

```kotlin
MessageRow(
    message = message,
    modifier = Modifier
        .fillMaxWidth()
        .background(
            if (message.selected) {
                MaterialTheme.colorScheme.secondaryContainer
            } else {
                MaterialTheme.colorScheme.surface
            }
        )
)
```

Не выносите modifier так, чтобы он скрывал item-specific behavior.

## Snapshot state granularity

Один огромный `UiState` проще, но иногда из-за него recomposes слишком много UI.

```kotlin
data class EditorUiState(
    val document: DocumentUi,
    val cursor: CursorUi,
    val toolbar: ToolbarUi,
    val collaborators: List<CollaboratorUi>
)
```

Если cursor меняется часто, не передавайте весь `EditorUiState` в every child. Передайте нужные parts:

```kotlin
EditorCanvas(
    document = state.document,
    cursor = state.cursor
)

EditorToolbar(
    toolbar = state.toolbar
)

CollaboratorsPanel(
    collaborators = state.collaborators
)
```

State object может оставаться одним в ViewModel, но UI parameters должны быть scoped.

## Practical refactoring workflow

Когда нашли лишнюю recomposition:

- подтвердите сценарий в Layout Inspector;
- найдите state read, который ее запускает;
- проверьте parameters stability;
- проверьте expensive work в composable;
- разделите composables по state boundaries;
- перенесите state read ниже;
- используйте lambda modifier для layout/draw hot values;
- добавьте `remember`/`derivedStateOf` только там, где есть причина;
- повторите измерение.

```kotlin
data class RecompositionRefactor(
    val cause: String,
    val change: String,
    val measurementBefore: String,
    val measurementAfter: String
)
```

## Типичные ошибки

### Дробить composables без причины

Мелкие функции помогают boundaries, но чрезмерная дробность ухудшает читаемость.

### Remember everything

`remember` с неправильными keys дает stale UI.

### Использовать `derivedStateOf` для простых выражений

Он нужен для reduction частых changes, не для любого computed value.

### Тащить state наверх слишком высоко

Hoisting не значит "все state в root".

### Игнорировать phase-specific reads

Scroll/animation values часто лучше читать в layout/draw lambdas.

### Обманывать stability annotations

Неверный `@Stable`/`@Immutable` может привести к stale UI.

## Итоги

Избегание лишней recomposition строится вокруг state boundaries:

- передавайте stable/immutable UI models;
- не читайте hot state выше, чем нужно;
- разделяйте static и dynamic UI scopes;
- используйте `remember` для expensive objects с правильными keys;
- используйте `derivedStateOf`, когда частый source state дает редко меняющийся result;
- defer state reads через lambdas и lambda modifiers;
- callbacks должны быть scoped и понятными;
- избегайте backwards writes;
- modifier reuse полезен в hot paths, но не должен скрывать item state;
- каждую оптимизацию проверяйте Layout Inspector/trace/benchmark до и после.
