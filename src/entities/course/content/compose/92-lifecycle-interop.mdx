## Где Compose встречается с Android lifecycle

Compose имеет собственную модель жизни: composition, recomposition, disposal. Android имеет lifecycle: `ON_CREATE`, `ON_START`, `ON_RESUME`, `ON_PAUSE`, `ON_STOP`, `ON_DESTROY`. Эти модели связаны, но не равны.

Например:

- composable может recomposition-иться много раз без lifecycle event;
- composable может исчезнуть из composition, пока Activity остается `RESUMED`;
- Fragment view может уничтожиться, пока Fragment instance еще жив;
- Flow collection должна останавливаться, когда screen не видим;
- camera/sensor/player должен освобождаться, когда composable ушел;
- обратный вызов внутри listener должен оставаться актуальным после recomposition.

Lifecycle interop нужен, когда Compose UI общается с Android объектами, которые живут вне composition.

```kotlin
import androidx.compose.material3.Text
import androidx.compose.runtime.Composable
import androidx.lifecycle.compose.LocalLifecycleOwner

@Composable
fun LifecycleOwnerName() {
    val lifecycleOwner = LocalLifecycleOwner.current

    Text(
        text = lifecycleOwner.lifecycle.currentState.name
    )
}
```

`LocalLifecycleOwner` дает текущий `LifecycleOwner`, который обычно приходит от Activity, Fragment view или `ComposeView` host.

<CatTip>
Composition lifecycle отвечает за "этот UI сейчас в дереве?". Android lifecycle отвечает за "этот host сейчас started/resumed/destroyed?". Для внешних ресурсов часто нужны оба ответа.
</CatTip>

## `collectAsStateWithLifecycle`

Для UI-состояние из `Flow` или `StateFlow` используйте lifecycle-aware collection.

```kotlin
import androidx.compose.runtime.Composable
import androidx.compose.runtime.getValue
import androidx.lifecycle.compose.collectAsStateWithLifecycle

@Composable
fun ProfileRoute(
    viewModel: ProfileViewModel
) {
    val state by viewModel.state.collectAsStateWithLifecycle()

    ProfileScreen(
        state = state,
        onRetry = viewModel::retry
    )
}
```

`collectAsStateWithLifecycle` собирает значения lifecycle-aware образом и представляет последнее значение как Compose `State`. Это базовый путь для state из ViewModel.

Плохой вариант:

```kotlin
import androidx.compose.runtime.Composable
import androidx.compose.runtime.LaunchedEffect

@Composable
fun BadProfileRoute(
    viewModel: ProfileViewModel
) {
    LaunchedEffect(Unit) {
        viewModel.state.collect { state ->
            viewModel.logVisibleState(state)
        }
    }
}
```

Этот код не является UI-состояние rendering, а collection не привязана к visible lifecycle так явно, как нужно. Для побочные эффекты существуют отдельные patterns.

## Минимальный lifecycle state

Иногда нужно собирать state только с определенного lifecycle state.

```kotlin
import androidx.compose.runtime.Composable
import androidx.compose.runtime.getValue
import androidx.lifecycle.Lifecycle
import androidx.lifecycle.compose.collectAsStateWithLifecycle

@Composable
fun StartedOnlyRoute(
    viewModel: DashboardViewModel
) {
    val state by viewModel.state.collectAsStateWithLifecycle(
        minActiveState = Lifecycle.State.STARTED
    )

    DashboardScreen(state = state)
}
```

`STARTED` обычно подходит для visible UI. `RESUMED` нужен реже: например, если state имеет смысл обновлять только пока screen получает user input.

## `currentStateAsState`

Если UI должен реагировать на lifecycle state, можно получить его как Compose state.

```kotlin
import androidx.compose.material3.Text
import androidx.compose.runtime.Composable
import androidx.compose.runtime.getValue
import androidx.lifecycle.compose.LocalLifecycleOwner
import androidx.lifecycle.compose.currentStateAsState

@Composable
fun LifecycleDebugLabel() {
    val lifecycleOwner = LocalLifecycleOwner.current
    val lifecycleState by lifecycleOwner.lifecycle.currentStateAsState()

    Text(
        text = "Lifecycle: $lifecycleState"
    )
}
```

Это полезно для debug UI, аналитика tooling и rare cases. Не стоит строить весь UI вокруг lifecycle state, если можно строить его вокруг domain state.

## `LifecycleEventEffect`

Для реакции на конкретное lifecycle event есть `LifecycleEventEffect`.

```kotlin
import androidx.compose.runtime.Composable
import androidx.lifecycle.Lifecycle
import androidx.lifecycle.compose.LifecycleEventEffect

@Composable
fun AnalyticsLifecycleEvents(
    screenName: String,
    analytics: Analytics
) {
    LifecycleEventEffect(Lifecycle.Event.ON_RESUME) {
        analytics.logScreenResumed(screenName)
    }

    LifecycleEventEffect(Lifecycle.Event.ON_PAUSE) {
        analytics.logScreenPaused(screenName)
    }
}
```

Этот API хорош для простых event reactions. Если нужен объект, который нужно создать на start и закрыть на stop, часто удобнее `LifecycleStartEffect` или `DisposableEffect` с observer.

## `DisposableEffect` с `LifecycleEventObserver`

Универсальный способ подписаться на lifecycle:

```kotlin
import androidx.compose.runtime.Composable
import androidx.compose.runtime.DisposableEffect
import androidx.lifecycle.Lifecycle
import androidx.lifecycle.LifecycleEventObserver
import androidx.lifecycle.compose.LocalLifecycleOwner

@Composable
fun LifecycleLogger(
    logger: Logger
) {
    val lifecycleOwner = LocalLifecycleOwner.current

    DisposableEffect(lifecycleOwner, logger) {
        val observer = LifecycleEventObserver { _, event ->
            when (event) {
                Lifecycle.Event.ON_START -> logger.log("Экран стал видимым")
                Lifecycle.Event.ON_STOP -> logger.log("Экран перестал быть видимым")
                else -> Unit
            }
        }

        lifecycleOwner.lifecycle.addObserver(observer)

        onDispose {
            lifecycleOwner.lifecycle.removeObserver(observer)
        }
    }
}
```

`DisposableEffect` нужен, потому что observer нужно снять. Любая подписка на внешний объект должна иметь cleanup.

<MouseTrap>
Если вы добавили listener/observer в composable и не можете показать, где он снимается, почти наверняка вы сделали lifecycle leak.
</MouseTrap>

## `rememberUpdatedState` для свежих обратные вызовы

Внешний listener часто создается один раз, но должен вызывать актуальную лямбда после recomposition. Для этого используется `rememberUpdatedState`.

```kotlin
import androidx.compose.runtime.Composable
import androidx.compose.runtime.DisposableEffect
import androidx.compose.runtime.getValue
import androidx.compose.runtime.rememberUpdatedState
import androidx.lifecycle.Lifecycle
import androidx.lifecycle.LifecycleEventObserver
import androidx.lifecycle.compose.LocalLifecycleOwner

@Composable
fun OnResumeAction(
    onResume: () -> Unit
) {
    val lifecycleOwner = LocalLifecycleOwner.current
    val currentOnResume by rememberUpdatedState(onResume)

    DisposableEffect(lifecycleOwner) {
        val observer = LifecycleEventObserver { _, event ->
            if (event == Lifecycle.Event.ON_RESUME) {
                currentOnResume()
            }
        }

        lifecycleOwner.lifecycle.addObserver(observer)

        onDispose {
            lifecycleOwner.lifecycle.removeObserver(observer)
        }
    }
}
```

Без `rememberUpdatedState` observer может захватить старую лямбда и вызвать устаревшее действие.

## Remembering external objects safely

`remember` создает объект, привязанный к composition. Если объект держит resource, нужен `DisposableEffect`.

```kotlin
import androidx.compose.runtime.Composable
import androidx.compose.runtime.DisposableEffect
import androidx.compose.runtime.remember
import androidx.compose.ui.platform.LocalContext

@Composable
fun AudioRecorderHost(
    enabled: Boolean
) {
    val context = LocalContext.current
    val recorder = remember {
        AudioRecorder(context.applicationContext)
    }

    DisposableEffect(recorder) {
        onDispose {
            recorder.release()
        }
    }

    if (enabled) {
        RecorderControls(
            onStart = recorder::start,
            onStop = recorder::stop
        )
    }
}
```

Здесь `remember` гарантирует, что recorder не создается на каждой recomposition. `DisposableEffect` гарантирует cleanup.

## Start/stop resources по lifecycle

Некоторые ресурсы должны существовать, пока composable в composition, но активно работать только между `ON_START` и `ON_STOP`.

```kotlin
import androidx.compose.runtime.Composable
import androidx.compose.runtime.DisposableEffect
import androidx.compose.runtime.remember
import androidx.compose.ui.platform.LocalContext
import androidx.lifecycle.Lifecycle
import androidx.lifecycle.LifecycleEventObserver
import androidx.lifecycle.compose.LocalLifecycleOwner

@Composable
fun LocationPreview(
    onLocation: (LocationPoint) -> Unit
) {
    val context = LocalContext.current
    val lifecycleOwner = LocalLifecycleOwner.current
    val locationClient = remember {
        LocationClient(context.applicationContext)
    }

    DisposableEffect(
        lifecycleOwner,
        locationClient,
        onLocation
    ) {
        val observer = LifecycleEventObserver { _, event ->
            when (event) {
                Lifecycle.Event.ON_START -> {
                    locationClient.start(onLocation)
                }
                Lifecycle.Event.ON_STOP -> {
                    locationClient.stop()
                }
                else -> Unit
            }
        }

        lifecycleOwner.lifecycle.addObserver(observer)

        onDispose {
            lifecycleOwner.lifecycle.removeObserver(observer)
            locationClient.stop()
            locationClient.release()
        }
    }
}
```

Но тут есть тонкость: `onLocation` в keys пересоздаст effect при смене лямбда. Если listener должен жить стабильно, используйте `rememberUpdatedState`.

```kotlin
import androidx.compose.runtime.Composable
import androidx.compose.runtime.DisposableEffect
import androidx.compose.runtime.getValue
import androidx.compose.runtime.remember
import androidx.compose.runtime.rememberUpdatedState
import androidx.compose.ui.platform.LocalContext
import androidx.lifecycle.Lifecycle
import androidx.lifecycle.LifecycleEventObserver
import androidx.lifecycle.compose.LocalLifecycleOwner

@Composable
fun StableLocationPreview(
    onLocation: (LocationPoint) -> Unit
) {
    val context = LocalContext.current
    val lifecycleOwner = LocalLifecycleOwner.current
    val currentOnLocation by rememberUpdatedState(onLocation)
    val locationClient = remember {
        LocationClient(context.applicationContext)
    }

    DisposableEffect(lifecycleOwner, locationClient) {
        val observer = LifecycleEventObserver { _, event ->
            when (event) {
                Lifecycle.Event.ON_START -> {
                    locationClient.start {
                        currentOnLocation(it)
                    }
                }
                Lifecycle.Event.ON_STOP -> {
                    locationClient.stop()
                }
                else -> Unit
            }
        }

        lifecycleOwner.lifecycle.addObserver(observer)

        onDispose {
            lifecycleOwner.lifecycle.removeObserver(observer)
            locationClient.stop()
            locationClient.release()
        }
    }
}
```

Так resource не перезапускается из-за каждой смены обратный вызов identity.

## `repeatOnLifecycle` outside Compose

В View/Fragment code collection обычно выглядит так:

```kotlin
import androidx.lifecycle.Lifecycle
import androidx.lifecycle.lifecycleScope
import androidx.lifecycle.repeatOnLifecycle
import kotlinx.coroutines.launch

class ProfileFragment : Fragment(R.layout.profile_fragment) {

    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        viewLifecycleOwner.lifecycleScope.launch {
            viewLifecycleOwner.repeatOnLifecycle(
                Lifecycle.State.STARTED
            ) {
                viewModel.state.collect { state ->
                    render(state)
                }
            }
        }
    }
}
```

В Compose route чаще пишут `collectAsStateWithLifecycle`, потому что UI хочет State value.

## One-off events

Одноразовые события требуют осторожности. Не храните snackbar или navigation как обычное поле `UiState`, которое будет заново срабатывать после configuration change.

```kotlin
sealed interface ProfileEvent {
    data class ShowSnackbar(
        val message: String
    ) : ProfileEvent

    data object NavigateBack : ProfileEvent
}
```

Collect events lifecycle-aware:

```kotlin
import androidx.compose.material3.SnackbarHostState
import androidx.compose.runtime.Composable
import androidx.compose.runtime.LaunchedEffect
import androidx.compose.runtime.remember
import androidx.lifecycle.Lifecycle
import androidx.lifecycle.compose.LocalLifecycleOwner
import androidx.lifecycle.repeatOnLifecycle

@Composable
fun ProfileEvents(
    viewModel: ProfileViewModel,
    onNavigateBack: () -> Unit
) {
    val lifecycleOwner = LocalLifecycleOwner.current
    val snackbarHostState = remember {
        SnackbarHostState()
    }

    LaunchedEffect(viewModel, lifecycleOwner) {
        lifecycleOwner.repeatOnLifecycle(Lifecycle.State.STARTED) {
            viewModel.events.collect { event ->
                when (event) {
                    is ProfileEvent.ShowSnackbar -> {
                        snackbarHostState.showSnackbar(event.message)
                    }
                    ProfileEvent.NavigateBack -> {
                        onNavigateBack()
                    }
                }
            }
        }
    }
}
```

Для UI-состояние используйте `collectAsStateWithLifecycle`. Для event stream используйте `LaunchedEffect` + `repeatOnLifecycle`, если событие должно обрабатываться только пока screen visible.

## `dropUnlessResumed` и защита navigation

Иногда обратный вызов не должен выполняться, если lifecycle уже не `RESUMED`. Например, пользователь дважды нажал на кнопку, screen уже начал navigation, а второй обратный вызов приходит позже.

```kotlin
import androidx.compose.material3.Button
import androidx.compose.material3.Text
import androidx.compose.runtime.Composable
import androidx.lifecycle.compose.dropUnlessResumed

@Composable
fun SafeNavigationButton(
    onOpenDetails: () -> Unit
) {
    val safeOpenDetails = dropUnlessResumed {
        onOpenDetails()
    }

    Button(onClick = safeOpenDetails) {
        Text(text = "Открыть")
    }
}
```

Это не заменяет idempotent navigation logic, но помогает не выполнять действия, когда lifecycle уже не готов.

## Lifecycle внутри `ComposeView`

Когда Compose живет внутри View hierarchy, `LocalLifecycleOwner` приходит из ViewTree. Во Fragment это должен быть view lifecycle owner.

```kotlin
class DetailsFragment : Fragment(R.layout.details_fragment) {

    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        val composeView = view.findViewById<ComposeView>(
            R.id.compose_details
        )

        composeView.setViewCompositionStrategy(
            ViewCompositionStrategy.DisposeOnViewTreeLifecycleDestroyed
        )
        composeView.setContent {
            DetailsRoute(
                viewModel = viewModel
            )
        }
    }
}
```

Если `ComposeView` живет дольше Fragment view, lifecycle-aware collection и effects могут держать старую View hierarchy.

## Camera, sensor, player

External objects обычно имеют три разные операции:

- create;
- start/stop;
- release.

Не смешивайте их.

```kotlin
@Composable
fun CameraPreviewHost(
    cameraControllerFactory: () -> CameraController
) {
    val lifecycleOwner = LocalLifecycleOwner.current
    val cameraController = remember {
        cameraControllerFactory()
    }

    DisposableEffect(lifecycleOwner, cameraController) {
        val observer = LifecycleEventObserver { _, event ->
            when (event) {
                Lifecycle.Event.ON_START -> cameraController.start()
                Lifecycle.Event.ON_STOP -> cameraController.stop()
                else -> Unit
            }
        }

        lifecycleOwner.lifecycle.addObserver(observer)

        onDispose {
            lifecycleOwner.lifecycle.removeObserver(observer)
            cameraController.stop()
            cameraController.release()
        }
    }

    CameraPreview(
        controller = cameraController
    )
}
```

Если controller зависит от permission, lens, session config, эти параметры должны быть keys. Но keys должны быть осознанными: каждое изменение key пересоздает lifecycle effect.

## Testing lifecycle interop

Что проверять:

- Flow не собирается, когда lifecycle ниже `STARTED`;
- observer снимается при dispose;
- external resource получает `stop` и `release`;
- обратный вызов после recomposition актуален;
- one-off events не повторяются после rotation;
- `ComposeView` во Fragment уничтожает Composition в `onDestroyView`;
- navigation action не выполняется после `ON_STOP`.

Unit-тест для resource wrapper:

```kotlin
class FakeCameraController : CameraController {
    var starts = 0
    var stops = 0
    var releases = 0

    override fun start() {
        starts += 1
    }

    override fun stop() {
        stops += 1
    }

    override fun release() {
        releases += 1
    }
}
```

Compose UI lifecycle tests обычно требуют `createAndroidComposeRule` и управление Activity state. Для чистой logic проще вынести lifecycle mapping в маленький adapter и тестировать его отдельно.

## Типичные ошибки

### Collect Flow через обычный `collectAsState`

На Android UI обычно нужен lifecycle-aware вариант.

### Использовать Activity lifecycle во Fragment view

Fragment view может быть уничтожена раньше Fragment/Activity lifecycle.

### Создавать external object на каждой recomposition

Без `remember` вы получите лишние подключения, observers или SDK instances.

### Не освобождать resource

Camera, sensor, player и recorder не исчезают магически вместе с composable.

### Захватывать старую лямбда в listener

Используйте `rememberUpdatedState`, если listener живет дольше recomposition.

### Смешивать state и events

Snackbar/navigation как поле `UiState` часто повторяется после восстановления.

### Делать lifecycle центральной бизнес-моделью

Lifecycle должен управлять подписками и ресурсами, а не заменять domain state.

## Итоги

Lifecycle interop в Compose строится на ясном разделении:

- `LocalLifecycleOwner` дает текущий host lifecycle;
- `collectAsStateWithLifecycle` подходит для UI-состояние из Flow/StateFlow;
- `currentStateAsState` превращает lifecycle state в Compose state;
- `LifecycleEventEffect` удобен для простых lifecycle event reactions;
- `DisposableEffect` нужен для observers, listeners и cleanup;
- `remember` создает external object один раз на composition lifetime;
- `rememberUpdatedState` защищает long-lived listeners от stale обратные вызовы;
- `repeatOnLifecycle` полезен для event streams и View/Fragment interop;
- external resources обычно имеют create/start/stop/release phases;
- во Fragment + `ComposeView` lifecycle должен быть привязан к ViewTree lifecycle;
- lifecycle-aware code должен быть тестируемым и не прятать business state.
