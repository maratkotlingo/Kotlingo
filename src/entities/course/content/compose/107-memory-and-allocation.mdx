## Почему allocations важны для Compose

Даже если каждая отдельная allocation маленькая, тысячи allocations в scroll/animation frame могут привести к GC и jank. Compose делает много работы за вас, но ваш код все еще может создавать:

- новые списки в composition;
- новые modifier chains в hot path;
- новые lambdas/objects в lazy item;
- `Path`, `Brush`, `Paint`, `ColorFilter` в draw;
- formatters и formatted strings;
- bitmap/painter objects;
- snapshot state updates слишком частыми копиями;
- temporary data во время sorting/filtering.

```kotlin
data class AllocationFinding(
    val screen: String,
    val hotPath: String,
    val allocatedObject: String,
    val suspectedImpact: String
)
```

Memory work не всегда виден в recomposition counters. Иногда UI recomposes мало, но draw создает objects каждый frame.

<CatTip>
GC jank неприятен тем, что выглядит как "случайный лаг". На самом деле он часто вызван вполне повторяемыми allocations в hot path.
</CatTip>

## Hot paths

Особенно внимательно смотрите:

- lazy list item composition;
- scroll observers;
- animation frame;
- `Canvas` draw;
- `drawBehind`;
- `graphicsLayer` updates;
- text input;
- pointer input loop;
- image loading placeholders;
- custom layout measurement.

Холодный path, например settings screen once per session, не требует такой же агрессии.

## Modifier reuse

Modifier chains immutable и дешевые, но в больших списках можно вынести общие части.

```kotlin
val feedRowModifier = Modifier
    .fillMaxWidth()
    .padding(horizontal = 16.dp, vertical = 8.dp)

LazyColumn {
    items(
        items = feedItems,
        key = { item ->
            item.id
        }
    ) { item ->
        FeedRow(
            item = item,
            modifier = feedRowModifier
        )
    }
}
```

Это помогает уменьшить repeated allocation и делает common layout contract явным.

Не выносите stateful или item-dependent modifier:

```kotlin
FeedRow(
    item = item,
    modifier = Modifier
        .fillMaxWidth()
        .background(
            if (item.selected) {
                MaterialTheme.colorScheme.secondaryContainer
            } else {
                MaterialTheme.colorScheme.surface
            }
        )
)
```

Если modifier зависит от `item`, он должен остаться рядом с item.

## Remember expensive objects

`remember` подходит для expensive objects:

```kotlin
import androidx.compose.runtime.Composable
import androidx.compose.runtime.remember
import androidx.compose.ui.graphics.Brush
import androidx.compose.ui.graphics.Color

@Composable
fun BrandGradientSurface(
    content: @Composable () -> Unit
) {
    val brush = remember {
        Brush.linearGradient(
            colors = listOf(
                Color(0xFF005B4F),
                Color(0xFF3A6EA5)
            )
        )
    }

    GradientSurface(
        brush = brush,
        content = content
    )
}
```

Если object зависит от parameters, keys должны быть полными:

```kotlin
val brush = remember(startColor, endColor) {
    Brush.linearGradient(
        colors = listOf(startColor, endColor)
    )
}
```

Неполные keys дают stale UI.

## Painter and image memory

Изображения — большой memory risk:

- decode original size для маленькой thumbnail;
- держать bitmap в composable state;
- создавать ImageRequest на каждый recomposition;
- не очищать heavy painter;
- использовать animated images без constraints;
- забывать content scale/aspect ratio.

```kotlin
@Composable
fun AvatarImage(
    url: String?,
    modifier: Modifier = Modifier
) {
    AsyncImage(
        model = ImageRequest.Builder(LocalContext.current)
            .data(url)
            .crossfade(true)
            .build(),
        contentDescription = null,
        contentScale = ContentScale.Crop,
        modifier = modifier
            .size(48.dp)
            .clip(CircleShape)
    )
}
```

Если request строится вручную и recomposes часто, remember:

```kotlin
val context = LocalContext.current
val request = remember(url, context) {
    ImageRequest.Builder(context)
        .data(url)
        .crossfade(true)
        .build()
}
```

Не передавайте `Painter` из ViewModel. ViewModel должен отдавать URL/resource id/domain state.

## `drawWithCache`

`drawWithCache` полезен, когда draw object зависит от size или stable parameters.

```kotlin
import androidx.compose.ui.Modifier
import androidx.compose.ui.draw.drawWithCache
import androidx.compose.ui.graphics.Brush
import androidx.compose.ui.graphics.Color

fun Modifier.cachedGradientBorder(): Modifier {
    return drawWithCache {
        val brush = Brush.linearGradient(
            colors = listOf(
                Color(0xFF006C4F),
                Color(0xFF88D4AB)
            ),
            start = topLeft,
            end = bottomRight
        )

        onDrawBehind {
            drawRect(brush = brush)
        }
    }
}
```

`drawWithCache` пересоздает cache, когда size/state dependencies меняются. Это лучше, чем создавать brush в каждой draw call.

## Path caching

Плохо:

```kotlin
Canvas(modifier = Modifier.fillMaxSize()) {
    val path = Path()
    path.moveTo(0f, size.height)
    path.lineTo(size.width, 0f)
    path.lineTo(size.width, size.height)
    path.close()

    drawPath(
        path = path,
        color = Color.Black
    )
}
```

Лучше:

```kotlin
fun Modifier.cachedTriangleBackground(
    color: Color
): Modifier {
    return drawWithCache {
        val path = Path().apply {
            moveTo(0f, size.height)
            lineTo(size.width, 0f)
            lineTo(size.width, size.height)
            close()
        }

        onDrawBehind {
            drawPath(
                path = path,
                color = color
            )
        }
    }
}
```

Если geometry depends on size, cache by size via `drawWithCache`.

## Snapshot churn

Snapshot churn — слишком частые state writes или слишком крупные immutable copies.

Плохо:

```kotlin
var points by mutableStateOf(emptyList<Offset>())

fun onDrag(point: Offset) {
    points = points + point
}
```

При каждом pointer move создается новый list. Для drawing это может быть слишком дорого.

Вариант лучше для in-progress stroke:

```kotlin
val points = mutableStateListOf<Offset>()

fun onDrag(point: Offset) {
    points.add(point)
}
```

Но это не значит, что mutable snapshot list всегда лучше. Для UI state высокого уровня immutable copy часто безопаснее и проще.

Практическое правило:

- business/UI screen state: immutable snapshots;
- hot interactive buffer: specialized mutable state holder;
- persistence boundary: serialize stable document model;
- после завершения gesture можно commit в immutable model.

## State write frequency

Если sensor/timer/input пишет state слишком часто, UI может захлебнуться.

```kotlin
class DragState {
    var offset by mutableStateOf(Offset.Zero)
        private set

    fun update(rawOffset: Offset) {
        offset = rawOffset
    }
}
```

Для high-frequency data подумайте:

- нужно ли каждое значение UI;
- можно ли throttle/debounce;
- можно ли читать в draw/layout phase;
- можно ли хранить buffer вне Compose state;
- можно ли commit-ить coarse state.

## Avoid allocations in pointer loops

Плохо:

```kotlin
pointerInput(Unit) {
    awaitPointerEventScope {
        while (true) {
            val event = awaitPointerEvent()
            val activePointers = event.changes.map { change ->
                change.position
            }
            handle(activePointers)
        }
    }
}
```

`map` создает list на каждое pointer event.

Лучше:

```kotlin
pointerInput(Unit) {
    awaitPointerEventScope {
        val buffer = ArrayList<Offset>(4)

        while (true) {
            val event = awaitPointerEvent()
            buffer.clear()

            event.changes.forEach { change ->
                buffer.add(change.position)
            }

            handle(buffer)
        }
    }
}
```

Здесь buffer живет внутри coroutine и переиспользуется. Не отдавайте mutable buffer наружу на хранение.

## String formatting

Formatting может быть дорогим и allocation-heavy.

Плохо:

```kotlin
@Composable
fun PriceLabel(
    cents: Long
) {
    Text(
        text = NumberFormat
            .getCurrencyInstance()
            .format(cents / 100.0)
    )
}
```

Лучше:

```kotlin
@Composable
fun PriceLabel(
    priceText: String
) {
    Text(text = priceText)
}
```

Formatter можно держать в mapper/state holder:

```kotlin
class PriceFormatter(
    private val locale: Locale
) {
    private val delegate = NumberFormat
        .getCurrencyInstance(locale)

    fun format(cents: Long): String {
        return delegate.format(cents / 100.0)
    }
}
```

UI получает готовый text.

## Lambdas and captures

Не нужно бояться каждой lambda. Но в hot lists следите за captures:

```kotlin
items(
    items = products,
    key = { it.id }
) { product ->
    ProductRow(
        product = product,
        onClick = {
            onProductClick(product.id)
        }
    )
}
```

Это обычно нормально. Проблема начинается, если внутри создаются большие event objects или тяжелые closures.

Плохо:

```kotlin
onClick = {
    analytics.log(
        ProductClickEvent(
            product = product,
            visibleFilters = filters.toList(),
            timestamp = clock.now()
        )
    )
    onProductClick(product.id)
}
```

Лучше вынести analytics выше или передавать компактные IDs.

## Snapshot state lists/maps

`mutableStateListOf` и `mutableStateMapOf` полезны, когда collection изменяется granularly и UI должен наблюдать эти изменения.

```kotlin
class SelectionState {
    private val selectedIds = mutableStateListOf<String>()

    fun isSelected(id: String): Boolean {
        return id in selectedIds
    }

    fun toggle(id: String) {
        if (id in selectedIds) {
            selectedIds.remove(id)
        } else {
            selectedIds.add(id)
        }
    }
}
```

Но для screen state часто лучше immutable set:

```kotlin
data class SelectionUiState(
    val selectedIds: Set<String>
)
```

Выбор зависит от frequency, size и ownership.

## Memory profiling workflow

Практический workflow:

- воспроизвести jank/leak/OOM;
- открыть Memory Profiler;
- записать allocation tracking для сценария;
- найти high allocation classes;
- проверить trace на GC pauses;
- проверить image memory;
- проверить lifecycle leaks;
- внести одну оптимизацию;
- повторить recording.

```kotlin
data class MemoryOptimizationNote(
    val scenario: String,
    val topAllocation: String,
    val change: String,
    val before: String,
    val after: String
)
```

Memory fixes без before/after часто превращаются в усложнение кода ради эстетики.

## Leak risks

Compose-specific leak risks:

- remember external object without dispose;
- listener captures Activity/Fragment;
- coroutine scope lives beyond expected lifecycle;
- `AndroidView` keeps old callback;
- image loader/request holds Context incorrectly;
- static singleton holds composable state.

```kotlin
@Composable
fun SensorHost(
    sensorClient: SensorClient
) {
    DisposableEffect(sensorClient) {
        sensorClient.start()

        onDispose {
            sensorClient.stop()
        }
    }
}
```

External resource должен иметь cleanup.

## Типичные ошибки

### Создавать `Path`/`Brush` в каждом draw

Используйте `drawWithCache`, если geometry зависит от size.

### Форматировать строки в каждом item

Готовьте UI text заранее или кэшируйте formatter.

### Копировать огромный list на каждый drag event

Разделяйте hot buffer и committed immutable model.

### Декодировать большие изображения для thumbnail

Image memory быстро съедает budget.

### Remember with wrong keys

Меньше allocations, но stale UI.

### Лечить несуществующую проблему

Не усложняйте код без profiler/trace signal.

## Итоги

Memory и allocations в Compose требуют внимания к hot paths:

- allocations в scroll/animation/draw могут вызвать GC jank;
- reusable modifier parts полезны в больших списках;
- expensive objects кэшируйте через `remember` с правильными keys;
- `drawWithCache` подходит для Brush/Path/geometry by size;
- images должны иметь правильный decode size и lifecycle;
- snapshot churn появляется от слишком частых state writes/copies;
- high-frequency gestures могут использовать mutable buffers/state holders;
- formatting лучше делать вне hot composition;
- external resources требуют `DisposableEffect`;
- оптимизируйте по Memory Profiler/System Trace, а не по догадкам.
