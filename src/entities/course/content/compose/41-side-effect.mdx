## Что делает SideEffect

`SideEffect` выполняет блок кода после каждой успешной recomposition.

Это значит:

- composable был выполнен;
- изменения composition успешно применены;
- только после этого запускается `SideEffect`;
- при следующей успешной recomposition блок выполнится снова.

`SideEffect` нужен, чтобы публиковать Compose state во внешний объект, которым Compose не управляет.

```kotlin
import androidx.compose.runtime.Composable
import androidx.compose.runtime.SideEffect

class ExternalTitleStore {
    var title: String = ""
}

@Composable
fun PublishTitle(
    title: String,
    externalTitleStore: ExternalTitleStore
) {
    SideEffect {
        externalTitleStore.title = title
    }
}
```

Если `title` изменится и recomposition успешно применится, внешний store получит актуальное значение.

<CatTip>
`SideEffect` — это не "любой побочный эффект". Это очень конкретный инструмент: синхронно опубликовать уже примененное Compose-состояние во внешний объект.
</CatTip>

## Почему не писать прямо в тело composable

Плохо:

```kotlin
import androidx.compose.runtime.Composable

class ExternalUserProperties {
    var userType: String = ""
}

@Composable
fun UserPropertiesWrong(
    userType: String,
    properties: ExternalUserProperties
) {
    properties.userType = userType
}
```

Проблема:

- запись происходит до гарантии успешного применения composition;
- recomposition может быть отменена;
- внешнее состояние может получить значение от UI, который не стал текущим;
- side effect спрятан в теле composable.

Лучше:

```kotlin
import androidx.compose.runtime.Composable
import androidx.compose.runtime.SideEffect

@Composable
fun UserProperties(
    userType: String,
    properties: ExternalUserProperties
) {
    SideEffect {
        properties.userType = userType
    }
}
```

`SideEffect` гарантирует, что запись происходит после успешной recomposition.

## Когда использовать

Используйте `SideEffect`, если:

- нужно обновить внешний mutable object;
- операция быстрая и синхронная;
- операция должна выполняться после каждой успешной recomposition;
- не нужна coroutine;
- не нужен cleanup;
- не нужна подписка;
- не нужен restart по ключу.

Примеры:

- обновить user property analytics SDK;
- записать текущий screen name в non-Compose holder;
- синхронизировать маленький adapter state;
- сообщить внешнему объекту current density/theme flag;
- обновить bridge object, который читает imperative слой.

Не используйте `SideEffect`, если:

- нужно вызвать suspend-функцию;
- нужна задержка;
- нужна навигация;
- нужно показать snackbar;
- нужна подписка с очисткой;
- нужно выполнить действие один раз;
- работа дорогая;
- запись может блокировать main thread.

## Analytics user properties

Классический пример — аналитика, у которой есть mutable user properties.

```kotlin
import androidx.compose.runtime.Composable
import androidx.compose.runtime.SideEffect
import androidx.compose.runtime.remember

class AnalyticsClient {
    private val properties = mutableMapOf<String, String>()

    fun setUserProperty(
        key: String,
        value: String
    ) {
        properties[key] = value
    }

    fun trackEvent(
        name: String
    ) {
        val eventProperties = properties.toMap()
        saveEvent(name, eventProperties)
    }

    private fun saveEvent(
        name: String,
        properties: Map<String, String>
    ) {
        // В реальном SDK здесь событие отправляется во внутреннюю очередь аналитики.
    }
}

@Composable
fun rememberAnalyticsClient(
    userType: String
): AnalyticsClient {
    val analyticsClient = remember {
        AnalyticsClient()
    }

    SideEffect {
        analyticsClient.setUserProperty(
            key = "user_type",
            value = userType
        )
    }

    return analyticsClient
}
```

Разбор:

- `AnalyticsClient` создается один раз через `remember`;
- после каждой успешной recomposition обновляется `user_type`;
- последующие events получат актуальное свойство;
- `SideEffect` не отправляет событие сам, а синхронизирует внешний объект.

<MouseTrap>
Не отправляйте analytics event через `SideEffect`, если событие должно случиться один раз. `SideEffect` выполняется после каждой успешной recomposition, значит event может уйти много раз.
</MouseTrap>

## Публикация state в imperative bridge

Представим, что старый View-based слой читает объект bridge.

```kotlin
import androidx.compose.runtime.Composable
import androidx.compose.runtime.SideEffect

class ToolbarBridge {
    var title: String = ""
    var actionsEnabled: Boolean = false
}

@Composable
fun ToolbarBridgePublisher(
    screenTitle: String,
    actionsEnabled: Boolean,
    toolbarBridge: ToolbarBridge
) {
    SideEffect {
        toolbarBridge.title = screenTitle
        toolbarBridge.actionsEnabled = actionsEnabled
    }
}
```

Этот подход может пригодиться во время миграции View UI на Compose. Когда миграция завершена, лучше заменить bridge на чистый Compose state.

## SideEffect и recomposition frequency

`SideEffect` выполняется после каждой успешной recomposition текущего composable. Если composable часто recomposes, запись тоже будет частой.

```kotlin
import androidx.compose.foundation.lazy.LazyListState
import androidx.compose.runtime.Composable
import androidx.compose.runtime.SideEffect

class ScrollDebugStore {
    var firstVisibleIndex: Int = 0
}

@Composable
fun PublishScrollIndex(
    listState: LazyListState,
    store: ScrollDebugStore
) {
    SideEffect {
        store.firstVisibleIndex = listState.firstVisibleItemIndex
    }
}
```

Такой код может выполняться очень часто при скролле. Если нужно реагировать на scroll changes для аналитики или debounce, лучше использовать `snapshotFlow` внутри `LaunchedEffect`.

## SideEffect против LaunchedEffect

`SideEffect`:

- синхронный;
- без suspend;
- после каждой успешной recomposition;
- без ключей;
- без cleanup;
- для публикации state наружу.

`LaunchedEffect`:

- запускает coroutine;
- может вызывать suspend;
- имеет ключи;
- отменяется и перезапускается;
- подходит для snackbar, navigation effect, delay, Flow collection.

```kotlin
import androidx.compose.runtime.Composable
import androidx.compose.runtime.LaunchedEffect
import androidx.compose.runtime.SideEffect

interface EventLogger {
    fun setCurrentUserType(userType: String)
    suspend fun trackScreenOpened(screenName: String)
}

@Composable
fun AnalyticsEffects(
    screenName: String,
    userType: String,
    logger: EventLogger
) {
    SideEffect {
        logger.setCurrentUserType(userType)
    }

    LaunchedEffect(screenName, logger) {
        logger.trackScreenOpened(screenName)
    }
}
```

Здесь:

- user property публикуется после каждой успешной recomposition;
- screen opened event отправляется как coroutine effect с ключом `screenName`.

## SideEffect против DisposableEffect

`DisposableEffect` нужен, когда есть cleanup.

```kotlin
import androidx.compose.runtime.Composable
import androidx.compose.runtime.DisposableEffect
import androidx.compose.runtime.SideEffect

interface ExternalModeController {
    fun setMode(mode: String)
    fun addListener(listener: () -> Unit)
    fun removeListener(listener: () -> Unit)
}

@Composable
fun ExternalModePublisher(
    mode: String,
    controller: ExternalModeController
) {
    SideEffect {
        controller.setMode(mode)
    }
}

@Composable
fun ExternalListenerRegistration(
    controller: ExternalModeController,
    onChanged: () -> Unit
) {
    DisposableEffect(controller, onChanged) {
        val listener = {
            onChanged()
        }

        controller.addListener(listener)

        onDispose {
            controller.removeListener(listener)
        }
    }
}
```

Если действие выглядит как register/unregister, нужен `DisposableEffect`, а не `SideEffect`.

## Production-пример: current route metadata

В приложении может быть non-Compose telemetry object, который хранит текущие metadata экрана для crash reports.

```kotlin
import androidx.compose.runtime.Composable
import androidx.compose.runtime.Immutable
import androidx.compose.runtime.SideEffect

@Immutable
data class ScreenMetadata(
    val routeName: String,
    val userSegment: String,
    val experimentGroup: String
)

class CrashMetadataStore {
    private val values = mutableMapOf<String, String>()

    fun set(
        key: String,
        value: String
    ) {
        values[key] = value
    }

    fun snapshot(): Map<String, String> {
        return values.toMap()
    }
}

@Composable
fun CrashMetadataPublisher(
    metadata: ScreenMetadata,
    store: CrashMetadataStore
) {
    SideEffect {
        store.set(
            key = "route_name",
            value = metadata.routeName
        )
        store.set(
            key = "user_segment",
            value = metadata.userSegment
        )
        store.set(
            key = "experiment_group",
            value = metadata.experimentGroup
        )
    }
}
```

Почему `SideEffect` подходит:

- операция быстрая;
- нет suspend;
- нет cleanup;
- нужно синхронизировать внешний holder с текущим Compose state;
- запись должна происходить только после успешного применения UI state.

## Production-пример: bridge для legacy toolbar

```kotlin
import androidx.compose.foundation.layout.Column
import androidx.compose.material3.Text
import androidx.compose.runtime.Composable
import androidx.compose.runtime.SideEffect

class LegacyToolbarController {
    var title: String = ""
        private set

    var subtitle: String = ""
        private set

    fun update(
        title: String,
        subtitle: String
    ) {
        this.title = title
        this.subtitle = subtitle
    }
}

@Composable
fun LegacyToolbarAwareScreen(
    title: String,
    subtitle: String,
    toolbarController: LegacyToolbarController
) {
    SideEffect {
        toolbarController.update(
            title = title,
            subtitle = subtitle
        )
    }

    Column {
        Text(text = title)
        Text(text = subtitle)
    }
}
```

В новом Compose-only экране лучше просто использовать Compose toolbar. Но во время миграции такой bridge может быть полезен.

## Нельзя вызывать suspend

`SideEffect` не является suspend-контекстом.

Плохо:

```kotlin
import androidx.compose.runtime.Composable
import androidx.compose.runtime.SideEffect

interface RemoteLogger {
    suspend fun sendState(value: String)
}

@Composable
fun RemoteLoggerWrong(
    value: String,
    remoteLogger: RemoteLogger
) {
    SideEffect {
        // Здесь нельзя вызвать remoteLogger.sendState(value), потому что функция suspend.
    }
}
```

Правильно выбрать другой API:

```kotlin
import androidx.compose.runtime.Composable
import androidx.compose.runtime.LaunchedEffect

@Composable
fun RemoteLoggerEffect(
    value: String,
    remoteLogger: RemoteLogger
) {
    LaunchedEffect(value, remoteLogger) {
        remoteLogger.sendState(value)
    }
}
```

## Когда SideEffect не нужен

Не нужен `SideEffect`, если вы просто хотите отобразить state.

```kotlin
import androidx.compose.material3.Text
import androidx.compose.runtime.Composable

@Composable
fun UserTypeLabel(
    userType: String
) {
    Text(text = "Тип пользователя: $userType")
}
```

Здесь нет внешнего объекта, поэтому нет side effect.

Не нужен `SideEffect` для:

- обновления локальной переменной;
- вычисления derived value;
- показа snackbar;
- navigation;
- loading данных;
- регистрации listener;
- записи в database.

## Типичные ошибки

### Отправлять событие аналитики

Если event должен уйти один раз, используйте `LaunchedEffect` с правильным ключом или effect stream из ViewModel.

### Делать дорогую работу

`SideEffect` выполняется на main thread после recomposition. Он должен быть быстрым.

### Использовать для suspend API

Suspend APIs требуют coroutine. Используйте `LaunchedEffect` или `rememberCoroutineScope`.

### Ожидать cleanup

У `SideEffect` нет `onDispose`. Если нужен cleanup, используйте `DisposableEffect`.

### Думать, что он выполняется один раз

Он выполняется после каждой успешной recomposition, где присутствует этот call site.

## Итоги

`SideEffect` публикует Compose state наружу:

- выполняется после каждой успешной recomposition;
- не имеет ключей;
- не является suspend;
- не имеет cleanup;
- подходит для быстрой синхронной записи во внешний object;
- хорош для analytics user properties, crash metadata и legacy bridge;
- не подходит для одноразовых events;
- не подходит для snackbar, navigation, Flow collection и listener subscriptions;
- если нужна coroutine, выбирайте `LaunchedEffect`;
- если нужен cleanup, выбирайте `DisposableEffect`.
