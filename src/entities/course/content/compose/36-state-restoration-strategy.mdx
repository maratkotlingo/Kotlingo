## Почему стратегия восстановления важна

Android-приложение может потерять UI state по нескольким причинам:

- configuration change, например поворот экрана;
- смена размера окна на foldable, tablet или desktop-режиме;
- system-initiated process death, когда система освобождает память;
- удаление composable из composition;
- navigation pop;
- пользователь вручную закрыл Activity из recent apps.

Compose дает несколько инструментов, но у каждого свой lifetime:

- `remember`;
- `rememberSaveable`;
- `ViewModel`;
- `SavedStateHandle`;
- persistent storage: база данных, DataStore, файлы, remote source.

Надежная стратегия начинается с вопроса: "Что именно мы восстанавливаем и почему пользователь ожидает это увидеть снова?"

<CatTip>
Сохранять нужно не максимум данных, а минимум, достаточный для восстановления UX. Часто это id, enum, query, scroll position или короткий черновик, а не весь объект экрана.
</CatTip>

## Карта lifetimes

| Инструмент | Переживает recomposition | Переживает configuration change | Может пережить system process death | Подходит для |
| --- | --- | --- | --- | --- |
| `remember` | Да | Нет | Нет | локальные объекты и состояние composition |
| `rememberSaveable` | Да | Да | Да, если state сохранен системой | маленький UI element state |
| `ViewModel` | Да | Да | Нет | screen state и business logic |
| `SavedStateHandle` | Да, через ViewModel | Да | Да, если state сохранен системой | минимальный state ViewModel |
| Persistent storage | Да | Да | Да | данные приложения, которые должны жить долго |

Эта таблица не означает, что нужно использовать все сразу. Она помогает выбрать правильное место.

## remember

`remember` хранит значение только пока соответствующее место остается в composition.

```kotlin
import androidx.compose.runtime.Composable
import androidx.compose.runtime.getValue
import androidx.compose.runtime.mutableStateOf
import androidx.compose.runtime.remember
import androidx.compose.runtime.setValue

@Composable
fun LocalExpansion() {
    var expanded by remember {
        mutableStateOf(false)
    }
}
```

Используйте для:

- локального раскрытия;
- временного объекта;
- formatter;
- interaction state;
- объектов, которые не должны переживать пересоздание Activity.

Не используйте для:

- текста формы, который пользователь будет раздраженно терять при повороте;
- выбранного таба важного экрана;
- scroll position, если восстановление важно;
- screen state из repository.

## rememberSaveable

`rememberSaveable` сохраняет маленький UI element state через saved instance state.

```kotlin
import androidx.compose.material3.OutlinedTextField
import androidx.compose.material3.Text
import androidx.compose.runtime.Composable
import androidx.compose.runtime.getValue
import androidx.compose.runtime.mutableStateOf
import androidx.compose.runtime.saveable.rememberSaveable
import androidx.compose.runtime.setValue

@Composable
fun SaveableSearchField() {
    var query by rememberSaveable {
        mutableStateOf("")
    }

    OutlinedTextField(
        value = query,
        onValueChange = { nextQuery ->
            query = nextQuery
        },
        label = {
            Text(text = "Поиск")
        }
    )
}
```

Используйте для:

- текста ввода;
- выбранного таба;
- id выбранного элемента;
- scroll position;
- состояния раскрытия важной секции;
- небольших primitive/String/Parcelable значений.

Не используйте для:

- больших списков;
- bitmap;
- полной модели экрана;
- данных из базы;
- объектов repository;
- состояния, которое должно жить после явного закрытия экрана пользователем.

<MouseTrap>
`rememberSaveable` не превращает Bundle в базу данных. Bundle ограничен по размеру, а большие объекты могут привести к runtime-падению.
</MouseTrap>

## ViewModel

ViewModel переживает configuration changes и держит screen-level state.

```kotlin
import androidx.lifecycle.ViewModel
import kotlinx.coroutines.flow.MutableStateFlow
import kotlinx.coroutines.flow.StateFlow
import kotlinx.coroutines.flow.asStateFlow

data class ReaderUiState(
    val articleId: String,
    val title: String,
    val loading: Boolean
)

class ReaderViewModel : ViewModel() {

    private val _uiState = MutableStateFlow(
        ReaderUiState(
            articleId = "",
            title = "",
            loading = true
        )
    )

    val uiState: StateFlow<ReaderUiState> = _uiState.asStateFlow()
}
```

ViewModel подходит для:

- загрузки данных;
- screen UI state;
- бизнес-логики;
- объединения repository flows;
- обработки UI events;
- сохранения state при повороте экрана.

Но ViewModel сам по себе не переживает system-initiated process death. Если процесс убит, ViewModel будет создан заново.

## SavedStateHandle

`SavedStateHandle` хранит маленький state, связанный с ViewModel, через saved state mechanism.

Он подходит для:

- navigation arguments;
- выбранного filter id;
- короткого query;
- enum выбранного режима;
- id открытой сущности;
- небольшого черновика, если он нужен ViewModel.

Он не подходит для:

- полной UI-модели экрана;
- списков объектов;
- больших JSON;
- кэша изображений;
- данных, которые должны лежать в database/DataStore.

## SavedStateHandle: getStateFlow

`getStateFlow` удобно использовать, если ViewModel хочет получать saveable value как `StateFlow`.

```kotlin
import androidx.lifecycle.SavedStateHandle
import androidx.lifecycle.ViewModel
import kotlinx.coroutines.flow.StateFlow

enum class ChannelFilter {
    All,
    Unread,
    Mentions
}

class ChannelListViewModel(
    private val savedStateHandle: SavedStateHandle
) : ViewModel() {

    val filterName: StateFlow<String> = savedStateHandle.getStateFlow(
        key = FILTER_KEY,
        initialValue = ChannelFilter.All.name
    )

    fun setFilter(filter: ChannelFilter) {
        savedStateHandle[FILTER_KEY] = filter.name
    }

    companion object {
        private const val FILTER_KEY = "channel_filter"
    }
}
```

UI может преобразовать имя обратно в enum или ViewModel может сделать отдельный mapped `StateFlow<ChannelFilter>`.

```kotlin
import kotlinx.coroutines.flow.Flow
import kotlinx.coroutines.flow.map

fun ChannelListViewModel.filter(): Flow<ChannelFilter> {
    return filterName.map { name ->
        ChannelFilter.valueOf(name)
    }
}
```

Почему сохраняется `String`, а не весь сложный объект:

- `String` легко положить в Bundle;
- enum name достаточно для восстановления;
- ViewModel может снова построить screen state из repository.

## SavedStateHandle: saveable

Если state использует Compose `Saver`, можно применять `SavedStateHandle.saveable`. Этот API помечен как экспериментальный, поэтому его нужно подключать осознанно.

```kotlin
import androidx.compose.runtime.getValue
import androidx.compose.runtime.mutableStateOf
import androidx.compose.runtime.setValue
import androidx.compose.ui.text.input.TextFieldValue
import androidx.lifecycle.SavedStateHandle
import androidx.lifecycle.ViewModel
import androidx.lifecycle.viewmodel.compose.SavedStateHandleSaveableApi
import androidx.lifecycle.viewmodel.compose.saveable

@OptIn(SavedStateHandleSaveableApi::class)
class MessageDraftViewModel(
    savedStateHandle: SavedStateHandle
) : ViewModel() {

    var message by savedStateHandle.saveable(
        stateSaver = TextFieldValue.Saver
    ) {
        mutableStateOf(TextFieldValue(""))
    }
        private set

    fun updateMessage(nextMessage: TextFieldValue) {
        message = nextMessage
    }
}
```

Такой подход полезен, если ViewModel должна хранить `TextFieldValue` вместе с selection и composition.

## SavedStateHandle как key-value map

`SavedStateHandle` также похож на map.

Связанные операции:

- `get<T>(key)`: прочитать значение;
- `set(key, value)`: записать значение;
- `operator get`: синтаксис `savedStateHandle[key]`;
- `operator set`: синтаксис `savedStateHandle[key] = value`;
- `contains(key)`: проверить наличие ключа;
- `remove<T>(key)`: удалить значение;
- `keys()`: получить набор ключей;
- `getStateFlow(key, initialValue)`: читать значение как StateFlow;
- `saveable`: сохранить Compose `MutableState` через Saver.

Пример:

```kotlin
import androidx.lifecycle.SavedStateHandle

class DraftStateStore(
    private val savedStateHandle: SavedStateHandle
) {

    fun saveDraftId(draftId: String) {
        savedStateHandle[DRAFT_ID_KEY] = draftId
    }

    fun readDraftId(): String? {
        return savedStateHandle[DRAFT_ID_KEY]
    }

    fun clearDraftId() {
        savedStateHandle.remove<String>(DRAFT_ID_KEY)
    }

    companion object {
        private const val DRAFT_ID_KEY = "draft_id"
    }
}
```

## Navigation arguments

Navigation arguments должны описывать identity или минимальный параметр маршрута, а не передавать весь объект.

Плохо:

```kotlin
import kotlinx.serialization.Serializable

@Serializable
data class ProductDetailsRouteWrong(
    val title: String,
    val description: String,
    val imageUrls: List<String>
)
```

Почему плохо:

- route становится большой;
- данные могут устареть;
- deep link сложнее;
- Bundle получает лишнюю нагрузку;
- источник истины раздваивается.

Хорошо:

```kotlin
import kotlinx.serialization.Serializable

@Serializable
data class ProductDetailsRoute(
    val productId: String
)
```

По `productId` ViewModel загружает актуальные данные.

## Type-safe Navigation и SavedStateHandle.toRoute

В Navigation Compose с type-safe routes аргументы можно получить в ViewModel через `SavedStateHandle.toRoute<T>()`.

```kotlin
import androidx.lifecycle.SavedStateHandle
import androidx.lifecycle.ViewModel
import androidx.lifecycle.viewModelScope
import androidx.navigation.toRoute
import kotlinx.coroutines.flow.SharingStarted
import kotlinx.coroutines.flow.StateFlow
import kotlinx.coroutines.flow.map
import kotlinx.coroutines.flow.stateIn
import kotlinx.serialization.Serializable

@Serializable
data class ProfileRoute(
    val userId: String
)

class ProfileViewModel(
    savedStateHandle: SavedStateHandle,
    repository: ProfileRepository
) : ViewModel() {

    private val route = savedStateHandle.toRoute<ProfileRoute>()

    val uiState: StateFlow<ProfileUiState> = repository
        .observeProfile(userId = route.userId)
        .map { profile ->
            ProfileUiState.Content(
                name = profile.name,
                subtitle = profile.subtitle
            )
        }
        .stateIn(
            scope = viewModelScope,
            started = SharingStarted.WhileSubscribed(5_000),
            initialValue = ProfileUiState.Loading
        )
}

interface ProfileRepository {
    fun observeProfile(userId: String): kotlinx.coroutines.flow.Flow<Profile>
}

data class Profile(
    val name: String,
    val subtitle: String
)

sealed interface ProfileUiState {
    data object Loading : ProfileUiState
    data class Content(
        val name: String,
        val subtitle: String
    ) : ProfileUiState
}
```

Так ViewModel получает аргумент из своего saved state, а composable не должен передавать id через `LaunchedEffect`.

## Process death

System-initiated process death означает, что Android может убить процесс приложения в фоне. Когда пользователь вернется, Activity и ViewModel будут созданы заново.

Что может восстановиться:

- saved instance state;
- `rememberSaveable`;
- `SavedStateHandle`;
- navigation back stack state;
- данные из persistent storage.

Что не восстановится само:

- обычный `remember`;
- поля старого ViewModel;
- in-memory repository cache без persistent source;
- coroutine jobs;
- временные one-off effects.

Важная nuance: state, записанный в `SavedStateHandle`, сохраняется механизмом saved state в определенные моменты жизненного цикла Activity. Не проектируйте критически важные пользовательские данные так, будто каждая запись мгновенно записана на диск.

<MouseTrap>
Если потеря данных недопустима, это не transient UI state. Сохраняйте такие данные в надежное persistent storage: database, DataStore или файл с явной стратегией записи.
</MouseTrap>

## Persisted app data vs transient UI state

Transient UI state:

- query в поиске;
- selected tab;
- scroll position;
- id раскрытой секции;
- короткий черновик сообщения;
- фильтр списка.

Persisted app data:

- пользовательский профиль;
- избранные товары;
- корзина;
- опубликованный комментарий;
- настройки приложения;
- offline cache;
- документы пользователя.

Transient state можно хранить в `rememberSaveable` или `SavedStateHandle`. Persisted data должна жить в durable source.

```kotlin
interface SettingsRepository {
    suspend fun setDarkThemeEnabled(enabled: Boolean)
    fun observeDarkThemeEnabled(): kotlinx.coroutines.flow.Flow<Boolean>
}
```

Настройка темы — не просто transient UI state. Пользователь ожидает, что она сохранится после полного закрытия приложения и перезагрузки устройства.

## Strategy by scenario

### Текст в поле поиска локального списка

Используйте `rememberSaveable`, если поиск локальный для composable.

### Query влияет на repository и бизнес-логику

Храните query в ViewModel. Если нужно пережить process death, сохраните query в `SavedStateHandle`.

### Открыт экран details

Передавайте `itemId` через navigation argument. Данные загружайте из repository.

### Пользователь редактирует длинный документ

Не полагайтесь только на Bundle. Нужен autosave в persistent storage.

### Выбранная вкладка на экране

Если это UI element state, используйте `rememberSaveable`. Если вкладка влияет на запросы данных, храните в ViewModel и при необходимости в `SavedStateHandle`.

### Scroll position списка

Используйте state list API вроде `rememberLazyListState`, который поддерживает сохранение, или явно сохраняйте минимальные scroll-параметры через Saver.

## Production-пример: список заказов с фильтром

ViewModel хранит фильтр в `SavedStateHandle`, а заказы берет из repository.

```kotlin
import androidx.lifecycle.SavedStateHandle
import androidx.lifecycle.ViewModel
import androidx.lifecycle.viewModelScope
import kotlinx.coroutines.flow.SharingStarted
import kotlinx.coroutines.flow.StateFlow
import kotlinx.coroutines.flow.combine
import kotlinx.coroutines.flow.map
import kotlinx.coroutines.flow.stateIn

enum class OrderFilter {
    Active,
    Completed,
    All
}

data class OrderRow(
    val id: String,
    val title: String,
    val completed: Boolean
)

data class OrdersUiState(
    val filter: OrderFilter,
    val orders: List<OrderRow>,
    val loading: Boolean
)

class OrdersViewModel(
    private val savedStateHandle: SavedStateHandle,
    repository: OrdersRepository
) : ViewModel() {

    private val filterName = savedStateHandle.getStateFlow(
        key = FILTER_KEY,
        initialValue = OrderFilter.Active.name
    )

    val uiState: StateFlow<OrdersUiState> = combine(
        filterName.map { name ->
            OrderFilter.valueOf(name)
        },
        repository.observeOrders()
    ) { filter, orders ->
        val visibleOrders = when (filter) {
            OrderFilter.Active -> {
                orders.filter { order ->
                    !order.completed
                }
            }

            OrderFilter.Completed -> {
                orders.filter { order ->
                    order.completed
                }
            }

            OrderFilter.All -> {
                orders
            }
        }

        OrdersUiState(
            filter = filter,
            orders = visibleOrders,
            loading = false
        )
    }.stateIn(
        scope = viewModelScope,
        started = SharingStarted.WhileSubscribed(5_000),
        initialValue = OrdersUiState(
            filter = OrderFilter.Active,
            orders = emptyList(),
            loading = true
        )
    )

    fun setFilter(filter: OrderFilter) {
        savedStateHandle[FILTER_KEY] = filter.name
    }

    companion object {
        private const val FILTER_KEY = "orders_filter"
    }
}

interface OrdersRepository {
    fun observeOrders(): kotlinx.coroutines.flow.Flow<List<OrderRow>>
}
```

Что здесь где хранится:

- `OrderFilter.name` в `SavedStateHandle`;
- список заказов в repository;
- производный `OrdersUiState` в ViewModel;
- UI получает готовую модель и callbacks.

## UI для production-примера

```kotlin
import androidx.compose.foundation.layout.Column
import androidx.compose.foundation.layout.Row
import androidx.compose.foundation.lazy.LazyColumn
import androidx.compose.foundation.lazy.items
import androidx.compose.material3.CircularProgressIndicator
import androidx.compose.material3.FilterChip
import androidx.compose.material3.ListItem
import androidx.compose.material3.Text
import androidx.compose.runtime.Composable

@Composable
fun OrdersScreen(
    state: OrdersUiState,
    onFilterChange: (OrderFilter) -> Unit
) {
    Column {
        Row {
            OrderFilter.entries.forEach { filter ->
                FilterChip(
                    selected = state.filter == filter,
                    onClick = {
                        onFilterChange(filter)
                    },
                    label = {
                        Text(text = filter.name)
                    }
                )
            }
        }

        if (state.loading) {
            CircularProgressIndicator()
        } else {
            LazyColumn {
                items(
                    items = state.orders,
                    key = { order ->
                        order.id
                    }
                ) { order ->
                    ListItem(
                        headlineContent = {
                            Text(text = order.title)
                        },
                        supportingContent = {
                            Text(
                                text = if (order.completed) {
                                    "Завершен"
                                } else {
                                    "Активен"
                                }
                            )
                        }
                    )
                }
            }
        }
    }
}
```

Для composable state используйте `StateRestorationTester`.

    OutlinedTextField(

        modifier = Modifier.testTag("draft"),

        tester.setContent {
            RestorableDraftField()
        }

        composeRule
            .onNodeWithTag("draft")
            .performTextInput("Текст черновика")

        tester.emulateSavedInstanceStateRestore()

        composeRule
            .onNodeWithTag("draft")
            .assertTextEquals("Текст черновика")
    }
}

## Типичные ошибки

### Сохранять весь screen state в SavedStateHandle

Screen state часто получается из business rules и repository. В `SavedStateHandle` кладите минимальные входные данные.

### Передавать большие объекты через navigation

Передавайте id. Загружайте данные из source of truth.

### Полагаться на ViewModel при process death

ViewModel переживает configuration change, но не является persistent storage.

### Путать user-initiated dismissal и system process death

Если пользователь явно закрыл Activity из recent apps, восстановление transient state обычно не гарантируется и не всегда ожидается.

### Сохранять критические данные только в Bundle

Если потеря недопустима, нужна надежная запись в database/DataStore/file.

## Итоги

Стратегия восстановления строится по lifetime:

- `remember` — память composition;
- `rememberSaveable` — маленький UI element state через Bundle;
- `ViewModel` — screen state и business logic через configuration changes;
- `SavedStateHandle` — минимальный ViewModel state для process recreation;
- navigation arguments — identity маршрута, чаще всего id;
- persistent storage — данные, которые пользователь ожидает сохранить надолго;
- Bundle не предназначен для больших объектов;
- process death требует восстановления state из минимальных ключей и durable sources;
- лучший saved state часто выглядит скучно: строка, id, enum name, индекс и scroll offset.
