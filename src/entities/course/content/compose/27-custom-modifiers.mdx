## Что такое custom modifier

`Modifier` — это неизменяемая цепочка операций, которые применяются к composable: размер, отступы, клики, фон, рисование, accessibility, layout-поведение, pointer input, тестовые теги и многое другое.

Custom modifier — это собственное звено или набор звеньев в этой цепочке.

Он нужен, когда:

- один и тот же набор модификаторов повторяется в разных местах;
- нужно скрыть детали визуального стиля за понятным именем;
- нужен modifier, который меняет measurement или placement;
- нужно рисовать поверх или под контентом;
- нужно добавить semantics для accessibility и UI-тестов;
- нужно обработать жесты;
- нужен производительный reusable modifier для design system или библиотеки.

Самый простой custom modifier — extension-функция над `Modifier`.

```kotlin
import androidx.compose.foundation.background
import androidx.compose.foundation.layout.padding
import androidx.compose.foundation.shape.RoundedCornerShape
import androidx.compose.ui.Modifier
import androidx.compose.ui.graphics.Color
import androidx.compose.ui.unit.dp

fun Modifier.infoPill(): Modifier {
    return this
        .background(
            color = Color(0xFFEAF4FF),
            shape = RoundedCornerShape(999.dp)
        )
        .padding(horizontal = 12.dp, vertical = 6.dp)
}
```

Использование:

```kotlin
import androidx.compose.material3.Text
import androidx.compose.runtime.Composable
import androidx.compose.ui.Modifier

@Composable
fun StatusLabel() {
    Text(
        text = "Синхронизировано",
        modifier = Modifier.infoPill()
    )
}
```

Такой modifier не создает новую низкоуровневую сущность. Он просто собирает уже существующие модификаторы в осмысленную функцию.

<CatTip>
Начинайте с extension-функции над `Modifier`. Переходите к `Modifier.Node` только тогда, когда действительно нужен низкоуровневый контроль, высокая переиспользуемость или оптимизация.
</CatTip>

## Правильная форма API

Хороший custom modifier обычно выглядит так:

```kotlin
fun Modifier.someBehavior(
    requiredParameter: String,
    optionalParameter: Int = 0
): Modifier {
    return this.then(
        Modifier
    )
}
```

Правила:

- функция должна быть extension-функцией над `Modifier`;
- она должна возвращать `Modifier`;
- первым действием нужно использовать текущий receiver `this`;
- параметры должны быть стабильными и понятными;
- необязательные параметры получают безопасные значения по умолчанию;
- имя должно описывать поведение, а не внутреннюю реализацию.

Плохо:

```kotlin
import androidx.compose.ui.Modifier

fun cardModifier(): Modifier {
    return Modifier
}
```

Почему плохо:

- функция не расширяет существующую цепочку;
- ее нельзя удобно вставить в середину цепочки;
- она может случайно стереть modifiers, переданные вызывающим кодом;
- имя описывает сущность слишком расплывчато.

Хорошо:

```kotlin
import androidx.compose.foundation.border
import androidx.compose.foundation.shape.RoundedCornerShape
import androidx.compose.ui.Modifier
import androidx.compose.ui.graphics.Color
import androidx.compose.ui.unit.Dp
import androidx.compose.ui.unit.dp

fun Modifier.focusRing(
    visible: Boolean,
    color: Color = Color(0xFF1A73E8),
    width: Dp = 2.dp
): Modifier {
    return if (visible) {
        this.border(
            width = width,
            color = color,
            shape = RoundedCornerShape(8.dp)
        )
    } else {
        this
    }
}
```

## Порядок в цепочке

Порядок modifier-звеньев влияет на результат. Это одна из самых важных тем.

```kotlin
import androidx.compose.foundation.background
import androidx.compose.foundation.layout.padding
import androidx.compose.material3.Text
import androidx.compose.runtime.Composable
import androidx.compose.ui.Modifier
import androidx.compose.ui.graphics.Color
import androidx.compose.ui.unit.dp

@Composable
fun ModifierOrderDemo() {
    Text(
        text = "Фон включает внутренний отступ",
        modifier = Modifier
            .background(Color(0xFFDFF7E2))
            .padding(16.dp)
    )

    Text(
        text = "Фон не включает внешний отступ",
        modifier = Modifier
            .padding(16.dp)
            .background(Color(0xFFFFF3D9))
    )
}
```

В первом случае фон применяется до padding, поэтому окрашивается область, которая затем получает внутренний отступ. Во втором случае padding создает внешнее пространство, а фон рисуется только внутри оставшейся области.

<MouseTrap>
Не прячьте в custom modifier такой порядок, который вызывающий код не ожидает. Если modifier добавляет и `padding`, и `clickable`, и `background`, его имя должно ясно говорить, что это уже готовое поведение компонента, а не просто цвет или отступ.
</MouseTrap>

## Modifier.then

`then` добавляет одну цепочку modifiers к другой.

```kotlin
import androidx.compose.ui.Modifier

fun Modifier.optional(
    enabled: Boolean,
    modifier: Modifier
): Modifier {
    return if (enabled) {
        this.then(modifier)
    } else {
        this
    }
}
```

Такой прием полезен, когда modifier собирается условно.

```kotlin
import androidx.compose.foundation.background
import androidx.compose.foundation.layout.padding
import androidx.compose.material3.Text
import androidx.compose.runtime.Composable
import androidx.compose.ui.Modifier
import androidx.compose.ui.graphics.Color
import androidx.compose.ui.unit.dp

@Composable
fun OptionalModifierDemo(
    highlighted: Boolean
) {
    Text(
        text = "Строка статуса",
        modifier = Modifier
            .padding(8.dp)
            .optional(
                enabled = highlighted,
                modifier = Modifier.background(Color(0xFFFFF1A8))
            )
    )
}
```

## Modifier.layout

`Modifier.layout` позволяет изменить измерение и размещение конкретного composable без создания нового layout-контейнера.

Он подходит, когда:

- нужно добавить нестандартное смещение;
- нужно измерить child с измененными constraints;
- нужно изменить размер, который элемент сообщает родителю;
- поведение относится к одному элементу, а не к группе детей.

```kotlin
import androidx.compose.ui.Modifier
import androidx.compose.ui.layout.layout
import androidx.compose.ui.unit.Dp

fun Modifier.reserveBadgeSpace(
    badgeSize: Dp
): Modifier {
    return this.layout { measurable, constraints ->
        val badgeSizePx = badgeSize.roundToPx()

        val placeable = measurable.measure(
            constraints.offset(
                horizontal = -badgeSizePx,
                vertical = -badgeSizePx
            )
        )

        val width = constraints.constrainWidth(placeable.width + badgeSizePx)
        val height = constraints.constrainHeight(placeable.height + badgeSizePx)

        layout(width = width, height = height) {
            placeable.placeRelative(
                x = 0,
                y = badgeSizePx / 2
            )
        }
    }
}
```

Разбор:

- `constraints.offset` уменьшает доступное место для child;
- child измеряется один раз;
- layout сообщает родителю увеличенный размер;
- child размещается с вертикальным сдвигом;
- место сверху остается под бейдж или декоративный элемент.

## Рисование: drawBehind

`drawBehind` рисует под содержимым элемента.

```kotlin
import androidx.compose.ui.Modifier
import androidx.compose.ui.draw.drawBehind
import androidx.compose.ui.geometry.Offset
import androidx.compose.ui.graphics.Color
import androidx.compose.ui.unit.Dp

fun Modifier.bottomAccentLine(
    color: Color,
    thickness: Dp
): Modifier {
    return this.drawBehind {
        val thicknessPx = thickness.toPx()
        val y = size.height - thicknessPx / 2f

        drawLine(
            color = color,
            start = Offset(x = 0f, y = y),
            end = Offset(x = size.width, y = y),
            strokeWidth = thicknessPx
        )
    }
}
```

`drawBehind` хорош для:

- линий;
- простых фонов;
- декоративных маркеров;
- легких shape-эффектов;
- индикаторов состояния.

## Рисование: drawWithContent

`drawWithContent` дает контроль над тем, когда рисовать контент: до собственного рисунка, после него или между несколькими слоями.

```kotlin
import androidx.compose.ui.Modifier
import androidx.compose.ui.draw.drawWithContent
import androidx.compose.ui.graphics.Brush
import androidx.compose.ui.graphics.Color

fun Modifier.readabilityFade(
    startColor: Color,
    endColor: Color
): Modifier {
    return this.drawWithContent {
        drawContent()

        drawRect(
            brush = Brush.verticalGradient(
                colors = listOf(
                    startColor.copy(alpha = 0f),
                    endColor.copy(alpha = 0.72f)
                )
            )
        )
    }
}
```

Этот modifier рисует контент, а затем полупрозрачный градиент поверх. Такой прием полезен для карточек с изображением и текстом поверх нижней части.

## Рисование: drawWithCache

`drawWithCache` позволяет вычислить дорогие объекты один раз и переиспользовать их между кадрами, пока не изменятся зависимости: размер, density, layout direction или состояния, прочитанные внутри cache-блока.

```kotlin
import androidx.compose.ui.Modifier
import androidx.compose.ui.draw.drawWithCache
import androidx.compose.ui.geometry.CornerRadius
import androidx.compose.ui.graphics.Brush
import androidx.compose.ui.graphics.Color

fun Modifier.cachedWarningBackground(
    topColor: Color,
    bottomColor: Color
): Modifier {
    return this.drawWithCache {
        val brush = Brush.verticalGradient(
            colors = listOf(topColor, bottomColor),
            startY = 0f,
            endY = size.height
        )

        val cornerRadius = CornerRadius(
            x = 14.dp.toPx(),
            y = 14.dp.toPx()
        )

        onDrawBehind {
            drawRoundRect(
                brush = brush,
                cornerRadius = cornerRadius
            )
        }
    }
}
```

Когда использовать `drawWithCache`:

- создается `Brush`, `Path`, `Shader`, `TextMeasurer` или другой объект, зависящий от размера;
- вычисления повторяются часто;
- размер элемента влияет на графику;
- нужен `onDrawBehind` или `onDrawWithContent`.

Когда не использовать:

- рисование совсем простое;
- вычисления дешевые;
- код становится труднее читать без выигрыша.

## Semantics modifier

`semantics` добавляет смысловую информацию для accessibility, UI-тестов и инструментов.

```kotlin
import androidx.compose.ui.Modifier
import androidx.compose.ui.semantics.Role
import androidx.compose.ui.semantics.contentDescription
import androidx.compose.ui.semantics.role
import androidx.compose.ui.semantics.semantics
import androidx.compose.ui.semantics.stateDescription

fun Modifier.accessibleFavoriteToggle(
    selected: Boolean,
    itemTitle: String
): Modifier {
    return this.semantics {
        role = Role.Checkbox
        contentDescription = "Избранное для элемента $itemTitle"
        stateDescription = if (selected) {
            "Добавлено в избранное"
        } else {
            "Не добавлено в избранное"
        }
    }
}
```

Semantics особенно важны, если визуальное состояние не выражено текстом. Например, иконка сердца без semantics для TalkBack может быть непонятной.

## Pointer input

`pointerInput` добавляет низкоуровневую обработку pointer-событий. Для обычных кликов, скролла и drag чаще лучше использовать готовые модификаторы: `clickable`, `combinedClickable`, `draggable`, `scrollable`, `transformable`.

```kotlin
import androidx.compose.foundation.gestures.detectTapGestures
import androidx.compose.ui.Modifier
import androidx.compose.ui.input.pointer.pointerInput

fun Modifier.onDoubleTapOnly(
    onDoubleTap: () -> Unit
): Modifier {
    return this.pointerInput(onDoubleTap) {
        detectTapGestures(
            onDoubleTap = {
                onDoubleTap()
            }
        )
    }
}
```

Ключи `pointerInput` работают как ключи `remember`: если ключ изменился, coroutine-обработчик pointer input перезапускается.

<MouseTrap>
Не передавайте в `pointerInput(Unit)` callback, который может измениться. Старый обработчик может продолжить видеть устаревшую лямбду. Либо используйте сам callback как ключ, либо оберните его через `rememberUpdatedState` внутри composable-уровня.
</MouseTrap>

## composed

`Modifier.composed` позволяет создать modifier, внутри которого можно использовать composable APIs, например `remember`. Сейчас для низкоуровневых reusable modifiers чаще предпочитают `Modifier.Node`, но `composed` все еще встречается в приложениях и библиотеках.

```kotlin
import androidx.compose.foundation.background
import androidx.compose.runtime.getValue
import androidx.compose.runtime.mutableStateOf
import androidx.compose.runtime.remember
import androidx.compose.runtime.setValue
import androidx.compose.ui.Modifier
import androidx.compose.ui.composed
import androidx.compose.ui.graphics.Color
import androidx.compose.ui.platform.debugInspectorInfo

fun Modifier.flashOnSelection(
    selected: Boolean
): Modifier {
    return composed(
        inspectorInfo = debugInspectorInfo {
            name = "flashOnSelection"
            properties["selected"] = selected
        }
    ) {
        var wasSelected by remember {
            mutableStateOf(selected)
        }

        val color = if (selected && !wasSelected) {
            Color(0xFFFFF1A8)
        } else {
            Color.Transparent
        }

        wasSelected = selected

        Modifier.background(color)
    }
}
```

Этот пример показывает идею, но для настоящей анимации лучше использовать animation APIs. `composed` полезен, когда modifier обязан помнить локальное состояние, но важно не превращать его в место для сложной логики экрана.

## Modifier.Node

`Modifier.Node` — современный низкоуровневый способ писать эффективные custom modifiers. Он отделяет элемент modifier-цепочки от node-объекта, который может переиспользоваться и обновляться без лишнего пересоздания.

Низкоуровневая модель обычно состоит из двух частей:

- `ModifierNodeElement`: описывает параметры modifier и умеет создать или обновить node;
- `Modifier.Node`: хранит изменяемые поля и реализует нужные node-интерфейсы.

```kotlin
import androidx.compose.ui.Modifier
import androidx.compose.ui.geometry.CornerRadius
import androidx.compose.ui.graphics.Color
import androidx.compose.ui.graphics.drawscope.ContentDrawScope
import androidx.compose.ui.node.DrawModifierNode
import androidx.compose.ui.node.ModifierNodeElement
import androidx.compose.ui.platform.InspectorInfo
import androidx.compose.ui.unit.Dp

fun Modifier.pillBackground(
    color: Color,
    cornerRadius: Dp
): Modifier {
    return this.then(
        PillBackgroundElement(
            color = color,
            cornerRadius = cornerRadius
        )
    )
}

private data class PillBackgroundElement(
    private val color: Color,
    private val cornerRadius: Dp
) : ModifierNodeElement<PillBackgroundNode>() {

    override fun create(): PillBackgroundNode {
        return PillBackgroundNode(
            color = color,
            cornerRadius = cornerRadius
        )
    }

    override fun update(node: PillBackgroundNode) {
        node.color = color
        node.cornerRadius = cornerRadius
    }

    override fun InspectorInfo.inspectableProperties() {
        name = "pillBackground"
        properties["color"] = color
        properties["cornerRadius"] = cornerRadius
    }
}

private class PillBackgroundNode(
    var color: Color,
    var cornerRadius: Dp
) : Modifier.Node(), DrawModifierNode {

    override fun ContentDrawScope.draw() {
        drawRoundRect(
            color = color,
            cornerRadius = CornerRadius(
                x = cornerRadius.toPx(),
                y = cornerRadius.toPx()
            )
        )
        drawContent()
    }
}
```

Разбор:

- `pillBackground` остается удобной extension-функцией;
- `PillBackgroundElement` является неизменяемым описанием параметров;
- `create` создает node при первом добавлении modifier;
- `update` обновляет уже существующий node при изменении параметров;
- `PillBackgroundNode` реализует `DrawModifierNode`;
- `drawContent()` сохраняет отрисовку реального содержимого.

<CatTip>
`Modifier.Node` особенно хорошо подходит для design system: один раз аккуратно реализовали modifier, дальше вся команда использует короткий и понятный API.
</CatTip>

## LayoutModifierNode

Через `LayoutModifierNode` можно создать node, который меняет measurement и placement.

```kotlin
import androidx.compose.ui.Modifier
import androidx.compose.ui.layout.Measurable
import androidx.compose.ui.layout.MeasureResult
import androidx.compose.ui.layout.MeasureScope
import androidx.compose.ui.node.LayoutModifierNode
import androidx.compose.ui.node.ModifierNodeElement
import androidx.compose.ui.platform.InspectorInfo
import androidx.compose.ui.unit.Dp

fun Modifier.minInteractiveSide(
    side: Dp
): Modifier {
    return this.then(
        MinInteractiveSideElement(side = side)
    )
}

private data class MinInteractiveSideElement(
    private val side: Dp
) : ModifierNodeElement<MinInteractiveSideNode>() {

    override fun create(): MinInteractiveSideNode {
        return MinInteractiveSideNode(side = side)
    }

    override fun update(node: MinInteractiveSideNode) {
        node.side = side
    }

    override fun InspectorInfo.inspectableProperties() {
        name = "minInteractiveSide"
        properties["side"] = side
    }
}

private class MinInteractiveSideNode(
    var side: Dp
) : Modifier.Node(), LayoutModifierNode {

    override fun MeasureScope.measure(
        measurable: Measurable,
        constraints: androidx.compose.ui.unit.Constraints
    ): MeasureResult {
        val sidePx = side.roundToPx()
        val placeable = measurable.measure(constraints)

        val width = constraints.constrainWidth(
            maxOf(placeable.width, sidePx)
        )
        val height = constraints.constrainHeight(
            maxOf(placeable.height, sidePx)
        )

        return layout(width = width, height = height) {
            val x = (width - placeable.width) / 2
            val y = (height - placeable.height) / 2
            placeable.placeRelative(x = x, y = y)
        }
    }
}
```

Этот modifier увеличивает интерактивную область маленькой иконки, но центрирует исходный контент. В реальном приложении его стоит сочетать с `clickable` и semantics.

## Production-пример: modifier для состояния карточки

```kotlin
import androidx.compose.foundation.border
import androidx.compose.foundation.clickable
import androidx.compose.foundation.interaction.MutableInteractionSource
import androidx.compose.foundation.layout.padding
import androidx.compose.foundation.shape.RoundedCornerShape
import androidx.compose.material3.Card
import androidx.compose.material3.Text
import androidx.compose.runtime.Composable
import androidx.compose.runtime.Immutable
import androidx.compose.runtime.remember
import androidx.compose.ui.Modifier
import androidx.compose.ui.graphics.Color
import androidx.compose.ui.semantics.Role
import androidx.compose.ui.semantics.contentDescription
import androidx.compose.ui.semantics.role
import androidx.compose.ui.semantics.selected
import androidx.compose.ui.semantics.semantics
import androidx.compose.ui.unit.dp

@Immutable
data class PaymentCardUi(
    val id: String,
    val title: String,
    val maskedNumber: String,
    val selected: Boolean
)

fun Modifier.paymentCardBehavior(
    card: PaymentCardUi,
    onSelect: (String) -> Unit
): Modifier {
    return this
        .semantics {
            role = Role.RadioButton
            selected = card.selected
            contentDescription = if (card.selected) {
                "Выбрана карта ${card.title}, номер ${card.maskedNumber}"
            } else {
                "Карта ${card.title}, номер ${card.maskedNumber}"
            }
        }
        .border(
            width = if (card.selected) {
                2.dp
            } else {
                1.dp
            },
            color = if (card.selected) {
                Color(0xFF1A73E8)
            } else {
                Color(0xFFD6DEE8)
            },
            shape = RoundedCornerShape(12.dp)
        )
        .clickable(
            interactionSource = MutableInteractionSource(),
            indication = null,
            role = Role.RadioButton,
            onClick = {
                onSelect(card.id)
            }
        )
}

@Composable
fun PaymentCardItem(
    card: PaymentCardUi,
    onSelect: (String) -> Unit,
    modifier: Modifier = Modifier
) {
    val itemModifier = remember(card, onSelect, modifier) {
        modifier.paymentCardBehavior(
            card = card,
            onSelect = onSelect
        )
    }

    Card(modifier = itemModifier) {
        Text(
            text = "${card.title}\n${card.maskedNumber}",
            modifier = Modifier.padding(16.dp)
        )
    }
}
```

В этом примере modifier объединяет:

- semantics для доступности;
- визуальное выделение;
- click behavior;
- передачу стабильного доменного идентификатора наружу.

Для `clickable` обычно не стоит отключать `indication`, если нет отдельной визуальной реакции. Здесь реакция показана через `focusRing`, но в реальном design system это решение нужно согласовать с accessibility и визуальными состояниями.

## Типичные ошибки

### Терять входящий modifier

Если extension-функция возвращает новый `Modifier` и не использует `this`, она обрывает цепочку вызывающего кода.

### Прятать слишком много поведения

Modifier с названием `blueBackground` не должен внезапно добавлять click, semantics и padding.

### Нарушать порядок

`padding`, `background`, `clickable`, `clip` и `border` сильно зависят от позиции в цепочке.

### Использовать pointerInput для простого клика

Для клика есть `clickable`. Для долгого нажатия есть `combinedClickable`. Для drag есть `draggable`. Низкоуровневый pointer input нужен не всегда.

### Делать stateful modifier без понимания жизненного цикла

Если modifier хранит состояние, нужно понимать, когда оно создается, обновляется и уничтожается. Для простого визуального стиля состояние обычно не нужно.

## Итоги

Custom modifiers помогают сделать UI выразительным и поддерживаемым:

- extension-функции подходят для композиции готовых modifiers;
- `then` соединяет цепочки;
- порядок modifier-звеньев влияет на measurement, drawing и input;
- `layout` меняет measurement одного элемента;
- `drawBehind`, `drawWithContent` и `drawWithCache` управляют отрисовкой;
- `semantics` добавляет смысл для accessibility и тестов;
- `pointerInput` нужен для низкоуровневых жестов;
- `composed` позволяет использовать composable APIs внутри modifier;
- `Modifier.Node` является современным инструментом для производительных reusable modifiers;
- хороший custom modifier сохраняет входящий `Modifier`, имеет ясное имя и не прячет неожиданные побочные эффекты.
