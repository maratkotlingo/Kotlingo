## Почему синхронизация важна

UI tests выполняются быстрее пользователя. Если тест нажал кнопку и сразу проверил результат, а UI еще обрабатывает coroutine, animation или Flow emission, тест может стать flaky.

Compose testing framework умеет ждать idle state, но не может угадать все внешние asynchronous operations:

- network fake delay;
- repository Flow;
- Paging load;
- WorkManager;
- custom coroutine scope;
- infinite animation;
- Android View interop;
- idling ресурсы вне Compose.

```kotlin
data class AsyncUiState(
    val loading: Boolean,
    val items: List<String>
)
```

Тест должен знать, когда state действительно достиг ожидаемого состояния.

<CatTip>
Flaky UI test почти всегда говорит: "я не знаю, чего именно жду". Хорошая синхронизация делает ожидание явным.
</CatTip>

## Automatic synchronization

Compose test rule автоматически ждет:

- composition;
- recomposition;
- pending frame work;
- Compose animations, если main clock управляется test rule;
- некоторые UI idle conditions.

Обычный тест не нуждается в manual waiting:

```kotlin
composeRule
    .onNodeWithText("Сохранить")
    .performClick()

composeRule
    .onNodeWithText("Сохранено")
    .assertExists()
```

Если state update синхронный или идет через Compose-managed work, rule часто сам дождется idle.

## `waitForIdle`

`waitForIdle()` явно ждет, пока Compose станет idle.

```kotlin
composeRule
    .onNodeWithText("Запустить")
    .performClick()

composeRule.waitForIdle()

composeRule
    .onNodeWithText("Готово")
    .assertExists()
```

Используйте `waitForIdle`, если нужно разделить action и assertion для читаемости. Но если результат появляется после внешнего Flow/network fake, одного `waitForIdle` может быть недостаточно.

## `runOnIdle`

`runOnIdle` выполняет block, когда Compose idle. Это удобно для чтения test state.

```kotlin
@Test
fun click_updatesCounter() {
    var count = 0

    composeRule.setContent {
        CounterButton(
            count = count,
            onIncrement = {
                count += 1
            }
        )
    }

    composeRule
        .onNodeWithText("0")
        .performClick()

    composeRule.runOnIdle {
        assertThat(count).isEqualTo(1)
    }
}
```

Не меняйте production state из `runOnIdle`, если тест должен имитировать пользователя. Для test setup это допустимо, но action лучше делать через UI.

## `waitUntil`

`waitUntil` ждет произвольное условие.

```kotlin
composeRule.waitUntil(
    timeoutMillis = 5_000
) {
    repository.loaded
}
```

Лучше ждать UI condition:

```kotlin
composeRule.waitUntil(
    timeoutMillis = 5_000
) {
    composeRule
        .onAllNodesWithText("Готово")
        .fetchSemanticsNodes()
        .isNotEmpty()
}
```

Если testing API предоставляет `waitUntilAtLeastOneExists` или похожие helpers в вашей версии, используйте их: они делают intent понятнее.

## Не используйте `Thread.sleep`

Плохо:

```kotlin
Thread.sleep(2_000)
```

Почему плохо:

- тест медленный;
- на слабом устройстве 2 секунды может не хватить;
- на быстром устройстве вы тратите время зря;
- sleep не связан с реальным condition.

Лучше:

```kotlin
composeRule.waitUntil(
    timeoutMillis = 5_000
) {
    composeRule
        .onAllNodesWithText("Данные загружены")
        .fetchSemanticsNodes()
        .isNotEmpty()
}
```

<MouseTrap>
`Thread.sleep` в UI test — почти всегда признание, что тест не понимает состояние системы.
</MouseTrap>

## Main clock

Compose test rule дает `mainClock`, которым можно управлять animations/time.

```kotlin
composeRule.mainClock.autoAdvance = false

composeRule
    .onNodeWithText("Показать")
    .performClick()

composeRule.mainClock.advanceTimeBy(300)

composeRule
    .onNodeWithText("Контент")
    .assertIsDisplayed()
```

Если `autoAdvance = false`, вы сами двигаете frame time. Это полезно для animation tests.

## Animation tests

Проверить промежуточное состояние:

```kotlin
@Test
fun panel_animatesIn() {
    composeRule.mainClock.autoAdvance = false

    composeRule.setContent {
        AnimatedPanel(visible = true)
    }

    composeRule.mainClock.advanceTimeBy(150)

    composeRule
        .onNodeWithText("Панель")
        .assertExists()
}
```

Для большинства tests лучше проверять final state, а не точные animation frames. Промежуточные кадры хрупкие.

## Infinite animations

Infinite animations могут мешать idle. Если тест зависает:

- отключите animation для test mode;
- используйте finite animation;
- управляйте mainClock;
- проверьте, не ждете ли idle, который никогда не наступит.

```kotlin
@Composable
fun LoadingIndicator(
    animated: Boolean
) {
    if (animated) {
        CircularProgressIndicator()
    } else {
        StaticLoadingIndicator()
    }
}
```

В tests:

```kotlin
LoadingIndicator(animated = false)
```

Не обязательно тестировать бесконечную визуальную анимацию в каждом screen test.

## Coroutines and flows

Если ViewModel использует coroutines/flows, тест должен контролировать dispatcher или fake repository.

```kotlin
class FakeLessonsRepository {
    private val _lessons = MutableStateFlow<List<String>>(emptyList())
    val lessons: StateFlow<List<String>> = _lessons

    fun emitLessons(value: List<String>) {
        _lessons.value = value
    }
}
```

Тест:

```kotlin
@Test
fun lessonsFlow_updatesUi() {
    val repository = FakeLessonsRepository()

    composeRule.setContent {
        val lessons by repository.lessons.collectAsStateWithLifecycle()

        LessonsList(lessons = lessons)
    }

    repository.emitLessons(
        listOf("Compose UI testing")
    )

    composeRule
        .onNodeWithText("Compose UI testing")
        .assertExists()
}
```

Если emission происходит не на main thread или через delay, используйте coroutine test utilities и явное ожидание.

## Testing dispatchers

Для ViewModel tests используйте `runTest` и test dispatchers. Для UI tests с real Activity осторожнее: Compose test rule управляет своим окружением, а production Dispatchers.Main уже может быть частью instrumentation.

Обычно проще:

- fake repository без real delay;
- expose `StateFlow`;
- emit state из test;
- assert UI.

Если нужно тестировать delay/debounce, выносите logic в unit test с `runTest`, а UI test проверяет итоговое состояние.

## Paging synchronization

Paging loads asynchronously. После `collectAsLazyPagingItems` нужно дождаться item.

```kotlin
composeRule.waitUntil(
    timeoutMillis = 5_000
) {
    composeRule
        .onAllNodesWithText("Первый урок")
        .fetchSemanticsNodes()
        .isNotEmpty()
}
```

Проверяйте UI, а не internal paging state, если тест screen-level.

## Idling resources

Если внешняя система не интегрирована с Compose idling, можно зарегистрировать idling resource.

```kotlin
composeRule.registerIdlingResource(
    repositoryIdlingResource
)

try {
    composeRule
        .onNodeWithText("Обновить")
        .performClick()

    composeRule
        .onNodeWithText("Готово")
        .assertExists()
} finally {
    composeRule.unregisterIdlingResource(
        repositoryIdlingResource
    )
}
```

Idling resource должен быть надежным. Неверная реализация делает тесты хуже, чем явное `waitUntil`.

## Waiting for absence

Иногда нужно дождаться исчезновения loading.

```kotlin
composeRule.waitUntil(
    timeoutMillis = 5_000
) {
    composeRule
        .onAllNodesWithText("Загрузка")
        .fetchSemanticsNodes()
        .isEmpty()
}
```

После этого проверяйте content:

```kotlin
composeRule
    .onNodeWithText("Готово")
    .assertIsDisplayed()
```

Не проверяйте только исчезновение spinner: экран может стать пустым.

## Flaky test debugging

Если тест иногда падает:

- распечатайте semantics tree в момент падения;
- увеличьте timeout временно для диагностики;
- проверьте infinite animations;
- уберите real network/time;
- замените sleep на condition;
- проверьте dispatcher;
- проверьте locale/font scale;
- проверьте lazy item scroll;
- сохраните screenshot/logcat, если возможно.

```kotlin
fun ComposeTestRule.dumpTree(
    label: String
) {
    onRoot(useUnmergedTree = true)
        .printToLog(label)
}
```

Debug helper должен быть test-only.

## Типичные ошибки

### Использовать `Thread.sleep`

Тест становится медленным и flaky.

### Ждать Compose idle вместо внешнего state

Repository/worker может еще работать.

### Делать infinite animation обязательной для screen test

Idle может не наступить.

### Проверять animation frame слишком точно

Тест ломается от небольших изменений duration/easing.

### Не контролировать fake data source

Real async делает тест nondeterministic.

### Ждать только исчезновение loading

Нужно еще проверить успешный или error state.

## Итоги

Synchronization в Compose tests строится на явных условиях:

- Compose test rule автоматически ждет многие Compose operations;
- `waitForIdle` ждет Compose idle;
- `runOnIdle` безопасно читает test state после idle;
- `waitUntil` нужен для внешних/asynchronous conditions;
- `Thread.sleep` почти всегда плохой выбор;
- `mainClock` управляет Compose time и animations;
- infinite animations требуют test strategy;
- flows/coroutines лучше тестировать через fake repositories и controlled state;
- Paging требует ожидания loaded nodes;
- idling resources полезны для внешних систем, если реализованы надежно;
- flaky tests лечатся явным condition, controlled data и debug semantics tree.
