## Зачем тестировать анимации

Анимация — это не только украшение. Она влияет на навигацию, обратная связь, accessibility, perceived performance и стабильность UI. Плохая анимация может:

- скрыть важное состояние;
- создать flaky UI test;
- вызвать jank;
- сломать screenshot test;
- сделать экран утомительным;
- задержать взаимодействие;
- потреблять ресурсы после завершения загрузка;
- конфликтовать с lazy lists и gestures.

Тестирование анимаций в Compose делится на несколько уровней:

- deterministic UI tests через test clock;
- screenshot tests конкретных animation frames;
- unit tests state machine, которая управляет motion;
- manual QA на устройстве;
- profiling jank и frame time;
- Macrobenchmark для важных journeys;
- tracing для поиска фаз Composition, Layout и Draw.

<CatTip>
Не обязательно тестировать каждый кадр каждой анимации. Обычно достаточно проверить стартовое состояние, конечное состояние, ключевые interaction points и отсутствие регрессий в важных motion-сценариях.
</CatTip>

## `MainTestClock`

Compose UI tests используют test clock. `MainTestClock` отвечает за продвижение времени для recomposition, animations и gestures внутри Compose test environment.

Ключевые операции:

- `mainClock.autoAdvance = false`: отключить автоматическое продвижение времени;
- `advanceTimeBy(milliseconds)`: продвинуть время на конкретный интервал;
- `advanceTimeByFrame()`: продвинуть время на один frame;
- `advanceTimeUntil(timeoutMillis) { condition }`: продвигать время до условия;
- `waitForIdle()`: дождаться idle state;
- `autoAdvance = true`: default режим, где test rule синхронизирует UI перед actions/assertions.

```kotlin
import androidx.compose.animation.core.animateFloatAsState
import androidx.compose.foundation.layout.Box
import androidx.compose.runtime.Composable
import androidx.compose.runtime.getValue
import androidx.compose.ui.Modifier
import androidx.compose.ui.draw.alpha
import androidx.compose.ui.platform.testTag

@Composable
fun FadingBox(
    visible: Boolean
) {
    val alpha by animateFloatAsState(
        targetValue = if (visible) {
            1f
        } else {
            0f
        },
        label = "fading box alpha"
    )

    Box(
        modifier = Modifier
            .testTag("fading-box")
            .alpha(alpha)
    )
}
```

```kotlin
import androidx.compose.ui.test.junit4.createComposeRule
import androidx.compose.ui.test.onNodeWithTag
import androidx.compose.ui.test.assertExists
import org.junit.Rule
import org.junit.Test

class FadingBoxTest {

    @get:Rule
    val composeTestRule = createComposeRule()

    @Test
    fun fadingBoxExistsAfterAnimationTimeAdvances() {
        composeTestRule.mainClock.autoAdvance = false

        composeTestRule.setContent {
            FadingBox(visible = true)
        }

        composeTestRule.mainClock.advanceTimeBy(1_000)

        composeTestRule
            .onNodeWithTag("fading-box")
            .assertExists()
    }
}
```

Этот тест не ждет реальную секунду. Он продвигает virtual time.

## `autoAdvance`

По умолчанию `autoAdvance = true`. Это удобно для обычных UI tests: Compose test rule сам продвигает время, пока UI не станет idle.

Для анимаций иногда нужно отключить auto advance:

- проверить intermediate frame;
- сделать screenshot на конкретной отметке времени;
- убедиться, что old content существует во время exit;
- вручную контролировать sequence;
- избежать ожидания бесконечной анимации.

```kotlin
@Test
fun animationCanBeCapturedHalfway() {
    composeTestRule.mainClock.autoAdvance = false

    var expanded = false

    composeTestRule.setContent {
        ExpandableTestCard(expanded = expanded)
    }

    composeTestRule.runOnIdle {
        expanded = true
    }

    composeTestRule.mainClock.advanceTimeBy(120)

    composeTestRule
        .onNodeWithTag("expandable-card")
        .assertExists()
}
```

Если `autoAdvance = false`, assertions не будут автоматически доводить animation до конца. Это делает тест детерминированным, но требует ручного управления временем.

## Тестирование `AnimatedVisibility`

`AnimatedVisibility` удаляет content только после exit animation. Тест должен учитывать это.

```kotlin
import androidx.compose.animation.AnimatedVisibility
import androidx.compose.animation.fadeIn
import androidx.compose.animation.fadeOut
import androidx.compose.material3.Text
import androidx.compose.runtime.Composable
import androidx.compose.ui.Modifier
import androidx.compose.ui.platform.testTag

@Composable
fun TestableAnimatedVisibility(
    visible: Boolean
) {
    AnimatedVisibility(
        visible = visible,
        enter = fadeIn(),
        exit = fadeOut(),
        label = "test visibility"
    ) {
        Text(
            text = "Видимый контент",
            modifier = Modifier.testTag("visibility-content")
        )
    }
}
```

```kotlin
import androidx.compose.ui.test.assertDoesNotExist

@Test
fun contentIsRemovedAfterExitAnimation() {
    composeTestRule.mainClock.autoAdvance = false

    var visible = true

    composeTestRule.setContent {
        TestableAnimatedVisibility(visible = visible)
    }

    composeTestRule.runOnIdle {
        visible = false
    }

    // До завершения exit content может еще существовать.
    composeTestRule.mainClock.advanceTimeBy(1_000)

    composeTestRule
        .onNodeWithTag("visibility-content")
        .assertDoesNotExist()
}
```

Если transition использует spring, точная длительность не фиксирована. В таком случае продвигайте clock достаточно долго или ждите idle, если нет бесконечной анимации.

## Тестирование `AnimatedContent`

Во время transition old и new content могут существовать одновременно. Поэтому tests должны быть устойчивыми к overlap.

```kotlin
import androidx.compose.animation.AnimatedContent
import androidx.compose.material3.Text
import androidx.compose.runtime.Composable
import androidx.compose.ui.Modifier
import androidx.compose.ui.platform.testTag

enum class TestLoadState {
    Loading,
    Content,
    Error
}

@Composable
fun TestableAnimatedContent(
    state: TestLoadState
) {
    AnimatedContent(
        targetState = state,
        label = "load state content"
    ) { targetState ->
        Text(
            text = targetState.name,
            modifier = Modifier.testTag(
                tag = when (targetState) {
                    TestLoadState.Loading -> "loading"
                    TestLoadState.Content -> "content"
                    TestLoadState.Error -> "error"
                }
            )
        )
    }
}
```

```kotlin
@Test
fun animatedContentShowsFinalState() {
    composeTestRule.mainClock.autoAdvance = false

    var state = TestLoadState.Loading

    composeTestRule.setContent {
        TestableAnimatedContent(state = state)
    }

    composeTestRule.runOnIdle {
        state = TestLoadState.Content
    }

    composeTestRule.mainClock.advanceTimeBy(1_000)

    composeTestRule
        .onNodeWithTag("content")
        .assertExists()
}
```

Для intermediate frame можно проверить, что old и new nodes существуют, если transition их одновременно держит. Но такие тесты более хрупкие.

## Бесконечные анимации в тестах

`rememberInfiniteTransition` может мешать `waitForIdle`, потому что animation никогда не завершится. Если тест не проверяет сам shimmer или pulse, лучше сделать анимацию отключаемой.

```kotlin
import androidx.compose.runtime.Composable

@Composable
fun LoadingRow(
    animated: Boolean
) {
    if (animated) {
        ShimmerLine()
    } else {
        StaticSkeletonLine()
    }
}

@Composable
fun StaticSkeletonLine() {
    androidx.compose.foundation.layout.Box(
        modifier = androidx.compose.ui.Modifier
            .testTag("static-skeleton")
    )
}
```

```kotlin
@Test
fun loadingRowCanBeTestedWithoutInfiniteAnimation() {
    composeTestRule.setContent {
        LoadingRow(animated = false)
    }

    composeTestRule
        .onNodeWithTag("static-skeleton")
        .assertExists()
}
```

<MouseTrap>
Не заставляйте каждый UI-тест жить с бесконечным shimmer. Добавьте параметр, theme flag или предпросмотр/test variant, который заменяет loop на статичный skeleton.
</MouseTrap>

## Screenshot frames

Для screenshot testing анимаций полезно фиксировать конкретные моменты:

- 0 мс: старт;
- середина transition;
- конец transition;
- состояние после interruption;
- состояние reduce motion, если оно есть.

```kotlin
@Test
fun capturesHalfExpandedFrame() {
    composeTestRule.mainClock.autoAdvance = false

    var expanded = false

    composeTestRule.setContent {
        ExpandableTestCard(expanded = expanded)
    }

    composeTestRule.runOnIdle {
        expanded = true
    }

    composeTestRule.mainClock.advanceTimeBy(150)

    // Здесь screenshot tool делает снимок текущего кадра.
    composeTestRule
        .onNodeWithTag("expandable-card")
        .assertExists()
}
```

Снимки intermediate frames хрупче, чем final state screenshots. Используйте их для критичных motion-паттернов, а не для каждой мелкой анимации.

## State machine tests

Часто проще тестировать не animation API, а state machine, которая выбирает motion state.

```kotlin
enum class PlayerUiState {
    Stopped,
    Loading,
    Playing,
    Paused
}

enum class PlayerMotionState {
    Hidden,
    Loading,
    Visible
}

fun PlayerUiState.toMotionState(): PlayerMotionState {
    return when (this) {
        PlayerUiState.Stopped -> PlayerMotionState.Hidden
        PlayerUiState.Loading -> PlayerMotionState.Loading
        PlayerUiState.Playing -> PlayerMotionState.Visible
        PlayerUiState.Paused -> PlayerMotionState.Visible
    }
}
```

```kotlin
import kotlin.test.Test
import kotlin.test.assertEquals

class PlayerMotionStateTest {

    @Test
    fun playingAndPausedUseVisibleMotion() {
        assertEquals(
            expected = PlayerMotionState.Visible,
            actual = PlayerUiState.Playing.toMotionState()
        )
        assertEquals(
            expected = PlayerMotionState.Visible,
            actual = PlayerUiState.Paused.toMotionState()
        )
    }
}
```

Если motion state выбран правильно, UI animation можно покрывать меньшим количеством screenshot/UI tests.

## Profiling: что такое jank

Jank — видимый рывок, когда приложение не успело подготовить frame к deadline. На 60 Hz frame budget примерно 16.6 мс, на 120 Hz — примерно 8.3 мс. Animation smoothness зависит от того, успевают ли Composition, Layout, Draw, RenderThread и GPU вовремя.

Анимация может jank-ать из-за:

- тяжелой recomposition на каждом кадре;
- изменения layout большого дерева;
- дорогого drawing без cache;
- allocation churn;
- image decoding на main thread;
- lazy list item churn;
- нестабильных параметров;
- отсутствия baseline profile;
- debug build overhead;
- слишком большого количества simultaneous animations.

## Что измерять

Для animation performance полезны:

- frame time;
- jank rate;
- missed frames;
- recomposition counts;
- layout passes;
- draw cost;
- allocations;
- startup cost, если animation происходит на первом экране;
- scroll jank для animated lazy items.

Не оптимизируйте по ощущению только на emulator. Проверяйте release build на реальном устройстве.

## Инструменты

Основные инструменты:

- Android Studio Layout Inspector: recomposition counters и tree inspection;
- System Trace: timeline frame pipeline;
- Macrobenchmark: повторяемые тесты производительности;
- Baseline Profile: прогрев критических путей кода;
- JankStats: runtime мониторинг jank в боевой среде-like окружении;
- Animation Предпросмотр/Inspector: визуальная отладка animations;
- профилировщик памяти: allocation churn.

## Macrobenchmark для animated journey

Macrobenchmark подходит, когда нужно измерить реальный user journey: открыть экран, нажать item, дождаться общий transition, проскроллить animated list.

```kotlin
import androidx.benchmark.macro.MacrobenchmarkScope
import androidx.benchmark.macro.junit4.MacrobenchmarkRule
import androidx.benchmark.macro.StartupMode
import androidx.benchmark.macro.FrameTimingMetric
import androidx.test.ext.junit.runners.AndroidJUnit4
import org.junit.Rule
import org.junit.Test
import org.junit.runner.RunWith

@RunWith(AndroidJUnit4::class)
class AnimationJourneyBenchmark {

    @get:Rule
    val benchmarkRule = MacrobenchmarkRule()

    @Test
    fun openDetailsSharedTransition() {
        benchmarkRule.measureRepeated(
            packageName = "com.example.app",
            metrics = listOf(
                FrameTimingMetric()
            ),
            iterations = 5,
            startupMode = StartupMode.WARM,
            setupBlock = {
                pressHome()
                startActivityAndWait()
            }
        ) {
            openFirstArticle()
            device.waitForIdle()
        }
    }
}

private fun MacrobenchmarkScope.openFirstArticle() {
    device.findObject(
        androidx.test.uiautomator.By.res("article-0")
    ).click()
}
```

Этот пример показывает структуру. В реальном проекте нужны stable resource IDs или semantics/тестовые теги, которые доступны automation layer.

## Layout Inspector

Layout Inspector помогает увидеть:

- какие composables часто recomposed;
- какие параметры вызывают обновления;
- как меняется tree;
- где layout глубже, чем ожидалось;
- какие элементы остаются в composition во время transitions.

Что проверить для анимаций:

- не recomposed ли весь экран на каждый frame;
- не пересоздаются ли expensive objects;
- не держится ли shimmer после загрузки;
- не меняется ли layout там, где можно было использовать `graphicsLayer`;
- не анимируется ли state слишком высокого уровня.

## System Trace

System Trace полезен, когда есть jank, но непонятно где. Он показывает:

- main thread work;
- render thread;
- frame deadlines;
- Choreographer frames;
- expensive slices;
- layout/draw work;
- coroutine scheduling;
- binder calls и I/O, если они попали в critical path.

Практический подход:

1. Запустите release или profileable build.
2. Воспроизведите конкретную animation.
3. Найдите janky frames.
4. Посмотрите, что происходило на main thread.
5. Сравните Composition/Layout/Draw cost.
6. Уберите лишнюю работу из frame path.

## Performance проверочный список для анимаций

Перед релизом проверьте:

- animation не запускает network/база данных work;
- на каждый кадр не сортируется список;
- heavy `Path`, `Brush`, `Shader` создаются в `drawWithCache` или `remember`;
- high-frequency чтения состояния отложены в лямбда modifiers;
- для movement используется `graphicsLayer` или лямбда `offset`, если подходит;
- бесконечные анимации удаляются из composition;
- lazy lists имеют stable keys;
- общий elements имеют stable keys;
- image загрузка не блокирует transition;
- screenshot tests фиксируют важные states;
- benchmark покрывает важные journeys.

## Оптимизация draw-heavy animation

```kotlin
import androidx.compose.animation.core.animateFloatAsState
import androidx.compose.foundation.Canvas
import androidx.compose.foundation.layout.size
import androidx.compose.runtime.Composable
import androidx.compose.runtime.getValue
import androidx.compose.ui.Modifier
import androidx.compose.ui.graphics.Color
import androidx.compose.ui.unit.dp

@Composable
fun AnimatedProgressRing(
    progress: Float
) {
    val animatedProgress by animateFloatAsState(
        targetValue = progress.coerceIn(0f, 1f),
        label = "ring progress"
    )

    Canvas(
        modifier = Modifier.size(48.dp)
    ) {
        drawArc(
            color = Color.Blue,
            startAngle = -90f,
            sweepAngle = 360f * animatedProgress,
            useCenter = false,
            style = androidx.compose.ui.graphics.drawscope.Stroke(
                width = 4.dp.toPx()
            )
        )
    }
}
```

Если ring становится сложным, содержит gradients и paths, переносите тяжелую подготовку в `drawWithCache`.

## Типичные ошибки

### Проверять анимацию через `Thread.sleep`

В Compose UI tests используйте `mainClock.advanceTimeBy`, а не реальное ожидание.

### Ждать idle при бесконечной анимации

Infinite transition не завершится. Отключайте loop в тесте или управляйте clock вручную.

### Считать debug build источником правды о performance

Debug build может jank-ать из-за overhead. Для performance нужны release/profileable builds.

### Тестировать только конечное состояние общий transition

Общий transition может приходить к правильному финалу, но прыгать в середине. Для важных переходов полезны screenshot frames или ручная QA-проверка.

### Игнорировать layout cost

Если animation меняет `padding`, `height`, `width`, она может вызывать layout. Иногда `graphicsLayer` дешевле.

### Профилировать без конкретного сценария

Фраза "анимации тормозят" слишком общая. Нужен воспроизводимый journey: какой экран, какое действие, какое устройство, какой build.

## Итоги

Тестирование и профилирование анимаций требует разных инструментов:

- `MainTestClock` делает animation tests детерминированными;
- `autoAdvance = false` позволяет проверять intermediate frames;
- `advanceTimeBy` и `advanceTimeByFrame` заменяют real-time ожидание;
- `AnimatedVisibility` и `AnimatedContent` могут держать old content во время transition;
- бесконечные анимации нужно отключать или изолировать в тестах;
- screenshot frames полезны для важных motion states;
- Macrobenchmark и `FrameTimingMetric` помогают измерять jank;
- Layout Inspector и System Trace помогают найти источник frame cost;
- performance нужно проверять на release/profileable build и реальном устройстве.
