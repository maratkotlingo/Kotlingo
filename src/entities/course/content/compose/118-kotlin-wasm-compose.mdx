## Что такое Kotlin/Wasm Compose

Kotlin/Wasm позволяет запускать Kotlin code в браузере через WebAssembly. Compose Multiplatform использует Wasm как основной путь для shared Compose UI на web. В отличие от DOM-first UI, Compose for Web/Wasm обычно рендерит UI через canvas/skia-like pipeline и переносит знакомую Compose model в browser target.

Важно: Web/Wasm target находится в beta-стадии для Compose Multiplatform. Это означает, что его можно изучать и применять осознанно, но production rollout требует отдельной оценки рисков:

- browser compatibility;
- startup/download size;
- accessibility;
- SEO;
- resource loading;
- debugging;
- hosting/CDN;
- JavaScript interop;
- performance на слабых devices;
- API changes.

```kotlin
data class WebAppCapabilities(
    val wasmSupported: Boolean,
    val offlineSupported: Boolean,
    val browserName: String
)
```

Web target должен иметь graceful failure или fallback strategy, если browser/runtime не поддерживает нужный path.

<CatTip>
Wasm Compose переносит Compose mental model в браузер, но браузер не перестает быть браузером: загрузка, URL, history, accessibility и network остаются web-проблемами.
</CatTip>

## Gradle target

Типичная настройка:

```kotlin
kotlin {
    wasmJs {
        browser {
            commonWebpackConfig {
                outputFileName = "app.js"
            }
        }
        binaries.executable()
    }

    sourceSets {
        commonMain.dependencies {
            implementation(compose.runtime)
            implementation(compose.foundation)
            implementation(compose.material3)
        }

        wasmJsMain.dependencies {
            implementation(compose.components.resources)
        }
    }
}
```

Exact DSL меняется вместе с Kotlin/Compose Multiplatform версиями. Проверяйте template, который генерирует актуальный JetBrains wizard.

## Browser entry point

Упрощенный entry point:

```kotlin
import androidx.compose.ui.ExperimentalComposeUiApi
import androidx.compose.ui.window.CanvasBasedWindow

@OptIn(ExperimentalComposeUiApi::class)
fun main() {
    CanvasBasedWindow(
        title = "Kotlingo Web"
    ) {
        AppTheme {
            WebAppRoot()
        }
    }
}
```

В новых версиях API names могут меняться. Архитектурно важно: web target имеет собственный entry point и shell.

## Browser target constraints

Compose UI в браузере отличается от Android/Desktop:

- startup зависит от download/compile time;
- browser zoom и DPI влияют на rendering;
- text input и IME имеют web-specific особенности;
- accessibility model отличается;
- URL/history нужно реализовать явно;
- browser APIs доступны через Kotlin/JS interop;
- file access ограничен browser security;
- service workers/offline требуют web-specific code;
- SEO может быть проблемой для canvas-rendered UI.

Не переносите Android assumptions в web target.

## Resources

Compose Multiplatform resources поддерживаются, но web target требует внимания к загрузке.

```kotlin
import org.jetbrains.compose.resources.painterResource
import org.jetbrains.compose.resources.stringResource
import my.app.generated.resources.Res
import my.app.generated.resources.app_logo
import my.app.generated.resources.app_title

@Composable
fun WebHeader() {
    Row {
        Image(
            painter = painterResource(Res.drawable.app_logo),
            contentDescription = null
        )
        Text(
            text = stringResource(Res.string.app_title)
        )
    }
}
```

Для web targets актуальные Compose Multiplatform версии добавляют APIs для preloading fonts/images, чтобы уменьшить visual flicker.

```kotlin
suspend fun preloadAppResources() {
    preloadFont(Res.font.inter_regular)
    preloadImageVector(Res.drawable.app_logo)
}
```

Exact API зависит от версии. Идея: ресурсы для первого экрана лучше подготовить заранее.

## Startup cost

Web/Wasm startup включает:

- загрузку HTML/JS/Wasm/assets;
- инициализацию runtime;
- загрузку fonts/resources;
- первую composition;
- первый draw.

Оптимизируйте:

- размер bundle;
- lazy loading feature modules, если возможно;
- CDN caching;
- resource preloading;
- первый экран;
- избегайте тяжелых initialization в `main`;
- показывайте lightweight loading shell.

```kotlin
@Composable
fun WebStartupShell(
    ready: Boolean,
    content: @Composable () -> Unit
) {
    if (ready) {
        content()
    } else {
        Text(text = "Загрузка")
    }
}
```

Не стройте весь app graph до первого meaningful paint, если пользователь сначала видит только splash.

## URL and navigation

Browser URL должен отражать navigation, если app ожидает web behavior:

- back/forward buttons;
- deep links;
- refresh;
- shareable URL;
- browser history;
- query params.

Common route model:

```kotlin
sealed interface WebRoute {
    data object Home : WebRoute
    data class Lesson(
        val id: String
    ) : WebRoute
}

fun WebRoute.toPath(): String {
    return when (this) {
        WebRoute.Home -> "/"
        is WebRoute.Lesson -> "/lessons/$id"
    }
}
```

Platform web shell синхронизирует `WebRoute` с browser history. Shared screen этого не должен знать.

## Browser APIs

Browser APIs живут в wasm/js source set:

```kotlin
interface BrowserClipboard {
    suspend fun writeText(text: String)
}
```

Web implementation:

```kotlin
class WebBrowserClipboard : BrowserClipboard {

    override suspend fun writeText(text: String) {
        window.navigator.clipboard.writeText(text).await()
    }
}
```

Shared UI:

```kotlin
@Composable
fun CopyButton(
    onCopy: () -> Unit
) {
    Button(onClick = onCopy) {
        Text(text = "Копировать")
    }
}
```

Browser permissions/errors должны обрабатываться в platform layer.

## Accessibility constraints

Canvas-based web UI может иметь accessibility limitations по сравнению с DOM-native UI. Проверяйте:

- screen reader behavior;
- keyboard navigation;
- focus visibility;
- browser zoom;
- text selection;
- semantics mapping;
- high contrast mode;
- reduced motion;
- tab order.

Если project требует strong SEO/accessibility для public content, оцените Web/Wasm target особенно строго.

<MouseTrap>
Если сайт должен индексироваться поисковиками как текстовый content, canvas-rendered Compose UI может быть неправильным инструментом для основной страницы.
</MouseTrap>

## Resource limitations

Ограничения web target:

- bundle size влияет на startup;
- fonts/images грузятся асинхронно;
- large bitmaps дорого скачивать и декодировать;
- browser cache rules важны;
- offline support нужно проектировать;
- CORS влияет на remote resources;
- file system access ограничен.

Для изображений:

```kotlin
data class WebImageAsset(
    val url: String,
    val width: Int,
    val height: Int,
    val preload: Boolean
)
```

Не грузите desktop-quality images для маленьких cards.

## Current constraints

Будьте готовы к:

- beta-level API changes;
- отличиям в text input;
- отличиям в accessibility;
- ограничениям browser APIs;
- differences between Chrome/Firefox/Safari;
- tooling gaps;
- larger startup payload than DOM-first apps;
- need for browser-specific QA.

Это не делает target плохим. Это означает: rollout должен быть осознанным.

## Deployment

Web deployment требует:

- build output;
- static hosting;
- correct MIME types for Wasm;
- caching headers;
- compression;
- HTTPS;
- fallback route for SPA;
- source maps policy;
- browser compatibility checks.

```text
build/dist/wasmJs/productionExecutable/
  index.html
  app.js
  app.wasm
  resources/
```

CI должен публиковать artifact и запускать smoke test в браузере.

## Testing

Тестовая стратегия:

- common unit tests;
- screenshot tests for shared components;
- browser smoke tests;
- keyboard/focus tests;
- route/deep link tests;
- performance startup budget;
- accessibility manual checks;
- resource loading failure checks.

```kotlin
data class WebSmokeCheck(
    val path: String,
    val expectedText: String,
    val maxStartupMillis: Long
)
```

Web target требует web-like QA, а не только Android Compose tests.

## Типичные ошибки

### Считать Web/Wasm stable как Android

Статус target другой, значит rollout risk другой.

### Игнорировать URL/history

Web app без back/deep links ощущается чужеродно.

### Тащить Android APIs в common UI

Web target сразу ломает такие assumptions.

### Не думать о bundle size

Startup в браузере начинается с download.

### Считать accessibility автоматической

Проверяйте screen readers и keyboard в браузере.

### Использовать Wasm Compose для SEO-first страниц

Для public content может быть лучше DOM/server-rendered подход.

## Итоги

Kotlin/Wasm Compose переносит shared Compose UI в браузер:

- Web/Wasm target находится в beta для Compose Multiplatform;
- Gradle target обычно `wasmJs { browser(); binaries.executable() }`;
- browser entry point отличается от Android/Desktop;
- URL/history/resources/accessibility требуют web-specific shell;
- resources нужно оптимизировать и preload-ить для первого экрана;
- browser APIs держите за adapter boundary;
- startup cost зависит от bundle/assets/runtime initialization;
- deployment требует корректных Wasm MIME/cache/HTTPS настроек;
- testing должно включать browser smoke, accessibility и performance checks;
- Web/Wasm полезен, но требует отдельной product/engineering оценки.
