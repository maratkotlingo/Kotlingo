## Что такое побочный эффект

Побочный эффект — это действие, которое меняет что-то за пределами текущей composable-функции или взаимодействует с внешним миром.

Примеры побочные эффекты:

- показать `Snackbar`;
- открыть другой экран;
- отправить событие аналитики;
- подписаться на `Lifecycle`;
- запустить coroutine;
- вызвать suspend-функцию;
- зарегистрировать обратный вызов внешнего SDK;
- записать свойство в объект, которым Compose не управляет;
- собрать `Flow`;
- загрузить данные через репозиторий.

Composable-функция в идеале должна быть чистой: получить параметры, описать UI и завершиться. Она может быть вызвана много раз, не выполнена из-за оптимизации, отменена, выполнена в другом порядке относительно соседних функций или выброшена до применения изменений. Поэтому прямое выполнение внешних действий в теле composable опасно.

Плохо:

```kotlin
import androidx.compose.material3.Text
import androidx.compose.runtime.Composable

interface AnalyticsLogger {
    fun logScreenOpened(screenName: String)
}

@Composable
fun ProfileScreenWrong(
    analyticsLogger: AnalyticsLogger
) {
    analyticsLogger.logScreenOpened("profile")

    Text(text = "Профиль")
}
```

Почему плохо:

- recomposition может вызвать `logScreenOpened` повторно;
- composition может быть отменена, а аналитика уже отправлена;
- порядок вызовов composable не является надежным контрактом для внешних действий;
- тесты и Предпросмотр получают скрытое поведение.

Правильнее:

```kotlin
import androidx.compose.material3.Text
import androidx.compose.runtime.Composable
import androidx.compose.runtime.LaunchedEffect

@Composable
fun ProfileScreen(
    analyticsLogger: AnalyticsLogger
) {
    LaunchedEffect(analyticsLogger) {
        analyticsLogger.logScreenOpened("profile")
    }

    Text(text = "Профиль")
}
```

`LaunchedEffect` привязывает побочный эффект к lifecycle конкретного места в composition.

<CatTip>
Главная мысль: UI описывается в composable, а внешняя работа запускается через API эффектов. Это делает поведение предсказуемым даже при recomposition.
</CatTip>

## Почему тело composable не место для эффектов

Compose runtime имеет право:

- повторно вызвать composable при изменении state;
- пропустить composable, если параметры стабильны и не изменились;
- начать recomposition и затем отменить ее;
- выполнить sibling composable в другом порядке;
- удалить часть дерева из composition;
- заново вставить часть дерева при изменении ключей.

Это делает прямые побочные эффекты в теле composable нестабильными.

```kotlin
import androidx.compose.material3.Text
import androidx.compose.runtime.Composable

interface Navigator {
    fun openHome()
}

@Composable
fun LoginResultWrong(
    loggedIn: Boolean,
    navigator: Navigator
) {
    if (loggedIn) {
        navigator.openHome()
    }

    Text(text = "Проверяем вход")
}
```

Если `loggedIn` остается `true`, recomposition может вызвать навигацию несколько раз. Правильнее обработать событие через controlled effect или route-level event stream.

```kotlin
import androidx.compose.material3.Text
import androidx.compose.runtime.Composable
import androidx.compose.runtime.LaunchedEffect

@Composable
fun LoginResult(
    loggedIn: Boolean,
    navigator: Navigator
) {
    if (loggedIn) {
        LaunchedEffect(loggedIn, navigator) {
            navigator.openHome()
        }
    }

    Text(text = "Проверяем вход")
}
```

Этот вариант запускает навигацию при входе соответствующей ветки в composition. В реальном приложении часто еще лучше собирать one-off navigation effects из ViewModel на route boundary.

## Effect API map

В Compose есть несколько effect APIs, и каждый решает свою задачу.

### LaunchedEffect

Запускает coroutine, привязанную к composition.

Используйте для:

- suspend-функций;
- `delay`;
- `SnackbarHostState.showSnackbar`;
- `animateScrollToItem`;
- сбора `Flow`, если это UI-specific collection;
- одноразового запуска при входе в composition;
- перезапуска работы при изменении ключей.

### rememberCoroutineScope

Возвращает `CoroutineScope`, привязанный к месту в composition.

Используйте для:

- запуска coroutine из event handler;
- ручного контроля нескольких jobs;
- snackbar по нажатию;
- scroll/animation после пользовательского действия.

### DisposableEffect

Запускает эффект с обязательной очисткой.

Используйте для:

- регистрации обратный вызов;
- подписки на listener;
- `LifecycleEventObserver`;
- receiver-like API;
- external SDK session;
- ресурсов, которые нужно снять в `onDispose`.

### SideEffect

Выполняет код после каждой успешной recomposition.

Используйте для:

- публикации Compose state в объект, которым Compose не управляет;
- обновления user properties аналитики;
- синхронизации маленького внешнего mutable holder после успешной composition.

### produceState

Преобразует внешний источник данных в Compose `State<T>`.

Используйте для:

- обратный вызов-based API;
- suspend-загрузки с returned `State`;
- адаптеров библиотек;
- локального bridge к внешнему источнику, если готовых extension API недостаточно.

### rememberUpdatedState

Сохраняет актуальную ссылку на значение или обратный вызов без перезапуска long-lived effect.

Используйте для:

- свежей лямбда внутри `LaunchedEffect(Unit)`;
- таймера, который не должен перезапускаться при смене обратный вызов;
- listener, который должен вызывать актуальную функцию.

### snapshotFlow

Преобразует snapshot reads в cold `Flow`.

Используйте для:

- аналитики по scroll state;
- Flow operators над Compose state;
- `distinctUntilChanged`, `debounce`, `filter`;
- побочные эффекты, которым нужен stream изменений snapshot state.

<MouseTrap>
Effect API не должен становиться местом для всей бизнес-логики. Если работа принадлежит экрану, данным или сценарий использования, чаще ее нужно поднять во ViewModel или domain layer, а effect оставить мостом между UI lifecycle и внешним действием.
</MouseTrap>

## Ключи API эффектов

Некоторые effects принимают keys:

- `LaunchedEffect(key1, key2)`;
- `DisposableEffect(key1, key2)`;
- `produceState(initialValue, key1, key2)`.

Ключи определяют restart behavior:

- effect входит в composition: запускается;
- key изменился: старый effect отменяется или очищается, затем запускается новый;
- effect выходит из composition: coroutine отменяется или вызывается `onDispose`.

```kotlin
import androidx.compose.runtime.Composable
import androidx.compose.runtime.LaunchedEffect

interface ArticleAnalytics {
    suspend fun trackArticleOpened(articleId: String)
}

@Composable
fun ArticleAnalyticsEffect(
    articleId: String,
    analytics: ArticleAnalytics
) {
    LaunchedEffect(articleId, analytics) {
        analytics.trackArticleOpened(articleId)
    }
}
```

Если `articleId` изменится, старый эффект отменится, а новый отправит событие для нового article.

## Что должно быть ключом

Общее правило: значения, которые используются внутри effect и должны менять lifecycle работы, должны быть ключами.

Нужно добавить ключ:

- id сущности, для которой выполняется загрузка;
- объект репозиторий, если он может замениться;
- `LifecycleOwner`, если listener регистрируется на нем;
- `NavBackStackEntry`, если effect привязан к нему;
- URL, если загрузка должна перезапуститься при смене URL.

Не нужно добавлять ключ:

- обратный вызов, если он должен обновиться без перезапуска эффекта и используется через `rememberUpdatedState`;
- stable object из `remember` без ключей;
- значение, которое не влияет на lifecycle работы;
- derived value, если исходные ключи уже покрывают lifecycle.

```kotlin
import androidx.compose.runtime.Composable
import androidx.compose.runtime.LaunchedEffect
import androidx.compose.runtime.getValue
import androidx.compose.runtime.rememberUpdatedState
import kotlinx.coroutines.delay

@Composable
fun SplashTimeout(
    onTimeout: () -> Unit
) {
    val latestOnTimeout by rememberUpdatedState(onTimeout)

    LaunchedEffect(Unit) {
        delay(1_200)
        latestOnTimeout()
    }
}
```

Здесь `onTimeout` не является ключом, потому что смена обратный вызов не должна перезапускать таймер. Но обратный вызов должен быть свежим, поэтому используется `rememberUpdatedState`.

## Константные ключи

Иногда effect должен жить ровно столько, сколько живет место вызова в composition. Тогда ключом может быть `Unit`.

```kotlin
import androidx.compose.runtime.Composable
import androidx.compose.runtime.LaunchedEffect

@Composable
fun ScreenEnteredEffect(
    onEntered: suspend () -> Unit
) {
    LaunchedEffect(Unit) {
        onEntered()
    }
}
```

Это валидно, но требует осторожности:

- effect не перезапустится при изменении параметров;
- long-lived operation может увидеть устаревшие значения;
- обратный вызов нужно обновлять через `rememberUpdatedState`, если он может измениться;
- иногда правильнее сделать ключом id экрана или сущности.

<MouseTrap>
`LaunchedEffect(Unit)` и `LaunchedEffect(true)` выглядят невинно, но они означают: "не перезапускать работу при recomposition". Всегда проверяйте, не захватили ли вы внутри устаревшие параметры.
</MouseTrap>

## Effect после успешной composition

Важная деталь: effect должен запускаться только тогда, когда composition успешно применена. Это особенно заметно с `SideEffect`, который выполняется после успешной recomposition.

```kotlin
import androidx.compose.runtime.Composable
import androidx.compose.runtime.SideEffect

class ExternalScreenState {
    var currentTitle: String = ""
}

@Composable
fun PublishTitle(
    title: String,
    externalState: ExternalScreenState
) {
    SideEffect {
        externalState.currentTitle = title
    }
}
```

Если recomposition была отменена, внешний state не должен получить значение из UI, который не был применен.

## Отмена effects

Cancellation — обычная часть работы Compose effects.

`LaunchedEffect`:

- запускает coroutine;
- coroutine отменяется при выходе из composition;
- coroutine отменяется при смене ключей;
- `finally`-блок в coroutine выполнится, если он корректно написан.

`DisposableEffect`:

- выполняет настройка;
- вызывает `onDispose` при выходе из composition;
- вызывает `onDispose` перед restart при смене ключей.

`produceState`:

- запускает producer coroutine;
- отменяет producer при выходе;
- отменяет и перезапускает producer при смене ключей;
- может использовать `awaitDispose` для cleanup обратный вызов-based source.

```kotlin
import androidx.compose.runtime.Composable
import androidx.compose.runtime.LaunchedEffect
import kotlinx.coroutines.delay
import kotlinx.coroutines.isActive
import kotlin.coroutines.coroutineContext

@Composable
fun HeartbeatEffect(
    screenName: String,
    analytics: AnalyticsLogger
) {
    LaunchedEffect(screenName, analytics) {
        while (coroutineContext.isActive) {
            analytics.logScreenOpened(screenName)
            delay(30_000)
        }
    }
}
```

Этот effect будет остановлен, когда соответствующий экран уйдет из composition.

## Unidirectional data flow и effects

Effects не должны ломать однонаправленный поток данных.

Хорошая схема:

1. UI показывает state.
2. Пользователь делает действие.
3. UI вызывает обратный вызов.
4. ViewModel обрабатывает event.
5. ViewModel обновляет state или emits effect.
6. Route собирает state/effect и вызывает UI-side API.

```kotlin
import androidx.compose.material3.SnackbarHostState
import androidx.compose.runtime.Composable
import androidx.compose.runtime.LaunchedEffect
import androidx.compose.runtime.getValue
import androidx.compose.runtime.remember
import androidx.lifecycle.compose.collectAsStateWithLifecycle
import kotlinx.coroutines.flow.SharedFlow
import kotlinx.coroutines.flow.StateFlow

data class SaveProfileUiState(
    val name: String,
    val saving: Boolean
)

sealed interface SaveProfileEffect {
    data class ShowSnackbar(
        val message: String
    ) : SaveProfileEffect
}

@Composable
fun SaveProfileRoute(
    stateFlow: StateFlow<SaveProfileUiState>,
    effects: SharedFlow<SaveProfileEffect>,
    onNameChange: (String) -> Unit,
    onSaveClick: () -> Unit
) {
    val state by stateFlow.collectAsStateWithLifecycle()
    val snackbarHostState = remember {
        SnackbarHostState()
    }

    LaunchedEffect(effects, snackbarHostState) {
        effects.collect { effect ->
            when (effect) {
                is SaveProfileEffect.ShowSnackbar -> {
                    snackbarHostState.showSnackbar(effect.message)
                }
            }
        }
    }

    SaveProfileScreen(
        state = state,
        onNameChange = onNameChange,
        onSaveClick = onSaveClick
    )
}
```

`SaveProfileScreen` остается stateless, а route выполняет побочный эффект.

```kotlin
import androidx.compose.foundation.layout.Column
import androidx.compose.material3.Button
import androidx.compose.material3.OutlinedTextField
import androidx.compose.material3.Text
import androidx.compose.runtime.Composable

@Composable
fun SaveProfileScreen(
    state: SaveProfileUiState,
    onNameChange: (String) -> Unit,
    onSaveClick: () -> Unit
) {
    Column {
        OutlinedTextField(
            value = state.name,
            onValueChange = { nextName ->
                onNameChange(nextName)
            },
            label = {
                Text(text = "Имя")
            }
        )

        Button(
            enabled = !state.saving,
            onClick = onSaveClick
        ) {
            Text(
                text = if (state.saving) {
                    "Сохраняем"
                } else {
                    "Сохранить"
                }
            )
        }
    }
}
```

## Когда effect не нужен

Effect не нужен, если задача решается декларативным UI.

Не нужен effect:

- чтобы показать `Text` при ошибке;
- чтобы переключить видимость прогресс;
- чтобы отфильтровать список для отображения;
- чтобы вычислить `enabled`;
- чтобы вызвать обратный вызов из `Button(onClick)`;
- чтобы сохранить локальный `remember` state при вводе.

```kotlin
import androidx.compose.material3.Text
import androidx.compose.runtime.Composable

@Composable
fun ErrorMessage(
    message: String?
) {
    if (message != null) {
        Text(text = message)
    }
}
```

Здесь не нужен `LaunchedEffect`, потому что UI просто описывает состояние.

## Типичные ошибки

### Запускать работу прямо в теле composable

Такой код может выполниться много раз или в неподходящий момент.

### Использовать effect для обычного отображения state

Если можно просто написать `if (loading)`, effect не нужен.

### Неверно выбирать ключи

Слишком мало ключей дает устаревшие данные. Слишком много ключей перезапускает работу без нужды.

### Игнорировать cancellation

Coroutine внутри `LaunchedEffect` должна корректно реагировать на cancellation. Не блокируйте поток долгой синхронной работой.

### Делать пустой cleanup

Пустой `onDispose` обычно означает, что выбран не тот API эффектов.

## Итоги

Побочные эффекты — это контролируемый мост между Compose и внешним миром:

- composable-функции должны оставаться чистыми;
- прямые внешние действия в теле composable опасны;
- `LaunchedEffect` запускает coroutine;
- `rememberCoroutineScope` запускает coroutine из event handlers;
- `DisposableEffect` регистрирует и очищает внешние ресурсы;
- `SideEffect` публикует Compose state после успешной recomposition;
- `produceState` превращает внешний источник в `State<T>`;
- ключи управляют restart behavior;
- `rememberUpdatedState` помогает избежать устаревших обратный вызов без restart;
- effects должны поддерживать unidirectional data flow, а не заменять архитектуру.
