## Главная идея

Compose проще всего понять так: экран — это результат текущего состояния.

```kotlin
@Composable
fun Greeting(name: String) {
    Text(text = "Привет, $name")
}
```

Если `name` равен `"Анна"`, на экране будет текст `"Привет, Анна"`. Если
`name` изменился на `"Илья"`, Compose снова выполнит нужный кусок UI-кода и
покажет новый текст.

В старом View-подходе ты часто писал: найди `TextView`, вызови `setText`,
спрячь один блок, покажи другой. В Compose ты описываешь результат:
если состояние такое, экран выглядит так.

<CatTip title="Формула на старте">
  State вошел в composable, UI вышел наружу. Не думай о ручном обновлении
  каждой кнопки и строки. Думай о том, какие данные нужны экрану прямо сейчас.
</CatTip>

## Что такое recomposition

Recomposition — это повторное выполнение части composable-кода после изменения
состояния.

```kotlin
import androidx.compose.foundation.layout.Column
import androidx.compose.material3.Button
import androidx.compose.material3.Text
import androidx.compose.runtime.Composable
import androidx.compose.runtime.getValue
import androidx.compose.runtime.mutableIntStateOf
import androidx.compose.runtime.remember
import androidx.compose.runtime.setValue

@Composable
fun CounterScreen() {
    var count by remember { mutableIntStateOf(0) }

    Column {
        Text(text = "Нажатий: $count")

        Button(
            onClick = {
                count += 1
            }
        ) {
            Text(text = "Увеличить")
        }
    }
}
```

Что происходит при клике:

1. `count += 1` меняет observable state.
2. Compose понимает, что UI, который читал `count`, надо пересчитать.
3. `CounterScreen` или его часть выполняется снова.
4. `Text` получает новую строку.
5. Compose применяет изменения на экране.

Тебе не нужно вручную искать текст и обновлять его. Тебе нужно правильно хранить
состояние.

## Почему обычная `var` не работает как состояние

Локальная переменная внутри composable пересоздается при новом выполнении
функции. Она не сообщает Compose, что экран надо обновить.

```kotlin
@Composable
fun BrokenCounterScreen() {
    var count = 0

    Button(
        onClick = {
            count += 1
        }
    ) {
        Text(text = "Нажатий: $count")
    }
}
```

При клике значение внутри lambda изменится, но Compose не узнает, что экран
надо перерисовать. Для UI-состояния используй `remember`, `rememberSaveable`,
состояние из ViewModel или `StateFlow`, собранный в Compose.

<MouseTrap title="Ловушка: переменная не равна состоянию экрана">
  Если значение должно менять UI, оно должно быть observable для Compose.
  Обычная локальная `var` подходит для временного вычисления, но не для состояния
  кнопки, поля ввода или выбранной строки.
</MouseTrap>

## Что делает `remember`

`remember` хранит значение между recomposition, пока этот composable остается на
экране.

```kotlin
import androidx.compose.material3.OutlinedTextField
import androidx.compose.material3.Text
import androidx.compose.runtime.Composable
import androidx.compose.runtime.getValue
import androidx.compose.runtime.mutableStateOf
import androidx.compose.runtime.remember
import androidx.compose.runtime.setValue

@Composable
fun SearchField() {
    var query by remember { mutableStateOf("") }

    OutlinedTextField(
        value = query,
        onValueChange = { newValue ->
            query = newValue
        },
        label = {
            Text(text = "Поиск")
        }
    )
}
```

Без `remember` значение `query` сбрасывалось бы при каждом новом выполнении
функции. С `remember` Compose сохраняет его, пока поле ввода находится в
composition.

Важно: `remember` не переживает пересоздание Activity, например поворот экрана.
Для простых полей ввода часто используют `rememberSaveable`, а для состояния
экрана в реальном приложении — ViewModel.

## Состояние лучше поднимать наверх

Если компонент сам хранит все внутри себя, его сложнее тестировать и
переиспользовать. Часто лучше передать значение и событие снаружи.

```kotlin
@Composable
fun LikeButton(
    liked: Boolean,
    onLikedChange: (Boolean) -> Unit
) {
    Button(
        onClick = {
            onLikedChange(!liked)
        }
    ) {
        Text(
            text = if (liked) {
                "Уже нравится"
            } else {
                "Нравится"
            }
        )
    }
}
```

Такой компонент не решает сам, где хранить `liked`. Он просто показывает
состояние и сообщает о клике. Это называется state hoisting, и это один из самых
важных навыков в Compose.

## Побочные действия не пишут прямо в теле composable

Composable-функция может выполниться много раз. Поэтому внутри обычного тела
нельзя напрямую отправлять аналитику, запускать навигацию, делать сетевой запрос
или регистрировать listener.

Плохо:

```kotlin
@Composable
fun ProfileScreen(userId: String, analytics: Analytics) {
    analytics.trackScreenOpened(userId)
    Text(text = "Профиль")
}
```

Лучше:

```kotlin
import androidx.compose.runtime.Composable
import androidx.compose.runtime.LaunchedEffect

@Composable
fun ProfileScreen(userId: String, analytics: Analytics) {
    LaunchedEffect(userId) {
        analytics.trackScreenOpened(userId)
    }

    Text(text = "Профиль")
}
```

`LaunchedEffect(userId)` запускается после попадания composable в composition и
перезапускается только при изменении ключа `userId`.

## Что знать про внутренности Compose

У Compose есть внутренние структуры, которые помогают помнить состояние и
обновлять UI эффективно. Новичку не нужно глубоко изучать Slot Table, группы,
skipping и другие детали runtime.

Практических правил достаточно:

- состояние должно быть observable;
- composable-функции должны быть быстрыми;
- не делай побочные действия в обычном теле composable;
- для списков используй стабильные `key`, если строки могут переставляться;
- выноси состояние наверх, когда компонент должен быть переиспользуемым;
- тяжелую работу делай вне composition или заворачивай в подходящий state/effect.

## Итоги

После этого урока нужно унести простую модель:

- UI в Compose зависит от текущего состояния;
- recomposition пересчитывает нужные части UI после изменения состояния;
- `remember` хранит локальное состояние между recomposition;
- обычная `var` внутри composable не является состоянием экрана;
- state hoisting делает компоненты проще и чище;
- побочные действия живут в `LaunchedEffect`, `DisposableEffect` и других effects;
- внутренности runtime можно изучить позже, когда базовые экраны уже пишутся уверенно.
