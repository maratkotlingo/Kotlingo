## Почему XR — это не просто еще один большой экран

Android XR переносит Android UI в spatial environment. Пользователь может видеть 2D panels, spatial panels, 3D content, hands/controllers, gaze-like targeting и surfaces вокруг себя. Обычный Compose UI остается полезным, но рабочий процесс меняется:

- UI может находиться не только внутри прямоугольного Activity window;
- расстояние до пользователя влияет на readable size;
- depth и motion могут утомлять;
- input может быть hand/controller/gaze/voice;
- pointer precision отличается от touch;
- controls должны быть крупнее и стабильнее;
- пользователь может одновременно видеть несколько panels;
- 2D app может быть embedded в spatial shell;
- API остается experimental и требует осторожной архитектуры.

```kotlin
data class XrUiMode(
    val spatialEnabled: Boolean,
    val comfortableDistanceMeters: Float,
    val inputKind: XrInputKind
)

enum class XrInputKind {
    TouchLike,
    Controller,
    HandTracking,
    Mixed
}
```

Такой state не обязан точно повторять platform API. Его задача — изолировать app UI от быстрых изменений experimental XR libraries.

<CatTip>
XR-интерфейс должен быть удобным для тела, а не только красивым для скриншота. Размер, расстояние, motion и положение controls здесь становятся частью accessibility.
</CatTip>

## Dependencies и experimental status

Android XR Compose APIs поставляются отдельными Jetpack XR артефакты и могут требовать предпросмотр SDK, emulator/device image и experimental opt-in.

```kotlin
dependencies {
    implementation("androidx.xr.compose:compose:1.0.0-alpha01")
    implementation("androidx.xr.scenecore:scenecore:1.0.0-alpha01")
}
```

Версии здесь иллюстративные. Для реального проекта:

- проверяйте Android XR release notes;
- не прячьте experimental APIs глубоко в common UI components;
- держите XR integration в отдельном module/package;
- создавайте fallback для обычного 2D mode;
- покрывайте platform mapping тестами.

<MouseTrap>
Experimental XR API не должен протекать в каждый composable приложения. Иначе обновление предпросмотр SDK превратится в правку всего UI.
</MouseTrap>

## 2D panel-first стратегия

Безопасный старт для существующего Compose приложения — 2D panel-first:

- обычные screen composables остаются 2D;
- XR shell решает, где panel живет в spatial environment;
- app state и ViewModel не знают про spatial position;
- XR-only controls добавляются вокруг panel, а не внутрь каждого screen.

```kotlin
@Composable
fun XrAppRoot(
    xrMode: XrUiMode,
    appState: AppState
) {
    if (xrMode.spatialEnabled) {
        SpatialAppShell(
            appState = appState,
            xrMode = xrMode
        )
    } else {
        FlatAppShell(
            appState = appState
        )
    }
}
```

2D fallback обязателен: XR availability, emulator/device support и user comfort не должны быть единственным путем к feature.

## Subspace mental model

XR Compose добавляет spatial composition concepts, часто называемые Subspace. Внутри обычного Compose вы описываете 2D UI; внутри spatial layer вы размещаете panels и spatial elements.

Псевдо-структура:

```kotlin
import androidx.compose.runtime.Composable

@Composable
fun SpatialAppShell(
    appState: AppState,
    xrMode: XrUiMode
) {
    Subspace {
        SpatialPanel(
            distanceMeters = xrMode.comfortableDistanceMeters
        ) {
            MainWorkspacePanel(
                appState = appState
            )
        }

        OrbitalControlsPanel(
            appState = appState
        )
    }
}
```

Конкретные names (`Subspace`, `SpatialPanel`, modifiers) зависят от версии Jetpack XR. Стабильная идея: spatial shell размещает 2D/3D surfaces, а обычные screen composables остаются максимально platform-independent.

## Panels

Panel — это прямоугольная поверхность в spatial environment. Для обычного app UI panel часто является главным контейнером.

```kotlin
data class XrPanelSpec(
    val id: String,
    val title: String,
    val widthDp: Int,
    val heightDp: Int,
    val distanceMeters: Float
)
```

Panel sizing должен учитывать:

- readable text size;
- distance;
- поле зрения;
- user posture;
- controller precision;
- количество контента;
- motion comfort;
- hand reach comfort.

```kotlin
fun defaultWorkspacePanelSpec(
    xrMode: XrUiMode
): XrPanelSpec {
    return XrPanelSpec(
        id = "workspace",
        title = "Рабочая область",
        widthDp = 960,
        heightDp = 640,
        distanceMeters = xrMode.comfortableDistanceMeters
    )
}
```

Не делайте XR panel размером с phone screen. В spatial UI у пользователя больше пространства, но не бесконечное внимание.

## Orbiter

Orbiter-like controls — это controls, прикрепленные рядом с panel или к ее краю. Они полезны для secondary actions, которые не должны занимать основной content.

```kotlin
@Composable
fun WorkspaceWithOrbiter(
    state: WorkspaceState,
    onClose: () -> Unit,
    onPin: () -> Unit
) {
    SpatialPanel {
        WorkspaceContent(state = state)
    }

    Orbiter {
        XrIconButton(
            label = "Закрепить",
            onClick = onPin
        )
        XrIconButton(
            label = "Закрыть",
            onClick = onClose
        )
    }
}
```

Не выносите основное действие в orbiter, если пользователь должен выполнить его часто. Primary рабочий процесс должен оставаться рядом с content и быть obvious.

## Spatial layout thinking

В XR нельзя просто добавить third dimension ко всему. Начните с вопросов:

- что является main content;
- какие controls должны быть attached;
- какие surfaces можно отложить;
- что должно быть visible одновременно;
- сколько panels пользователь реально может обработать;
- где безопасная зона внимания;
- может ли motion вызвать discomfort;
- что произойдет при возврате в 2D mode.

```kotlin
sealed interface WorkspaceSurface {
    data object MainEditor : WorkspaceSurface
    data object Inspector : WorkspaceSurface
    data object Timeline : WorkspaceSurface
    data object Help : WorkspaceSurface
}

data class SpatialWorkspaceState(
    val visibleSurfaces: Set<WorkspaceSurface>,
    val focusedSurface: WorkspaceSurface
)
```

Spatial state не должен быть набором случайных booleans. Пользовательская рабочая область нуждается в явной модели.

## Input differences

XR input может отличаться от touch:

- hand pinch не равен tap;
- controller ray имеет hover-like targeting;
- gaze targeting требует dwell/confirmation;
- text input может быть voice/virtual keyboard;
- drag в воздухе утомляет;
- small controls сложнее нажимать;
- hover/target states важнее.

```kotlin
data class XrActionTarget(
    val label: String,
    val minSizeDp: Int = 56,
    val requiresConfirmation: Boolean = false
)
```

Для destructive actions полезно требовать confirmation:

```kotlin
@Composable
fun XrDeleteAction(
    onDelete: () -> Unit
) {
    ConfirmingActionButton(
        label = "Удалить",
        confirmationLabel = "Подтвердить удаление",
        onConfirm = onDelete
    )
}
```

XR controls лучше делать крупнее, спокойнее и стабильнее, чем phone controls.

## Text и readability

Text в XR зависит от distance. Если panel далеко, обычный phone `bodyMedium` может быть нечитаем.

```kotlin
fun typographyScaleForDistance(
    distanceMeters: Float
): Float {
    return when {
        distanceMeters < 1.0f -> 1.0f
        distanceMeters < 1.8f -> 1.15f
        else -> 1.3f
    }
}
```

Compose theme adapter:

```kotlin
@Composable
fun XrReadableTheme(
    distanceMeters: Float,
    content: @Composable () -> Unit
) {
    val scale = typographyScaleForDistance(distanceMeters)

    AppTheme(
        typographyScale = scale,
        content = content
    )
}
```

Не масштабируйте все blindly. Icon sizes, layout density и line lengths тоже требуют проверки.

## Motion comfort

Motion в XR должен быть осторожным:

- избегайте резких перемещений panels;
- не анимируйте depth без необходимости;
- не вращайте UI вокруг пользователя;
- не используйте постоянные parallax effects для текста;
- делайте transitions короткими и предсказуемыми;
- давайте reduce motion path.

```kotlin
data class XrMotionSettings(
    val reduceMotion: Boolean,
    val allowSpatialTransitions: Boolean
)
```

```kotlin
fun spatialTransitionDurationMillis(
    settings: XrMotionSettings
): Int {
    return if (settings.reduceMotion) {
        0
    } else {
        180
    }
}
```

Motion comfort — это не вкусовщина, а часть безопасного XR UX.

## Embedded surfaces

Embedded surfaces — это случаи, когда Android/Compose UI встраивается в другую среду:

- XR panel;
- car/embedded device surface;
- kiosk;
- secondary display;
- пользовательский shell;
- system-controlled container.

Общее правило: screen не должен предполагать, что он full-screen phone Activity.

```kotlin
data class EmbeddedSurfaceConstraints(
    val maxWidthDp: Int,
    val maxHeightDp: Int,
    val inputKind: EmbeddedInputKind,
    val canOpenSystemDialogs: Boolean
)

enum class EmbeddedInputKind {
    Touch,
    Rotary,
    Dpad,
    Pointer,
    Spatial
}
```

UI:

```kotlin
@Composable
fun EmbeddedAwareScreen(
    constraints: EmbeddedSurfaceConstraints,
    state: ScreenState
) {
    when (constraints.inputKind) {
        EmbeddedInputKind.Dpad -> DpadScreen(state)
        EmbeddedInputKind.Rotary -> RotaryScreen(state)
        EmbeddedInputKind.Spatial -> SpatialPanelScreen(state)
        else -> TouchScreen(state)
    }
}
```

Это не значит, что нужно писать четыре разных приложения. Нужны разные interaction shells вокруг общей state/model.

## Lifecycle и resources

XR/embedded shell может иметь lifecycle, отличающийся от phone Activity assumptions:

- panel может быть скрыта, но app session живет;
- spatial surface может быть recreated;
- input device может измениться;
- Activity может перейти в background;
- 3D resources нужно release-ить;
- sensors/controllers требуют cleanup.

```kotlin
@Composable
fun SpatialResourceHost(
    enabled: Boolean,
    factory: () -> SpatialRenderer
) {
    val renderer = remember {
        factory()
    }

    DisposableEffect(renderer) {
        onDispose {
            renderer.release()
        }
    }

    LaunchedEffect(enabled) {
        if (enabled) {
            renderer.resume()
        } else {
            renderer.pause()
        }
    }
}
```

External XR resources должны иметь явные `pause/resume/release`, а не жить "пока composable виден где-то".

## Architecture boundary

Рекомендуемая структура:

- `core-ui`: обычные Compose components;
- `feature-*`: state, routes, screens без XR imports;
- `xr-shell`: spatial panels, Subspace, Orbiter, XR-specific modifiers;
- `embedded-shell`: kiosk/car/secondary-display adaptations;
- `platform-xr`: mapping XR sessions/input/features в app model.

```kotlin
interface WorkspaceShell {
    @Composable
    fun Content(
        state: WorkspaceState,
        onEvent: (WorkspaceEvent) -> Unit
    )
}

class FlatWorkspaceShell : WorkspaceShell {

    @Composable
    override fun Content(
        state: WorkspaceState,
        onEvent: (WorkspaceEvent) -> Unit
    ) {
        FlatWorkspaceScreen(
            state = state,
            onEvent = onEvent
        )
    }
}
```

XR shell:

```kotlin
class XrWorkspaceShell : WorkspaceShell {

    @Composable
    override fun Content(
        state: WorkspaceState,
        onEvent: (WorkspaceEvent) -> Unit
    ) {
        SpatialWorkspaceScreen(
            state = state,
            onEvent = onEvent
        )
    }
}
```

Так feature state остается стабильным, а spatial implementation можно менять вместе с SDK.

## Тестирование XR и встроенного пользовательского интерфейса

Что тестировать:

- 2D fallback;
- spatial shell mapping;
- panel size decisions;
- input mode differences;
- reduce motion settings;
- focus/target states;
- читаемость текста на ожидаемом расстоянии;
- resource cleanup;
- непрерывность состояния, когда панель скрывается/отображается;
- emulator/device behavior;
- accessibility alternatives.

Unit-тест для decision logic:

```kotlin
import com.google.common.truth.Truth.assertThat
import org.junit.Test

class XrPanelSpecTest {

    @Test
    fun workspacePanel_usesComfortableDistance() {
        val spec = defaultWorkspacePanelSpec(
            XrUiMode(
                spatialEnabled = true,
                comfortableDistanceMeters = 1.4f,
                inputKind = XrInputKind.Controller
            )
        )

        assertThat(spec.distanceMeters)
            .isEqualTo(1.4f)
    }
}
```

Большую часть spatial UX нельзя доказать обычным Compose UI test. Нужны device/emulator checks и design review в headset.

## Типичные ошибки

### Тащить XR API во все composables

SDK меняется, и весь UI становится experimental.

### Делать слишком много panels

Spatial clutter утомляет быстрее, чем обычный dashboard.

### Использовать phone-sized controls

Hand/controller targeting требует больших hit areas.

### Игнорировать 2D fallback

XR availability не должна блокировать core feature.

### Анимировать depth ради эффекта

Motion discomfort дороже визуального wow-эффекта.

### Считать embedded surface полноценной Activity

Container может иметь другие input, lifecycle и constraints.

## Итоги

Compose для Android XR и embedded surfaces требует отдельной boundary:

- XR не является просто large screen;
- 2D panel-first стратегия безопасна для существующих Compose приложений;
- spatial shell должен жить отдельно от feature screens;
- panels, orbiters и spatial layouts требуют явной state model;
- input отличается от touch: hand/controller/gaze требуют больших targets и обратная связь;
- text readability зависит от distance;
- motion comfort важнее эффектности;
- embedded surfaces имеют свои constraints/input/lifecycle;
- experimental XR APIs нельзя распространять по всему UI;
- testing требует не только unit/UI tests, но и headset/emulator review.
