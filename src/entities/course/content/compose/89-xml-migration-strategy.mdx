## Почему нужна стратегия, а не просто переписывание

Миграция с XML/View на Compose — это не только синтаксическая замена `TextView` на `Text`. У старого UI есть:

- navigation graph;
- Fragment lifecycle;
- XML themes;
- styles;
- пользовательский Views;
- RecyclerView adapters;
- ViewBinding/DataBinding;
- Espresso tests;
- accessibility behavior;
- аналитика hooks;
- feature flags;
- release process;
- команда, которая привыкла к старой модели.

Если начать переписывать экраны без стратегии, можно быстро получить два UI-фреймворка, две дизайн-системы, два способа навигации и непонятный источник state. Хорошая миграция уменьшает риск постепенно.

```kotlin
data class MigrationCandidate(
    val screenName: String,
    val userTraffic: Int,
    val uiComplexity: Int,
    val businessRisk: Int,
    val hasCustomViews: Boolean,
    val hasExistingComposePieces: Boolean
) {
    val migrationScore: Int
        get() = uiComplexity + businessRisk + if (hasCustomViews) 3 else 0
}
```

Даже такая грубая модель помогает не начинать с самого опасного экрана только потому, что он раздражает разработчиков.

<CatTip>
Хорошая миграция выглядит скучно: маленькие шаги, ясные границы, много проверок. Зато пользователи не замечают, что под экраном поменялся фундамент.
</CatTip>

## Основные paths миграции

Есть несколько способов вводить Compose:

- новые экраны сразу на Compose;
- новые компоненты внутри существующих XML screens через `ComposeView`;
- legacy Views внутри Compose через `AndroidView`;
- screen-by-screen переписывание;
- bottom-up миграция отдельных blocks;
- top-down миграция shell/navigation;
- полная замена feature module.

Обычно лучший путь комбинированный:

- новые features пишутся на Compose;
- дизайн-система получает Compose implementation;
- простые старые экраны мигрируются первыми;
- сложные экраны получают interop bridge;
- navigation переводится после того, как screen-level composables готовы.

## Design system first

Если в приложении есть дизайн-система, начните не с экрана, а с базовых компонентов и tokens:

- colors;
- typography;
- spacing;
- shapes;
- icons;
- buttons;
- text fields;
- панели приложения;
- list items;
- cards;
- empty/error/загрузка states.

Без этого каждый мигрированный экран создаст свою маленькую дизайн-систему.

```kotlin
import androidx.compose.material3.MaterialTheme
import androidx.compose.material3.Shapes
import androidx.compose.material3.Typography
import androidx.compose.material3.lightColorScheme
import androidx.compose.runtime.Composable

@Composable
fun KotlingoTheme(
    content: @Composable () -> Unit
) {
    MaterialTheme(
        colorScheme = lightColorScheme(
            primary = AppColors.Primary,
            secondary = AppColors.Secondary,
            surface = AppColors.Surface
        ),
        typography = Typography(),
        shapes = Shapes(),
        content = content
    )
}
```

В реальном проекте `AppColors` должны приходить из общих design tokens, а не быть случайным набором Compose-only цветов.

## XML theme bridge

Пока приложение смешанное, Compose theme и XML theme должны совпадать визуально.

```xml
<style name="Theme.App" parent="Theme.Material3.DayNight.NoActionBar">
    <item name="colorPrimary">@color/brand_primary</item>
    <item name="colorSecondary">@color/brand_secondary</item>
    <item name="android:fontFamily">@font/inter</item>
</style>
```

Compose side:

```kotlin
object AppColors {
    val Primary = androidx.compose.ui.graphics.Color(0xFF006C4F)
    val Secondary = androidx.compose.ui.graphics.Color(0xFF52635B)
    val Surface = androidx.compose.ui.graphics.Color(0xFFFBFDF8)
}
```

Лучше генерировать оба набора из одного источника tokens или держать явную таблицу соответствий.

<MouseTrap>
Если XML и Compose используют разные primary color, разные corner radius и разные text sizes, пользователь видит не миграцию, а визуальный разлом.
</MouseTrap>

## Начинать с простых экранов

Хорошие первые кандидаты:

- static settings screen;
- about screen;
- empty state;
- confirmation screen;
- onboarding step;
- simple form без сложного пользовательский input;
- feature flag screen;
- экран, уже содержащий Compose block.

Плохие первые кандидаты:

- главный денежный flow;
- сложный экран с nested fragments;
- экран с кастомным drawing View;
- экран с тяжелым RecyclerView и drag-and-drop;
- экран, где нет тестов;
- экран с большим количеством hidden business rules.

## Screen-by-screen migration

Screen-by-screen подход переводит весь экран на Compose, сохраняя внешние contracts:

- тот же route/deep link;
- тот же ViewModel или держатель состояния;
- те же аналитика events;
- те же permissions;
- те же UI states;
- те же accessibility labels;
- те же тестовые сценарии.

Пример старой Fragment boundary:

```kotlin
class OrdersFragment : Fragment(R.layout.orders_fragment) {

    private val viewModel: OrdersViewModel by viewModels()

    override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
        val binding = OrdersFragmentBinding.bind(view)

        binding.retryButton.setOnClickListener {
            viewModel.retry()
        }

        viewLifecycleOwner.lifecycleScope.launch {
            viewModel.state.collect { state ->
                binding.progress.isVisible = state.loading
                binding.errorText.isVisible = state.error != null
                binding.errorText.text = state.error
            }
        }
    }
}
```

Compose boundary:

```kotlin
class OrdersFragment : Fragment() {

    private val viewModel: OrdersViewModel by viewModels()

    override fun onCreateView(
        inflater: LayoutInflater,
        container: ViewGroup?,
        savedInstanceState: Bundle?
    ): View {
        return ComposeView(requireContext()).apply {
            id = R.id.orders_compose_root
            setViewCompositionStrategy(
                ViewCompositionStrategy.DisposeOnViewTreeLifecycleDestroyed
            )
            setContent {
                OrdersRoute(
                    viewModel = viewModel
                )
            }
        }
    }
}
```

Route:

```kotlin
import androidx.compose.runtime.Composable
import androidx.lifecycle.compose.collectAsStateWithLifecycle

@Composable
fun OrdersRoute(
    viewModel: OrdersViewModel
) {
    val state = viewModel.state.collectAsStateWithLifecycle()

    OrdersScreen(
        state = state.value,
        onRetry = viewModel::retry,
        onOrderClick = viewModel::openOrder
    )
}
```

UI:

```kotlin
import androidx.compose.foundation.layout.Box
import androidx.compose.foundation.layout.fillMaxSize
import androidx.compose.material3.Button
import androidx.compose.material3.CircularProgressIndicator
import androidx.compose.material3.Text
import androidx.compose.runtime.Composable
import androidx.compose.ui.Alignment
import androidx.compose.ui.Modifier

@Composable
fun OrdersScreen(
    state: OrdersState,
    onRetry: () -> Unit,
    onOrderClick: (String) -> Unit
) {
    Box(modifier = Modifier.fillMaxSize()) {
        when {
            state.loading -> {
                CircularProgressIndicator(
                    modifier = Modifier.align(Alignment.Center)
                )
            }
            state.error != null -> {
                Button(
                    modifier = Modifier.align(Alignment.Center),
                    onClick = onRetry
                ) {
                    Text(text = "Повторить")
                }
            }
            else -> {
                OrdersList(
                    orders = state.orders,
                    onOrderClick = onOrderClick
                )
            }
        }
    }
}
```

Здесь ViewModel и domain behavior не переписываются. Меняется только rendering layer.

## Bottom-up migration

Bottom-up подход полезен, когда экран слишком большой. Сначала заменяют leaf components:

- price badge;
- product card;
- toolbar action;
- empty state;
- form field;
- item row.

XML screen остается, но внутри появляются `ComposeView`.

```xml
<androidx.compose.ui.platform.ComposeView
    android:id="@+id/compose_price_badge"
    android:layout_width="wrap_content"
    android:layout_height="wrap_content" />
```

```kotlin
binding.composePriceBadge.setViewCompositionStrategy(
    ViewCompositionStrategy.DisposeOnViewTreeLifecycleDestroyed
)
binding.composePriceBadge.setContent {
    AppTheme {
        PriceBadge(
            price = state.price,
            discount = state.discount
        )
    }
}
```

Плюс: маленький риск. Минус: можно надолго застрять в interop и получить много `ComposeView`.

## Top-down migration

Top-down подход начинает с Compose root/shell и оставляет старые Views внутри через `AndroidView`.

Он подходит, если:

- navigation уже переводится на Compose;
- экранный layout проще переписать целиком;
- legacy пользовательский View занимает маленькую область;
- старые Views имеют ясный lifecycle.

```kotlin
@Composable
fun ProductDetailsScreen(
    state: ProductDetailsState,
    onBack: () -> Unit
) {
    ProductDetailsScaffold(
        title = state.title,
        onBack = onBack
    ) {
        LegacyProductChart(
            points = state.priceHistory
        )
    }
}
```

Здесь Compose становится host, а legacy View превращается во временный island.

## Mixed navigation

На переходном этапе могут сосуществовать:

- Fragment Navigation component;
- Navigation Compose;
- Activity routes;
- deep links;
- modal flows;
- legacy диалогах.

Не пытайтесь мигрировать navigation одновременно со всеми экранами, если это увеличивает риск. Лучше определить boundary.

Fragment host вызывает Compose route:

```kotlin
@Composable
fun ProfileRoute(
    state: ProfileState,
    onOpenSettings: () -> Unit,
    onOpenAvatarPicker: () -> Unit
) {
    ProfileScreen(
        state = state,
        onSettingsClick = onOpenSettings,
        onAvatarClick = onOpenAvatarPicker
    )
}
```

Fragment связывает обратные вызовы:

```kotlin
composeView.setContent {
    AppTheme {
        ProfileRoute(
            state = currentState,
            onOpenSettings = {
                findNavController().navigate(
                    R.id.action_profile_to_settings
                )
            },
            onOpenAvatarPicker = {
                avatarPicker.launch("image/*")
            }
        )
    }
}
```

Когда navigation переедет в Compose, `ProfileScreen` не придется менять.

## State migration

Состояние лучше переносить отдельно от UI:

- сначала сделать явную `UiState`;
- затем подключить ее к старому UI;
- затем использовать ту же `UiState` в Compose;
- потом удалить старые bindings.

```kotlin
data class CheckoutUiState(
    val items: List<CartItemUi>,
    val deliveryAddress: String,
    val paymentMethod: String,
    val loading: Boolean,
    val errorMessage: String?
)
```

События:

```kotlin
sealed interface CheckoutEvent {
    data object Retry : CheckoutEvent
    data object Pay : CheckoutEvent
    data class RemoveItem(val id: String) : CheckoutEvent
    data class ChangeQuantity(
        val id: String,
        val quantity: Int
    ) : CheckoutEvent
}
```

ViewModel:

```kotlin
fun onEvent(event: CheckoutEvent) {
    when (event) {
        CheckoutEvent.Pay -> pay()
        CheckoutEvent.Retry -> retry()
        is CheckoutEvent.RemoveItem -> removeItem(event.id)
        is CheckoutEvent.ChangeQuantity -> changeQuantity(
            id = event.id,
            quantity = event.quantity
        )
    }
}
```

Такой event model одинаково хорошо подходит View layer и Compose.

## RecyclerView migration

Для списков есть три пути:

- оставить `RecyclerView` и заменить отдельные ViewHolder на `ComposeView`;
- переписать item UI на Compose, но оставить adapter;
- заменить весь список на `LazyColumn`/`LazyVerticalGrid`.

Если список простой и уже имеет Compose items, лучше быстрее перейти к lazy layout. Если список использует сложные ItemTouchHelper, ConcatAdapter, paging, sticky behaviors и legacy decorations, миграция может быть постепенной.

Проверочный список:

- есть stable keys;
- scroll position сохраняется;
- empty/загрузка/error states совпадают;
- item animations не ломают UX;
- accessibility collection semantics не ухудшилась;
- performance проверена на release build;
- paging не создает лишние recompositions;
- все item click events сохранили аналитика.

## Testing during migration

Перед заменой экрана полезно зафиксировать behavior:

- screenshot/golden для основных состояний;
- accessibility checks;
- Espresso tests для старого экрана;
- Compose tests для нового экрана;
- contract tests для ViewModel;
- navigation tests для deep links;
- ручной exploratory проверочный список.

```kotlin
class OrdersViewModelTest {

    @Test
    fun retry_afterNetworkError_loadsOrdersAgain() = runTest {
        val repository = FakeOrdersRepository()
        val viewModel = OrdersViewModel(repository)

        repository.nextResult = Result.failure(
            IllegalStateException("Нет сети")
        )
        viewModel.load()

        repository.nextResult = Result.success(
            listOf(Order(id = "42"))
        )
        viewModel.retry()

        assertThat(viewModel.state.value.orders)
            .containsExactly(OrderUi(id = "42"))
    }
}
```

Если ViewModel contract покрыт тестами, UI миграция становится менее страшной.

## Feature flags и rollout

Для важных экранов полезен feature flag:

```kotlin
class OrdersFragment : Fragment() {

    override fun onCreateView(
        inflater: LayoutInflater,
        container: ViewGroup?,
        savedInstanceState: Bundle?
    ): View {
        return if (featureFlags.composeOrdersScreenEnabled) {
            createComposeOrdersView()
        } else {
            inflater.inflate(
                R.layout.orders_fragment,
                container,
                false
            )
        }
    }
}
```

Feature flag должен быть временным. У него должны быть:

- owner;
- дата удаления;
- метрики сравнения;
- rollback path;
- тесты для обеих веток, пока обе живы.

<MouseTrap>
Флаг без срока удаления становится постоянной развилкой. Через полгода команда будет чинить два экрана и бояться удалить любой из них.
</MouseTrap>

## Аналитика и accessibility parity

Compose screen должен сохранять важные non-visual contracts:

- screen view event;
- click events;
- error events;
- experiment exposure;
- content descriptions;
- тестовые теги, если они часть test infrastructure;
- deep link behavior;
- back behavior;
- focus order;
- keyboard actions.

Сравнивайте не только пиксели. Иногда новый экран выглядит правильно, но аналитика больше не видит checkout failure или TalkBack читает элементы в другом порядке.

## Командный рабочий процесс

Миграция проще, если договориться о правилах:

- новые UI components пишутся на Compose;
- legacy XML правится только для bugfixes;
- каждый новый Compose screen имеет предпросмотр states;
- ViewModel не зависит от Compose;
- composables принимают неизменяемое состояние;
- interop wrappers лежат в отдельном package;
- feature flags удаляются после rollout;
- дизайн-система развивается в одном месте;
- code review проверяет modifier order, accessibility и владение состоянием.

## Когда миграция завершена

Экран можно считать мигрированным, если:

- XML layout удален;
- ViewBinding/DataBinding для экрана удален;
- Fragment остался только как host или тоже удален;
- screen-level composable покрывает все states;
- tests обновлены;
- accessibility parity проверена;
- аналитика parity проверена;
- feature flag удален;
- old resources/styles больше не используются;
- команда знает новый owner path.

## Типичные ошибки

### Переписывать UI и архитектуру одновременно

Если сломался flow, трудно понять, виноват Compose, новый state layer или новая navigation.

### Начинать со сложнейшего экрана

Команда получает максимум риска до того, как освоит patterns.

### Не мигрировать дизайн-систему

Каждый экран начинает изобретать свои buttons, spacing и colors.

### Оставлять interop навсегда

`ComposeView` и `AndroidView` должны иметь понятную причину существования.

### Не сравнивать accessibility

Визуально экран может совпасть, а программа чтения с экрана flow ухудшиться.

### Держать два источника state

XML часть обновляется из binding, Compose часть из ViewModel, и они расходятся.

### Не удалять старый код

Миграция без удаления legacy code только увеличивает проект.

## Итоги

Миграция XML на Compose — это управляемый engineering project:

- начинайте с design system и state contracts;
- выбирайте screen candidates по риску, а не по эмоциям;
- новые features лучше писать на Compose;
- простые screens мигрируйте screen-by-screen;
- сложные screens можно переводить bottom-up или top-down;
- mixed navigation должна иметь ясную boundary;
- ViewModel/держатель состояния не должен зависеть от Compose;
- interop wrappers временные и изолированные;
- testing, accessibility и аналитика parity обязательны;
- feature flags помогают rollout, но должны удаляться;
- миграция завершена только после удаления старых XML/binding paths.
