## Зачем общий design system

Когда приложение выходит за пределы Android, дизайн-система становится способом удержать продуктовую целостность без копирования UI по платформам. Общий design system задает:

- цветовые tokens;
- типографику;
- spacing;
- shapes;
- icons;
- common components;
- motion tokens;
- стандартные настройки доступности;
- layout density;
- API contracts;
- правила платформенной адаптации.

Но общий design system не должна превращать все платформы в одинаковую картинку. Хорошая система разделяет:

- общий product language;
- platform conventions;
- form factor adaptations;
- технические boundaries.

```kotlin
object AppSpacing {
    val xxs = 2.dp
    val xs = 4.dp
    val sm = 8.dp
    val md = 16.dp
    val lg = 24.dp
    val xl = 32.dp
}
```

Tokens в common code помогают Android, iOS, Desktop и Web говорить на одном визуальном языке.

<CatTip>
Общий design system — это не фабрика одинаковых экранов. Это общий словарь, который каждая платформа произносит своим акцентом.
</CatTip>

## Theme tokens

Разделяйте raw tokens и semantic tokens.

Raw token:

```kotlin
object BrandPalette {
    val Green600 = Color(0xFF006C4F)
    val Blue600 = Color(0xFF2F5D8C)
    val Red600 = Color(0xFFB3261E)
}
```

Semantic token:

```kotlin
@Immutable
data class AppColorTokens(
    val primaryAction: Color,
    val dangerAction: Color,
    val pageBackground: Color,
    val cardBackground: Color,
    val textPrimary: Color,
    val textSecondary: Color
)
```

Semantic colors говорят, где цвет используется, а не как он называется в палитре.

## CompositionLocal tokens

```kotlin
val LocalAppColors = staticCompositionLocalOf<AppColorTokens> {
    error("AppColorTokens не предоставлены")
}

val LocalAppDimensions = staticCompositionLocalOf<AppDimensions> {
    error("AppDimensions не предоставлены")
}

@Immutable
data class AppDimensions(
    val minTouchTarget: Dp,
    val cardCornerRadius: Dp,
    val contentMaxWidth: Dp
)
```

Theme:

```kotlin
@Composable
fun SharedAppTheme(
    darkTheme: Boolean,
    platform: PlatformKind,
    content: @Composable () -> Unit
) {
    val colors = if (darkTheme) {
        darkAppColors()
    } else {
        lightAppColors()
    }

    val dimensions = dimensionsFor(platform)

    CompositionLocalProvider(
        LocalAppColors provides colors,
        LocalAppDimensions provides dimensions
    ) {
        MaterialTheme(
            colorScheme = colors.toMaterialColorScheme(),
            content = content
        )
    }
}
```

Так common components получают tokens без прямой зависимости от platform.

## Platform kind

```kotlin
enum class PlatformKind {
    Android,
    Ios,
    Desktop,
    Web
}

expect fun currentPlatformKind(): PlatformKind
```

Android:

```kotlin
actual fun currentPlatformKind(): PlatformKind {
    return PlatformKind.Android
}
```

Desktop:

```kotlin
actual fun currentPlatformKind(): PlatformKind {
    return PlatformKind.Desktop
}
```

Не используйте platform kind для каждой мелочи. Сначала попробуйте выразить различие через size class, input mode или component parameter.

## Common composables

Хороший common component:

```kotlin
@Composable
fun AppPrimaryButton(
    text: String,
    onClick: () -> Unit,
    enabled: Boolean = true,
    modifier: Modifier = Modifier
) {
    Button(
        onClick = onClick,
        enabled = enabled,
        modifier = modifier.heightIn(
            min = LocalAppDimensions.current.minTouchTarget
        )
    ) {
        Text(text = text)
    }
}
```

Component API должен быть:

- small;
- stable;
- state-driven;
- platform-neutral;
- accessible по умолчанию;
- easy to предпросмотр/test.

## Платформенная реализация за общим API

Иногда common API нужен, а реализация разная.

```kotlin
@Composable
expect fun PlatformTooltipArea(
    tooltip: String,
    content: @Composable () -> Unit
)
```

Desktop actual:

```kotlin
@Composable
actual fun PlatformTooltipArea(
    tooltip: String,
    content: @Composable () -> Unit
) {
    TooltipArea(
        tooltip = {
            Surface {
                Text(text = tooltip)
            }
        },
        content = content
    )
}
```

Android actual:

```kotlin
@Composable
actual fun PlatformTooltipArea(
    tooltip: String,
    content: @Composable () -> Unit
) {
    content()
}
```

На Android tooltip может быть не нужен или реализован иначе. Common API сохраняет intent.

## Resources

Общий design system должна использовать multiplatform resources:

```text
design-system/
  src/commonMain/composeResources/
    values/strings.xml
    drawable/ic_close.xml
    font/inter_regular.ttf
```

Common icon:

```kotlin
@Composable
fun AppCloseIcon(
    contentDescription: String?
) {
    Icon(
        painter = painterResource(Res.drawable.ic_close),
        contentDescription = contentDescription
    )
}
```

Не смешивайте Android `R.drawable` с common `Res.drawable` внутри common components.

## Typography

Typography должна учитывать:

- platform font availability;
- fallback fonts;
- desktop readability;
- web загрузка;
- iOS font rendering;
- large font accessibility;
- content density.

```kotlin
@Immutable
data class AppTypographyTokens(
    val titleLarge: TextStyle,
    val body: TextStyle,
    val label: TextStyle,
    val code: TextStyle
)
```

Platform adaptation:

```kotlin
fun typographyFor(
    platform: PlatformKind
): AppTypographyTokens {
    return when (platform) {
        PlatformKind.Desktop -> desktopTypography()
        PlatformKind.Web -> webTypography()
        PlatformKind.Android,
        PlatformKind.Ios -> mobileTypography()
    }
}
```

Не считайте, что один font size одинаково читаем на desktop monitor, phone и browser canvas.

## Icons

Icon system:

- common resource names;
- semantic wrappers;
- content description policy;
- sizes;
- stroke/fill style;
- при необходимости замены для конкретной платформы.

```kotlin
enum class AppIcon {
    Close,
    Search,
    Settings,
    Back
}

@Composable
fun AppIcon(
    icon: AppIcon,
    contentDescription: String?,
    modifier: Modifier = Modifier
) {
    Icon(
        painter = painterResource(icon.resource),
        contentDescription = contentDescription,
        modifier = modifier
    )
}
```

Back icon может отличаться для RTL/platform conventions, поэтому wrapper полезнее прямого resource access.

## Adaptive components

Общий component должен принимать layout intent:

```kotlin
enum class NavigationPresentation {
    BottomBar,
    Rail,
    Drawer
}

@Composable
fun AppNavigation(
    presentation: NavigationPresentation,
    destinations: List<AppDestination>,
    selected: AppDestination,
    onSelect: (AppDestination) -> Unit
) {
    when (presentation) {
        NavigationPresentation.BottomBar -> BottomNavigationContent(
            destinations = destinations,
            selected = selected,
            onSelect = onSelect
        )
        NavigationPresentation.Rail -> NavigationRailContent(
            destinations = destinations,
            selected = selected,
            onSelect = onSelect
        )
        NavigationPresentation.Drawer -> NavigationDrawerContent(
            destinations = destinations,
            selected = selected,
            onSelect = onSelect
        )
    }
}
```

Решение о presentation принимает shell, component рендерит выбранную форму.

## API stability

Design system — это internal library. Ее API нужно проектировать:

- не раскрывать implementation details;
- избегать boolean explosion;
- использовать state classes для сложных components;
- документировать значения по умолчанию;
- deprecate вместо удаления;
- покрывать screenshots/tests;
- держать changelog.

Плохо:

```kotlin
@Composable
fun AppCard(
    blue: Boolean,
    big: Boolean,
    clickable: Boolean,
    outlined: Boolean,
    desktop: Boolean
) {
}
```

Лучше:

```kotlin
enum class AppCardTone {
    Default,
    Accent,
    Warning
}

enum class AppCardDensity {
    Comfortable,
    Compact
}

@Composable
fun AppCard(
    tone: AppCardTone,
    density: AppCardDensity,
    onClick: (() -> Unit)?,
    content: @Composable () -> Unit
) {
}
```

## Testing design system

Для общий design system нужны:

- модульные тесты для отображения токенов;
- скриншот-тесты компонентов;
- accessibility semantics tests;
- platform smoke tests;
- font scale variants;
- dark/light variants;
- RTL variants;
- API binary/source compatibility checks, если library общий между командами.

```kotlin
class AppButtonSemanticsTest {

    @get:Rule
    val composeRule = createComposeRule()

    @Test
    fun primaryButton_hasClickAction() {
        composeRule.setContent {
            SharedAppTheme(
                darkTheme = false,
                platform = PlatformKind.Android
            ) {
                AppPrimaryButton(
                    text = "Сохранить",
                    onClick = {}
                )
            }
        }

        composeRule
            .onNodeWithText("Сохранить")
            .assertHasClickAction()
    }
}
```

Android tests не доказывают desktop/web behavior, но защищают common semantics.

## Governance

Design system требует правил:

- кто владеет tokens;
- как добавлять component;
- как deprecate old API;
- кто approves visual diffs;
- как версионировать;
- как документировать platform exceptions;
- как проверять accessibility;
- как синхронизировать с Figma/design tokens.

Без governance design system превращается в папку случайных composables.

## Типичные ошибки

### Добиваться pixel-perfect одинаковости

Платформы имеют разные conventions и input models.

### Прятать platform differences внутри if-ов

Если differences значимые, нужен явный shell или expect/actual.

### Делать component API boolean-heavy

Компонент быстро становится непредсказуемым.

### Использовать Android resources в common code

Это ломает multiplatform boundary.

### Не тестировать font scale/RTL

Общий component ломается сразу на нескольких платформах.

### Не иметь владельца design system

API и visual language расползаются.

## Итоги

Общий design system удерживает продуктовый UI across platforms:

- tokens должны быть semantic, а не только raw palette;
- CompositionLocals удобны для colors/dimensions/typography;
- expect/actual и interfaces прячут platform capabilities;
- common composables должны быть state-driven и accessible по умолчанию;
- resources используйте через Compose Multiplatform `Res`;
- typography/icons требуют platform-aware policy;
- adaptive components принимают presentation, выбранную shell-ом;
- API design system нужно версионировать и deprecate осознанно;
- screenshot, semantics и platform smoke tests обязательны;
- governance важен не меньше, чем код.
