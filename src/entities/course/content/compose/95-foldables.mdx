## Почему foldable — это не просто большой экран

Foldable может быть телефоном, планшетом, двумя областями вокруг hinge, устройством в tabletop posture, book posture или обычным flat expanded screen. Пользователь может менять posture прямо во время сценария: открыть, сложить, поставить на стол, повернуть, перейти в split-screen.

Обычные adaptive rules по ширине важны, но foldables добавляют новую информацию:

- есть ли folding feature;
- где находится hinge/fold;
- разделяет ли hinge content;
- перекрывает ли hinge pixels;
- горизонтальная или вертикальная ориентация fold;
- flat или half-opened state;
- tabletop posture;
- book posture;
- dual-screen behavior;
- continuity между folded/unfolded.

```kotlin
data class FoldableState(
    val posture: FoldablePosture,
    val hinge: HingeInfo? = null
)

enum class FoldablePosture {
    Normal,
    Tabletop,
    Book
}

data class HingeInfo(
    val isSeparating: Boolean,
    val orientation: HingeOrientation,
    val occluding: Boolean
)

enum class HingeOrientation {
    Horizontal,
    Vertical
}
```

Как и с size classes, полезно мапить platform details в UI-friendly model и передавать ее вниз.

<CatTip>
Foldable layout должен уважать физику устройства. Если hinge разделяет экран, это не декоративная линия, а граница, через которую неудобно читать, нажимать и перетаскивать.
</CatTip>

## Jetpack WindowManager

Foldable information обычно приходит через Jetpack WindowManager. Он сообщает `WindowLayoutInfo`, внутри которого могут быть `DisplayFeature`, включая `FoldingFeature`.

Gradle dependency:

```kotlin
dependencies {
    implementation("androidx.window:window:1.4.0")
}
```

Базовая collection модель:

```kotlin
import android.app.Activity
import androidx.compose.runtime.Composable
import androidx.compose.runtime.getValue
import androidx.compose.ui.platform.LocalContext
import androidx.lifecycle.compose.collectAsStateWithLifecycle
import androidx.window.layout.WindowInfoTracker
import androidx.window.layout.WindowLayoutInfo

@Composable
fun rememberWindowLayoutInfo(): WindowLayoutInfo? {
    val context = LocalContext.current
    val activity = context.findActivity() ?: return null
    val tracker = WindowInfoTracker.getOrCreate(context)
    val layoutInfo by tracker
        .windowLayoutInfo(activity)
        .collectAsStateWithLifecycle(initialValue = null)

    return layoutInfo
}

fun android.content.Context.findActivity(): Activity? {
    var current = this

    while (current is android.content.ContextWrapper) {
        if (current is Activity) {
            return current
        }
        current = current.baseContext
    }

    return null
}
```

В production коде helper `findActivity` лучше держать в platform module и покрыть тестами/inspection. Не стоит разбрасывать его по composables.

## `FoldingFeature`

`FoldingFeature` описывает fold/hinge:

- `state`: flat или half-opened;
- `orientation`: horizontal или vertical;
- `isSeparating`: разделяет ли display на области;
- `occlusionType`: есть ли физическое перекрытие pixels;
- `bounds`: rectangle folding feature в window coordinates.

```kotlin
import androidx.window.layout.DisplayFeature
import androidx.window.layout.FoldingFeature
import androidx.window.layout.WindowLayoutInfo

fun WindowLayoutInfo.foldingFeatureOrNull(): FoldingFeature? {
    return displayFeatures
        .filterIsInstance<FoldingFeature>()
        .firstOrNull()
}
```

Дальше можно построить UI model:

```kotlin
import androidx.window.layout.FoldingFeature

fun FoldingFeature?.toFoldableState(): FoldableState {
    if (this == null) {
        return FoldableState(posture = FoldablePosture.Normal)
    }

    val hingeOrientation = when (orientation) {
        FoldingFeature.Orientation.HORIZONTAL -> HingeOrientation.Horizontal
        FoldingFeature.Orientation.VERTICAL -> HingeOrientation.Vertical
        else -> HingeOrientation.Vertical
    }

    val posture = when {
        isTabletopPosture(this) -> FoldablePosture.Tabletop
        isBookPosture(this) -> FoldablePosture.Book
        else -> FoldablePosture.Normal
    }

    return FoldableState(
        posture = posture,
        hinge = HingeInfo(
            isSeparating = isSeparating,
            orientation = hingeOrientation,
            occluding = occlusionType == FoldingFeature.OcclusionType.FULL
        )
    )
}
```

## Tabletop posture

Tabletop posture обычно означает:

- device half-opened;
- hinge horizontal;
- верхняя половина хорошо подходит для preview/video;
- нижняя половина подходит для controls;
- пользователь может не держать device руками.

```kotlin
import androidx.window.layout.FoldingFeature

fun isTabletopPosture(
    feature: FoldingFeature?
): Boolean {
    return feature?.state == FoldingFeature.State.HALF_OPENED &&
        feature.orientation == FoldingFeature.Orientation.HORIZONTAL
}
```

UI:

```kotlin
import androidx.compose.foundation.layout.Column
import androidx.compose.runtime.Composable
import androidx.compose.ui.Modifier

@Composable
fun TabletopVideoCall(
    callState: VideoCallState,
    modifier: Modifier = Modifier
) {
    Column(modifier = modifier) {
        VideoParticipantsPane(
            participants = callState.participants,
            modifier = Modifier.weight(1f)
        )

        VideoCallControlsPane(
            muted = callState.muted,
            cameraEnabled = callState.cameraEnabled,
            modifier = Modifier.weight(1f)
        )
    }
}
```

В настоящем tabletop layout hinge bounds должны влиять на разделение panes. Простое `weight(1f)` работает только как учебная модель.

## Book posture

Book posture:

- device half-opened;
- hinge vertical;
- две стороны работают как разворот;
- подходит для чтения, сравнения, dual preview;
- content не должен пересекать hinge.

```kotlin
import androidx.window.layout.FoldingFeature

fun isBookPosture(
    feature: FoldingFeature?
): Boolean {
    return feature?.state == FoldingFeature.State.HALF_OPENED &&
        feature.orientation == FoldingFeature.Orientation.VERTICAL
}
```

Reader:

```kotlin
import androidx.compose.foundation.layout.Row
import androidx.compose.runtime.Composable
import androidx.compose.ui.Modifier

@Composable
fun BookPostureReader(
    leftPage: ReaderPage,
    rightPage: ReaderPage,
    modifier: Modifier = Modifier
) {
    Row(modifier = modifier) {
        ReaderPagePane(
            page = leftPage,
            modifier = Modifier.weight(1f)
        )
        ReaderPagePane(
            page = rightPage,
            modifier = Modifier.weight(1f)
        )
    }
}
```

Book posture не значит "любые две колонки". Разворот должен иметь смысл: страница/страница, список/detail, compare, preview/controls.

<MouseTrap>
Если текстовая строка пересекает hinge, пользователь физически читает через складку. Даже если pixels видны, UX уже сломан.
</MouseTrap>

## Hinge bounds

`FoldingFeature.bounds` помогает понять, где находится hinge. Для custom layout можно зарезервировать gutter.

```kotlin
import android.graphics.Rect
import androidx.compose.foundation.layout.Row
import androidx.compose.foundation.layout.Spacer
import androidx.compose.foundation.layout.width
import androidx.compose.runtime.Composable
import androidx.compose.ui.Modifier
import androidx.compose.ui.platform.LocalDensity
import androidx.compose.ui.unit.dp

@Composable
fun VerticalHingeTwoPane(
    hingeBounds: Rect?,
    left: @Composable () -> Unit,
    right: @Composable () -> Unit,
    modifier: Modifier = Modifier
) {
    val density = LocalDensity.current
    val hingeWidth = with(density) {
        (hingeBounds?.width() ?: 0).toDp()
    }

    Row(modifier = modifier) {
        PaneBox(
            modifier = Modifier.weight(1f),
            content = left
        )
        Spacer(
            modifier = Modifier.width(
                hingeWidth.coerceAtLeast(16.dp)
            )
        )
        PaneBox(
            modifier = Modifier.weight(1f),
            content = right
        )
    }
}
```

Если `occlusionType == FULL`, в hinge area реально нельзя рисовать meaningful content. Если occlusion отсутствует, область все равно может быть неудобной для touch и чтения.

## Separating hinge

`isSeparating` говорит, что folding feature делит display на logical areas. Даже если fold flat, dual-screen device может иметь separating hinge.

Практические правила:

- не размещать primary button на hinge;
- не центрировать modal action прямо на fold;
- не делать drag target через hinge;
- не прокручивать list так, чтобы row постоянно пересекал hinge;
- добавлять gutter;
- использовать panes вместо одного непрерывного canvas.

```kotlin
fun shouldUseTwoPhysicalPanes(
    foldableState: FoldableState,
    windowState: AdaptiveWindowState
): Boolean {
    return foldableState.hinge?.isSeparating == true ||
        windowState.widthClass == WindowWidthClass.Expanded
}
```

Expanded width может дать два panes и без foldable. Separating hinge делает panes особенно важными.

## Fold-aware app root

Root собирает adaptive state и выбирает layout:

```kotlin
import androidx.compose.runtime.Composable

@Composable
fun FoldAwareAppRoot(
    appState: AppState
) {
    val layoutInfo = rememberWindowLayoutInfo()
    val foldingFeature = layoutInfo?.foldingFeatureOrNull()
    val foldableState = foldingFeature.toFoldableState()
    val windowState = rememberAdaptiveWindowState()

    when (foldableState.posture) {
        FoldablePosture.Tabletop -> {
            TabletopAppShell(
                appState = appState,
                hinge = foldableState.hinge
            )
        }
        FoldablePosture.Book -> {
            BookAppShell(
                appState = appState,
                hinge = foldableState.hinge
            )
        }
        FoldablePosture.Normal -> {
            AdaptiveAppShell(
                appState = appState,
                windowState = windowState
            )
        }
    }
}
```

Не заставляйте каждый screen самостоятельно читать `WindowInfoTracker`, если posture является app-level concern.

## Continuity при fold/unfold

Fold/unfold — это configuration change и layout change. Пользователь не должен терять:

- выбранный item;
- введенный текст;
- scroll position;
- playback state;
- call state;
- current route;
- open pane;
- media controls visibility;
- camera mode.

```kotlin
data class ReaderUiState(
    val documentId: String,
    val currentChapterId: String,
    val currentPage: Int,
    val selectedTextRange: TextRange?,
    val controlsVisible: Boolean
)
```

Reader layout может меняться, но `ReaderUiState` остается тем же.

```kotlin
@Composable
fun ReaderScreen(
    state: ReaderUiState,
    foldableState: FoldableState,
    onPageChange: (Int) -> Unit
) {
    when (foldableState.posture) {
        FoldablePosture.Book -> {
            BookReaderLayout(
                state = state,
                onPageChange = onPageChange
            )
        }
        FoldablePosture.Tabletop -> {
            TabletopReaderLayout(
                state = state,
                onPageChange = onPageChange
            )
        }
        FoldablePosture.Normal -> {
            SinglePaneReaderLayout(
                state = state,
                onPageChange = onPageChange
            )
        }
    }
}
```

## Camera и tabletop

Camera apps часто выигрывают от tabletop:

- preview сверху;
- controls снизу;
- hands-free capture;
- rear/front screen workflows;
- aspect ratio changes.

```kotlin
@Composable
fun FoldAwareCameraScreen(
    state: CameraUiState,
    foldableState: FoldableState,
    onCapture: () -> Unit
) {
    if (foldableState.posture == FoldablePosture.Tabletop) {
        TabletopCameraLayout(
            preview = {
                CameraPreview(
                    stream = state.previewStream
                )
            },
            controls = {
                CameraControls(
                    mode = state.mode,
                    onCapture = onCapture
                )
            }
        )
    } else {
        FullscreenCameraLayout(
            state = state,
            onCapture = onCapture
        )
    }
}
```

Camera preview не должен растягиваться без учета aspect ratio. Fold-aware layout часто требует отдельной camera pipeline настройки.

## Input на foldables

Foldable input может быть неоднородным:

- touch на верхней/нижней части;
- stylus на unfolded screen;
- keyboard в tabletop;
- mouse/trackpad на large foldable;
- drag-and-drop между panes.

Не делайте жест через hinge обязательным. Например, drag из list pane в detail pane через hinge может быть неудобен. Добавьте menu/action alternative.

```kotlin
@Composable
fun FoldAwareMoveAction(
    onMoveToDetail: () -> Unit
) {
    MoveButton(
        onClick = onMoveToDetail
    )
}
```

Gesture может быть быстрым path, но кнопка/action остается accessible path.

## Testing foldables

Тестировать нужно:

- folded compact;
- unfolded expanded;
- tabletop;
- book;
- separating hinge;
- flat with separating hinge;
- orientation changes;
- split-screen на foldable;
- IME in tabletop;
- drag/touch возле hinge;
- TalkBack traversal между panes;
- state continuity при fold/unfold.

В unit-тестах тестируйте mapper:

```kotlin
import com.google.common.truth.Truth.assertThat
import org.junit.Test

class FoldableStateMapperTest {

    @Test
    fun horizontalHalfOpened_isTabletop() {
        val feature = fakeFoldingFeature(
            state = FoldingFeature.State.HALF_OPENED,
            orientation = FoldingFeature.Orientation.HORIZONTAL
        )

        val state = feature.toFoldableState()

        assertThat(state.posture)
            .isEqualTo(FoldablePosture.Tabletop)
    }
}
```

UI screenshot tests должны смотреть, что content не пересекает hinge. Это трудно доказать простым semantics test.

## Типичные ошибки

### Считать foldable обычным tablet

Hinge/posture может полностью менять interaction model.

### Рисовать важный content через hinge

Даже без full occlusion fold area часто неудобна для чтения и touch.

### Сбрасывать route при fold/unfold

Пользователь не должен терять сценарий из-за физического изменения device.

### Использовать только width class

Width class не говорит, что устройство стоит в tabletop posture.

### Делать tabletop layout без учета hinge bounds

Верхняя и нижняя области могут быть не ровно 50/50.

### Не давать альтернативу жестам через hinge

Foldable gestures должны иметь accessible action path.

## Итоги

Foldables добавляют физическую модель к adaptive UI:

- Jetpack WindowManager сообщает `WindowLayoutInfo` и `FoldingFeature`;
- `FoldingFeature` описывает state, orientation, bounds, occlusion и separation;
- tabletop posture обычно half-opened + horizontal hinge;
- book posture обычно half-opened + vertical hinge;
- separating hinge должен влиять на pane layout;
- content и touch targets не должны попадать в hinge area;
- fold/unfold не должен сбрасывать user state;
- camera, video, reading и productivity workflows могут получить специальные layouts;
- foldable info лучше мапить в UI-friendly state на верхнем уровне;
- testing требует эмуляторов/устройств и проверки posture, hinge и continuity.
