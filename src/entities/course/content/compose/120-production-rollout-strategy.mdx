## Почему rollout strategy нужна заранее

Compose adoption редко заканчивается на "мы написали экран". В боевой появляются вопросы:

- как выкатывать новые Compose screens;
- как мигрировать XML постепенно;
- как обновлять Compose BOM/Kotlin/AGP;
- как deprecate старые components;
- как мониторить regressions;
- как проверять performance;
- как поддерживать несколько платформ;
- как откатывать feature;
- кто владеет design system API.

Rollout strategy превращает Compose из эксперимента в управляемую инженерную практику.

```kotlin
data class RolloutPlan(
    val featureName: String,
    val owner: String,
    val rolloutPercent: Int,
    val rollbackEnabled: Boolean,
    val monitoredSignals: List<String>
)
```

Если у feature нет owner, metrics и rollback path, это не rollout, а надежда.

<CatTip>
Боевой rollout — это не момент релиза. Это весь путь от первого Compose component до поддержки версий, мониторинга и удаления старого кода.
</CatTip>

## Incremental adoption

Безопасные уровни adoption:

- new leaf components;
- new cards/list items;
- new isolated screens;
- new feature module;
- design system migration;
- navigation shell migration;
- multiplatform extraction;
- full общий UI.

Для существующего Android app часто работает путь:

```text
1. Добавить Compose в проект
2. Создать theme bridge
3. Переписать isolated component
4. Добавить screenshot/semantics tests
5. Внедрить ComposeView в один screen
6. Написать новый screen на Compose
7. Постепенно заменить feature
8. Удалить XML/ViewBinding legacy
```

Не начинайте с самого сложного checkout/player/editor screen, если команда только входит в Compose.

## Feature flags

Feature flag помогает выкатывать screen постепенно.

```kotlin
interface FeatureFlags {
    val composeCheckoutEnabled: Boolean
}

fun createCheckoutView(
    flags: FeatureFlags
): View {
    return if (flags.composeCheckoutEnabled) {
        createComposeCheckoutView()
    } else {
        inflateLegacyCheckoutView()
    }
}
```

Flag должен иметь:

- owner;
- rollout plan;
- rollback rule;
- removal date;
- dashboards;
- тесты на активные пути, пока оба живы.

<MouseTrap>
Feature flag без даты удаления становится вторым приложением внутри первого. Через полгода никто не помнит, какая ветка настоящая.
</MouseTrap>

## Version upgrades

Compose stack связан с:

- Kotlin version;
- Compose Compiler Gradle Plugin;
- Compose BOM;
- Android Gradle Plugin;
- AndroidX libraries;
- Material 3;
- lifecycle/navigation/activity compose;
- Kotlin Multiplatform plugin;
- Compose Multiplatform plugin;
- IDE support.

Upgrade plan:

- читать release notes;
- обновлять в feature branch;
- запускать unit/UI/screenshot tests;
- запускать Macrobenchmark для critical flows;
- проверять compiler warnings/deprecations;
- проверять предпросмотры;
- проверить app startup;
- rollout на internal/beta track;
- мониторить crashes/jank.

```kotlin
data class ComposeUpgradeChecklist(
    val kotlinUpdated: Boolean,
    val composeBomUpdated: Boolean,
    val compilerPluginUpdated: Boolean,
    val testsGreen: Boolean,
    val benchmarksCompared: Boolean
)
```

Не обновляйте весь Android/KMP toolchain в пятницу вечером одним коммитом без rollback strategy.

## Compatibility policy

Для design system и общий UI нужна compatibility policy:

- какие APIs stable;
- какие experimental;
- как долго deprecated API живет;
- как маркируются breaking changes;
- кто approves removal;
- как документируются migrations.

```kotlin
@RequiresOptIn(
    message = "Этот компонент экспериментальный и может измениться без migration window."
)
annotation class ExperimentalAppDesignSystemApi
```

Experimental API:

```kotlin
@ExperimentalAppDesignSystemApi
@Composable
fun AppAdaptiveToolbar(
    state: ToolbarState
) {
    // Компонент еще не стабилизирован.
}
```

Так callers явно принимают риск.

## API deprecation

Deprecated component должен вести к replacement.

```kotlin
@Deprecated(
    message = "Используйте AppPrimaryButton с tone = Primary.",
    replaceWith = ReplaceWith(
        "AppPrimaryButton(text = text, onClick = onClick)"
    )
)
@Composable
fun LegacyGreenButton(
    text: String,
    onClick: () -> Unit
) {
    AppPrimaryButton(
        text = text,
        onClick = onClick
    )
}
```

Deprecation plan:

- announce;
- provide replacement;
- migrate internal usages;
- при необходимости добавить проверку/поиск;
- удалить после согласованного окна.

## Monitoring regressions

Compose rollout monitor:

- crash-free sessions;
- ANR rate;
- startup time;
- jank/slow frames;
- screen load time;
- memory usage;
- image OOMs;
- accessibility complaints;
- UI test flakiness;
- screenshot diffs;
- user funnel metrics.

```kotlin
data class ComposeScreenMetrics(
    val screenName: String,
    val composeVersion: String,
    val startupMsP95: Long?,
    val jankPercent: Float?,
    val crashCount: Int
)
```

Metric без baseline мало полезна. Сравнивайте до/после и legacy/Compose ветки.

## Performance budgets

Для critical screens задайте budgets:

```kotlin
data class PerformanceBudget(
    val screen: String,
    val coldStartupP95Ms: Long?,
    val firstContentP95Ms: Long,
    val scrollJankPercentMax: Float,
    val memoryMbMax: Int?
)
```

Budgets должны быть реалистичными и измеряться одинаково. Иначе они станут декоративной таблицей.

## Crash rollback

Feature flag rollback:

```kotlin
class ComposeCheckoutRolloutController(
    private val remoteConfig: RemoteConfig,
    private val crashMonitor: CrashMonitor
) {
    fun shouldEnable(): Boolean {
        if (crashMonitor.isCheckoutCrashRateHigh()) {
            return false
        }

        return remoteConfig.composeCheckoutEnabled
    }
}
```

Rollback должен быть простым. Если rollback требует релиз нового APK, rollout риск выше.

## Multiplatform rollout

Compose Multiplatform добавляет:

- разные release trains;
- platform-specific bugs;
- source set compatibility;
- web beta risk;
- desktop packaging;
- iOS review/distribution;
- общий UI API versioning;
- platform QA matrix.

Rollout по платформам может быть разным:

```text
Android: production 100%
iOS: beta 25%
Desktop: internal
Web/Wasm: preview behind flag
```

Не обязано быть "всё общий сразу на всех платформах".

## Наблюдаемость в пользовательском интерфейсе

Добавьте lightweight events:

- screen rendered;
- first content visible;
- error state shown;
- retry clicked;
- slow load;
- используется резервная ветка;
- Compose feature flag state.

```kotlin
interface UiTelemetry {
    fun screenRendered(name: String)
    fun errorShown(name: String, type: String)
    fun featureFlagEvaluated(name: String, enabled: Boolean)
}
```

UI telemetry не должна собирать sensitive data и не должна тормозить composition.

## Release проверочный список

Перед rollout:

- tests green;
- screenshot diffs reviewed;
- accessibility checks complete;
- Macrobenchmark compared;
- baseline profile updated;
- crash dashboard ready;
- feature flag configured;
- rollback tested;
- old path still works, если flag нужен;
- docs/changelog updated;
- owner assigned.

```kotlin
data class ReleaseReadiness(
    val testsGreen: Boolean,
    val rollbackTested: Boolean,
    val metricsDashboardReady: Boolean,
    val accessibilityChecked: Boolean
) {
    val ready: Boolean
        get() = testsGreen &&
            rollbackTested &&
            metricsDashboardReady &&
            accessibilityChecked
}
```

## Removing legacy code

Migration не завершена, пока legacy path жив.

Удалить:

- XML layouts;
- ViewBinding classes;
- old styles;
- old adapters;
- unused resources;
- old flags;
- duplicate tests;
- dead navigation actions;
- old аналитика events, если заменены.

Проверьте:

- no references via `rg`;
- APK size;
- resource shrinker;
- screenshots;
- deep links;
- rollback no longer expected.

## Documentation

Документируйте:

- Compose versions;
- design system component policy;
- предпросмотр/test conventions;
- migration patterns;
- common pitfalls;
- upgrade process;
- platform-specific exceptions;
- rollout history.

Документация должна быть короткой и живой. Лучше проверочный список, чем трактат, который никто не читает.

## Типичные ошибки

### Переписать всё сразу

Риск высок, диагностика сложная, rollback почти невозможен.

### Обновить toolchain без benchmarks

Performance регрессия может прийти не из вашего кода.

### Не удалить feature flag

Две ветки продолжают жить и расходиться.

### Не мониторить jank

Crash-free не значит smooth.

### Считать общий UI одинаково готовым для всех платформ

Статусы и риски Android/iOS/Desktop/Web отличаются.

### Не иметь owner design system

API ломается хаотично.

## Итоги

Боевой rollout strategy делает Compose adoption управляемым:

- начинайте incremental, а не с полного rewrite;
- feature flags требуют owner, metrics, rollback и removal date;
- version upgrades должны включать tests, предпросмотры, benchmarks и release notes;
- design system APIs требуют compatibility/deprecation policy;
- monitoring должен включать crashes, ANR, startup, jank, memory и funnel metrics;
- performance budgets нужны для critical flows;
- rollback должен быть реально проверен;
- Compose Multiplatform rollout можно вести по платформам разными темпами;
- migration завершена только после удаления legacy code;
- документация и governance удерживают качество после первого релиза.
