## Что такое Compose Multiplatform

Compose Multiplatform — это декларативный UI framework от JetBrains, построенный на идеях Jetpack Compose и Kotlin Multiplatform. Он позволяет писать shared Compose UI в common Kotlin code и запускать его на нескольких платформах.

Ключевая мысль: Compose Multiplatform не является "Android Compose, который случайно запустили везде". Это родственный framework с общим mental model:

- composable functions;
- state → UI;
- recomposition;
- modifiers;
- layout;
- Material-like components;
- resources;
- platform-specific integrations.

Но платформы отличаются: windowing, input, lifecycle, accessibility, navigation, packaging, fonts, resources и native API access имеют свои правила.

```kotlin
@Composable
fun SharedCounter(
    count: Int,
    onIncrement: () -> Unit
) {
    Column {
        Text(text = "Значение: $count")
        Button(onClick = onIncrement) {
            Text(text = "Увеличить")
        }
    }
}
```

Такой component может жить в `commonMain`, если использует только common Compose APIs.

<CatTip>
Shared UI работает лучше всего, когда вы делите не пиксели, а намерение: state, component contracts, theme tokens и behavior. Platform shell всё равно остается важным.
</CatTip>

## Текущий статус платформ

По состоянию на актуальные JetBrains docs:

- Android: Stable;
- iOS: Stable;
- Desktop JVM: Stable;
- Web на Kotlin/Wasm: Beta.

Это важная продуктовая информация. Stable платформы можно рассматривать для production при нормальной engineering дисциплине. Beta web target требует отдельной оценки риска, особенно для accessibility, browser integration, resources, performance и team tooling.

## Relationship with Jetpack Compose

Jetpack Compose:

- поддерживается Google;
- ориентирован на Android;
- глубоко интегрирован с Android lifecycle, resources, navigation, testing, accessibility.

Compose Multiplatform:

- поддерживается JetBrains;
- основан на Kotlin Multiplatform;
- позволяет shared UI на Android, iOS, desktop и web;
- имеет platform-specific APIs для desktop/iOS/web;
- не всегда имеет 1:1 соответствие Android APIs.

Практический вывод: Android Compose знания переносятся, но Android-only APIs не должны попадать в common UI.

```kotlin
// Этот код не подходит для общего исходного набора, потому что использует Android-контекст.
@Composable
fun AndroidOnlyShareButton(
    context: android.content.Context
) {
    Button(
        onClick = {
            context.startActivity(createShareIntent())
        }
    ) {
        Text(text = "Поделиться")
    }
}
```

Shared UI должен получить событие:

```kotlin
@Composable
fun SharedShareButton(
    onShare: () -> Unit
) {
    Button(onClick = onShare) {
        Text(text = "Поделиться")
    }
}
```

Platform shell решает, как открыть share sheet.

## Source sets

Типичная структура:

```text
shared/
  src/
    commonMain/
    androidMain/
    iosMain/
    desktopMain/
    wasmJsMain/
```

`commonMain`:

- shared UI;
- shared state holders;
- theme tokens;
- mappers;
- common interfaces;
- pure Kotlin logic.

Platform source sets:

- Android Activity/Context integrations;
- iOS UIKit/AppKit bridges;
- Desktop windows/menus/tray;
- Web browser APIs;
- platform resources/adapters;
- expect/actual implementations.

## Shared UI boundaries

Хороший shared component:

```kotlin
@Immutable
data class SettingsItemUi(
    val id: String,
    val title: String,
    val subtitle: String?,
    val enabled: Boolean
)

@Composable
fun SettingsList(
    items: List<SettingsItemUi>,
    onItemClick: (String) -> Unit
) {
    LazyColumn {
        items(
            items = items,
            key = { item ->
                item.id
            }
        ) { item ->
            SettingsRow(
                item = item,
                onClick = {
                    onItemClick(item.id)
                }
            )
        }
    }
}
```

Плохой shared component:

```kotlin
@Composable
fun SettingsList(
    activity: android.app.Activity,
    repository: AndroidSettingsRepository
) {
    // Общий интерфейс не должен знать про Android-активити и Android-репозиторий.
}
```

Platform dependencies остаются на краю.

## What differs from Android Compose

Отличия могут быть в:

- lifecycle;
- navigation libraries;
- accessibility behavior;
- text input and IME;
- pointer/keyboard shortcuts;
- platform menus;
- window APIs;
- dialogs;
- system theme;
- resources;
- font loading;
- image loading;
- testing tools;
- packaging.

Например, Android `stringResource(R.string.title)` не существует как common API. Compose Multiplatform имеет собственную resource system.

## Multiplatform resources

Compose Multiplatform resources генерируют `Res` class и работают в common code.

Пример:

```kotlin
import org.jetbrains.compose.resources.stringResource
import my.app.generated.resources.Res
import my.app.generated.resources.app_title

@Composable
fun AppTitle() {
    Text(
        text = stringResource(Res.string.app_title)
    )
}
```

Resources живут в common resources:

```text
commonMain/composeResources/
  values/strings.xml
  drawable/logo.xml
  font/inter.ttf
```

Это не тот же API, что Android `R.string`. Не смешивайте resource systems без boundary.

## expect/actual bridges

Когда common UI нуждается в platform capability, используйте interface или expect/actual.

Interface approach:

```kotlin
interface ShareController {
    fun shareText(text: String)
}

@Composable
fun SharePanel(
    shareController: ShareController,
    text: String
) {
    Button(
        onClick = {
            shareController.shareText(text)
        }
    ) {
        Text(text = "Поделиться")
    }
}
```

expect/actual approach:

```kotlin
expect class PlatformInfo {
    val name: String
}
```

Android actual:

```kotlin
actual class PlatformInfo {
    actual val name: String = "Android"
}
```

Use expect/actual for platform facts and adapters. Use interfaces when you want dependency injection and testability.

## Shared design system

Shared design system обычно включает:

- colors;
- typography tokens;
- spacing;
- shapes;
- icons;
- common components;
- motion tokens;
- content density rules.

```kotlin
object AppSpacing {
    val xs = 4.dp
    val sm = 8.dp
    val md = 16.dp
    val lg = 24.dp
}

@Composable
fun AppTheme(
    content: @Composable () -> Unit
) {
    MaterialTheme(
        colorScheme = appColorScheme(),
        typography = appTypography(),
        content = content
    )
}
```

Но platform conventions тоже важны. Desktop menu, iOS back behavior, Android system bars и web browser constraints не должны игнорироваться ради "одинаковости".

## State sharing

Shared UI лучше работает с shared state:

```kotlin
class CounterStateHolder {
    var count by mutableStateOf(0)
        private set

    fun increment() {
        count += 1
    }
}
```

Но lifecycle of state holder может отличаться:

- Android ViewModel;
- iOS lifecycle;
- Desktop window lifecycle;
- browser page lifecycle.

Практический подход: common state holder + platform owner.

## Navigation

Navigation в multiplatform app требует решения:

- common navigation model;
- platform navigation shell;
- third-party multiplatform navigation library;
- per-platform navigation with shared screens.

Простая common model:

```kotlin
sealed interface AppRoute {
    data object Home : AppRoute
    data class LessonDetails(
        val lessonId: String
    ) : AppRoute
}
```

Shared UI:

```kotlin
@Composable
fun AppContent(
    route: AppRoute,
    onNavigate: (AppRoute) -> Unit
) {
    when (route) {
        AppRoute.Home -> HomeScreen(
            onOpenLesson = { id ->
                onNavigate(AppRoute.LessonDetails(id))
            }
        )
        is AppRoute.LessonDetails -> LessonDetailsScreen(
            lessonId = route.lessonId,
            onBack = {
                onNavigate(AppRoute.Home)
            }
        )
    }
}
```

Для production navigation учитывайте deep links, browser history, back stack и platform expectations.

## Testing shared UI

Shared UI можно тестировать на нескольких уровнях:

- pure common unit tests;
- screenshot tests for common components;
- Android Compose UI tests;
- Desktop UI/manual tests;
- iOS integration tests;
- Web/browser tests, если target включен.

Не ожидайте, что один test runner докажет корректность всех platforms. Shared code снижает duplication, но не отменяет platform QA.

## Adoption strategy

Начинать можно по-разному:

- shared domain/data only;
- shared design tokens;
- shared small UI components;
- shared full screens;
- shared app shell.

Для existing Android app часто безопаснее:

- вынести UI state models;
- выделить design tokens;
- создать common components;
- затем мигрировать selected screens;
- держать Android-specific shell отдельно.

## Типичные ошибки

### Считать Multiplatform бесплатным

Shared code уменьшает duplication, но добавляет source sets, tooling, platform QA и release coordination.

### Тащить Android APIs в commonMain

Так shared UI перестает быть shared.

### Игнорировать platform conventions

Одинаковый UI не всегда означает хороший UI.

### Начинать с полного app rewrite

Риск слишком высокий. Лучше incremental adoption.

### Не фиксировать status платформ

Web/Wasm beta требует другого risk management, чем stable Android/Desktop/iOS.

### Смешивать Android resources и MPP resources

Resource boundary должен быть явным.

## Итоги

Compose Multiplatform расширяет Compose mental model за пределы Android:

- Android, iOS и Desktop JVM stable; Web/Wasm beta;
- общий UI возможен через common Kotlin code;
- Android-only APIs не должны попадать в commonMain;
- source sets разделяют common и platform code;
- resources используют Compose Multiplatform resource system;
- platform capabilities лучше прятать за interfaces или expect/actual;
- shared design system должен уважать platform conventions;
- state и navigation можно разделять, но lifecycle/back behavior платформ отличается;
- adoption лучше делать постепенно;
- shared UI требует platform QA, а не только common tests.
